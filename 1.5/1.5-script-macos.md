## PART 1 — SEEING THE CPU

### What is the CPU?

**SPEAK**
> "The CPU is the chip that runs your code. It has an architecture — what instruction set it speaks — a number of cores, and a cache hierarchy. Lecture 1.4 drew the memory pyramid. L1, L2, L3 caches sit right next to the cores. RAM sits below that. Each tier is bigger and slower. The OS knows every one of these numbers; you just have to ask it."

### Run the demo

**TYPE**
```bash
sysctl -n machdep.cpu.brand_string
```

**OUTPUT**
```
Apple M1 Pro
```

**EXPLAIN**
> "That's the chip inside this laptop — an Apple M1 Pro, read directly from a kernel variable. `sysctl` is the tool that asks the kernel for these facts; we'll use it a few more times today. Yours will show whatever chip you have."

**TYPE**
```bash
sysctl -n hw.physicalcpu
sysctl -n hw.logicalcpu
```

**OUTPUT**
```
8
8
```

**EXPLAIN**
> "Eight and eight. Physical cores are real, independent execution units on the chip. Logical cores are what the kernel scheduler sees. On an Intel Mac with Hyper-Threading you'd see 4 physical and 8 logical — one real core pretending to be two. Apple Silicon doesn't do that, so the numbers match."

**TYPE**
```bash
sysctl -n hw.perflevel0.l1dcachesize
sysctl -n hw.perflevel0.l2cachesize
```

**OUTPUT**
```
131072
12582912
```

**EXPLAIN**
> "Two numbers in bytes. 131,072 is 128 KiB — the L1 data cache per performance core. 12,582,912 is 12 MiB — the L2 cache shared across the performance cluster. `perflevel0` is Apple's name for the fast P-cores; there's also a `perflevel1` for the slower efficiency cores. And no L3 on Apple Silicon — that's expected, the chip uses a system-level cache Apple doesn't expose here."

**TYPE**
```bash
system_profiler SPHardwareDataType | grep -E "Model Name|Chip|Total Number of Cores|Memory"
```

**OUTPUT**
```
      Model Name: MacBook Pro
      Chip: Apple M1 Pro
      Total Number of Cores: 8 (6 performance and 2 efficiency)
      Memory: 32 GB
```

**EXPLAIN**
> "Same facts About This Mac shows. Model, chip, core breakdown, memory. `Total Number of Cores: 8 (6 performance and 2 efficiency)` confirms the P/E split we just saw. I'm grepping down to four lines because the full `system_profiler` output includes your serial number and hardware UUID — safe at home, not safe on a recording."

### Exercise — cache ratio

**SPEAK**
> "Let's see how much bigger L2 is than L1."

**TYPE**
```bash
python3.12 -c "print(12582912 / 131072)"
```

**OUTPUT**
```
96.0
```

**EXPLAIN**
> "L2 is 96× bigger than L1 on this chip. That's the pyramid in one number — each tier down multiplies capacity by roughly 10-100×. This is why data that fits in L1 runs so much faster than data that spills into RAM."

## PART 2 — SEEING RAM

### What is RAM?

**SPEAK**
> "RAM is the scratchpad. Your programs, their variables, their open files — all live in RAM while they run. macOS tracks it in categories: free, active, inactive, wired, compressed. We'll see each."

### Run the demo

**TYPE**
```bash
sysctl -n hw.memsize
```

**OUTPUT**
```
34359738368
```

**EXPLAIN**
> "That's the total RAM in bytes. It looks weird because it's exactly 32 × 1024³ — RAM is always sized in powers of two. We'll see why that differs from the 'GB' on the spec sheet in the exercise."

**TYPE**
```bash
vm_stat | head -7
```

**OUTPUT**
```
Mach Virtual Memory Statistics: (page size of 16384 bytes)
Pages free:                               78745.
Pages active:                            507575.
Pages inactive:                          444596.
Pages speculative:                        82603.
Pages throttled:                              0.
Pages wired down:                        152544.
```

**EXPLAIN**
> "`vm_stat` reports in *pages*, not bytes. Header says the page size is 16384 bytes — 16 KiB on Apple Silicon. 

A page is the fixed-size chunk that the operating system uses to manage RAM. Instead of tracking memory byte-by-byte, the kernel divides RAM into equal-sized blocks called pages and accounts for memory in whole-page units — allocating, freeing, protecting, and swapping memory a page at a time.

Multiply any row by 16384 for bytes. `Pages free`: truly untouched. `Pages active`: in active use. `Pages inactive`: in use but cold and reclaimable. `Pages wired down`: kernel-pinned, never reclaimed. `free` looks tiny — that's normal. Inactive memory is effectively available the moment anything needs it."

### Exercise — convert units

**SPEAK**
> "Let's turn that huge byte count into something human-readable — and see why the number on your spec sheet is a little bit of a fib."

**TYPE**
```bash
python3.12 -c "b = 34359738368; print(f'{b / 1000**3:.2f} GB, {b / 1024**3:.2f} GiB')"
```

**OUTPUT**
```
34.36 GB, 32.00 GiB
```

**EXPLAIN**
> "Quick tour of the code. `b = 34359738368` is the byte count we read from `hw.memsize` a moment ago. We divide it two ways. `1000**3` is one billion — the normal 'giga-' you know from gigahertz. That gives us **GB**, the decimal ruler. `1024**3` is 1024 cubed, a bit more than a billion — the computer's native ruler, because everything inside is built from bits and bits come in powers of two. That gives us **GiB**, the binary ruler. Same pile of bytes, two different labels. The `:.2f` just rounds each result to 2 decimal places so the output is clean.
>
> Now the reason this matters. My spec sheet says this Mac has '32 GB' of RAM. But the true decimal number is 34.36 GB. Apple is using 'GB' loosely to mean GiB, because '32' is the clean, recognizable number and RAM is always built in powers of two. Storage companies do the opposite — a '500 GB' SSD really holds 500 billion bytes, and your OS shows it as 465 GiB. Same trick, different direction: whoever labels the box picks whichever ruler makes the number look best.
>
> The habit to build: don't trust the units on the box. Trust the byte count. It's the one thing that's physically real — there are literally that many storage cells in the chip. GB and GiB are just labels we put on top of it."

### What is a disk?

**SPEAK**
> "Below RAM in the pyramid: disk. Huge, slow, persistent. Not one flat slab — sliced into partitions, each holding a filesystem, each mounted at a directory. `diskutil list` shows the whole tree."

### Run the demo

**TYPE**
```bash
diskutil list | head -20
```

**OUTPUT**
```
/dev/disk0 (internal, physical):
   #:                       TYPE NAME                    SIZE       IDENTIFIER
   0:      GUID_partition_scheme                        *500.3 GB   disk0
   1:             Apple_APFS_ISC Container disk1         524.3 MB   disk0s1
   2:                 Apple_APFS Container disk3         494.4 GB   disk0s2
   3:        Apple_APFS_Recovery Container disk2         5.4 GB     disk0s3

/dev/disk3 (synthesized):
   #:                       TYPE NAME                    SIZE       IDENTIFIER
   0:      APFS Container Scheme -                      +494.4 GB   disk3
                                 Physical Store disk0s2
   1:                APFS Volume Macintosh HD - Data     218.0 GB   disk3s1
   2:                APFS Volume Macintosh HD            11.3 GB    disk3s3
   3:              APFS Snapshot com.apple.os.update-... 11.3 GB    disk3s3s1
   4:                APFS Volume Preboot                 8.1 GB     disk3s4
   5:                APFS Volume Recovery                2.0 GB     disk3s5
   6:                APFS Volume VM                      4.3 GB     disk3s6
```

**EXPLAIN**
> "Two sections. `/dev/disk0 (internal, physical)` is the real SSD — 500 GB, carved into three APFS containers. The big one is `disk0s2`. Below that, `/dev/disk3 (synthesized)` is *not* a second disk — it's a logical view of that container, with six volumes (Macintosh HD, Data, Preboot, Recovery, VM, and a snapshot) all sharing one 494 GB pool. That's APFS's container model. Classical partitioning on Windows or Linux locks each partition to a fixed size; APFS drops that."

**TYPE**
```bash
df -h | head -3
```

**OUTPUT**
```
Filesystem        Size    Used   Avail Capacity iused ifree %iused  Mounted on
/dev/disk3s3s1   460Gi    10Gi   233Gi     5%    426k  2.4G    0%   /
devfs            202Ki   202Ki     0Bi   100%     698     0  100%   /dev
```

**EXPLAIN**
> "Different question: for each *mounted* filesystem, how big and how full? The top row — root, `/`, backed by `disk3s3s1` — is the one that matters. 460 GiB total, 10 used, 233 available. `Avail` is the column you care about when asking 'do I have room for this?' `df -h` works identically on Linux and macOS — same command, same columns."

```

**EXPLAIN**
> "Different question than `diskutil list`: for each *mounted* filesystem, how big and how full? Every row is one filesystem. `Avail` is the column you care about when asking 'do I have room for this?' The row that matters is the first one — `/dev/disk3s1s1` mounted at `/`, the root of the whole filesystem, with 157 GiB available. That's where your files actually live. The other `disk3sN` rows are system housekeeping volumes (Data, Preboot, Update, VM) sharing the same 460 GiB APFS container we saw in `diskutil list`.
>
> Now the weird rows, because this is where an important idea lives. `devfs` at `/dev` is 200 KiB total, 100% full, and isn't storage at all — it's a **virtual filesystem**. Unix has a core design principle: *everything is a file*. Your keyboard, mic, disks, terminal — all appear as entries under `/dev/`. When a program wants to read from the mic, it opens a file in `/dev/` and reads bytes; the kernel translates that into a real hardware operation behind the scenes. `devfs` is the driver that makes that illusion work. `map auto_home` at the bottom is similar — it's not a filesystem, just a rule the kernel follows to mount home directories on demand.
>
> The pattern: the kernel presents things that aren't storage as if they were filesystems, because once everything looks like a filesystem, every program can talk to it the same way. `disk3s1s1` is real flash; `devfs` is hardware dressed up as files; `map auto_home` is a rule pretending to be a directory. All three show up in `df` because from a program's point of view, they all just look like places you can read and write bytes. `df -h` works identically on Linux and macOS."

## PART 4 — SEEING A FILE AS BYTES

### What is a file at the byte level?

**SPEAK**
> "Lecture 1.1 said everything is bytes. Now you see it. `xxd` takes any file and prints the raw hex. Text, images, anything — every file on your disk is just a sequence of bytes. Let's open a text file and prove it."

### Run the demo

**TYPE**
```bash
cd /tmp
echo 'Hello!' > hello.txt
xxd hello.txt
```

**OUTPUT**
```
00000000: 4865 6c6c 6f21 0a                        Hello!.
```

**EXPLAIN**
> "The whole file on one line. Three columns: offset on the left (`00000000` = byte zero), hex bytes in the middle (grouped in pairs for readability — really seven distinct bytes: 48 65 6c 6c 6f 21 0a), and the ASCII column on the right. 48 is `H`, 65 is `e`, 6c is `l`, 6f is `o`, 21 is `!`. The trailing `.` in the gutter isn't a period — it's xxd's placeholder for byte `0a`, the newline `echo` added at the end. This file is literally those seven numbers — nothing more."

**TYPE**
```bash
xxd -b hello.txt
```

**OUTPUT**
```
00000000: 01001000 01100101 01101100 01101100 01101111 00100001  Hello!
00000006: 00001010                                               .
```

**EXPLAIN**
> "Same bytes, now as literal bits. `01001000` is H — eight flash memory cells on the SSD storing that pattern right now. This is what lecture 1.2 was pointing at: bits are physical state, bytes are groups of eight bits, files are sequences of bytes. The slide and the screen are the same thing."

### Exercise — peek inside a PNG

**SPEAK**
> "Text files are a friendly case — the bytes map one-to-one to characters you can read. Let's try a file where the bytes aren't text at all. We'll download a PNG image and look at its first 16 bytes."

**TYPE**
```bash
curl -sO https://www.libpng.org/pub/png/img_png/pnglogo-blk.png
xxd -l 16 pnglogo-blk.png
```

**OUTPUT**
```
00000000: 8950 4e47 0d0a 1a0a 0000 000d 4948 4452  .PNG........IHDR
```

**EXPLAIN**
> "Sixteen bytes of a PNG, every byte deliberate. Bytes 1-8 — `89 50 4e 47 0d 0a 1a 0a` — are the PNG signature, a fixed sequence every PNG file on Earth starts with. Byte 1 (`89`) has the high bit set to catch 7-bit-stripping file transfers. Bytes 2-4 spell `PNG` in ASCII — you can see it in the gutter on the right. Bytes 5-6 (`0d 0a`) are CR-LF, to catch line-ending mangling. Byte 7 (`1a`) is a DOS control-Z. Byte 8 is another LF. After the signature: `00 00 00 0d` is a length field (13 in decimal), and `49 48 44 52` spells `IHDR` — the name of the first chunk inside the PNG. Every image viewer on Earth checks these bytes to recognize a PNG. Not the filename — the bytes. You could rename this file `cat.jpg` and Preview would still open it as a PNG, because the bytes don't lie."

### Exercise — identify a file by its magic number

**SPEAK**
> "Almost every file format announces itself in its first few bytes. Let's build the habit of reading that signature. I'll copy a file from macOS itself — the `ls` command — and look at its first four bytes."

**TYPE**
```bash
xxd -l 4 /bin/ls
```

**OUTPUT**
```
00000000: cffa edfe                                ....
```

**EXPLAIN**
> "`cf fa ed fe` is the Mach-O magic number — the signature every macOS executable starts with. `Mach-O` is Apple's executable format, same way PNG is the image format. When you double-click an app or type `ls` at the prompt, the kernel reads these four bytes first, sees Mach-O, and knows how to load the file into memory and run it. The ASCII gutter shows `....` because none of these bytes are printable characters — that's expected for a binary file. Try it at home with other files: a JPEG starts with `ff d8 ff`, a PDF with `25 50 44 46` (which spells `%PDF` in ASCII), a Python script with `23 21` (`#!`). The `file` command you may have used before? It works by reading these exact bytes and matching against a magic-number table. You now know the trick it uses."

## PART 5 — SAME BYTES, NOW IN PYTHON

### What does it mean for a number to be in a base?

**SPEAK**
> "You saw `50 4e 47` spell PNG in the xxd output. Let's prove it in Python, one one-liner at a time."

### Run the demo

**TYPE**
```bash
python3.12 -c "print(hex(80))"
```

**OUTPUT**
```
0x50
```

**TYPE**
```bash
python3.12 -c "print(bin(80))"
```

**OUTPUT**
```
0b1010000
```

**TYPE**
```bash
python3.12 -c "print(ord('P'))"
python3.12 -c "print(chr(80))"
```

**OUTPUT**
```
80
P
```

**EXPLAIN**
> "One number, four views. `hex(80)` → `'0x50'` (the `0x` is the literal-prefix Python uses in source). `bin(80)` → `'0b1010000'`. `ord('P')` → 80 (P's ASCII code). `chr(80)` → `'P'`. Same 80 every time. `hex`/`bin` return strings; `ord`/`chr` bridge characters and numbers."

**TYPE**
```bash
python3.12 -c "print(int('50', 16))"
python3.12 -c "print(int('01010000', 2))"
```

**OUTPUT**
```
80
80
```

**EXPLAIN**
> "`int(string, base)` parses a string in any base 2–36. Feed it `'50'` with base 16, or `'01010000'` with base 2 — both give back 80. This is the inverse of the `hex` and `bin` display functions. The round-trip is lossless."

**TYPE**
```bash
python3.12 -c "print(f'{15:02x}')"
python3.12 -c "print(f'{5:08b}')"
```

**OUTPUT**
```
0f
00000101
```

**EXPLAIN**
> "f-string format specs. `:02x` means 'hex, 2 digits minimum, zero-padded' — matches xxd's per-byte format. `:08b` means 'binary, 8 digits, zero-padded' — one full byte of bits. Without the padding, `hex(15)` and `bin(5)` would give you `0xf` and `0b101` — too short to line up in a byte-oriented dump."

**TYPE**
```bash
python3.12 -c "print(bytes.fromhex('89504e47'))"
```

**OUTPUT**
```
b'\x89PNG'
```

**EXPLAIN**
> "This is the payoff. `89504e47` is the first four bytes of the PNG from Part 4. `bytes.fromhex()` turns that hex string into actual bytes. Python's repr shows printable bytes as characters (`P`, `N`, `G`) and non-printable ones as `\\xNN` escapes (`\\x89`). Same four bytes xxd showed us earlier — two tools, same physical file content."

### Exercise — your own initial

**TYPE**
```bash
python3.12 -c "c='J'; print(f'{c!r} -> dec={ord(c)}, hex={ord(c):02x}, bin={ord(c):08b}')"
```

**OUTPUT**
```
'J' -> dec=74, hex=4a, bin=01001010
```

**EXPLAIN**
> "`J` is decimal 74, hex 4a, binary 01001010. Try lowercase `j` and you'll get 106, 6a, 01101010. Compare the binary: exactly one bit different — the 0x20 bit. That single bit is the ASCII case-toggle; XOR it to flip between upper and lower."
