## PART 1 — SEEING THE CPU

### What is the CPU?

**SPEAK**
> "The CPU is the chip that runs your code. It has an architecture — what instruction set it speaks — a number of cores, and a cache hierarchy. Lecture 1.4 drew the memory pyramid. L1, L2, L3 caches sit right next to the cores. RAM sits below that. Each tier is bigger and slower. The OS knows every one of these numbers; you just have to ask it."

### Run the demo

**TYPE**
```bash
sysctl -n machdep.cpu.brand_string
```

**OUTPUT**
```
Apple M1 Pro
```

**EXPLAIN**
> "That's the chip. On my Mac, an M1 Pro. Yours could be M2, M3, M4, an Intel i7 — whatever you have, `sysctl` will tell you."

**TYPE**
```bash
sysctl -n hw.physicalcpu
sysctl -n hw.logicalcpu
```

**OUTPUT**
```
8
8
```

**EXPLAIN**
> "Eight physical cores, eight logical cores. On an Apple Silicon chip those always match because there's no simultaneous multithreading. On an Intel Mac with Hyper-Threading you'd often see 4 physical and 8 logical — one real core pretending to be two."

**TYPE**
```bash
sysctl -n hw.perflevel0.l1dcachesize
sysctl -n hw.perflevel0.l2cachesize
```

**OUTPUT**
```
131072
12582912
```

**EXPLAIN**
> "Two numbers in bytes. First: 131,072 — that's 128 KiB of L1 data cache per performance core. Second: 12.5 MiB of L2 shared across the performance cluster. Apple Silicon splits cores into `perflevel0` — the fast cluster — and `perflevel1` — the efficiency cluster. This is the pyramid. Tiny L1. Bigger L2. RAM is going to be bigger still."

**TYPE**
```bash
system_profiler SPHardwareDataType | grep -E "Model Name|Chip|Total Number of Cores|Memory"
```

**OUTPUT**
```
      Model Name: MacBook Pro
      Chip: Apple M1 Pro
      Total Number of Cores: 8 (6 performance and 2 efficiency)
      Memory: 32 GB
```

**EXPLAIN**
> "One command, the whole summary card. Notice `6 performance and 2 efficiency` — that's the P/E breakdown. This is what the spec sheet says."

### Exercise — cache ratio

**SPEAK**
> "Let's see how much bigger L2 is than L1."

**TYPE**
```bash
python3.12 -c "print(12582912 / 131072)"
```

**OUTPUT**
```
96.0
```

**EXPLAIN**
> "L2 is 96 times bigger than L1 on this chip. That's the pyramid — every tier you drop down, the storage jumps roughly 2 orders of magnitude."

## PART 2 — SEEING RAM

### What is RAM?

**SPEAK**
> "RAM is the scratchpad. Your programs, their variables, their open files — all live in RAM while they run. macOS tracks it in categories: free, active, inactive, wired, compressed. We'll see each."

### Run the demo

**TYPE**
```bash
sysctl -n hw.memsize
```

**OUTPUT**
```
34359738368
```

**EXPLAIN**
> "That's the total installed RAM in bytes. Divide by 1024 cubed to get GiB — it's 32."

**TYPE**
```bash
vm_stat | head -7
```

**OUTPUT**
```
Mach Virtual Memory Statistics: (page size of 16384 bytes)
Pages free:                               78745.
Pages active:                            507575.
Pages inactive:                          444596.
Pages speculative:                        82603.
Pages throttled:                              0.
Pages wired down:                        152544.
```

**EXPLAIN**
> "`vm_stat` reports in pages. That first line: `page size of 16384 bytes` — 16 KiB on Apple Silicon, not 4 KiB. Multiply any row by 16384 for bytes. `Pages free`: truly unused, usually small. `Pages active`: being read or written right now. `Pages inactive`: in use but cold — droppable. `Pages wired down`: kernel-pinned, never reclaimed."

### Exercise — convert units

**SPEAK**
> "Let me prove the OS's GiB and the spec sheet's GB are different numbers for the same thing."

**TYPE**
```bash
python3.12 -c "b = 34359738368; print(f'{b / 1024**3:.2f} GiB, {b / 1000**3:.2f} GB')"
```

**OUTPUT**
```
32.00 GiB, 34.36 GB
```

**EXPLAIN**
> "Spec sheets use decimal GB — 1 billion bytes. The OS uses GiB — 1024 cubed bytes. That's why a 'SIZE GB' disk always shows as a smaller number in Finder. The difference is 7%."

## PART 3 — SEEING DISKS

### What is a disk?

**SPEAK**
> "Below RAM in the pyramid: disk. Huge, slow, persistent. Not one flat slab — sliced into partitions, each holding a filesystem, each mounted at a directory. `diskutil list` shows the whole tree."

### Run the demo

**TYPE**
```bash
diskutil list | head -20
```

**OUTPUT**
```
/dev/disk0 (internal, physical):
   #:                       TYPE NAME                    SIZE       IDENTIFIER
   0:      GUID_partition_scheme                        *500.3 GB   disk0
   1:             Apple_APFS_ISC Container disk1         524.3 MB   disk0s1
   2:                 Apple_APFS Container disk3         494.4 GB   disk0s2
   3:        Apple_APFS_Recovery Container disk2         5.4 GB     disk0s3

/dev/disk3 (synthesized):
   #:                       TYPE NAME                    SIZE       IDENTIFIER
   0:      APFS Container Scheme -                      +494.4 GB   disk3
                                 Physical Store disk0s2
   1:                APFS Volume Macintosh HD - Data     218.0 GB   disk3s1
   2:                APFS Volume Macintosh HD            11.3 GB    disk3s3
   3:              APFS Snapshot com.apple.os.update-... 11.3 GB    disk3s3s1
   4:                APFS Volume Preboot                 8.1 GB     disk3s4
   5:                APFS Volume Recovery                2.0 GB     disk3s5
   6:                APFS Volume VM                      4.3 GB     disk3s6
```

**EXPLAIN**
> "`disk0` is the real physical SSD — 500 GB. It's cut into three APFS containers. Then `disk3` is *synthesized* — it's not a second disk, it's a logical view of the big APFS container holding Macintosh HD, Data, Preboot, Recovery, and VM. All those volumes share the same pool of free space. That's the APFS model."

**TYPE**
```bash
df -h | head -3
```

**OUTPUT**
```
Filesystem        Size    Used   Avail Capacity iused ifree %iused  Mounted on
/dev/disk3s3s1   460Gi    10Gi   233Gi     5%    426k  2.4G    0%   /
devfs            202Ki   202Ki     0Bi   100%     698     0  100%   /dev
```

**EXPLAIN**
> "`df -h` answers: for each mounted filesystem, how big and how full? Top row is root — `/` — backed by `disk3s3s1`, the snapshot volume. 10 GiB used, 233 available. That's where every file you create lives."

### Exercise — find your home partition

**TYPE**

**OUTPUT**

## PART 4 — SEEING A FILE AS BYTES

### What is a file at the byte level?

**SPEAK**
> "Lecture 1.1 said everything is bytes. Now you see it. `xxd` takes any file and prints the raw hex. Text, images, anything — all bytes on your disk. Let's open a text file."

### Run the demo

**TYPE**
```bash
cd /tmp
echo 'Hello!' > hello.txt
xxd hello.txt
```

**OUTPUT**
```
00000000: 4865 6c6c 6f21 0a                        Hello!.
```

**EXPLAIN**
> "Three columns. Offset — byte 0. Hex bytes — `48 65 6c 6c 6f 21 0a`. ASCII — `Hello!` plus a dot for the newline, because newline isn't a printable character. `48` is `H`. `65` is `e`. Every letter is a byte."

**TYPE**
```bash
xxd -b hello.txt
```

**OUTPUT**
```
00000000: 01001000 01100101 01101100 01101100 01101111 00100001  Hello!
00000006: 00001010                                               .
```

**EXPLAIN**
> "Same file, bits not hex. `01001000` is H. Eight bits. Lecture 1.2 said gates combine into flip-flops, each holding one bit. Those eight zeros and ones are the actual state of eight flip-flops somewhere on my SSD."

**TYPE**
```bash
curl -sO https://www.libpng.org/pub/png/img_png/pnglogo-blk.png
xxd -l 16 pnglogo-blk.png
```

**OUTPUT**
```
00000000: 8950 4e47 0d0a 1a0a 0000 000d 4948 4452  .PNG........IHDR
```

**EXPLAIN**
> "The first 8 bytes of every PNG on Earth: `89 50 4e 47 0d 0a 1a 0a`. That's the PNG magic number. The ASCII column shows `.PNG........IHDR` — the dot is byte `89`, which isn't printable. Then `PNG`. Then control bytes for format detection. Then the IHDR chunk. Every image viewer checks these bytes to decide what it's looking at. Not the filename — the bytes."

### Exercise — dump your own file

**TYPE**
```bash
xxd -l 32 /tmp/hello.txt
```

**OUTPUT**
```
00000000: 4865 6c6c 6f21 0a                        Hello!.
```

**EXPLAIN**
> "Or try any file in your home. A source file. An image. A PDF. `-l 32` limits to the first 32 bytes. The first 2-4 usually identify the format."

## PART 5 — SAME BYTES, NOW IN PYTHON

### What does it mean for a number to be in a base?

**SPEAK**
> "You saw `50 4e 47` spell PNG in the xxd output. Let's prove it in Python, one one-liner at a time."

### Run the demo

**TYPE**
```bash
python3.12 -c "print(hex(80))"
```

**OUTPUT**
```
0x50
```

**TYPE**
```bash
python3.12 -c "print(bin(80))"
```

**OUTPUT**
```
0b1010000
```

**TYPE**
```bash
python3.12 -c "print(ord('P'))"
python3.12 -c "print(chr(80))"
```

**OUTPUT**
```
80
P
```

**EXPLAIN**
> "One number, 80, shown four ways. Hex `0x50`. Binary `0b1010000`. Decimal `80`. Character `P`. `ord` and `chr` are the pair that go back and forth between character and integer. Same number every time."

**TYPE**
```bash
python3.12 -c "print(int('50', 16))"
python3.12 -c "print(int('01010000', 2))"
```

**OUTPUT**
```
80
80
```

**EXPLAIN**
> "`int` with a base parses a string. Parse `50` as base 16, get 80. Parse `01010000` as base 2, get 80. Every representation round-trips."

**TYPE**
```bash
python3.12 -c "print(f'{15:02x}')"
python3.12 -c "print(f'{5:08b}')"
```

**OUTPUT**
```
0f
00000101
```

**EXPLAIN**
> "f-string format specs. `:02x` means hex, minimum 2 digits, zero-padded. That's what `xxd` uses — always two hex digits per byte. `:08b` means binary, 8 digits, zero-padded. One full byte of bits."

**TYPE**
```bash
python3.12 -c "print(bytes.fromhex('89504e47'))"
```

**OUTPUT**
```
b'\x89PNG'
```

**EXPLAIN**
> "The exact bytes you saw at the start of that PNG. `\x89` for the non-printable byte, then literal `PNG` because those bytes are printable characters. Python and xxd describing the same bytes from two directions."

### Exercise — your own initial

**TYPE**
```bash
python3.12 -c "c='J'; print(f'{c!r} -> dec={ord(c)}, hex={ord(c):02x}, bin={ord(c):08b}')"
```

**OUTPUT**
```
'J' -> dec=74, hex=4a, bin=01001010
```

**EXPLAIN**
> "J is decimal 74, hex 4a, binary 01001010. Change `J` to your own initial and run it. If you try uppercase and lowercase of the same letter, the difference is always exactly one bit — 0x20. That bit is the case toggle."

## Recording notes

- Keep each command visible for ~3 seconds before moving on.
- At the `89 50 4e 47` PNG magic number, pause 2 seconds — this is the "aha" moment.
- Cross-reference: when showing the PNG magic number, verbally call back to the Python section where `bytes.fromhex('89504e47')` will produce `b'\x89PNG'`.

- Python one-liners in separate `TYPE` blocks (not chained with `;`) — easier for viewers to copy.
