## PART 1 — SEEING THE CPU

### What is a CPU introspection command?

**SPEAK**
> "Every program you run becomes instructions executed by the chip inside your Mac. Before we can reason about what's happening, we need to be able to name the chip: how many cores it has, what model it is, how much fast on-chip memory called cache it owns. macOS exposes all of that through a dotted namespace called sysctl. The kernel already tracks every one of these numbers to do its job — sysctl is the front door we use to ask."

### Run the demo

**TYPE**
```bash
sysctl -n machdep.cpu.brand_string
```

**OUTPUT**
```
Apple M1 Pro
```
*(Your chip name will look different — an M2, M3, or Intel model.)*

**EXPLAIN**
> "That's the exact model name the kernel knows my CPU by. No drivers, no API — just a name lookup in the kernel's MIB tree."

**TYPE**
```bash
sysctl -n hw.ncpu
```

**OUTPUT**
```
8
```

**EXPLAIN**
> "Eight logical CPUs. Each one is one independent stream of instructions the kernel can schedule work onto."

**TYPE**
```bash
sysctl hw machdep.cpu | head -20
```

**OUTPUT**
```
hw.ncpu: 8
hw.byteorder: 1234
hw.memsize: 34359738368
hw.activecpu: 8
hw.perflevel0.physicalcpu: 6
hw.perflevel0.physicalcpu_max: 6
hw.perflevel0.logicalcpu: 6
hw.perflevel0.logicalcpu_max: 6
hw.perflevel0.l1icachesize: 196608
hw.perflevel0.l1dcachesize: 131072
hw.perflevel0.l2cachesize: 12582912
hw.perflevel0.cpusperl2: 3
hw.perflevel0.name: Performance
hw.perflevel1.physicalcpu: 2
hw.perflevel1.physicalcpu_max: 2
hw.perflevel1.logicalcpu: 2
hw.perflevel1.logicalcpu_max: 2
hw.perflevel1.l1icachesize: 131072
hw.perflevel1.l1dcachesize: 65536
hw.perflevel1.l2cachesize: 4194304
```

**EXPLAIN**
> "Every line is one fact the kernel knows about this chip. Notice the two `perflevel` sections — that's because Apple Silicon physically has two kinds of cores on the same die. `perflevel0` is the six fast Performance cores; `perflevel1` is the two power-efficient Efficiency cores. `l1dcachesize: 131072` means each P-core has 128 kilobytes of L1 data cache. That is the fastest memory on the whole chip."

### Exercise — L2 cache in KiB

**SPEAK**
> "Now you try. Grab the L2 cache size for your Performance cores and print it in kilobytes. The raw value is in bytes, so you'll need to divide by 1024."

**TYPE**
```bash
echo "$(( $(sysctl -n hw.perflevel0.l2cachesize) / 1024 )) KiB"
```

**OUTPUT**
```
12288 KiB
```

**EXPLAIN**
> "12288 KiB — that's 12 mebibytes of L2 cache shared across the P-cores on my M1 Pro. The kernel gave us bytes. Division is our job. That's a theme — OS tells the truth, human units are always somebody else's problem."

## PART 2 — SEEING THE RAM

### What is memory introspection?

**SPEAK**
> "RAM is the pool of fast storage every running program lives in. Two questions come before anything interesting: how much RAM does this machine have, and how much is in use right now. macOS answers the first with one sysctl entry, and the second with a lower-level tool called `vm_stat`."

### Run the demo

**TYPE**
```bash
sysctl -n hw.memsize
```

**OUTPUT**
```
34359738368
```

**EXPLAIN**
> "That is the exact number of bytes of physical RAM this Mac has. A ten-digit integer. Unreadable — on purpose. The kernel's job is correctness, not presentation."

**TYPE**
```bash
echo "scale=2; $(sysctl -n hw.memsize) / 1073741824" | bc
```

**OUTPUT**
```
32.00
```

**EXPLAIN**
> "Divide by 2 to the 30th — that's 1073741824 — and you get gibibytes. 32.00. That's exactly what Activity Monitor shows you. Now you know what Activity Monitor is doing behind the scenes."

**TYPE**
```bash
vm_stat | head -12
```

**OUTPUT**
```
Mach Virtual Memory Statistics: (page size of 16384 bytes)
Pages free:                                3803.
Pages active:                            483734.
Pages inactive:                          482071.
Pages speculative:                          947.
Pages throttled:                              0.
Pages wired down:                        160882.
Pages purgeable:                              4.
"Translation faults":               11107401219.
Pages copy-on-write:                  115147619.
Pages zero filled:                   5606101891.
Pages reactivated:                     75600017.
```

**EXPLAIN**
> "Read the header — `page size of 16384 bytes`. That's the Apple Silicon page size; Intel Macs use 4096. Every number below is a count of pages, not bytes. `Pages wired down` is kernel memory that can't be swapped out. `Pages active` is what your running apps are holding. The rest is various flavors of reclaimable."

### Exercise — free memory in MiB

**SPEAK**
> "Your turn. Compute the free memory in mebibytes by multiplying the `Pages free` count by the page size and dividing by 2 to the 20th."

**TYPE**
```bash
echo "$(( $(vm_stat | awk '/Pages free/ {gsub("\\.","",$3); print $3}') * 16384 / 1048576 )) MiB"
```

**OUTPUT**
```
101 MiB
```

**EXPLAIN**
> "101 MiB free. Surprisingly small on a 32 GiB machine — that's because macOS uses almost all of RAM for caches and compressed pages. It's not actually low; Activity Monitor's Memory Pressure gauge is the honest measure."

## PART 3 — SEEING THE DISK

### What is a disk layout tool?

**SPEAK**
> "Your physical SSD is one device, but what you interact with — folders, apps, the Macintosh HD icon — lives inside a nested tree of APFS containers and volumes on top of it. Two commands pull the curtain back."

### Run the demo

**TYPE**
```bash
df -h /
```

**OUTPUT**
```
Filesystem        Size    Used   Avail Capacity iused ifree %iused  Mounted on
/dev/disk3s3s1   460Gi    10Gi   232Gi     5%    426k  2.4G    0%   /
```

**EXPLAIN**
> "`df` reads the mount table. The root filesystem is mounted at `/` on the device `/dev/disk3s3s1` — that weird name is the first snapshot of the third slice of the third disk. The next command is where that makes sense."

**TYPE**
```bash
diskutil list
```

**OUTPUT**
```
/dev/disk0 (internal, physical):
   #:                       TYPE NAME                    SIZE       IDENTIFIER
   0:      GUID_partition_scheme                        *500.3 GB   disk0
   1:             Apple_APFS_ISC Container disk1         524.3 MB   disk0s1
   2:                 Apple_APFS Container disk3         494.4 GB   disk0s2
   3:        Apple_APFS_Recovery Container disk2         5.4 GB     disk0s3

/dev/disk3 (synthesized):
   #:                       TYPE NAME                    SIZE       IDENTIFIER
   0:      APFS Container Scheme -                      +494.4 GB   disk3
                                 Physical Store disk0s2
   1:                APFS Volume Macintosh HD - Data     219.4 GB   disk3s1
   2:                APFS Volume Macintosh HD            11.3 GB    disk3s3
   3:              APFS Snapshot com.apple.os.update-... 11.3 GB    disk3s3s1
   4:                APFS Volume Preboot                 8.1 GB     disk3s4
   5:                APFS Volume Recovery                2.0 GB     disk3s5
   6:                APFS Volume VM                      4.3 GB     disk3s6
```

**EXPLAIN**
> "Two sections. `disk0` is the real 500-gigabyte SSD. `disk3` is labeled `synthesized` — it doesn't physically exist. It's the APFS container that APFS exports so you can refer to the pool of volumes under it. Inside that container are six volumes — Macintosh HD, Data, Preboot, Recovery, VM — and crucially they do not have fixed sizes. They all share the container's free space. That is what APFS is for."

### Exercise — how many System volumes

**SPEAK**
> "Your turn. List every mounted filesystem and count how many are mounted under `/System/Volumes/`."

**TYPE**
```bash
df -h | grep "/System/Volumes/" | wc -l
```

**OUTPUT**
```
       8
```

**EXPLAIN**
> "Eight mounts under `/System/Volumes/` on this machine. What you think of as one disk is many filesystems stitched together at boot. macOS uses that structure to separate the read-only system volume from the read-write data volume and keep firmware updates isolated."

## PART 4 — SEEING FILES AS BYTES

### What is `xxd`?

**SPEAK**
> "Every file on your disk is a sequence of bytes. A byte is a number from zero to 255. When Preview opens a PNG it interprets those bytes as pixels; when TextEdit opens a text file it interprets them as characters. The file itself doesn't know the difference. `xxd` lets us stop interpreting and just see the numbers."

### Detour — working directory

**SPEAK**
> "Quick detour before we dump any files, because relative paths matter."

**TYPE**
```bash
pwd
```

**OUTPUT**
```
/Users/giacobbe/src/courses/anyone-can-code/1.5
```

**EXPLAIN**
> "`pwd` prints where I am. That's where `xxd sample.png` will look when I don't give a full path."

**TYPE**
```bash
cd
pwd
```

**OUTPUT**
```
/Users/giacobbe
```

**EXPLAIN**
> "`cd` with no argument is a shortcut to home. I'm now in `/Users/giacobbe`."

**TYPE**
```bash
cd -
pwd
```

**OUTPUT**
```
/Users/giacobbe/src/courses/anyone-can-code/1.5
```

**EXPLAIN**
> "`cd -` jumps back to the previous directory. On zsh it does it silently, but `pwd` confirms. That's the pair that lets you bounce between two places without retyping paths."

### Run the demo

**TYPE**
```bash
echo "Hello, world!" > hello.txt
xxd hello.txt
```

**OUTPUT**
```
00000000: 4865 6c6c 6f2c 2077 6f72 6c64 210a       Hello, world!.
```

**EXPLAIN**
> "Three columns. Offset on the left — we're at byte zero. Sixteen bytes of hex in the middle, though only fourteen are shown because that's how long the file is. ASCII rendering on the right. `0x48` is the letter H. `0x0a` at the end is the newline `echo` added."

**TYPE**
```bash
xxd -l 16 sample.png
```

**OUTPUT**
```
00000000: 8950 4e47 0d0a 1a0a 0000 000d 4948 4452  .PNG........IHDR
```

**EXPLAIN**
> "Look at the right column. `.PNG` — that's the magic signature that marks this file as a PNG. Every PNG on every operating system starts with exactly these eight bytes: 89 50 4e 47 0d 0a 1a 0a. The file extension is a hint; this is the truth."

**TYPE**
```bash
xxd sample.png | head -6
```

**OUTPUT**
```
00000000: 8950 4e47 0d0a 1a0a 0000 000d 4948 4452  .PNG........IHDR
00000010: 0000 0010 0000 0010 0802 0000 0090 9168  ...............h
00000020: 3600 0000 1749 4441 5478 9c63 f8cf c040  6....IDATx.c...@
00000030: 1222 4df5 a886 510d 434a 0300 90f9 ff01  ."M...Q.CJ......
00000040: f9e1 fa78 0000 0000 4945 4e44 ae42 6082  ...x....IEND.B`.
```

**EXPLAIN**
> "Five rows for the whole file — that's the advantage of shipping a tiny 80-byte PNG. Look at the ASCII column: `IHDR` at the end of row one, `IDAT` in row two, `IEND` in row four. Those are the three PNG chunks the spec requires in that exact order, and we're looking at them directly. No libraries, no parsers."

### Exercise — uppercase vs lowercase

**SPEAK**
> "Your turn. Write a file containing just the letter A uppercase, dump it, then write a file with lowercase a and dump that. Explain the difference."

**TYPE**
```bash
printf 'A' > a.txt
xxd a.txt
printf 'a' > b.txt
xxd b.txt
rm a.txt b.txt
```

**OUTPUT**
```
00000000: 41                                       A
00000000: 61                                       a
```

**EXPLAIN**
> "`A` is `0x41`; `a` is `0x61`. That's bit 5 flipped — 32 in decimal. Every ASCII letter works the same way: uppercase-to-lowercase is one bit. You just saw a 50-year-old encoding decision with your own eyes."

## PART 5 — SEEING BYTES AS NUMBERS, LETTERS, AND BITS

### What is base conversion?

**SPEAK**
> "Humans count in base ten. Computers store numbers in base two. Hex is the shorthand programmers use for base two because one hex digit equals exactly four bits — a byte is always two hex digits. All three are the same number, written differently. Python has five built-in functions that convert between them."

### Run the demo

**TYPE**
```bash
python3.12 -c "print(hex(72))"
```

**OUTPUT**
```
0x48
```

**EXPLAIN**
> "`hex()` takes an integer and gives you a string with a `0x` prefix. 72 in base 16 is 48 — four times sixteen is 64, plus 8, equals 72."

**TYPE**
```bash
python3.12 -c "print(bin(72))"
```

**OUTPUT**
```
0b1001000
```

**EXPLAIN**
> "Same number in binary. Seven bits — one 64, one 8, zeros everywhere else."

**TYPE**
```bash
python3.12 -c "print(int('0x48', 16))"
```

**OUTPUT**
```
72
```

**EXPLAIN**
> "Round trip. `int(s, 16)` parses a string as base 16. We just took the string `0x48` and got back the integer 72."

**TYPE**
```bash
python3.12 -c "print(ord('H'))"
python3.12 -c "print(chr(72))"
```

**OUTPUT**
```
72
H
```

**EXPLAIN**
> "`ord` gives you the code point of a character; `chr` goes the other way. Capital H is the number 72. And that number is `0x48`. And `0x48` was the first byte of `hello.txt`. Four representations, one number."

**TYPE**
```bash
python3.12 -c "print(chr(int('48', 16)))"
```

**OUTPUT**
```
H
```

**EXPLAIN**
> "Composed two functions in one line. The hex string goes to an integer, the integer goes to a character. This is how you'd decode a file header by hand."

**TYPE**
```bash
python3.12 -c "print(''.join(chr(int(b,16)) for b in ['48','65','6c','6c','6f']))"
```

**OUTPUT**
```
Hello
```

**EXPLAIN**
> "Those five hex values came straight from the `xxd` dump of `hello.txt`. We just rebuilt the word `Hello` from its bytes, with stdlib Python, in one line. That's the whole payoff of the lab."

### Exercise — 255 in three bases

**SPEAK**
> "Your turn. Print the number 255 in decimal, hex, and binary on three separate lines using one Python invocation."

**TYPE**
```bash
python3.12 -c "n=255; print(n); print(hex(n)); print(bin(n))"
```

**OUTPUT**
```
255
0xff
0b11111111
```

**EXPLAIN**
> "255 is `0xff` is `0b11111111` — the largest value a single byte can hold. All three mean the exact same thing."

## Recording notes

- Use an M1 or newer Apple Silicon Mac for the capture — the `perflevel0` / `perflevel1` split is a key teaching beat in Part 1 that Intel Macs don't show.
- Pause 2 seconds after each OUTPUT block before narrating, so the viewer has time to read.
- Terminal font size 18+ recommended; `sysctl hw machdep.cpu | head -20` is the widest output in the script — make sure it fits without wrapping.
- Part 4's detour about `pwd` / `cd` / `cd -` is a pacing-sensitive segment. Go slow; these commands are muscle memory for many students but new for others.
- The PNG dump in Part 4 is the "aha" moment. Pause longer there.
- Total captured elapsed time on a full clean run: ~19 minutes. Budget ~22 with pauses.
