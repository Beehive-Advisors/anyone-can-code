# Research Report: Lab 1.5 — Seeing the Machine

## Overview

This lab gives students a way to inspect the machine they are actually using. It turns earlier lectures about CPUs, caches, memory, storage, and bytes into something visible in the terminal. The goal is not to memorize commands. The goal is to connect abstract systems ideas to real output from a real computer.

The lab covers four linked concepts. First, students inspect CPU architecture, model, core count, and cache sizes. Second, they inspect memory from the operating system's point of view and see that memory is tracked in pages, not as one giant blob. Third, they inspect storage layout and then read a file as raw bytes. Fourth, they convert the same value between decimal, binary, hexadecimal, and text with Python.

This is a direct-typing lab. Every meaningful step is a short terminal command or a short Python snippet. There is no need for demo scripts, a virtual environment, or external services. The only temporary file students create is `hello.txt`, which exists so they can see a real file as bytes.

## 1. CPU Identity And Architecture

### Background

A program does not run on a generic machine. It runs on a processor that understands a specific instruction set. That is what architecture names like `arm64` and `x86_64` are describing. They are not marketing labels. They tell you which machine instructions the CPU can execute.

The processor also has physical structure beyond its architecture name. It has cores, and it has multiple cache levels because the CPU is much faster than main memory. A useful beginner lab should expose both: the broad identity of the machine and some of the specific hardware facts that shape performance.

### How it works

Commands such as `uname`, `lscpu`, and `sysctl` are not benchmarking the processor. They are reading information the operating system already knows. On Linux, `lscpu` summarizes CPU information from kernel interfaces such as `sysfs` and `/proc/cpuinfo`. On macOS, `sysctl` exposes named kernel parameters such as `hw.physicalcpu` and `machdep.cpu.brand_string`.

The main teaching point is that hardware facts are surfaced through the operating system. Students are not peeking behind the OS. They are learning how the OS describes the machine it is running on.

### Implementation: `uname`, `lscpu`, and `sysctl`

The lab uses:

- `uname -m` to show the machine architecture
- `sysctl -n machdep.cpu.brand_string` on macOS for the CPU model name
- `sysctl hw.physicalcpu hw.logicalcpu hw.l1dcachesize hw.l1icachesize hw.l2cachesize` on macOS for core and cache information
- `lscpu` as the Linux and WSL equivalent teaching command

**Sources:**
- [lscpu(1) — Linux manual page](https://man7.org/linux/man-pages/man1/lscpu.1.html) — Documents how Linux exposes CPU architecture and topology through `lscpu`.
- [uname — POSIX utility specification](https://pubs.opengroup.org/onlinepubs/9699919799/utilities/uname.html) — Defines the portable meaning of `uname`.
- [sysctl(8) macOS manual mirror](https://www.manpagez.com/man/8/sysctl/) — Documents the macOS `sysctl` interface used for hardware facts.

## 2. Memory As Pages

### Background

When people talk about RAM informally, they usually talk about a single number such as 16 GB or 24 GB. Operating systems do not manage memory that way. They manage memory in fixed-size chunks called pages.

That difference matters because page-based accounting explains why tools report categories such as free, active, inactive, and wired memory. Those names are not arbitrary. They reflect the kernel's bookkeeping for what memory is doing right now.

### How it works

On Linux, tools such as `free` summarize kernel memory counters from `/proc/meminfo`. On macOS, `vm_stat` reports page-based statistics from the virtual memory subsystem. In both cases, the tool is reading counters the kernel maintains continuously.

This makes a good beginner lab sequence: first read total installed memory, then inspect page-based counters, then convert those counters into MiB. That teaches students that the human-friendly summary comes after the lower-level accounting.

### Implementation: `free`, `vm_stat`, and `sysctl`

The lab uses:

- `sysctl hw.memsize` on macOS for total installed memory in bytes
- `vm_stat` on macOS for page-level statistics
- short Python snippets to convert bytes and page counts into GiB or MiB
- `free -h` as the Linux and WSL equivalent teaching command

**Sources:**
- [free(1) — Debian manpage](https://manpages.debian.org/testing/procps/free.1.en.html) — Documents how `free` reports system memory on Linux.
- [Viewing Virtual Memory Usage — Apple](https://developer.apple.com/library/archive/documentation/Performance/Conceptual/ManagingMemory/Articles/VMPages.html) — Explains `vm_stat` and page-based memory statistics on macOS.
- [Performance Tools — Apple](https://developer.apple.com/library/archive/documentation/Performance/Conceptual/PerformanceOverview/PerformanceTools/PerformanceTools.html) — Lists `vm_stat` as a tool for Mach virtual memory statistics.

## 3. Storage Layout And Raw Bytes

### Background

Files feel high-level because most of the time students open them in an editor or app. Underneath that, files live inside a file system, which lives on top of volumes, containers, partitions, and physical devices.

That is why it is useful to pair a disk-layout command with a hex dump in the same lab. One command shows how the operating system organizes storage. The other shows what file content looks like once you zoom all the way down to bytes.

### How it works

On Linux, `lsblk` shows block devices and partitions in a tree. On macOS, `diskutil list` shows disks, APFS containers, and volumes. To inspect file bytes, `xxd` prints a hexadecimal or binary view of a file.

The lab uses a tiny file, `Hello\n`, because each byte can be reasoned about by hand. The student can map the visible text back to its hexadecimal, binary, and decimal values.

### Implementation: `lsblk`, `diskutil`, and `xxd`

The lab uses:

- `diskutil list | head -12` on macOS for storage layout
- `lsblk` as the Linux and WSL equivalent teaching command
- `xxd -g 1 -l 16 hello.txt` for a byte-oriented hex dump
- `xxd -b -l 6 hello.txt` for a binary view

**Sources:**
- [lsblk(8) — Linux manual page](https://man7.org/linux/man-pages/man8/lsblk.8.html) — Documents Linux block-device inspection with `lsblk`.
- [diskutil(8) macOS manual mirror](https://keith.github.io/xcode-man-pages/diskutil.8.html) — Documents disk and APFS inspection on macOS.
- [xxd(1) — Ubuntu manpage](https://manpages.ubuntu.com/manpages/focal/man1/xxd.1.html) — Documents hex and binary dumps with `xxd`.

## 4. Converting Between Number Systems And Text

### Background

A byte does not come with a display format attached. Humans can write the same value in decimal, binary, or hexadecimal. They can also interpret that value as text if both sides agree on an encoding.

That is the bridge between the earlier binary lessons and real file inspection. Students stop seeing binary and hex as classroom notation and start seeing them as alternate views of the same underlying value.

### How it works

Python integers are not base-specific. Functions such as `bin()` and `hex()` produce string representations in different bases. The function `int(text, base)` parses text in a given base back into an integer. The functions `ord()` and `chr()` convert between a character and its Unicode code point.

For ASCII characters such as `H`, the Unicode code point matches the familiar ASCII value. That makes this part especially useful for connecting old text encodings, modern Unicode terminology, and the byte values seen in the hex dump.

### Implementation: Python built-ins

The lab uses only Python built-ins:

- `bin()` and `hex()` to format integers
- `int(text, base)` to parse binary and hexadecimal text
- `ord()` and `chr()` to move between characters and code points

**Sources:**
- [Built-in Functions — Python documentation](https://docs.python.org/3/library/functions.html) — Documents `bin()`, `hex()`, `int()`, `ord()`, and `chr()`.
- [Built-in Types — Python documentation](https://docs.python.org/3/library/stdtypes.html) — Documents integer behavior and text parsing details.
- [Unicode HOWTO — Python documentation](https://docs.python.org/3/howto/unicode.html) — Explains code points and Python string handling.

## Lab Design Decisions

### No scaffolding

This lab does not need demo files. Every meaningful action is a one-liner or a very short Python snippet. Students learn more by typing those commands themselves than by running a wrapper script.

### Platform strategy

The syllabus examples name Linux commands such as `lscpu`, `free -h`, and `lsblk`, but this worktree is being built on macOS. The lab therefore documents Linux and WSL equivalents in setup and inside each relevant part, while using macOS-tested output in the captured blocks that appear in the handout and script.

### Local-only demo

No external services are needed. The lab is entirely local because the teaching goal is to inspect the learner's own machine, not a remote system. That keeps the environment simple and reinforces that the computer in front of the student is the primary teaching object.
