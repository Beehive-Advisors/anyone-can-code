# Research Report: Lab 1.5 — Seeing the Machine

## Overview

This lab gives students their first hands-on look at the physical machine underneath their code. By running a handful of built-in command-line tools, students see exactly what hardware they are working with: how many CPUs are actually available, how much memory the kernel has divided into categories, how a disk is sliced into partitions, and what the raw bytes inside a file look like. The second half of the lab crosses from the machine into mathematics: students use Python to convert the same number among decimal, binary, hexadecimal, and ASCII — making concrete the idea that all data is just numbers interpreted differently.

The four concepts in this lab are not independent. CPU registers and cache hold numbers. RAM holds programs and data as numbers. Files on disk are sequences of bytes (numbers 0–255). And ASCII is simply an agreement that the number 65 means the letter 'A'. The lab is designed so that Part 4 feels like a revelation — the hex codes Python computes in the last exercise are identical to the bytes xxd printed in Part 3.

All commands in Parts 1–3 are single-line Linux built-in tools. Part 4 uses only Python 3 builtins — no `pip install`, no imports. The entire lab runs with zero setup beyond a working Ubuntu terminal.

## 1. CPU Inspection with `lscpu`

### Background

A CPU is not a single thing. It is a layered structure of physical and logical components, and the numbers on the spec sheet — cores, threads, cache — each describe a different layer. Students who have heard "my laptop has 8 cores" often do not know what that means concretely. This section gives them the vocabulary and the tools to read their own machine's specs directly from the kernel.

The relationship: one physical **socket** (chip slot on the motherboard) can contain multiple **cores** — independent compute engines. Each core can run multiple **threads** simultaneously using a technique called Simultaneous Multithreading (SMT, marketed by Intel as Hyper-Threading). The operating system sees each thread as a separate CPU. The formula is:

```
logical CPUs = sockets × cores per socket × threads per core
```

Between the CPU and main memory sit layers of small, fast **caches**: L1 (32–64 KB per core, ~1–4 clock cycles), L2 (256 KB–1 MB per core, ~4–15 cycles), L3 (4–64+ MB, shared across cores, ~20–50 cycles). Every memory access that would cost 100+ cycles from RAM is potentially served in 1–4 cycles from L1. Caches exploit the observation that programs repeatedly access the same data (*temporal locality*) and access data near recently-used addresses (*spatial locality*). Data moves between cache levels in 64-byte **cache lines** — touching any byte in a line loads the entire 64 bytes.

### How it works

`lscpu` reads from two kernel interfaces: `/proc/cpuinfo` (one entry per logical CPU with raw fields like `cpu cores`, `siblings`, and `physical id`) and the sysfs topology directory `/sys/devices/system/cpu/cpu*/topology/` (which maps which logical CPUs share a core and which share a socket). It aggregates these into a human-readable table. Running `cat /proc/cpuinfo` shows the raw data `lscpu` is summarizing.

Cache details come from `/sys/devices/system/cpu/cpu*/cache/index*/`, where each `index*` directory represents one cache level with files for `level`, `type`, `size`, and `shared_cpu_list`. The `lscpu -C` flag formats this into a table with ONE-SIZE (per-core) vs ALL-SIZE (aggregate).

### Implementation: `lscpu`

```bash
lscpu                    # full table
lscpu | grep -E "^CPU\(s\):|Thread|Core|Socket|Model name"  # key fields only
lscpu -C                 # cache hierarchy table
nproc                    # just the total logical CPU count
cat /proc/cpuinfo        # raw kernel data lscpu summarizes
```

**macOS equivalents:**

```bash
sysctl -n machdep.cpu.brand_string     # CPU model name
sysctl -n hw.physicalcpu              # physical cores
sysctl -n hw.logicalcpu               # logical CPUs
# Apple Silicon — performance vs efficiency core caches:
sysctl -n hw.perflevel0.l1dcachesize  # P-core L1D in bytes (e.g. 131072 = 128KB)
sysctl -n hw.perflevel0.l2cachesize   # P-core L2 in bytes (e.g. 16777216 = 16MB)
sysctl -n hw.perflevel1.l1dcachesize  # E-core L1D in bytes (e.g. 65536 = 64KB)
```

## 2. Memory Inspection with `free` and `lsmem`

### Background

RAM (Random Access Memory) is the computer's working scratchpad. Unlike disk, it loses content when power is cut. Unlike cache, it is large enough to hold entire running programs. The kernel manages RAM as a shared resource — every process gets a slice, the kernel keeps some for its own structures, and any leftover is quietly used for **disk cache**: recently-read file data stored in RAM so repeat reads skip the disk entirely.

The confusing part: a Linux system showing only a small amount of "free" RAM is not in trouble. Most of the apparently "used" memory is disk cache, which the kernel will silently reclaim the instant any process needs more. The `available` column — not `free` — tells students what new programs can actually use.

### How it works

`free` reads from `/proc/meminfo`, a virtual file the kernel updates continuously. Key fields:

| `/proc/meminfo` field | Meaning |
|-----------------------|---------|
| `MemTotal` | Physical RAM installed |
| `MemFree` | Completely unoccupied RAM |
| `MemAvailable` | Estimate of what new allocations can use without swapping (includes reclaimable cache) |
| `Buffers` | Kernel buffer cache (filesystem metadata) |
| `Cached` | Page cache (actual file contents cached in RAM) |
| `SwapTotal` / `SwapFree` | Swap partition size and available space |

`lsmem` reads from the kernel's memory block subsystem to show physical memory address ranges and their online/offline status. On typical laptops and VMs all memory appears as one contiguous online range. On servers with hardware memory hot-plug, individual 2 GB blocks can be taken offline without rebooting.

DRAM internals: each bit is one capacitor (holding charge=1 or no charge=0) gated by one transistor. Capacitors leak — every row must be refreshed every ~64 ms. This continuous maintenance is why it is called **Dynamic** RAM.

### Implementation: `free`, `lsmem`, `/proc/meminfo`

```bash
cat /proc/meminfo | head -10  # raw kernel counters in kB
free -h                        # human-readable (-h), same data
lsmem                          # physical memory address ranges
```

**macOS equivalents:**

```bash
sysctl -n hw.memsize           # total RAM in bytes
python3 -c "print(25769803776 / 2**30, 'GB')"  # convert to GB
vm_stat                        # page statistics (pages × 16384 bytes = bytes on Apple Silicon)
```

**Sources:**
- [lsmem(1) — Linux manual page](https://man7.org/linux/man-pages/man1/lsmem.1.html) — flag reference and output format
- [Free vs. Available Memory in Linux — linuxblog.io](https://linuxblog.io/free-vs-available-memory-in-linux/) — explains why available > free
- [Dissecting the free command — Red Hat Blog](https://www.redhat.com/en/blog/dissecting-free-command) — maps free columns to /proc/meminfo fields

## 3. Disk Layout with `lsblk` and Raw Bytes with `xxd`

### Background

A disk is a sequence of fixed-size locations called **sectors** (typically 512 or 4096 bytes each). To make a disk usable, the OS first divides it into **partitions** using a partition table at the very start of the disk. Modern disks use **GPT** (GUID Partition Table): sector 0 holds a protective MBR, sector 1 holds the GPT header (disk GUID, number of partition entries, CRC32 checksums), and sectors 2–33 hold 128 partition entries, each recording a starting sector, ending sector, type GUID, and name.

The kernel exposes each disk as a **block device** — a file in `/dev/` (like `/dev/nvme0n1` or `/dev/sda`) that can be read in fixed-size chunks. Partitions appear as sub-devices: `nvme0n1p1`, `sda1`, etc. The kernel assigns each a **major:minor number** it uses internally to route I/O to the correct driver.

Every file — whether a document, an image, or a compiled program — is ultimately a sequence of bytes. `xxd` makes this visible by displaying raw file contents in hexadecimal alongside an ASCII interpretation column. Two hex digits represent exactly one byte (values 0–255), which is why hex is preferred over decimal: `48` is one byte, but decimal `72` takes up three characters. The **magic number** (first few bytes of a file) identifies the file type at the byte level — the OS uses these bytes, not the filename extension, to know how to handle the file.

### How it works

`lsblk` reads the kernel's sysfs block subsystem (`/sys/block/`) and the udev database rather than the disk itself — making it fast and non-destructive. The tree indentation shows which block devices are partitions of which disk.

`xxd` output format: each line shows an 8-digit hex **offset** (byte position of the first byte on that line, in hexadecimal), followed by 16 bytes displayed as 8 groups of 2 bytes, followed by the ASCII interpretation of those 16 bytes (non-printable bytes show as `.`).

Common magic numbers:

| File type | First bytes (hex) | ASCII column |
|-----------|-------------------|--------------|
| ELF binary (Linux) | `7f 45 4c 46` | `.ELF` |
| Mach-O Universal (macOS) | `ca fe ba be` | `....` |
| PNG image | `89 50 4e 47 0d 0a 1a 0a` | `.PNG....` |
| PDF document | `25 50 44 46 2d` | `%PDF-` |
| ZIP / JAR / DOCX | `50 4b 03 04` | `PK..` |

### Implementation: `lsblk`, `xxd`

```bash
lsblk                          # disk and partition tree
lsblk -f                       # adds FSTYPE, UUID, MOUNTPOINTS
xxd filename                   # full hex dump
xxd -g 1 filename              # one byte per group (easier to count)
xxd -l 16 /bin/ls              # first 16 bytes only (magic number)
echo "48656c6c6f0a" | xxd -r -p  # reverse: hex → binary
```

**macOS equivalent:**

```bash
diskutil list                  # disk and partition tree (replaces lsblk)
xxd -l 16 /bin/ls              # xxd pre-installed on macOS; Mach-O magic: cafebabe
```

## 4. Number Systems and Python Conversions

### Background

Computers store everything as numbers. The challenge is representation: the same number has many names depending on the base you use to write it. Decimal uses digits 0–9. Binary uses only 0 and 1 — matching the physical switches in hardware. Hexadecimal uses 0–9 and A–F, which is convenient because exactly two hex digits represent one byte (0–255), and each hex digit maps to exactly four bits, making mental conversion trivial once you know the table.

ASCII is an agreement: the integer 65 means the letter 'A', 66 means 'B', 48 means '0' (not zero — the *character* zero), and 10 means "newline". This mapping was standardized in 1963 and preserved unchanged into Unicode. When `xxd` prints `48` next to 'H' in the ASCII column, it is showing this agreement in action.

Python makes the four representations — decimal, binary, hex, ASCII — trivially accessible through built-in functions that need no imports.

### How it works

Positional notation: in any base B, the digit in position N from the right contributes `digit × B^N`. The number 255 is `2×10²+5×10¹+5×10⁰` in decimal, `1×2⁷+…+1×2⁰` in binary (= `11111111`), and `15×16¹+15×16⁰` in hex (= `FF`). All three are the same integer; only the notation differs.

ASCII design decisions worth knowing: uppercase `A`–`Z` are at codes 65–90; lowercase `a`–`z` are at 97–122. The difference is exactly 32 (= 0x20 = bit 5). Flipping bit 5 of any letter's ASCII code converts its case — `XOR 0x20` works for any letter. Digit characters `'0'`–`'9'` live at codes 48–57, so `ord('0')` = 48, not 0.

A Python `bytes` object is an immutable sequence of integers 0–255. Indexing it returns an `int` (not a character). This is the direct bridge between text and bytes.

### Implementation: Python builtins

```python
# Decimal → other bases
bin(65)              # '0b1000001'
hex(65)              # '0x41'
format(65, '08b')    # '01000001'  — no prefix, padded to 8 bits
format(65, '02x')    # '41'        — no prefix, padded to 2 hex chars

# Other bases → decimal
int('01000001', 2)   # 65  — binary string
int('41', 16)        # 65  — hex string
int('0b01000001', 0) # 65  — auto-detect prefix

# ASCII ↔ integer
ord('A')             # 65
chr(65)              # 'A'

# bytes
b'Hi!'[0]            # 72   — indexing returns int
b'Hi!'.hex()         # '486921'
bytes([72,101,108,108,111]).decode('ascii')  # 'Hello'
```

One-liners for direct terminal typing:

```bash
python3 -c "print(format(65, '08b'))"
python3 -c "print(ord('A'), hex(ord('A')), format(ord('A'), '08b'), chr(65))"
python3 -c "print([format(b,'02x') for b in open('test.txt','rb').read()])"
```

## Lab Design Decisions

### No scaffolding
Every command in this lab is a single terminal one-liner. `lscpu`, `free -h`, `lsmem`, `lsblk`, and `xxd` are all pre-installed on Ubuntu 22.04 with no additional install step. Python one-liners use only builtins. The STANDARDS.md explicitly names this lab as the canonical no-scaffolding example.

### Linux-primary with macOS callouts
Parts 1–3 use Linux-only tools (`lscpu`, `free`, `lsmem`, `lsblk`). Students on macOS run these labs on their Ubuntu VM or WSL. However, macOS callout boxes let Mac-native students relate the Linux output to their own machine using `sysctl`, `vm_stat`, and `diskutil list`. Part 4 (Python) is identical on both platforms.

### xxd offset is hexadecimal — explicit teaching moment
Students consistently read `00000010` (second xxd line) as "10 bytes in" rather than "16 bytes in". The lab explicitly calls this out and has students verify with `python3 -c "print(0x10)"`.

### Connecting Parts 3 and 4
The lab bridges xxd and Python by having students xxd a file and then read the same file's bytes in Python — both produce identical hex values. This is the lab's key unifying moment.
