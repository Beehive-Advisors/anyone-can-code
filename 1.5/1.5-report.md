# Research Report: Lab 1.5 — Seeing the Machine

## Overview

This is the first hands-on lab of the "Anyone Can Code" course. It follows the Computer Fundamentals lectures — binary/bits (1.1), logic gates (1.2), fetch-decode-execute (1.3), and the memory hierarchy (1.4) — and turns those abstract claims into real output the student produces on their own machine. The goal is to let students *see* the pyramid the lectures drew: CPU cores with tiny caches beside them, a pool of RAM, a disk with partitions, and files that are really just numbers all the way down.

The lab ships two tracks in one file. The 🍎 macOS track uses native Mac commands (`sysctl`, `vm_stat`, `diskutil`, `system_profiler`) that query the XNU kernel's Management Information Base. The 🐧 Linux / WSL track uses `lscpu`, `free -h`, `lsblk` — all of which read the kernel-exposed `/proc` and `/sys` virtual filesystems on an Ubuntu 22.04 host (WSL2 students see exactly the same output as a native Linux machine). `xxd` and Python's base-conversion built-ins are cross-platform, so Parts 4 and 5 use identical commands on both tracks.

There is no scaffolding. Every demo is one command the student types and one output they read — no scripts, no multi-file programs, no packages to install beyond the OS defaults. The pedagogy hinges on the student typing each command themselves and watching the result arrive, so the lab skill explicitly forbids wrapping these in a shell script.

## 1. Inspecting CPU Architecture, Cores, and Caches

### Background

Every modern OS publishes a live catalog of the CPU it's running on: the chip model, the core count, the cache sizes at each level. Students read it with one command. That single command is the payoff for lecture 1.3 (fetch-decode-execute happens on *this* CPU) and lecture 1.4 (the cache pyramid has *these* sizes).

### How it works

Linux exposes CPU data through two kernel-maintained virtual filesystems. `/proc/cpuinfo` is a flat key-value list, one entry per logical CPU. `/sys/devices/system/cpu/*` is a tree with per-core and per-cache directories. `lscpu` is a small C program that aggregates both into a formatted summary; `lscpu -C` specifically walks the cache directories and prints the L1d/L1i/L2/L3 hierarchy.

macOS exposes the same facts through the `sysctl` MIB — a key/value tree maintained by the XNU kernel. The relevant keys are:
- `machdep.cpu.brand_string` — the chip's marketing name
- `hw.physicalcpu` / `hw.logicalcpu` — core counts (on Apple Silicon these match because there is no SMT)
- `hw.perflevel0.l1dcachesize` / `hw.perflevel0.l2cachesize` — cache sizes in bytes for the Performance-core cluster (Apple Silicon splits cores into Performance and Efficiency tiers; `perflevel1` is the Efficiency tier)
- `system_profiler SPHardwareDataType` — a prose summary covering chip name, core breakdown, and installed RAM

Apple Silicon chips don't have a traditional L3. They use a large shared L2 and a system-level cache on the fabric that `sysctl` doesn't expose under the classical name. `hw.perflevel0.l3cachesize` returns 0 or empty — that is expected.

### Implementation: `lscpu` (Linux) and `sysctl` (macOS)

```bash
# Linux
lscpu                  # full topology table
lscpu -C               # cache hierarchy only

# macOS
sysctl -n machdep.cpu.brand_string
sysctl -n hw.physicalcpu
sysctl -n hw.perflevel0.l1dcachesize   # bytes
system_profiler SPHardwareDataType     # prose summary
```

- [How to Find the Size of L1, L2, and L3 Cache in Linux (Baeldung)](https://www.baeldung.com/linux/check-cache-size-l1-l2-l3) — lscpu cache walkthrough.

## 2. Inspecting RAM

### Background

A running computer has memory in constant motion. Some is held by the kernel, some by applications, some is "technically used but reclaimable" (disk block cache). Students come in thinking of RAM as one number — "my laptop has 16 GB." The real picture is layered, and `free -h` / `vm_stat` are the tools that print the layers.

### How it works

Linux tracks memory as counted pages in `/proc/meminfo`. Key fields:
- `MemTotal` — physical RAM installed (in kB)
- `MemFree` — truly unused (usually small)
- `MemAvailable` — free plus reclaimable cache; this is what matters
- `Buffers` + `Cached` — disk blocks cached in RAM; drop-able on pressure

`free -h` is a thin parser that formats these with human-readable units. The student's confusion point is always `MemFree` ≠ `MemAvailable`: almost all RAM is "used" as disk cache, but most of it is free for the taking.

macOS uses Mach kernel statistics. `vm_stat` prints counts of pages in each category:
- `Pages free` — truly unused
- `Pages active` — in use recently
- `Pages inactive` — in use but cold (first to be reclaimed)
- `Pages wired down` — kernel-pinned, never reclaimed
- `Pages occupied by compressor` — memory actively compressed to avoid swapping

Page size on Apple Silicon is 16384 bytes (vs 4096 on older Intel Macs). `vm_stat`'s first line declares the page size explicitly. Bytes = pages × page-size.

`sysctl -n hw.memsize` returns total installed RAM in raw bytes — a single integer.

### Implementation: `free -h` (Linux) and `vm_stat` + `sysctl` (macOS)

```bash
# Linux
free -h
grep MemTotal /proc/meminfo

# macOS
sysctl -n hw.memsize
vm_stat
top -l 1 | grep PhysMem       # optional one-line summary
```

**Sources:**
- [proc_meminfo(5) — Linux manual page (man7)](https://man7.org/linux/man-pages/man5/proc_meminfo.5.html) — `/proc/meminfo` field reference.
- [Dissecting the free command (Red Hat Sysadmin)](https://www.redhat.com/sysadmin/dissecting-free-command) — free -h column walkthrough.
- [vm_stat(1) macOS manual](https://www.unix.com/man_page/osx/1/vm_stat/) — vm_stat reference.
- [Viewing Virtual Memory Usage — Apple Developer Archive](https://developer.apple.com/library/archive/documentation/Performance/Conceptual/ManagingMemory/Articles/VMPages.html) — active/inactive/wired semantics.

## 3. Inspecting Disks and Storage

### Background

RAM forgets everything on power-off; disk remembers. A single physical SSD is almost never used as one unbroken surface — it's sliced into partitions, each usually holding a filesystem, each mounted at a directory somewhere in the tree. This Part prints the topology so students see: physical device → partitions/volumes → mount points → "is it full?"

### How it works

Linux represents every block device as a file under `/sys/block/`. `lsblk` walks that directory and prints a tree: disk → partitions → optionally LVM/LUKS layers. Typical names: `sda` / `sdb` (SATA), `nvme0n1` (NVMe SSD). `lsblk -f` adds filesystem type, label, and UUID columns.

macOS uses the DiskArbitration framework. `diskutil list` prints a tree of physical disks, APFS containers, and synthesized volumes. On modern Macs: `disk0` is the physical SSD with a GUID partition scheme, `disk3` is the synthesized APFS container holding Macintosh HD, Data, Preboot, Recovery, and VM as separate volumes that all share free space.

`df -h` works identically on both OSes. It asks each mounted filesystem "how big are you and how full are you?" via the `statvfs(2)` system call and prints one row per mount point.

### Implementation: `lsblk` (Linux) and `diskutil list` (macOS), `df -h` (both)

```bash
# Linux
lsblk
lsblk -f

# macOS
diskutil list

# Both
df -h
```

**Sources:**
- [lsblk(8) — Linux manual page (man7)](https://man7.org/linux/man-pages/man8/lsblk.8.html) — canonical lsblk reference.
- [DISKUTIL Command reference (SS64)](https://ss64.com/mac/diskutil.html) — diskutil documentation.
- [Disks, partitions, volumes, containers — Eclectic Light Company](https://eclecticlight.co/2018/08/01/disks-partitions-volumes-containers/) — APFS container/volume structure explained.
- [df command in Linux with Examples (GeeksforGeeks)](https://www.geeksforgeeks.org/linux-unix/df-command-linux-examples/) — df column reference.

## 4. Reading Raw Bytes with `xxd`

### Background

Lecture 1.1 claimed: "everything is bits, and the only question is what you agree they mean." This Part is where that claim becomes concrete. `xxd` prints any file as hexadecimal bytes with an ASCII sidecar. Students hex-dump a plain-text file and see their letters as bytes (`H`=0x48, `e`=0x65, …). Then they hex-dump a PNG image and see the file starts with the bytes `89 50 4e 47` — the PNG "magic number" that every image viewer on Earth checks to identify the format.

### How it works

`xxd` reads bytes and emits three columns per line:
- **Offset** (8 hex digits) — position in the file
- **Hex bytes** (16 per line by default, grouped in 2-byte chunks)
- **ASCII rendering** — printable bytes (0x20–0x7E) shown directly; everything else shown as `.`

Flags: `-l N` (first N bytes), `-c N` (bytes per line), `-b` (binary instead of hex — shows 8-bit patterns), `-r` (reverse: hex back to binary).

PNG signature (first 8 bytes of every PNG): `89 50 4E 47 0D 0A 1A 0A`.
- Byte 1 (`89`) has the high bit set — catches 7-bit-stripping transfers.
- Bytes 2–4 (`50 4E 47`) spell `PNG` in ASCII.
- Bytes 5–6 (`0D 0A`) are CR-LF — catch newline-mangling transfers.
- Byte 7 (`1A`) is Ctrl-Z — stops display on ancient DOS.
- Byte 8 (`0A`) is LF.

After the signature comes the IHDR chunk: 4-byte length, 4-byte type tag (`49 48 44 52` = `IHDR`), then width and height.

### Implementation: `xxd` (cross-platform — same on both OSes)

```bash
# Both macOS and Linux
echo 'Hello!' > hello.txt
xxd hello.txt               # default hex dump
xxd -b hello.txt            # binary (bits) instead of hex
curl -sO https://www.libpng.org/pub/png/img_png/pnglogo-blk.png
xxd -l 16 pnglogo-blk.png   # first 16 bytes — see the PNG signature
```

**Sources:**
- [xxd(1) — Linux manual page (die.net)](https://linux.die.net/man/1/xxd) — canonical xxd reference.
- [How Hexdump works (Opensource.com)](https://opensource.com/article/19/8/dig-binary-files-hexdump) — pedagogical hex-dump walkthrough.
- [PNG Specification: Rationale (libpng.org)](https://www.libpng.org/pub/png/spec/1.2/PNG-Rationale.html) — authoritative PNG 8-byte signature design notes.
- [List of file signatures (Wikipedia)](https://en.wikipedia.org/wiki/List_of_file_signatures) — reference table of magic bytes.

## 5. Python Conversions by Hand

### Background

Students just saw the bytes `89 50 4e 47` at the start of a PNG. They know from lecture 1.1 that `50 4e 47` is supposed to spell `PNG`. This Part is where they *prove* it, one Python one-liner at a time. Once `ord('P')` returns 80, and `hex(80)` returns `'0x50'`, and `chr(80)` returns `'P'`, the loop closes: the number on the disk, the hex in the dump, and the letter on the screen are all the same thing in three costumes.

### How it works

Python's built-ins do all four conversions:
- `bin(n)` / `hex(n)` — int → prefixed string (`'0b1010000'`, `'0x50'`)
- `int(s, base)` — string → int, parsed in any base 2–36
- `ord(c)` / `chr(n)` — character ↔ Unicode code point (ASCII-compatible for 0–127)
- `bytes.hex()` / `bytes.fromhex()` — bytes ↔ hex string (no prefix, no spaces)

f-string format specs cover the dump-friendly output:
- `{n:02x}` — hex, zero-padded to 2 digits (matches xxd's per-byte format)
- `{n:08b}` — binary, zero-padded to 8 digits (one full byte of bits)

The mental model students internalize: a byte is a number 0–255. That number has three equivalent *written* forms (binary, hex, decimal) and, for values 32–126, a fourth as an ASCII character. The built-ins are the four directions of that compass.

### Implementation: Python 3 built-ins (cross-platform)

```bash
# macOS
python3.12 -c "print(hex(80))"            # 0x50
python3.12 -c "print(bin(80))"            # 0b1010000
python3.12 -c "print(int('50', 16))"      # 80
python3.12 -c "print(ord('P'))"           # 80
python3.12 -c "print(chr(80))"            # P
python3.12 -c "print(f'{15:02x}')"        # 0f
python3.12 -c "print(f'{5:08b}')"         # 00000101
python3.12 -c "print(bytes.fromhex('89504e47'))"  # b'\x89PNG'

# Linux / WSL (same expressions; Ubuntu 22.04's default `python3` is 3.10)
python3 -c "print(hex(80))"
# ... same sequence
```

**Sources:**
- [Convert Binary, Octal, Decimal, and Hexadecimal in Python (note.nkmk.me)](https://note.nkmk.me/en/python-bin-oct-hex-int-format/) — comprehensive bin/oct/hex/int guide with f-strings.
- [Python ord() and chr() (DigitalOcean)](https://www.digitalocean.com/community/tutorials/python-ord-chr) — ord/chr with an ASCII table.
- [Format Specification Mini-Language (Python 3 docs)](https://docs.python.org/3/library/string.html#format-specification-mini-language) — official format-spec reference.
- [Convert Hex String to Bytes (GeeksforGeeks)](https://www.geeksforgeeks.org/python/convert-hex-string-to-bytes-in-python/) — `bytes.hex()` / `bytes.fromhex()` reference.

## Lab Design Decisions

### No scaffolding

Every demo is one command and one output. The pedagogical goal is precisely that students type each command and see the result themselves — there is no value in wrapping `sysctl -n hw.memsize` in a shell script. The Scaffolding Test in `STANDARDS.md` says to default to direct-typing, and this lab is a textbook case.

### Single toggle at the top, not per-Part switching

The `## Pick your track` section at the very top of the lab contains two `<details>` blocks — one for macOS, one for Linux / WSL — each holding the full standalone lab for that platform. The student clicks one toggle, reads top to bottom, and never sees the other track's commands. An earlier iteration used per-Part dual blocks (a 🍎 block and a 🐧 block inside every Part), which required the student to mentally filter throughout the whole lab. That got noisy. The single-toggle layout trades some shared-content duplication (conceptual intros, WHY questions, Putting-it-together) for a cleaner reading experience that matches the actual student workflow: pick your OS once, then work.

### Apple Silicon assumed on the macOS track

The lab uses `hw.perflevel0.*` sysctl keys which are Apple Silicon only. Intel Macs use a different key layout (`hw.l2cachesize`, no `perflevel`). Since every 2021+ Mac is Apple Silicon, this is the right default. A callout in Part 1 can direct Intel Mac students to the older key names if needed.

### `python3.12` on macOS, `python3` on Linux

The user's global CLAUDE.md specifies `python3.12` on macOS. Ubuntu 22.04 ships with `python3` (3.10). Every built-in used in Part 5 — `bin`, `hex`, `int`, `ord`, `chr`, `bytes.hex`, `bytes.fromhex`, f-string format specs — has been stable since Python 3.6, so the 3.10 vs 3.12 gap is pedagogically invisible. The lab uses the platform-native binary name on each track and notes this once in Before-you-begin.

### Local-only; one network dependency

The entire lab runs against the student's own hardware with one exception: Part 4 fetches a known PNG from libpng.org with `curl`. This gives reproducible hex-dump output across every student's machine. Any PNG on disk would work if that request fails.

### `lsmem` skipped

The original syllabus mentions `lsmem` as an option alongside `free -h`. `lsmem` is designed for memory-hotplug-aware servers (mainframes, s390x); on a typical laptop or WSL VM it returns minimal output. `free -h` is the canonical Linux RAM tool. The lab uses `free -h` and `/proc/meminfo` and doesn't teach `lsmem` — mentioning it would add complexity without insight.
