# Research Report: Lab 1.5 — Seeing the Machine

## Overview

This lab turns the abstract claims from lectures 1.1–1.4 into things students can see on their own hardware. The prior four lectures cover binary representation, logic gates, the fetch-decode-execute cycle, and the memory hierarchy — concepts that can feel like theory until a student types one command and watches their own CPU, RAM, and disks reveal themselves in the terminal. Every machine already contains this information; the OS has discovered it at boot and stores it in kernel data structures that a one-line command can read.

The lab uses only a shell and Python — no packages, no frameworks, no accounts, no installs. The single non-trivial design decision is cross-platform support. The Linux commands in the syllabus (`lscpu`, `lsmem`, `lsblk`) do not exist on macOS, and telling the course's many Mac-based students to install a Linux VM for lab 1 would be a poor introduction. The lab instead teaches the concept ("how do I see my CPU?") with parallel commands for each platform, so every student sees real data about the machine in front of them.

The closing section connects the two sides of the course's "everything is bytes" story. Students open a plain-text file with `xxd`, see its content as hex, and then write a few Python one-liners that convert the same characters to decimal, hex, and binary. The bytes the hex dump shows and the numbers the Python code prints match exactly — proof that two very different tools are describing the same underlying reality.

## 1. Inspecting CPU Architecture, Cores, and Caches

### Background

Every program eventually runs on a physical chip. That chip has an instruction set (x86_64, ARM64), a fixed number of cores, and a cache hierarchy sitting between the cores and main memory. The lecture on the memory hierarchy introduced these as concepts. This section lets students read those numbers off their own laptop and confirm that L1 is truly kilobytes while L3 is truly megabytes.

### How it works

Linux exposes CPU data through two kernel-provided virtual filesystems. `/proc/cpuinfo` is a human-readable text dump with one block per logical CPU. `/sys/devices/system/cpu/` is a directory tree with one file per attribute. `lscpu` is a small reader that walks both trees and formats the result.

macOS uses the `sysctl` MIB (Management Information Base) — a tree of named kernel variables like `hw.ncpu`, `hw.physicalcpu`, and `hw.l1dcachesize`. The XNU kernel populates these at boot by walking the device tree. On Apple Silicon, cache keys split by performance level: `hw.perflevel0.*` for performance cores, `hw.perflevel1.*` for efficiency cores. There is no `hw.l3cachesize` on Apple Silicon — the system-level cache is not exposed the same way.

### Implementation

**Linux / WSL:**
```bash
lscpu                       # full summary: arch, cores, caches, flags
lscpu | grep -i cache       # just the cache lines
cat /proc/cpuinfo           # raw, unformatted
```

**macOS:**
```bash
sysctl -n machdep.cpu.brand_string   # "Apple M1 Pro"
sysctl -n hw.physicalcpu             # real cores
sysctl -n hw.logicalcpu              # cores × SMT threads
sysctl -n hw.perflevel0.l1dcachesize # L1d per performance core (Apple Silicon)
sysctl -n hw.l1dcachesize            # L1d (Intel Macs)
system_profiler SPHardwareDataType   # prose overview
```

- [How to Find the Size of L1, L2, and L3 Cache in Linux — Baeldung](https://www.baeldung.com/linux/check-cache-size-l1-l2-l3) — how to read the "(N instances)" annotation

## 2. Inspecting RAM

### Background

RAM is the scratchpad every running program uses. A process's variables, its code, and the data it is currently touching all live there. From the memory-hierarchy lecture: RAM is much bigger than cache but roughly 100× slower. The questions this section answers for students: "How much RAM does my machine have? How much is in use right now? What is that weird 'available' column?"

### How it works

Linux tracks memory in the kernel's `mm` subsystem. `/proc/meminfo` is a text snapshot exposing roughly 50 counters — `MemTotal`, `MemFree`, `MemAvailable`, `Buffers`, `Cached`, `SwapTotal`. `free` parses that file and prints a two-row table. The most important distinction in the output is between `free` (RAM sitting empty) and `available` (free plus reclaimable cache). On a healthy system `free` is small because the kernel aggressively uses free RAM as disk cache; `available` is what applications can realistically grab.

macOS exposes memory facts in two places. `sysctl hw.memsize` returns total RAM in bytes. `vm_stat` is the Mach-layer statistics tool, which reports live usage in *pages* (not bytes). To convert pages to bytes, multiply by the page size, which `vm_stat`'s first line reports: 4096 on Intel Macs, 16384 on Apple Silicon.

### Implementation

**Linux / WSL:**
```bash
free -h                     # human-readable: total/used/free/available
cat /proc/meminfo           # the raw source
```

**macOS:**
```bash
sysctl -n hw.memsize        # total bytes
sysctl -n hw.pagesize       # page size (for interpreting vm_stat)
vm_stat                     # live page counts
```

**Sources:**
- [free(1) — Linux manual page](https://man7.org/linux/man-pages/man1/free.1.html) — every column explained
- [proc_meminfo(5) — Linux manual page](https://man7.org/linux/man-pages/man5/proc_meminfo.5.html) — the 50+ counters, what each one means
- [Free Command in Linux — Linuxize](https://linuxize.com/post/free-command-in-linux/) — `-h`, `-b`, `-s`, and when to use each
- [vm_stat(1) — macOS man page](https://keith.github.io/xcode-man-pages/vm_stat.1.html) — page categories and their meanings
- [Viewing Virtual Memory Usage — Apple Developer Archive](https://developer.apple.com/library/archive/documentation/Performance/Conceptual/ManagingMemory/Articles/VMPages.html) — active/inactive/wired semantics
- [How unified memory blows the SoCs off the M1 Macs — Eclectic Light](https://eclecticlight.co/2020/11/11/how-unified-memory-blows-the-socs-off-the-m1-macs/) — why `hw.memsize` reports one number shared by CPU and GPU on Apple Silicon

## 3. Inspecting Disks and Mount Points

### Background

Below RAM in the hierarchy is persistent storage. The lecture made two points worth grounding in real output: (1) disks are much larger but much slower than RAM, and (2) the data on them is organized into partitions and filesystems. This section separates three layers students often conflate:

1. **Physical device** — an actual SSD or HDD (`/dev/nvme0n1`, `/dev/sda`, `disk0`).
2. **Partition / volume** — a logical slice of the device (`/dev/nvme0n1p1`, `disk0s2`).
3. **Mount point** — the directory in the filesystem tree where a partition is attached (`/`, `/home`, `/Volumes/Backup`).

A path like `/Users/alice/notes.txt` is really "volume `disk3s1` mounted at `/`, then `Users`, then…" — the chain from silicon to filename.

### How it works

Linux represents every block device as a file in `/dev`. The kernel populates this directory at boot and on hot-plug by walking discovered storage hardware. `lsblk` reads `/sys/block/` and the udev database to assemble a tree of devices → partitions → mount points.

macOS uses the IOKit `DiskArbitration` framework. On modern macOS with APFS there is an extra layer: a physical disk contains an "APFS container" which contains multiple "APFS volumes" that share space dynamically. `diskutil list` shows all three layers: `/dev/disk0` (physical), `/dev/disk3` (synthesized APFS container), and `disk3s1, disk3s2, …` (volumes).

`df -h` works identically on both platforms and reports per-mount-point free and used space.

### Implementation

**Linux / WSL:**
```bash
lsblk                       # tree view of devices and partitions
lsblk -f                    # adds FSTYPE, LABEL, UUID columns
df -h                       # free/used per mount point
```

**macOS:**
```bash
diskutil list               # physical disks, containers, volumes
df -h                       # free/used per mount point (same as Linux)
```

**Sources:**
- [lsblk(8) — Linux manual page](https://man7.org/linux/man-pages/man8/lsblk.8.html) — every output column
- [lsblk Command Guide — ioflood](https://ioflood.com/blog/lsblk-linux-command/) — `-e 7` to hide loopback devices and other practical filters
- [diskutil — macOS command reference — ss64.com](https://ss64.com/mac/diskutil.html) — verb reference for `list`, `info`, `unmount`
- [Mac Diskutil Commands — iBoysoft](https://iboysoft.com/wiki/diskutil.html) — reading APFS container output
- [Exploring Block Devices in Linux — Medium](https://fahadahammed.medium.com/exploring-block-devices-in-linux-and-beginner-experiments-with-filesystems-1ebc757876ff) — the distinction between block and character devices

## 4. Reading Raw Bytes with `xxd`

### Background

The lecture on binary representation made the central claim that everything in a computer is bytes — text, images, programs, all of it. `xxd` is the tool that drops the pretense and shows the bytes directly. Open a plain-text file with `xxd` and you see the exact ASCII codes, in hex, for each character. Open a PNG and you see the magic bytes `89 50 4e 47` at the start — the PNG file signature, on disk, at offset zero, right now.

This is the moment "everything is bytes" stops being a claim from a slide and becomes a thing the student has seen with their own eyes.

### How it works

`xxd` reads a file byte by byte and emits three columns per line:

1. **Offset** — a hex counter of how far into the file we are. Each line is 16 bytes, so the counter advances by 0x10: `00000000`, `00000010`, `00000020`.
2. **Hex pairs** — each byte as two hex digits, grouped in 2-byte chunks: `4865 6c6c 6f2c …`.
3. **ASCII gutter** — each byte shown as its printable ASCII character when possible, or `.` for bytes that are not printable characters: `Hello, world!.`.

A line like
```
00000000: 4865 6c6c 6f2c 2077 6f72 6c64 210a       Hello, world!.
```
reads: starting at byte 0, the file contains the byte values 0x48, 0x65, 0x6c, 0x6c, 0x6f, 0x2c, 0x20, 0x77, 0x6f, 0x72, 0x6c, 0x64, 0x21, 0x0a — which spell `Hello, world!` followed by a newline (`0a` is the ASCII code for line feed).

`xxd -b` replaces the hex column with binary (8 bits per byte), turning the same `H` into `01001000`. That is lecture's "text is binary" claim, made visible.

### Implementation

Both platforms ship `xxd` by default (it travels with Vim).

```bash
xxd hello.txt              # default: 16 bytes per line, hex + ASCII
xxd -c 8 hello.txt         # 8 bytes per line (easier for beginners)
xxd -b hello.txt           # binary view instead of hex
xxd -l 64 file.bin         # first 64 bytes only
xxd -p hello.txt           # plain hex, one long string, no offsets
```

## 5. Python Conversions by Hand

### Background

The closing section of the lab proves that what `xxd` showed is not a special tool trick — it is the actual numeric value of each byte. Students write a few lines of Python that convert a character to its decimal code, then to hex, then to binary, then back to the character. The hex column their Python loop prints matches the hex column `xxd` printed for the same text file byte for byte.

### How it works

Integers in Python are stored as arbitrary-precision signed binary numbers internally. Built-ins like `bin()`, `hex()`, and `str()` do not change the stored value — they produce a **string** that represents the integer in a given base. `int("1001000", 2)` goes the other way, parsing a base-2 string into the integer it represents. Same number, different clothes.

ASCII characters are a mapping between integers (0–127) and symbols. `ord(c)` returns the integer code point of a single-character string. `chr(n)` is its inverse. Python strings are Unicode, but for pure-ASCII characters the first 128 code points match ASCII exactly.

Bytes objects are sequences of integers 0–255. `b"Hi".hex()` returns `"4869"`. `bytes.fromhex("4869")` returns `b"Hi"`. This is the machine-readable mirror of what `xxd` showed visually.

### Implementation

Standard library only — no install.

```python
ord("H")                 # 72         char -> decimal
chr(72)                  # "H"        decimal -> char
hex(72)                  # "0x48"     decimal -> hex string
bin(72)                  # "0b1001000" decimal -> binary string
int("48", 16)            # 72         hex string -> decimal
int("1001000", 2)        # 72         binary string -> decimal

f"{72:02x}"              # "48"       two hex digits, no prefix (matches xxd)
f"{72:08b}"              # "01001000" eight-bit binary

b"Hello".hex()           # "48656c6c6f"
bytes.fromhex("48656c6c6f")  # b"Hello"
```

**Sources:**
- [Python built-in functions — official docs](https://docs.python.org/3/library/functions.html) — `bin`, `hex`, `int`, `ord`, `chr`
- [binascii — Convert between binary and ASCII — Python docs](https://docs.python.org/3/library/binascii.html) — lower-level helpers for bytes/hex conversion
- [Python ord() and chr() — DigitalOcean](https://www.digitalocean.com/community/tutorials/python-ord-chr) — worked examples
- [bytes.fromhex() Method — GeeksforGeeks](https://www.geeksforgeeks.org/python/bytes-fromhex-method-python/) — edge cases (odd-length input, whitespace handling)
- [F-String Python Hex, Oct, and Bin — Finxter](https://blog.finxter.com/f-string-python-hex-oct-and-bin-efficient-number-conversions/) — format specs `:02x`, `:08b` for clean output

## Lab Design Decisions

### Cross-platform commands, not Linux-only

The syllabus lists `lscpu`, `lsmem`, `lsblk` — all Linux-only. The course supports both macOS and Windows (via WSL2), and a large share of students are on Macs. Teaching lab 1 as "install a Linux VM first" would be a poor introduction. Every Linux command in this lab has a macOS equivalent (`sysctl`, `vm_stat`, `diskutil list`) that reads the same underlying kernel-discovered hardware data. The lab pairs them so every student sees real output from the machine in front of them. `xxd`, `df -h`, and Python are identical across platforms, which covers roughly half the lab already.

### Skip `lsmem`

The syllabus mentions `lsmem` as an option alongside `free -h`. `lsmem` lists memory hot-plug blocks and typically returns a single row on a laptop. It covers no ground `free -h` does not, has no macOS equivalent, and would spend attention better spent on the `xxd` demo. Dropped.

### One shell script per concept, with `uname` branching

Three demo files total: `hardware_demo.sh` (CPU + RAM + disk), `bytes_demo.sh` (xxd hex dump), `conversions_demo.py` (Python). Each hardware section uses `if [[ "$(uname)" == "Darwin" ]]` to pick the right command, so macOS and Linux students both see a working demo on their own hardware without reading "skip this if Mac." The shell branching is a visible pattern (not hidden in a helper function) so students see it once and understand how cross-platform shell scripts are typically structured.

### `xxd` section uses `Hello, world!\n` specifically

The text is 14 bytes of pure ASCII. That fits on one line of `xxd` output and maps one-to-one onto bytes students can match against an ASCII table in under two minutes. The `0a` newline at the end is a teachable moment for line endings without needing to introduce CR/LF across platforms.

### PNG magic bytes, written inline

Rather than ship a sample PNG, the demo writes the 8-byte PNG signature directly with `printf '\x89PNG\r\n\x1a\n'`. The file is not a valid PNG (no chunks), but it is byte-for-byte what every real PNG starts with. Students see file-format signatures as a concept in ~3 lines without carrying sample images in the repo.

### Python stays in the standard library

The research identified `psutil` as a useful cross-platform library for CPU/RAM/disk introspection. It is not required for the lab — the shell commands already show the same data, and introducing a pip install in the first hands-on lab raises setup friction for no pedagogical gain. The `conversions_demo.py` script uses built-ins only (`ord`, `chr`, `bin`, `hex`, `int`, f-strings, `bytes.fromhex`) so the lab runs on a vanilla Python 3 install.

### Capstone exercise: the hex column matches

The Python demo closes by looping over `"Hello"` and printing a four-column table: char, decimal, hex, binary. Students compare the hex column to what `xxd hello.txt` printed and verify the numbers are the same. That cross-reference is the moment the lab's two halves click together — two very different tools describing the same bytes, the same way, from the same machine.
