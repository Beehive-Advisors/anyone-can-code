# Research Report: Lab 1.5 - Seeing the Machine

## Overview

This lab lets students inspect the computer they are using instead of treating it like a sealed box. The point is not to memorize a list of commands. The point is to connect abstract ideas from the previous lectures to real output from a real machine.

The lab focuses on four concrete questions. What processor architecture is this machine using. How much memory does the system have and how does the operating system account for it. How does the operating system describe storage devices and partitions. How do bytes in a file connect back to binary, hexadecimal, decimal, and text.

This is a direct-typing lab. Every core step is a short terminal command or a short Python one-liner. No demo files are needed. The only temporary file students create is `hello.txt`, which exists to make raw bytes visible with `xxd`.

## 1. CPU And Architecture

### Background

Programs do not run in a vacuum. They run on a processor with a specific instruction set and a specific physical design. When a machine says it is `arm64` or `x86_64`, that is not branding. It tells you which family of machine instructions the processor can understand.

Students have already seen the ideas of fetch, decode, execute, and caches. This lab makes those ideas concrete by asking the operating system what kind of CPU is present and how many cores and cache levels it exposes.

### How it works

On Unix-like systems, commands such as `uname`, `lscpu`, and `sysctl` surface kernel-maintained facts about the hardware. They are not measuring the processor by benchmarking it. They are reading values the operating system already knows from firmware tables, kernel data structures, and platform-specific interfaces.

On Linux, `lscpu` aggregates data from sources such as `/proc/cpuinfo` and `sysfs`. On macOS, `sysctl` exposes named kernel parameters such as `hw.physicalcpu` and `machdep.cpu.brand_string`. The important teaching point is that the command output is a view into the operating system's model of the machine.

### Implementation

On macOS, the lab uses:

- `uname -m` to show the machine architecture
- `sysctl -n machdep.cpu.brand_string` to show the CPU model name
- `sysctl hw.physicalcpu hw.logicalcpu hw.l1dcachesize hw.l1icachesize hw.l2cachesize` to show core and cache facts

On Linux or WSL, the equivalent teaching command is `lscpu`.

**Sources:**
- [lscpu(1) - Linux manual page](https://man7.org/linux/man-pages/man1/lscpu.1.html)
- [uname - POSIX utility specification](https://pubs.opengroup.org/onlinepubs/9699919799/utilities/uname.html)
- [sysctl user-space manual reference via Apple Developer](https://developer.apple.com/library/archive/documentation/System/Conceptual/ManPages_iPhoneOS/man3/sysctlnametomib.3.html)

## 2. Memory As Pages

### Background

When people say a machine has 24 GB of RAM, that sounds like one big continuous bucket. Operating systems do not manage memory that way. They manage it in pages, which are fixed-size chunks that the kernel can track, move, protect, and reclaim.

That is why memory tools often look less intuitive than disk tools. They are reporting categories such as free, active, inactive, and wired pages. Those labels are the operating system's bookkeeping for what memory is doing right now.

### How it works

Linux tools such as `free` summarize data from `/proc/meminfo`. macOS tools such as `vm_stat` report page-based counters from the virtual memory subsystem. In both cases, the tool is not scanning every byte of RAM. It is reading kernel-maintained counters.

The lab has students look at page counts first and then convert those counts into a human-sized number. That matters pedagogically because it shows that the friendly number comes after the lower-level accounting, not before it.

### Implementation

On macOS, the lab uses:

- `sysctl hw.memsize` for total installed memory in bytes
- `vm_stat` for page-based memory accounting
- a short Python snippet to turn page counts into MiB

On Linux or WSL, the equivalent teaching command is `free -h`.

**Sources:**
- [free(1) - Debian manpage](https://manpages.debian.org/testing/procps/free.1.en.html)
- [Viewing Virtual Memory Usage - Apple](https://developer.apple.com/library/archive/documentation/Performance/Conceptual/ManagingMemory/Articles/VMPages.html)
- [vm_statistics_data_t - Apple Developer Documentation](https://developer.apple.com/documentation/kernel/vm_statistics_data_t)

## 3. Storage Layout And Raw Bytes

### Background

Files feel high-level because we usually open them with editors, players, browsers, or image viewers. Underneath that, storage is organized as devices, partitions, containers, volumes, and blocks. A file is still just bytes stored somewhere in that structure.

Showing students both the disk layout and a small byte dump matters because it connects the physical story to the file story. The disk layout tells you where storage is carved up. The hex dump shows what one tiny piece of file content actually looks like at the byte level.

### How it works

Linux uses block-device tools such as `lsblk` to present disks and partitions in a tree. macOS uses `diskutil` to expose disks, APFS containers, and volumes. For file bytes, `xxd` prints hexadecimal views of file contents, optionally including a binary view.

The lab deliberately uses a six-byte file, `Hello\n`, because it is small enough for students to reason about by hand. Each byte can be mapped to a decimal value, a hexadecimal value, and a character.

### Implementation

On macOS, the lab uses:

- `diskutil list | head -12` to inspect storage layout
- `xxd -g 1 -l 16 hello.txt` for a byte-oriented hex dump
- `xxd -b -l 6 hello.txt` for a bit-level view

On Linux or WSL, the equivalent storage command is `lsblk`.

**Sources:**
- [lsblk(8) - Linux manual page](https://man7.org/linux/man-pages/man8/lsblk.8.html)
- [diskutil(8) macOS manual mirror](https://keith.github.io/xcode-man-pages/diskutil.8.html)
- [xxd(1) - Ubuntu manpage](https://manpages.ubuntu.com/manpages/questing/en/man1/xxd.1.html)

## 4. Converting Between Bases And Characters

### Background

The machine does not care whether a human writes `72`, `0x48`, or `0b1001000`. Those are different notations for the same numeric value. The same byte can also be interpreted as a character such as `H` if both sides agree on an encoding.

This is the bridge from abstract number systems to practical computing. Students stop seeing binary and hex as classroom notation and start seeing them as multiple views of the same underlying value.

### How it works

Python integers are base-independent values. The functions `bin()` and `hex()` create string representations of those values in different bases. The function `int(text, base)` parses text back into a numeric value. The functions `ord()` and `chr()` convert between a character and its Unicode code point.

For ASCII characters such as `H`, the Unicode code point matches the traditional ASCII value. That makes this lab a good place to show the historical continuity: text in a file becomes bytes, and those bytes can be reasoned about numerically.

### Implementation

The lab uses only Python built-ins:

- `bin()` and `hex()` for decimal-to-string conversion
- `int(text, base)` for parsing binary and hexadecimal text
- `ord()` and `chr()` for character and code point conversion

## Lab Design Decisions

### Why this is a direct-typing lab

Every meaningful demo step fits in a one-liner or a very short heredoc. The student learns more by typing `uname`, `sysctl`, `vm_stat`, `diskutil`, `xxd`, and short Python snippets directly than by running a prewritten script that hides the commands.

### Why the primary tested flow is macOS

This repo is being built from a macOS machine, and the standards require verbatim tested output. The lab therefore uses macOS commands for every output block that appears in the handout and script. Linux and WSL equivalents are still documented in setup so the lab remains useful to non-macOS students.

### Why there is no committed demo file

The only temporary artifact is `hello.txt`, created during the lab so students can inspect a real file at the byte level. That file is not part of the repo. It is part of the learner's terminal session.
