# Research Report: Lab 1.5 — Seeing the Machine

## Overview

This lab is a beginner's first direct look at their own computer's hardware. Until now, phrases like "my laptop has 8 cores" or "I have 32 GB of RAM" have been abstract numbers on a spec sheet. The goal of this lab is to let the student pull those numbers out of the operating system with their own hands, on their own machine, using commands that have been on every Unix-family system for decades.

Five small concepts are covered: reading the CPU, reading system memory, reading the disk layout, dumping the raw bytes of a file, and converting between decimal, hexadecimal, binary, and ASCII using Python. Each concept is a single-command investigation. None of them require installing anything beyond the tools that already ship with macOS and Ubuntu. Together they close the gap between "a computer runs programs" and "a computer is a specific chip with specific memory holding a specific sequence of numbered bytes that I can look at right now."

The lab is intentionally the canonical example of a *no-scaffolding* lab. The commands themselves — `sysctl`, `lscpu`, `free -h`, `xxd`, `python3.12 -c "..."` — are the teaching artifact. Wrapping them inside a `demo.sh` would rob the student of exactly the muscle memory they are here to build. A single reference file (`sample.png`) ships with the lab so every student's hex dump produces an identical result.

## 1. CPU introspection

### Background

Every program eventually becomes instructions executed by a chip. Before you can reason about *what code runs*, you should be able to name the chip running it: how many cores it has, what architecture it is, and how much fast on-chip memory (cache) it owns. The operating system already tracks all of this — it has to, to schedule your programs — so there is always a way to ask.

### How it works

Linux exposes CPU facts through two kernel-generated pseudo-file trees. `/proc/cpuinfo` is a flat, per-logical-CPU text dump; `/sys/devices/system/cpu/` is a structured tree with one file per attribute. Neither is stored on disk — the kernel synthesizes them on read. The `lscpu` command (part of the `util-linux` package) reads both, deduplicates, and prints a human-friendly summary: architecture, total CPU count, threads per core, L1/L2/L3 cache sizes, model name.

macOS does not have `/proc`. The Darwin kernel instead exposes a **Management Information Base (MIB)** — a dotted hierarchical namespace like `hw.ncpu`, `hw.memsize`, `machdep.cpu.brand_string`. The `sysctl` command walks this namespace by calling `sysctlbyname(3)` and printing whatever the kernel returns. There is no single `lscpu`-style summary on macOS; the closest approximation is `sysctl hw machdep.cpu`, which dumps every CPU-related MIB entry as `key: value` pairs.

On Apple Silicon, a single physical chip contains both Performance cores and Efficiency cores, exposed as `hw.perflevel0.*` and `hw.perflevel1.*`. A student on an M-series Mac who runs `sysctl hw machdep.cpu | head -20` will see both levels side by side — not a bug, a feature of the architecture.

### Implementation: sysctl (macOS) / lscpu (Linux)

On macOS:

```
sysctl -n machdep.cpu.brand_string   # Apple M1 Pro
sysctl -n hw.ncpu                    # 8
sysctl hw machdep.cpu | head -20     # the full summary, one MIB per line
```

On Linux:

```
grep -m1 "model name" /proc/cpuinfo  # model name : AMD Ryzen 7 5700G with Radeon Graphics
nproc                                # 16
lscpu                                # aligned summary report
```

- [lscpu(1) man page](https://man7.org/linux/man-pages/man1/lscpu.1.html) — authoritative reference for the util-linux tool
- [SS64: macOS sysctl](https://ss64.com/mac/sysctl.html) — practical guide to the MIB tree
- [eclecticlight.co — sysctl information](https://eclecticlight.co/sysctl-information/) — explains how sysctl works on modern macOS

## 2. Memory introspection

### Background

RAM is finite. Everything — the OS, open apps, filesystem caches — competes for it. Before reasoning about "is my program using too much memory," the student needs to answer two simpler questions: *how much RAM does this machine have*, and *how much is currently in use*. Both are free data — the OS is already tracking them to do its job.

### How it works

Linux again uses a kernel-synthesized pseudo-file: `/proc/meminfo`. Every line is `Key: value kB`. Key fields include `MemTotal` (physical RAM the kernel sees), `MemFree` (literally unused right now), `MemAvailable` (an estimate of how much a new app could grab without swapping — equal to free + reclaimable cache + reclaimable slab), `Buffers` / `Cached` (the page cache), and `SwapTotal` / `SwapFree`. The `free -h` command reads `/proc/meminfo`, applies human-readable units (GiB / MiB), and prints a compact table. The `available` column is the single most useful number on a healthy Linux system — `MemFree` alone is misleading because the kernel uses "unused" RAM as page cache.

macOS exposes total RAM as a single MIB entry: `sysctl -n hw.memsize` returns an `int64_t` in bytes, with no unit suffix — a 32 GiB machine returns `34359738368`. Live memory breakdown comes from `vm_stat`, a Mach-layer tool whose output values are counted in **pages**, not bytes. The first line of `vm_stat` prints the page size — typically 4096 B on Intel Macs, **16384 B on Apple Silicon**. A student doing the math with the wrong page size will be 4× off on an M-series Mac.

### Implementation

On macOS:

```
sysctl -n hw.memsize                                            # 34359738368  (bytes)
echo "scale=2; $(sysctl -n hw.memsize) / 1073741824" | bc       # 32.00         (GiB)
vm_stat | head -12                                              # Mach VM breakdown
```

On Linux:

```
grep MemTotal /proc/meminfo                                     # MemTotal: 65178768 kB
free -h                                                         # total / used / free / available table
```

**Sources:**
- [proc_meminfo(5) man page](https://man7.org/linux/man-pages/man5/proc_meminfo.5.html) — every field documented
- [What is MemAvailable?](https://sitano.github.io/memory/kernel/2021/08/10/what-is-kernel-mavailable/) — explains the estimate
- [Apple VMPages documentation](https://developer.apple.com/library/archive/documentation/Performance/Conceptual/ManagingMemory/Articles/VMPages.html) — Mach page accounting

## 3. Disk layout

### Background

A "disk" you think of as one drive is typically divided into partitions, and each partition can hold a different filesystem (or be encrypted, or hold swap). When you double-click your "Macintosh HD" icon you are looking at one volume inside a container inside a partition inside a physical disk. Three tools reveal this: one for macOS's APFS hierarchy, one for Linux's block-device tree, and one cross-platform view of mounted filesystems.

### How it works

Linux's `lsblk` reads `/sys/dev/block/` (sysfs) plus the udev database. It prints a tree rooted at each physical disk, descending through partitions to any logical volumes (LVM) or RAID layers above them. Columns report size, removable flag, filesystem type, and mount point. `lsblk` never shows a disk that the kernel does not know about.

macOS's `diskutil list` walks the IOKit device tree and the APFS container layer. APFS introduces **synthesized disks** — a synthesized `diskN` is not physical storage; it is the container APFS exports so you can refer to a group of volumes that dynamically share free space. Inside the synthesized container, each volume is `diskNsM` and they do not have pre-sized partitions. The output makes this explicit — entries are tagged `(internal, physical)` or `(synthesized)`.

`df -h` is the only tool in this lab that works identically on both platforms. It reads the mount table (`/proc/self/mountinfo` on Linux, `getfsstat()` on Darwin) and prints one row per mounted filesystem: Size, Used, Avail, Use%, Mount point. The `-h` flag switches units from 1-KiB blocks to human-readable.

### Implementation

On macOS:

```
df -h /            # the root filesystem, Size / Used / Avail
diskutil list      # physical disk0 + synthesized disk3 with APFS volumes
```

On Linux:

```
df -h /            # identical command shape; different device naming
lsblk              # tree of disks → partitions
```

**Sources:**
- [lsblk(8) man page](https://man7.org/linux/man-pages/man8/lsblk.8.html) — canonical Linux reference
- [SS64: macOS diskutil](https://ss64.com/mac/diskutil.html) — all subcommands documented
- [APFS command tools](https://eclecticlight.co/2024/04/22/apfs-command-tools/) — synthesized disks explained

## 4. Raw bytes with xxd

### Background

Every file on disk is a sequence of bytes. A byte is a number from 0 to 255. When you open a `.txt` file, your editor interprets those numbers as characters. When you open a `.png`, your image viewer interprets them as pixels. The file itself does not know the difference — the interpretation lives in the program. `xxd` is the tool that lets you stop interpreting and just see the numbers, in hexadecimal, side by side with a best-effort ASCII rendering.

### How it works

`xxd` prints three columns per row: the byte offset (in hex), the next 16 bytes (as 8 space-separated pairs), and the same 16 bytes rendered as ASCII characters with non-printable bytes replaced by `.`. Sixteen bytes per row is a convention that fits inside an 80-column terminal.

Useful flags — all work identically on BSD `xxd` (macOS) and Vim `xxd` (Linux):

| Flag | Effect |
|------|--------|
| `-l N` | stop after N bytes |
| `-s N` | seek N bytes into the file before starting |
| `-c N` | bytes per row (default 16) |
| `-g N` | bytes per space-separated group (default 2) |
| `-p` | plain mode — continuous hex, no offset or ASCII |
| `-r` | reverse — rebuild a binary file from a hexdump |

`xxd` is shipped with `vim`. It is present by default on macOS and on most Linux distros, but on minimal Ubuntu 22.04 containers the `xxd` package must be installed separately (`apt install xxd`).

### The `sample.png` reference file

The lab ships an 80-byte, 16×16-pixel solid-red PNG generated programmatically with Python stdlib (`struct`, `zlib`). The small size means the full hex dump fits in 5 lines, exposing the PNG signature (`89 50 4e 47 0d 0a 1a 0a`), a single IHDR chunk, a single IDAT chunk, and the IEND terminator without drowning the student in compressed data. Because the bytes are pinned at commit time, every student running `xxd sample.png` sees identical output — the lab's captured outputs match what they see on their own machine.

### Implementation

```
echo "Hello, world!" > hello.txt
xxd hello.txt              # 00000000: 4865 6c6c 6f2c 2077 6f72 6c64 210a   Hello, world!.
xxd -l 16 sample.png       # 00000000: 8950 4e47 0d0a 1a0a 0000 000d 4948 4452  .PNG........IHDR
```

The first command shows that plain text is just a sequence of ASCII byte values, one per character, terminated by `0x0a` (line feed). The second command reveals that a PNG file starts with a fixed 8-byte magic signature followed by structured chunks — the file type is literally encoded in its first few bytes.

**Sources:**
- [xxd(1) man page](https://manpages.ubuntu.com/manpages/jammy/man1/xxd.1.html) — all flags and formats
- [Hex dump — Wikipedia](https://en.wikipedia.org/wiki/Hex_dump) — the three-column convention and its history
- [PNG file structure — W3C](https://www.w3.org/TR/PNG-Structure.html) — the 8-byte signature and chunk layout

## 5. Base conversion in Python

### Background

Humans count in base 10 because we have 10 fingers. Computers store numbers in base 2 because a bit is either on or off. Base 16 (hexadecimal) is the human-readable shorthand for base 2: one hex digit packs 4 bits, so a byte (8 bits) fits neatly in 2 hex digits (`0xff` = `11111111` = 255). All three are the *same number*, written differently. Python ships five built-in functions that convert between these representations — plus the character each number corresponds to in ASCII.

### How it works

| Function | Input | Output | Example |
|----------|-------|--------|---------|
| `bin(n)` | `int` | `str` with `'0b'` prefix | `bin(5)` → `'0b101'` |
| `hex(n)` | `int` | `str` with `'0x'` prefix, lowercase hex | `hex(255)` → `'0xff'` |
| `int(s, base)` | `str` + base 2–36 | `int` | `int('0x48', 16)` → `72` |
| `ord(c)` | 1-char `str` | `int` Unicode code point | `ord('H')` → `72` |
| `chr(n)` | `int` (0 to `0x10FFFF`) | 1-char `str` | `chr(72)` → `'H'` |

Round-trip identities hold: `chr(ord(c)) == c` and `int(hex(n), 16) == n`. The quirks to note for students: `bin()` and `hex()` include the `'0b'` / `'0x'` prefix in their output, `int()` accepts the prefix but does not require it when the base is explicit, and `ord()` rejects multi-character strings with a `TypeError`.

### The payoff: tying xxd back to Python

The most powerful moment in the lab is the closing one-liner. Students have just seen `xxd hello.txt` print `48 65 6c 6c 6f 2c 20 77 6f 72 6c 64 21 0a`. Feeding those same hex bytes into `chr(int(b, 16))` rebuilds the original string:

```
python3.12 -c "print(''.join(chr(int(b,16)) for b in ['48','65','6c','6c','6f']))"
# Hello
```

This closes the loop between "xxd showed me the numbers" and "those numbers were the letters all along."

### Implementation

```
python3.12 -c "print(hex(72))"             # 0x48
python3.12 -c "print(bin(72))"             # 0b1001000
python3.12 -c "print(int('0x48', 16))"     # 72
python3.12 -c "print(ord('H'))"            # 72
python3.12 -c "print(chr(72))"             # H
python3.12 -c "print(chr(int('48', 16)))"  # H
```

**Sources:**
- [Python built-in functions — bin, hex, int, ord, chr](https://docs.python.org/3/library/functions.html) — the canonical reference
- [Hexadecimal — Wikipedia](https://en.wikipedia.org/wiki/Hexadecimal) — why base 16 became the programmer's shorthand for base 2
- [ASCII — Wikipedia](https://en.wikipedia.org/wiki/ASCII) — the 128-code-point character set that `ord()` / `chr()` agree with for the printable range

## Lab Design Decisions

### No scaffolding — direct terminal typing only

Every command in this lab is a one-liner under 60 characters. There are no loops, no functions, no persistent state between commands. Wrapping them inside a `demo.sh` would turn "the student learned sysctl" into "the student ran a script that happened to call sysctl." The tool invocation is the learning artifact. This is why `STANDARDS.md` names Lab 1.5 as the canonical example of a no-scaffolding lab.

The exception: the lab ships one reference file, `sample.png`. A shipped PNG is not scaffolding — it is a deterministic inspection target. Without it, every student would pick their own image, every hex dump would look different, and the captured lab outputs would no longer match what the student sees.

### Self-contained lab directory

The student `cd`s into `1.5/` once at Setup and stays there for the rest of the lab. Scratch files they create (`hello.txt`) land in `1.5/` and are cleaned up at the end. `sample.png` ships with the folder and stays in place. Anyone handed `1.5/` on a USB drive can work end-to-end without network access or external downloads.

The `cd`-teaching-moment is placed in Part 4 — the first time the working directory actually matters for a command (creating `hello.txt`, reading `sample.png` by relative path). Setup mentions `cd 1.5/` but does not explain it; Part 4 is where we slow down and teach `pwd`, `cd` with no argument, and `cd -`.

### Two tracks, parallel structure

The macOS track captures live on the instructor's M1 Pro (Apple Silicon, 8 cores, 32 GiB RAM, APFS). The Linux track captures inside a kubectl pod on the bare-metal k0s cluster (x86_64 Ryzen 7, 16 threads, 62 GiB RAM). The two tracks have the same Part count, the same exercises, the same conceptual questions. Only the commands and the captured outputs differ — by design, because the *point* of the lab is to let the student see their own machine through their OS's native dialect.

### Python 3.12, no venv, no pip

Every Python invocation in this lab is `python3.12 -c "..."` in the top-level shell. No virtual environment is created. No packages are installed. This is deliberate: the student should finish the lab believing that base conversion requires zero dependencies. Future labs will introduce `uv venv` and `pip install`; this one explicitly does not.
