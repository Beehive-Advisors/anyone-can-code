## Concept: What we're "seeing"

**SPEAK**
> "Lectures 1.1 through 1.4 made some big claims. Everything is bits. The CPU fetches, decodes, executes. There's a memory hierarchy with caches, RAM, and disk. Those are all true, but they're the kind of thing you can nod at and then forget, because you've never looked at them."
>
> "This lab's goal is to let you look. Your OS has already discovered every fact about your hardware — how many cores you have, how big your L1 cache is, what disks are attached, how much RAM is installed. It stores those facts in kernel data structures you can read with one command each. By the end of this video you'll understand the three pieces of code that read them, plus a small Python script that turns characters into the bits they really are."

**SPEAK**
> "Quick primer on the four things we'll touch. The CPU is the chip that runs code. It has cores — independent workers — and caches, which are tiny pockets of very fast memory sitting right next to the cores. RAM is the working scratchpad, bigger than cache but slower. Disk holds everything that survives a reboot, bigger than RAM but much slower. And a file on disk — any file — is a sequence of numeric byte values. A text file's bytes happen to match letters in the ASCII table. A PNG's bytes happen to match the PNG format's conventions. Once you see a file as its bytes, a lot of other concepts later in the course stop being mysterious."

## hardware_demo.sh — Section A: CPU

**TYPE**
```bash

```

**EXPLAIN** (lines 1–14 — header and platform detection)
```bash
#!/usr/bin/env bash
# hardware_demo.sh — read CPU, memory, and disk facts the kernel already knows
#
# What this shows:
#   - CPU architecture, core count, and cache sizes
#   - Total RAM and current usage
#   - Physical disks vs partitions vs mount points
#   - The scale ratio across the memory hierarchy (cache → RAM → disk)
#
# Run:
#   bash hardware_demo.sh
#
# Works on macOS and Linux. Uses `uname` to pick the right command per OS.

set -u
OS="$(uname)"
```
> "Line 1 is the shebang — tells the shell to use bash when this file runs. Lines 2 through 12 are a header comment; every demo file in this course has one, and it's the first thing you should write when you start a new script."
>
> "Line 14 is where the real work starts. `set -u` makes bash yell if we reference an undefined variable — cheap insurance. `OS=$(uname)` captures the operating system name — `Darwin` on macOS, `Linux` on Linux and WSL — so we can use it to pick commands later. The whole reason this script works cross-platform is that one line."

**EXPLAIN** (lines 22–46 — the macOS CPU branch)
```bash
if [[ "$OS" == "Darwin" ]]; then
  echo "  --- Chip and core counts (via sysctl) ---"
  echo "  Brand:          $(sysctl -n machdep.cpu.brand_string 2>/dev/null || echo '(not exposed on this chip)')"
  echo "  Physical cores: $(sysctl -n hw.physicalcpu)"
  echo "  Logical cores:  $(sysctl -n hw.logicalcpu)"
  echo "  Cache line:     $(sysctl -n hw.cachelinesize) bytes"
```
> "Line 22 opens the macOS branch. On macOS the hardware-info API is called `sysctl`, and you query it with dotted key names like `hw.physicalcpu`. Line 24 gets the chip name — `machdep.cpu.brand_string` returns something like 'Apple M1 Pro' on Apple Silicon or 'Intel Core i7' on an Intel Mac. The `2>/dev/null || echo '(not exposed...)'` fallback is there because on some Apple Silicon chips this key is empty; we don't want the script to crash."
>
> "Lines 25 and 26 are the core counts. Physical is the real number of cores. Logical can be higher on Intel Macs because of hyperthreading, which lets one physical core pretend to be two. Apple Silicon chips don't have hyperthreading, so physical equals logical."
>
> "Line 27 is the cache line size — that's the chunk of bytes the CPU fetches from RAM into cache at one time. 64 bytes on most Intel, 128 bytes on Apple Silicon. This is why it matters how data is laid out in memory: touching one byte pulls in 64 or 128 neighbors, and if you use those neighbors right away, they're already in cache."

**EXPLAIN** (lines 32–46 — cache size detection on Apple Silicon vs Intel)
```bash
  if sysctl -n hw.perflevel0.l1dcachesize >/dev/null 2>&1; then
    echo "  L1d (perf):  $(sysctl -n hw.perflevel0.l1dcachesize) bytes per core"
    echo "  L1i (perf):  $(sysctl -n hw.perflevel0.l1icachesize) bytes per core"
    echo "  L2  (perf):  $(sysctl -n hw.perflevel0.l2cachesize) bytes per cluster"
    if sysctl -n hw.perflevel1.l1dcachesize >/dev/null 2>&1; then
      echo "  L1d (eff):   $(sysctl -n hw.perflevel1.l1dcachesize) bytes per core"
      echo "  L2  (eff):   $(sysctl -n hw.perflevel1.l2cachesize) bytes per cluster"
    fi
  else
    echo "  L1d: $(sysctl -n hw.l1dcachesize 2>/dev/null || echo 'n/a') bytes"
    echo "  L1i: $(sysctl -n hw.l1icachesize 2>/dev/null || echo 'n/a') bytes"
    echo "  L2:  $(sysctl -n hw.l2cachesize  2>/dev/null || echo 'n/a') bytes"
    echo "  L3:  $(sysctl -n hw.l3cachesize  2>/dev/null || echo 'n/a') bytes"
  fi
```
> "This block is feature detection. Line 32 asks: does the key `hw.perflevel0.l1dcachesize` exist? If it does, we're on an Apple Silicon chip with performance and efficiency cores, and the caches are reported per cluster. Lines 33 through 39 query those keys."
>
> "If we're not on Apple Silicon — the `else` branch starting at line 40 — we're on an Intel Mac and the traditional keys apply: `hw.l1dcachesize`, `hw.l2cachesize`, `hw.l3cachesize`. Same idea, different key names. This pattern of feature-detect-then-query shows up constantly in cross-platform code. Don't assume; ask first."

## bytes_demo.sh — hex dumps and file structure

**TYPE**
```bash
code bytes_demo.sh
```

**EXPLAIN** (lines 1–14 — header and setup)
```bash
#!/usr/bin/env bash
# bytes_demo.sh — look at a file's raw bytes with xxd
#
# What this shows:
#   - A plain-text file is a sequence of numeric byte values
#   - Hex dump: offset | hex bytes | ASCII gutter
#   - The same bytes viewed as hex (xxd) and as bits (xxd -b)
#   - File-format magic bytes: a PNG starts with a specific signature
#
# Run:
#   bash bytes_demo.sh

set -e

DIR="$(cd "$(dirname "$0")" && pwd)"
HELLO="$DIR/hello.txt"
PNG="$DIR/tiny.png"
```
> "Header comment, then `set -e` on line 13 — that tells bash to stop immediately if any command fails, which is what you want for a demo. No silent errors."
>
> "Line 15 gets the absolute path to the directory this script lives in. `dirname "$0"` is the directory of the script itself; `cd` plus `pwd` resolves it to an absolute path. That way `hello.txt` and `tiny.png` are written next to the script no matter where you run the script from."

**EXPLAIN** (lines 20–27 — writing and dumping hello.txt)
```bash
printf 'Hello, world!\n' > "$HELLO"
echo "  Wrote $HELLO  ($(wc -c < "$HELLO" | tr -d ' ') bytes)"
echo ""
echo "  --- xxd hello.txt ---"
xxd "$HELLO"
```
> "Line 20 uses `printf` to write exactly 14 bytes — 13 printable characters plus one trailing newline. `\n` is the escape sequence for the newline byte, which is decimal 10 or hex 0x0a. `printf` is more predictable than `echo` for this — `echo`'s behavior around newlines varies across systems, `printf` is defined by the POSIX spec."
>
> "Line 21 counts the bytes in the file using `wc -c` to confirm we wrote 14. The `tr -d ' '` strips leading whitespace from `wc`'s output. Line 24 runs `xxd` with no flags — that's the three-column default: offset, hex pairs, ASCII gutter."

**EXPLAIN** (lines 37–45 — computing the binary representation by hand in bash)
```bash
BIN=""
N=$((16#48))
for ((i=7; i>=0; i--)); do
  BIN+=$(( (N >> i) & 1 ))
done
echo "    As binary:  $BIN"
```
> "This block converts the number 0x48 into its binary string, purely in bash, without any Python. Line 38 takes the number 72 — `16#48` is bash syntax for 'parse 48 as base 16.' Lines 39 through 41 loop over bit positions 7 down to 0, right-shift the number by that many positions, and mask the lowest bit. That gives us the value of each bit. We concatenate them into a string."
>
> "I'm showing you this because the same mental model will come up later when you do it in Python — bits are extractable one at a time with shifts and masks. That's what the CPU actually does in hardware, and it's what languages do to implement `bin()`."

**EXPLAIN** (lines 53–58 — binary view with xxd -b)
```bash
echo "  --- xxd -c 4 -b hello.txt  (4 bytes per line, binary view) ---"
xxd -c 4 -b "$HELLO" | head -n 4
```
> "Line 54 re-runs `xxd` with two flags: `-c 4` means '4 bytes per line' and `-b` means 'binary view.' So instead of hex pairs in the middle column, you see eight zeros-and-ones per byte. On line 1 of that output you'll see `01001000 01100101 01101100 01101100` — that's the bits for 'Hell,' literally. Lecture 1.1's 'text is binary' claim, visible."

**EXPLAIN** (lines 65–71 — writing a PNG file signature)
```bash
printf '\x89PNG\r\n\x1a\n' > "$PNG"
echo "  Wrote $PNG  ($(wc -c < "$PNG" | tr -d ' ') bytes — just the file signature)"
echo ""
echo "  --- xxd tiny.png ---"
xxd "$PNG"
```
> "Line 65 is sneaky. We use `printf` again, but this time with `\x` escape sequences — `\x89` writes the single byte with value 0x89. So this line writes eight bytes: 0x89, then the three ASCII characters P-N-G, then a carriage return (0x0d), a linefeed (0x0a), a control character 0x1a, and another linefeed. Those eight bytes are the PNG file signature defined in the PNG specification."
>
> "The file we created isn't a valid image — a real PNG has data chunks after the signature — but the *first 8 bytes* are exactly what every real PNG on your disk starts with. When your image viewer opens a file, it reads these 8 bytes and uses them to decide 'yes, this is a PNG.' Most file formats work this way."

## conversions_demo.py — Python number bases

**TYPE**
```bash
code conversions_demo.py
```

**EXPLAIN** (lines 1–15 — docstring and header)
```python
#!/usr/bin/env python3.12
"""
conversions_demo.py — convert between char, decimal, hex, and binary in Python

What this shows:
  - One byte has four equivalent representations: character, decimal, hex, binary
  - ord() / chr() bridge characters and integers
  - bin() / hex() / int() / f-string format specs produce and parse number strings
  - bytes.hex() / bytes.fromhex() do the same trip at the bytes-object level (the
    machine-readable equivalent of `xxd` and `xxd -r`)

Run:
    python3.12 conversions_demo.py
"""
```
> "Python docstring instead of a comment block — same purpose, different syntax. Triple-quoted string at the top of a module is both the module documentation and a human-readable header. I always put the run command in the docstring so you never have to guess."

**EXPLAIN** (lines 22–34 — four representations of 'H')
```python
ch = "H"
n = ord(ch)
print(f"  Character:     {ch!r}")
print(f"  ord('{ch}')        = {n}            (decimal)")
print(f"  hex({n})        = {hex(n)!r}          (hex string, with '0x' prefix)")
print(f"  bin({n})        = {bin(n)!r}     (binary string, with '0b' prefix)")
print(f"  chr({n})        = {chr(n)!r}            (back to the character)")

hex_str = f"{n:02x}"
bin_str = f"{n:08b}"
print(f"  f\"{{n:02x}}\"      = {hex_str!r}          (two hex digits — like xxd)")
print(f"  f\"{{n:08b}}\"      = {bin_str!r}    (eight bits — one full byte)")
```
> "Line 22 assigns the single-character string 'H' to `ch`. Line 23 calls `ord(ch)` — that returns the integer code point. For 'H' that's 72. Every printable ASCII character has a code point between 32 and 126."
>
> "Lines 25 through 28 are f-strings that print the four representations: decimal 72, hex `0x48`, binary `0b1001000`, character 'H'. The `!r` suffix in f-strings means 'use repr' — that's why the string appears with quotes around it. Without `!r`, we'd just print `H`, not `'H'`."
>
> "Lines 31 and 32 are the clean formats. `:02x` inside an f-string means 'format as hex, two digits, zero-padded.' That matches how `xxd` shows a byte — always two digits, 01 through FF. `:08b` means 'binary, eight digits, zero-padded' — one full byte of bits. These f-string format specs are the same ones you'd use with the older `format()` function."

**EXPLAIN** (lines 41–58 — round trip)
```python
step1 = ord("H")
step2 = f"{step1:02x}"
step3 = f"{step1:08b}"
step4 = int(step2, 16)
step5 = int(step3, 2)
step6 = chr(step4)
```
> "Lines 41 through 46 prove the conversions are inverses — round-trip. Line 41 goes from character to integer. Lines 42 and 43 format the integer as hex and binary strings. Lines 44 and 45 parse those strings back into integers. `int(step2, 16)` means 'parse step2 as base 16.' `int(step3, 2)` means 'parse step3 as base 2.' Line 46 converts the integer back to a character."
>
> "The whole point is: if you end up at the character you started with, no information was lost. You can freely convert between these four views — decimal, hex, binary, character — and none of them is more or less real. They're all the same number in different clothes."

**EXPLAIN** (lines 69–77 — looping over a word)
```python
word = "Hello"
print(f"  {'char':<6}{'decimal':<10}{'hex':<6}{'binary':<10}")
print(f"  {'----':<6}{'-------':<10}{'---':<6}{'------':<10}")
for c in word:
    code = ord(c)
    print(f"  {c!r:<6}{code:<10}{code:02x}    {code:08b}")
```
> "Line 69 is the word we're going to decompose. Lines 70 and 71 print a header row — `:<6` means 'left-align in a 6-character-wide field.' That's how we line up the columns."
>
> "Line 72 iterates over the string — in Python, iterating over a string gives you one character at a time. Line 73 gets the integer code. Line 74 prints one row: character in quotes, decimal, two-digit hex, eight-digit binary. When students look at the hex column in the output — 48 65 6c 6c 6f — they should see the exact same five hex pairs that xxd printed for hello.txt. That's the moment the two halves of the lab click together."

**EXPLAIN** (lines 86–93 — bytes round trip)
```python
original = b"Hello, world!"
as_hex   = original.hex()
restored = bytes.fromhex(as_hex)

print(f"  Original bytes:     {original!r}")
print(f"  original.hex():     {as_hex!r}")
print(f"  bytes.fromhex(...): {restored!r}")
print(f"  Round trip holds:   {original == restored}")
```
> "Line 86 declares a `bytes` object — that's what the `b` prefix means. Bytes are a sequence of integers 0 through 255, exactly like a file on disk. Line 87 calls `.hex()` on it, which flattens the whole sequence into one long hex string. Line 88 calls `bytes.fromhex()` to rebuild the bytes from the string."
>
> "This is the programmatic version of what `xxd` and `xxd -r` do. Later in the course, when you read files byte-by-byte for networking or binary parsing, you'll use these same two methods constantly. It's the same conversion we just did for one character, scaled up to a sequence."
