## PART 1 — SEEING THE CPU

### What is the CPU?

**SPEAK**
> "The CPU is the chip that runs your code. It has an architecture — what instruction set it speaks — a number of cores, and a cache hierarchy. Lecture 1.4 drew the memory pyramid. L1, L2, L3 caches sit right next to the cores. RAM sits below that. Each tier is bigger and slower. The Linux kernel knows every one of these numbers; you just have to ask it with `lscpu`."

### Run the demo

**TYPE**
```bash
lscpu | grep -E 'Architecture|Model name|^CPU\(s\):'
```

**OUTPUT**
```
Architecture:                         x86_64
CPU(s):                               16
Model name:                           AMD Ryzen 7 5700G with Radeon Graphics
```

**EXPLAIN**
> "Architecture tells us the instruction set — x86_64 on most laptops and desktops. Model name is the chip. CPU(s) is the logical CPU count. 16 here because this is an 8-core chip with SMT, so the kernel sees 16 independent execution streams."

**TYPE**
```bash
nproc
```

**OUTPUT**
```
16
```

**EXPLAIN**
> "`nproc` is the no-frills version — just the count. Same number."

**TYPE**
```bash
lscpu -C
```

**OUTPUT**
```
NAME ONE-SIZE ALL-SIZE WAYS TYPE        LEVEL  SETS PHY-LINE COHERENCY-SIZE
L1d       32K     256K    8 Data            1    64        1             64
L1i       32K     256K    8 Instruction     1    64        1             64
L2       512K       4M    8 Unified         2  1024        1             64
L3        16M      16M   16 Unified         3 16384        1             64
```

**EXPLAIN**
> "The cache hierarchy. L1d and L1i each 32 KiB per core — eight cores, 256 KiB total. L2 is 512 KiB per core — 4 MiB total. L3 is 16 MiB, one instance shared across all cores. That's the pyramid from lecture 1.4. Tiny L1, bigger L2, bigger L3."

### Exercise — cache ratio

**SPEAK**
> "Let's see how much bigger L2 is than L1."

**TYPE**
```bash
python3 -c "print(512 / 32)"
```

**OUTPUT**
```
16.0
```

**EXPLAIN**
> "L2 is 16 times bigger than L1d on this chip. The exact ratio varies by manufacturer, but the pattern — L1 is tiny, L2 is 10-20× bigger, L3 is bigger still — holds everywhere."

## PART 2 — SEEING RAM

### What is RAM?

**SPEAK**
> "RAM is the scratchpad. Your programs, their variables, their open files — all live in RAM while they run. Linux tracks it in categories — free, used, shared, cache, available — and `free -h` prints the whole breakdown."

### Run the demo

**TYPE**
```bash
grep MemTotal /proc/meminfo
```

**OUTPUT**
```
MemTotal:       65178768 kB
```

**EXPLAIN**
> "Total installed RAM — the kernel reports it in kibibytes. Divide by 1024 squared to get GiB."

**TYPE**
```bash
free -h
```

**OUTPUT**
```
               total        used        free      shared  buff/cache   available
Mem:            62Gi       6.6Gi        26Gi        48Mi        28Gi        54Gi
Swap:             0B          0B          0B
```

**EXPLAIN**
> "The `free -h` output in one look. Total: 62 GiB. Used: 6.6 GiB. Free: 26 GiB. `buff/cache`: 28 GiB — disk blocks the kernel cached in RAM, reclaimable the moment something wants them. `available`: 54 GiB — that's free plus reclaimable. That's the number you actually care about when asking 'can I open this?' Not `free`."

**TYPE**
```bash
head -5 /proc/meminfo
```

**OUTPUT**
```
MemTotal:       65178768 kB
MemFree:        28204856 kB
MemAvailable:   57427248 kB
Buffers:         1717748 kB
Cached:         25486936 kB
```

**EXPLAIN**
> "Straight from the kernel. `/proc/meminfo` is the source — `free` is just a formatter over these rows. The difference between `MemFree` and `MemAvailable` is the reclaimable cache."

### Exercise — convert units

**SPEAK**
> "Let me prove the OS's GiB and the spec sheet's GB are different numbers for the same thing."

**TYPE**
```bash
python3 -c "kb = 65178768; print(f'{kb / 1024**2:.2f} GiB, {kb * 1024 / 1000**3:.2f} GB')"
```

**OUTPUT**
```
62.16 GiB, 66.74 GB
```

**EXPLAIN**
> "Spec sheets use decimal GB — a billion bytes. The OS uses GiB — 1024 cubed bytes. That's why a 'SIZE GB' module always shows as a smaller number in a file manager. The difference is 7%."

## PART 3 — SEEING DISKS

### What is a disk?

**SPEAK**
> "Below RAM in the pyramid: disk. Huge, slow, persistent. Not one flat slab — sliced into partitions, each holding a filesystem, each mounted at a directory. `lsblk` prints the whole tree."

### Run the demo

**TYPE**
```bash
lsblk
```

**OUTPUT**
```
NAME        MAJ:MIN RM   SIZE RO TYPE MOUNTPOINTS
sda           8:0    0     5G  0 disk 
sdb           8:16   0    10G  0 disk 
sdc           8:32   0     1G  0 disk 
nvme0n1     259:0    0 931.5G  0 disk 
|-nvme0n1p1 259:1    0     1G  0 part 
|-nvme0n1p2 259:2    0     2G  0 part 
`-nvme0n1p3 259:3    0 928.5G  0 part 
```

**EXPLAIN**
> "Each top-level row with `TYPE disk` is a physical device. `nvme0n1` is the laptop's main SSD — 931 GiB. Nested under it are the partitions: a small EFI boot partition, a boot partition, and the big one for `/`. On a WSL2 system you'd typically see just `sda` with one big partition — the virtual disk Microsoft gives the Linux VM."

**TYPE**
```bash
df -h | head -3
```

**OUTPUT**
```
Filesystem                         Size  Used Avail Use% Mounted on
overlay                            913G   40G  828G   5% /
tmpfs                               64M     0   64M   0% /dev
```

**EXPLAIN**
> "`df -h` answers: for each mounted filesystem, how big and how full? Top row is the root — `/` — 913 gigabytes, 40 used, 828 free. That's where every file you create lives."

### Exercise — find your home partition

**TYPE**
```bash
df -h ~
```

**OUTPUT**
```
Filesystem                         Size  Used Avail Use% Mounted on
overlay                            913G   40G  828G   5% /
```

**EXPLAIN**
> "`~` expands to your home directory. `df -h ~` tells you which filesystem it's on. Here, same as root — one filesystem holding everything. Some desktop Linux installs separate `/home` onto its own partition; this would show a different row in that case."

## PART 4 — SEEING A FILE AS BYTES

### What is a file at the byte level?

**SPEAK**
> "Lecture 1.1 said everything is bytes. Now you see it. `xxd` takes any file and prints the raw hex. Text, images, anything — all bytes on your disk. Let's open a text file."

### Run the demo

**TYPE**
```bash
cd /tmp
echo 'Hello!' > hello.txt
xxd hello.txt
```

**OUTPUT**
```
00000000: 4865 6c6c 6f21 0a                        Hello!.
```

**EXPLAIN**
> "Three columns. Offset — byte 0. Hex bytes — `48 65 6c 6c 6f 21 0a`. ASCII — `Hello!` plus a dot for the newline, because newline isn't a printable character. `48` is `H`. `65` is `e`. Every letter is a byte."

**TYPE**
```bash
xxd -b hello.txt
```

**OUTPUT**
```
00000000: 01001000 01100101 01101100 01101100 01101111 00100001  Hello!
00000006: 00001010                                               .
```

**EXPLAIN**
> "Same file, bits not hex. `01001000` is H. Eight bits. Lecture 1.2 said gates combine into flip-flops, each holding one bit. Those eight zeros and ones are the actual state of eight flip-flops somewhere on my SSD."

**TYPE**
```bash
curl -sO https://www.libpng.org/pub/png/img_png/pnglogo-blk.png
xxd -l 16 pnglogo-blk.png
```

**OUTPUT**
```
00000000: 8950 4e47 0d0a 1a0a 0000 000d 4948 4452  .PNG........IHDR
```

**EXPLAIN**
> "The first 8 bytes of every PNG on Earth: `89 50 4e 47 0d 0a 1a 0a`. That's the PNG magic number. The ASCII column shows `.PNG........IHDR` — the dot is byte `89`, which isn't printable. Then `PNG`. Then control bytes for format detection. Then the IHDR chunk. Every image viewer checks these bytes to decide what it's looking at. Not the filename — the bytes."

### Exercise — dump your own file

**TYPE**
```bash
xxd -l 32 /tmp/hello.txt
```

**OUTPUT**
```
00000000: 4865 6c6c 6f21 0a                        Hello!.
```

**EXPLAIN**
> "Or try any file in your home. A source file. An image. A PDF. `-l 32` limits to the first 32 bytes. The first 2-4 usually identify the format."

## PART 5 — SAME BYTES, NOW IN PYTHON

### What does it mean for a number to be in a base?

**SPEAK**
> "You saw `50 4e 47` spell PNG in the xxd output. Let's prove it in Python, one one-liner at a time."

### Run the demo

**TYPE**
```bash
python3 -c "print(hex(80))"
```

**OUTPUT**
```
0x50
```

**TYPE**
```bash
python3 -c "print(bin(80))"
```

**OUTPUT**
```
0b1010000
```

**TYPE**
```bash
python3 -c "print(ord('P'))"
python3 -c "print(chr(80))"
```

**OUTPUT**
```
80
P
```

**EXPLAIN**
> "One number, 80, shown four ways. Hex `0x50`. Binary `0b1010000`. Decimal `80`. Character `P`. `ord` and `chr` are the pair that go back and forth between character and integer. Same number every time."

**TYPE**
```bash
python3 -c "print(int('50', 16))"
python3 -c "print(int('01010000', 2))"
```

**OUTPUT**
```
80
80
```

**EXPLAIN**
> "`int` with a base parses a string. Parse `50` as base 16, get 80. Parse `01010000` as base 2, get 80. Every representation round-trips."

**TYPE**
```bash
python3 -c "print(f'{15:02x}')"
python3 -c "print(f'{5:08b}')"
```

**OUTPUT**
```
0f
00000101
```

**EXPLAIN**
> "f-string format specs. `:02x` means hex, minimum 2 digits, zero-padded. That's what `xxd` uses — always two hex digits per byte. `:08b` means binary, 8 digits, zero-padded. One full byte of bits."

**TYPE**
```bash
python3 -c "print(bytes.fromhex('89504e47'))"
```

**OUTPUT**
```
b'\x89PNG'
```

**EXPLAIN**
> "The exact bytes you saw at the start of that PNG. `\x89` for the non-printable byte, then literal `PNG` because those bytes are printable characters. Python and xxd describing the same bytes from two directions."

### Exercise — your own initial

**TYPE**
```bash
python3 -c "c='J'; print(f'{c!r} -> dec={ord(c)}, hex={ord(c):02x}, bin={ord(c):08b}')"
```

**OUTPUT**
```
'J' -> dec=74, hex=4a, bin=01001010
```

**EXPLAIN**
> "J is decimal 74, hex 4a, binary 01001010. Change `J` to your own initial and run it. If you try uppercase and lowercase of the same letter, the difference is always exactly one bit — 0x20. That bit is the case toggle."

## Recording notes

- Record from a real Ubuntu 22.04 environment or a kubectl pod on a real Linux host. Do NOT record from Docker on macOS — sysfs-dependent commands (`lsblk`) produce reduced output there.
- Keep each command visible for ~3 seconds before moving on.
- At the `89 50 4e 47` PNG magic number, pause 2 seconds — this is the "aha" moment.
- Cross-reference: when showing the PNG magic number, verbally call back to the Python section where `bytes.fromhex('89504e47')` will produce `b'\x89PNG'`.
- On WSL2 specifically: `lsblk` may show only `sda` (the Linux VM's virtual disk) rather than the Windows host's drives. Mention this once — WSL2 is a virtual Linux machine, not direct hardware access.
- Ubuntu 22.04's default `python3` is 3.10. Every built-in used here has been stable since Python 3.6, so 3.10 vs 3.12 is pedagogically invisible.
- If the `curl` call fails (no network), substitute any PNG already in the filesystem.
