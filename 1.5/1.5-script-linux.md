## PART 1 ‚Äî SEEING THE CPU

### What is the CPU?

**SPEAK**
> "The CPU is the chip that runs your code. It has an architecture ‚Äî what instruction set it speaks ‚Äî a number of cores, and a cache hierarchy. Lecture 1.4 drew the memory pyramid. L1, L2, L3 caches sit right next to the cores. RAM sits below that. Each tier is bigger and slower. The Linux kernel knows every one of these numbers; you just have to ask it with `lscpu`."

### Run the demo

**TYPE**
```bash
lscpu | grep -E 'Architecture|Model name|^CPU\(s\):'
```

**OUTPUT**
```
Architecture:                         x86_64
CPU(s):                               16
Model name:                           AMD Ryzen 7 5700G with Radeon Graphics
```

**EXPLAIN**
> "Three labeled lines. Architecture: `x86_64` ‚Äî the instruction set this CPU speaks. On a Raspberry Pi or Windows-on-ARM laptop you'd see `aarch64` instead. CPU(s): 16. That's logical CPUs. The chip has 8 physical cores doing SMT (AMD's term) or Hyper-Threading (Intel's term) ‚Äî each physical core pretends to be two, so the kernel sees 16. All three facts came from the kernel's cached answer to the CPUID query."

**TYPE**
```bash
nproc
```

**OUTPUT**
```
16
```

**EXPLAIN**
> "Same count as `lscpu` ‚Äî same underlying kernel data. `nproc` is the no-frills version, what shell scripts call when they just need 'how many workers?'."

**TYPE**
```bash
lscpu -C
```

**OUTPUT**
```
NAME ONE-SIZE ALL-SIZE WAYS TYPE        LEVEL  SETS PHY-LINE COHERENCY-SIZE
L1d       32K     256K    8 Data            1    64        1             64
L1i       32K     256K    8 Instruction     1    64        1             64
L2       512K       4M    8 Unified         2  1024        1             64
L3        16M      16M   16 Unified         3 16384        1             64
```

**EXPLAIN**
> "The memory pyramid from lecture 1.4, as a table. Four rows, four levels. ONE-SIZE is per-instance; ALL-SIZE is the total across instances. L1d and L1i are 32 KiB each, per core (8 instances ‚Üí 256 KiB total). The split between instruction and data caches lets the CPU fetch the next instruction in parallel with reading its data. L2 is 512 KiB per core, unified. L3 is 16 MiB shared across the whole chip. That's the pyramid, in numbers, on this specific silicon."

### Exercise ‚Äî cache ratio

**SPEAK**
> "Let's see how much bigger L2 is than L1. Instead of copying numbers by hand, we'll capture the cache sizes into shell variables and let Python do the math ‚Äî so this runs on *your* machine with *your* numbers, not mine."

**TYPE**
```bash
L1=$(lscpu -C -B | awk '$1=="L1d"{print $2}')
L2=$(lscpu -C -B | awk '$1=="L2"{print $2}')
python3 -c "print($L2 / $L1)"
```

**OUTPUT**
```
16.0
```
*(Your ratio will differ by chip.)*

**EXPLAIN**
> "`lscpu -C -B` prints cache sizes in raw bytes ‚Äî the `-B` flag turns off the human-readable K/M suffixes. `awk` picks the L1d and L2 rows by name and grabs the second column (ONE-SIZE). `$(...)` captures each value into a shell variable, which Python then divides. L2 is 16√ó bigger than L1 on this chip. The pattern ‚Äî each tier multiplies capacity by roughly 10-100√ó ‚Äî is why data that fits in L1 runs so much faster than data that spills into RAM."

## PART 2 ‚Äî SEEING RAM

### What is RAM?

**SPEAK**
> "RAM is the scratchpad. Your programs, their variables, their open files ‚Äî all live in RAM while they run. Linux tracks it in categories ‚Äî free, used, shared, cache, available ‚Äî and `free -h` prints the whole breakdown."

### Run the demo

**TYPE**
```bash
grep MemTotal /proc/meminfo
```

**OUTPUT**
```
MemTotal:       65178768 kB
```

**EXPLAIN**
> "One row from `/proc/meminfo` ‚Äî the raw total RAM. The `kB` label on Linux actually means 1024 bytes, not 1000 (an old inconsistency). Divide by 1024¬≤ for GiB. The exact number is a little under 64 GiB because some memory is reserved for the kernel and firmware at boot ‚Äî normal on every Linux system."

**TYPE**
```bash
free -h
```

**OUTPUT**
```
               total        used        free      shared  buff/cache   available
Mem:            62Gi       6.6Gi        26Gi        48Mi        28Gi        54Gi
Swap:             0B          0B          0B
```

**EXPLAIN**
> "Six columns. `total` is installed RAM. `used` is what running programs committed. `free` is truly untouched. `buff/cache` ‚Äî 28 GiB ‚Äî is disk blocks the kernel keeps in RAM in case you re-read them; reclaimable the moment something needs memory. `available` is `free + reclaimable cache` ‚Äî the number you actually care about. The common mistake is looking at `free` alone and panicking. Empty memory is wasted memory; a healthy system keeps `free` small."

**TYPE**
```bash
head -5 /proc/meminfo
```

**OUTPUT**
```
MemTotal:       65178768 kB
MemFree:        28204856 kB
MemAvailable:   57427248 kB
Buffers:         1717748 kB
Cached:         25486936 kB
```

**EXPLAIN**
> "The same five counters `free -h` read, straight from the source. `/proc/meminfo` isn't a real file on disk ‚Äî it's a virtual file the kernel generates every time you read it, which is why the numbers change between runs. `MemFree` matches the `free` column; `MemAvailable` matches `available`; `Buffers + Cached` is most of `buff/cache`. Every Linux observability tool is a parser over files like these."

### Exercise ‚Äî convert units

**SPEAK**
> "Let me prove the OS's GiB and the spec sheet's GB are different numbers for the same thing."

**TYPE**
```bash
kb=$(awk '/^MemTotal/{print $2}' /proc/meminfo)
python3 -c "kb = $kb; print(f'{kb / 1024**2:.2f} GiB, {kb * 1024 / 1000**3:.2f} GB')"
```

**OUTPUT**
```
62.16 GiB, 66.74 GB
```
*(Your GiB / GB values will differ by machine; the 7% gap between them will not.)*

**EXPLAIN**
> "`awk` pulls the MemTotal number (in kB) directly out of `/proc/meminfo`, and we pass it into Python as the variable `kb`. Same RAM, two conventions: 62.16 GiB uses 1024¬≤ (binary, what the OS reports). 66.74 GB uses 1000¬≥ (decimal, what marketing prints). 7% gap. It's why a 500 GB SSD shows as 465 GiB in a file manager."

## PART 3 ‚Äî SEEING DISKS

### What is a disk?

**SPEAK**
> "Below RAM in the pyramid: disk. Huge, slow, persistent. Not one flat slab ‚Äî sliced into partitions, each holding a filesystem, each mounted at a directory. `lsblk` prints the whole tree."

### Run the demo

**TYPE**
```bash
lsblk
```

**OUTPUT**
```
NAME        MAJ:MIN RM   SIZE RO TYPE MOUNTPOINTS
sda           8:0    0     5G  0 disk 
sdb           8:16   0    10G  0 disk 
sdc           8:32   0     1G  0 disk 
nvme0n1     259:0    0 931.5G  0 disk 
|-nvme0n1p1 259:1    0     1G  0 part 
|-nvme0n1p2 259:2    0     2G  0 part 
`-nvme0n1p3 259:3    0 928.5G  0 part 
```

**EXPLAIN**
> "Each row with `TYPE disk` is a physical device. `nvme0n1` is the main NVMe SSD on this machine, 931 GiB, cut into three partitions (the small `p1` and `p2` are boot and EFI; the big `p3` holds `/`). The `sda/sdb/sdc` rows are extra block devices visible to this container. On a typical laptop you'd see one disk; on WSL2 you'd see a single `sda` ‚Äî the virtual disk Microsoft gives the Linux VM."

**TYPE**
```bash
df -h | head -3
```

**OUTPUT**
```
Filesystem                         Size  Used Avail Use% Mounted on
overlay                            913G   40G  828G   5% /
tmpfs                               64M     0   64M   0% /dev
```

**EXPLAIN**
> "Different question than `lsblk`: for each *mounted* filesystem, how big and how full? The top row ‚Äî `/`, backed by `overlay` ‚Äî is the one that matters: 913 GiB total, 40 used, 828 free. `Avail` is the column you check when asking 'do I have room for this?' Same command, same columns, on macOS and Linux ‚Äî one of the few places they speak the same dialect."

### Exercise ‚Äî find your home partition

**TYPE**
```bash
df -h ~
```

**OUTPUT**
```
Filesystem                         Size  Used Avail Use% Mounted on
overlay                            913G   40G  828G   5% /
```

**EXPLAIN**
> "`~` expands to your home directory before `df` runs. The single row names the filesystem actually backing your home ‚Äî here the same one as `/`. Some Linux installs put `/home` on its own partition so the OS can be reinstalled without losing user data; you'd see a different row for that. When you save a file, this is where its bytes land."

## PART 4 ‚Äî SEEING A FILE AS BYTES

### What is a file at the byte level?

**SPEAK**
> "Lecture 1.1 said everything is bytes. Now you see it. `xxd` takes any file and prints the raw hex. Text, images, anything ‚Äî all bytes on your disk. Let's open a text file."

### Run the demo

**SPEAK**
> "Quick aside on `cd` before we get started. We `cd`'d into the 1.5 folder back at the intro and we're staying here ‚Äî that's where `sample.png` lives, and where we'll create `hello.txt` in a second. Your shell stays in whatever directory you last `cd`'d to: running `echo`, `xxd`, `python3` does *not* move you, only `cd` does. If you ever lose track of where you are, `pwd` prints the current path; `cd` with no argument snaps back to your home folder. And `/tmp` ‚Äî which you'll see in other tutorials ‚Äî is the Unix convention for scratch files when you don't have a dedicated workspace like this one."

**TYPE**
```bash
pwd
```

**OUTPUT**
```
/path/to/course/1.5
```

**EXPLAIN**
> "There we are, inside the lab folder. Any file we create from here on lives right here, next to the lab markdown."

**TYPE**
```bash
echo 'Hello!' > hello.txt
xxd hello.txt
```

**OUTPUT**
```
00000000: 4865 6c6c 6f21 0a                        Hello!.
```

**EXPLAIN**
> "The whole file on one line. Three columns: offset on the left (`00000000` = byte zero), hex bytes in the middle (grouped in pairs for readability ‚Äî really seven distinct bytes), and the ASCII column on the right. 48 is `H`, 65 is `e`, 6c is `l`, 6f is `o`, 21 is `!`. The trailing `.` isn't a period ‚Äî it's xxd's placeholder for byte `0a`, the newline `echo` added on the end. This file is literally those seven numbers."

**TYPE**
```bash
xxd -b hello.txt
```

**OUTPUT**
```
00000000: 01001000 01100101 01101100 01101100 01101111 00100001  Hello!
00000006: 00001010                                               .
```

**EXPLAIN**
> "Same bytes, now as literal bits. `01001000` is H ‚Äî eight flash memory cells on the SSD storing that pattern right now. This is what lecture 1.2 was pointing at: bits are physical state, bytes are groups of eight, files are sequences of bytes. The slide and the screen are the same thing."

### Exercise ‚Äî peek inside a PNG

**SPEAK**
> "Text files are a friendly case ‚Äî the bytes map one-to-one to characters you can read. Let's try a file where the bytes aren't text at all. The `sample.png` that ships with this lab is a 69-byte PNG ‚Äî a tiny 1√ó1 red pixel, a real openable image. We'll look at its first 16 bytes."

**TYPE**
```bash
xxd -l 16 sample.png
```

**OUTPUT**
```
00000000: 8950 4e47 0d0a 1a0a 0000 000d 4948 4452  .PNG........IHDR
```

**EXPLAIN**
> "Sixteen bytes of a PNG, every byte deliberate. Bytes 1-8 ‚Äî `89 50 4e 47 0d 0a 1a 0a` ‚Äî are the PNG signature, a fixed sequence every PNG file on Earth starts with. Byte 1 (`89`) has the high bit set to catch 7-bit-stripping file transfers. Bytes 2-4 spell `PNG` in ASCII ‚Äî visible in the gutter on the right. Bytes 5-6 (`0d 0a`) are CR-LF, to catch line-ending mangling. Byte 7 (`1a`) is a DOS control-Z. After the signature: `00 00 00 0d` is a length field (13 in decimal), and `49 48 44 52` spells `IHDR`. Every image viewer on Earth checks these bytes to recognize a PNG. Not the filename ‚Äî the bytes. You could rename this file `cat.jpg` and an image viewer would still open it as a PNG, because the bytes don't lie."

### Exercise ‚Äî identify a file by its magic number

**SPEAK**
> "Almost every file format announces itself in its first few bytes. Let's build the habit of reading that signature on a compiled binary ‚Äî the `ls` command ‚Äî and look at its first four bytes."

**TYPE**
```bash
xxd -l 4 /bin/ls
```

**OUTPUT**
```
00000000: 7f45 4c46                                .ELF
```

**EXPLAIN**
> "`7f 45 4c 46` is the ELF magic number ‚Äî Executable and Linkable Format, the signature every Linux executable starts with. When you type `ls` at the prompt, the kernel reads these four bytes first, sees ELF, and knows how to load the file into memory and run it. The ASCII gutter shows `.ELF` because bytes 2-4 are literally the letters E, L, F ‚Äî byte 1 (`7f`) is non-printable, hence the dot. Try at home with other files: a JPEG starts with `ff d8 ff`, a PDF with `25 50 44 46` (`%PDF`), a Python script with `23 21` (`#!`). The `file` command you may have used before? It works by reading these exact bytes and matching against a magic-number table. You now know the trick it uses."

## PART 5 ‚Äî SAME BYTES, NOW IN PYTHON

### What does it mean for a number to be in a base?

**SPEAK**
> "You saw `50 4e 47` spell PNG in the xxd output. Let's prove it in Python, one one-liner at a time."

### Run the demo

**TYPE**
```bash
python3 -c "print(hex(80))"
```

**OUTPUT**
```
0x50
```

**TYPE**
```bash
python3 -c "print(bin(80))"
```

**OUTPUT**
```
0b1010000
```

**TYPE**
```bash
python3 -c "print(ord('P'))"
python3 -c "print(chr(80))"
```

**OUTPUT**
```
80
P
```

**EXPLAIN**
> "One number, four views. `hex(80)` ‚Üí `'0x50'` (the `0x` is the literal-prefix Python uses in source). `bin(80)` ‚Üí `'0b1010000'`. `ord('P')` ‚Üí 80 (P's ASCII code). `chr(80)` ‚Üí `'P'`. Same 80 every time. `hex`/`bin` return strings; `ord`/`chr` bridge characters and numbers."

**TYPE**
```bash
python3 -c "print(int('50', 16))"
python3 -c "print(int('01010000', 2))"
```

**OUTPUT**
```
80
80
```

**EXPLAIN**
> "`int(string, base)` parses a string in any base 2‚Äì36. Feed it `'50'` with base 16, or `'01010000'` with base 2 ‚Äî both give back 80. This is the inverse of the `hex` and `bin` display functions. The round-trip is lossless."

**TYPE**
```bash
python3 -c "print(f'{15:02x}')"
python3 -c "print(f'{5:08b}')"
```

**OUTPUT**
```
0f
00000101
```

**EXPLAIN**
> "f-string format specs. `:02x` means 'hex, 2 digits minimum, zero-padded' ‚Äî matches xxd's per-byte format. `:08b` means 'binary, 8 digits, zero-padded' ‚Äî one full byte of bits. Without the padding, `hex(15)` and `bin(5)` would give you `0xf` and `0b101` ‚Äî too short to line up in a byte-oriented dump."

**TYPE**
```bash
python3 -c "print(bytes.fromhex('89504e47'))"
```

**OUTPUT**
```
b'\x89PNG'
```

**EXPLAIN**
> "This is the payoff. `89504e47` is the first four bytes of the PNG from Part 4. `bytes.fromhex()` turns that hex string into actual bytes. Python's repr shows printable bytes as characters (`P`, `N`, `G`) and non-printable ones as `\\xNN` escapes (`\\x89`). Same four bytes xxd showed us earlier ‚Äî two tools, same physical file content."

### Exercise ‚Äî your own initial

**TYPE**
```bash
python3 -c "c='J'; print(f'{c!r} -> dec={ord(c)}, hex={ord(c):02x}, bin={ord(c):08b}')"
```

**OUTPUT**
```
'J' -> dec=74, hex=4a, bin=01001010
```

**EXPLAIN**
> "`J` is decimal 74, hex 4a, binary 01001010. Try lowercase `j` and you'll get 106, 6a, 01101010. Compare the binary: exactly one bit different ‚Äî the 0x20 bit. That single bit is the ASCII case-toggle; XOR it to flip between upper and lower."
