## PART 1 — SEEING THE CPU

### What is a CPU introspection command?

**SPEAK**
> "Every program you run becomes instructions executed by the chip inside your computer. Before we can reason about what's happening, we need to be able to name the chip: how many cores it has, what model it is, how much fast on-chip memory called cache it owns. Linux exposes all of that through a pseudo-file at `/proc/cpuinfo` and a structured tree at `/sys/devices/system/cpu/`. `lscpu` reads both and prints an aggregated summary."

### Run the demo

**TYPE**
```bash
grep -m1 "model name" /proc/cpuinfo
```

**OUTPUT**
```
model name	: AMD Ryzen 7 5700G with Radeon Graphics
```
*(Your CPU will look different — Intel Core, AMD Ryzen, or ARM depending on your machine.)*

**EXPLAIN**
> "`/proc/cpuinfo` is a pseudo-file — it doesn't exist on disk. The kernel generates it on read. Every logical CPU gets a block of key-value lines, and `grep -m1` just grabs the first `model name`."

**TYPE**
```bash
nproc
```

**OUTPUT**
```
16
```

**EXPLAIN**
> "Sixteen logical CPUs. Each one is one independent stream of instructions the kernel can schedule work onto."

**TYPE**
```bash
lscpu
```

**OUTPUT**
```
Architecture:                         x86_64
CPU op-mode(s):                       32-bit, 64-bit
Address sizes:                        48 bits physical, 48 bits virtual
Byte Order:                           Little Endian
CPU(s):                               16
On-line CPU(s) list:                  0-15
Vendor ID:                            AuthenticAMD
Model name:                           AMD Ryzen 7 5700G with Radeon Graphics
CPU family:                           25
Model:                                80
Thread(s) per core:                   2
Core(s) per socket:                   8
Socket(s):                            1
Stepping:                             0
CPU max MHz:                          4673.0000
CPU min MHz:                          400.0000
BogoMIPS:                             7599.86
Flags:                                fpu vme de pse tsc msr pae mce cx8 apic sep mtrr pge mca ...
(output continues...)
L1d cache:                            256 KiB (8 instances)
L1i cache:                            256 KiB (8 instances)
L2 cache:                             4 MiB (8 instances)
L3 cache:                             16 MiB (1 instance)
```

**EXPLAIN**
> "`lscpu` did the work of reading `/proc/cpuinfo` and `/sys/`, deduplicating across all sixteen logical CPUs, and producing one report. Look at the bottom: four cache lines. L1 instruction cache, L1 data cache, L2, and L3. Per-core sizes are summed into the `(N instances)` total. L1d is 256 KiB total across eight cores, which means each core has 32 KiB."

### Exercise — L1d cache

**SPEAK**
> "Now you try. Extract just the L1d cache line from `lscpu`'s output."

**TYPE**
```bash
lscpu | grep "L1d cache"
```

**OUTPUT**
```
L1d cache:                            256 KiB (8 instances)
```

**EXPLAIN**
> "256 KiB total across eight cores. If you want the per-core size, the sysfs path is `/sys/devices/system/cpu/cpu0/cache/index0/size` — but `lscpu` is usually what you want."

## PART 2 — SEEING THE RAM

### What is memory introspection?

**SPEAK**
> "RAM is the pool of fast storage every running program lives in. Two questions come before anything interesting: how much RAM does this machine have, and how much is in use right now. Linux answers both through the pseudo-file `/proc/meminfo`. The `free` command reads that file and prints a table."

### Run the demo

**TYPE**
```bash
grep MemTotal /proc/meminfo
```

**OUTPUT**
```
MemTotal:       65178768 kB
```

**EXPLAIN**
> "Total physical RAM, reported in kilobytes. Every line in `/proc/meminfo` is labeled with its unit — always `kB`, even on a machine with terabytes of memory. Human-readable conversion is somebody else's problem."

**TYPE**
```bash
free -h
```

**OUTPUT**
```
               total        used        free      shared  buff/cache   available
Mem:            62Gi       6.7Gi        26Gi        49Mi        28Gi        54Gi
Swap:             0B          0B          0B
```

**EXPLAIN**
> "Six columns. `total` is the same number we just got from `/proc/meminfo`, just converted to gibibytes. `used` is RAM held by processes. `free` is literally unused. `buff/cache` is the filesystem cache — looks used, can be reclaimed instantly. `available` is the honest answer to 'how much could I allocate right now' — it adds `free` plus the reclaimable portion of `buff/cache`. On this machine, `free` says 26 gigs but `available` says 54."

### Exercise — available memory in MiB

**SPEAK**
> "Your turn. Print just the `available` column for the Mem row in MiB."

**TYPE**
```bash
free -m | awk '/Mem:/ {print $7 " MiB available"}'
```

**OUTPUT**
```
55905 MiB available
```

**EXPLAIN**
> "55905 mebibytes. That's the shape of one-liner monitoring scripts and health checks use in the wild."

## PART 3 — SEEING THE DISK

### What is a disk layout tool?

**SPEAK**
> "Your physical disk is usually divided into partitions. Two commands pull the curtain back: `df -h` shows the mounted filesystems with usage; `lsblk` shows the full tree of block devices below those mounts."

### Run the demo

**TYPE**
```bash
df -h /
```

**OUTPUT**
```
Filesystem      Size  Used Avail Use% Mounted on
overlay         913G   40G  828G   5% /
```

**EXPLAIN**
> "`df` reads the mount table. The root filesystem is mounted at `/`. The device name you see here will look different on a bare-metal install — something like `/dev/sda2` or `/dev/nvme0n1p2`. `overlay` appears inside containers and on WSL2. The `Use%` column is the first number you look at when a machine misbehaves."

**TYPE**
```bash
lsblk
```

**OUTPUT**
```
NAME        MAJ:MIN RM   SIZE RO TYPE MOUNTPOINTS
sda           8:0    0     5G  0 disk 
sdb           8:16   0    10G  0 disk 
sdc           8:32   0     1G  0 disk 
nvme0n1     259:0    0 931.5G  0 disk 
|-nvme0n1p1 259:1    0     1G  0 part 
|-nvme0n1p2 259:2    0     2G  0 part 
`-nvme0n1p3 259:3    0 928.5G  0 part
```

**EXPLAIN**
> "A tree. Top-level entries `sda`, `sdb`, `sdc`, and `nvme0n1` are physical disks. The indented entries under `nvme0n1` are partitions. `TYPE` distinguishes `disk` from `part`. `MOUNTPOINTS` is empty here — but on your own machine you'll see `/boot/efi`, `/`, and so on next to each mounted partition."

**TYPE**

```

**OUTPUT**
```
NAME        FSTYPE FSVER LABEL UUID FSAVAIL FSUSE% MOUNTPOINTS

sdc

|-nvme0n1p1

*(Captured inside a container, so FSTYPE values are blank. On a bare-metal Linux install or WSL2 you will see `ext4`, `xfs`, `btrfs`, `vfat`, or `swap` populated in the FSTYPE column.)*

**EXPLAIN**
> "On your own machine you'll see `ext4`, `xfs`, `btrfs`, `vfat`, or `swap` in the `FSTYPE` column. This is how installers figure out what's already on a disk before offering to partition it."

## PART 4 — SEEING FILES AS BYTES

### What is `xxd`?

**SPEAK**
> "Every file on your disk is a sequence of bytes. A byte is a number from zero to 255. When your image viewer opens a PNG it interprets those bytes as pixels; when `cat` prints a text file it interprets them as characters. The file itself doesn't know the difference. `xxd` lets us stop interpreting and just see the numbers."

### Detour — working directory

**SPEAK**
> "Quick detour before we dump any files, because relative paths matter."

**TYPE**
```bash
pwd
```

**OUTPUT**
```
/root/1.5
```

**EXPLAIN**
> "`pwd` prints where I am. That's where `xxd sample.png` will look when I don't give a full path."

**TYPE**
```bash
cd
pwd
```

**OUTPUT**
```
/root
```

**EXPLAIN**
> "`cd` with no argument is a shortcut to home. I'm now in `/root`."

**TYPE**
```bash
cd -
```

**OUTPUT**
```
/root/1.5
```

**EXPLAIN**
> "`cd -` jumps back to the previous directory. Bash prints the new path automatically. That's the pair that lets you bounce between two places without retyping paths."

### Run the demo

**TYPE**
```bash
echo "Hello, world!" > hello.txt
xxd hello.txt
```

**OUTPUT**
```
00000000: 4865 6c6c 6f2c 2077 6f72 6c64 210a       Hello, world!.
```

**EXPLAIN**
> "Three columns. Offset on the left — we're at byte zero. Sixteen bytes of hex in the middle, though only fourteen are shown because that's how long the file is. ASCII rendering on the right. `0x48` is the letter H. `0x0a` at the end is the newline `echo` added."

**TYPE**
```bash
xxd -l 16 sample.png
```

**OUTPUT**
```
00000000: 8950 4e47 0d0a 1a0a 0000 000d 4948 4452  .PNG........IHDR
```

**EXPLAIN**
> "Look at the right column. `.PNG` — that's the magic signature that marks this file as a PNG. Every PNG on every operating system starts with exactly these eight bytes: 89 50 4e 47 0d 0a 1a 0a. The file extension is a hint; this is the truth."

**TYPE**
```bash
xxd sample.png | head -6
```

**OUTPUT**
```
00000000: 8950 4e47 0d0a 1a0a 0000 000d 4948 4452  .PNG........IHDR
00000010: 0000 0010 0000 0010 0802 0000 0090 9168  ...............h
00000020: 3600 0000 1749 4441 5478 9c63 f8cf c040  6....IDATx.c...@
00000030: 1222 4df5 a886 510d 434a 0300 90f9 ff01  ."M...Q.CJ......
00000040: f9e1 fa78 0000 0000 4945 4e44 ae42 6082  ...x....IEND.B`.
```

**EXPLAIN**
> "Five rows for the whole file — that's the advantage of shipping a tiny 80-byte PNG. Look at the ASCII column: `IHDR` at the end of row one, `IDAT` in row two, `IEND` in row four. Those are the three PNG chunks the spec requires in that exact order, and we're looking at them directly. No libraries, no parsers."

### Exercise — uppercase vs lowercase

**SPEAK**
> "Your turn. Write a file containing just the letter A uppercase, dump it, then write a file with lowercase a and dump that. Explain the difference."

**TYPE**
```bash
printf 'A' > a.txt
xxd a.txt
printf 'a' > b.txt
xxd b.txt
rm a.txt b.txt
```

**OUTPUT**
```
00000000: 41                                       A
00000000: 61                                       a
```

**EXPLAIN**
> "`A` is `0x41`; `a` is `0x61`. That's bit 5 flipped — 32 in decimal. Every ASCII letter works the same way: uppercase-to-lowercase is one bit. You just saw a 50-year-old encoding decision with your own eyes."

## PART 5 — SEEING BYTES AS NUMBERS, LETTERS, AND BITS

### What is base conversion?

**SPEAK**
> "Humans count in base ten. Computers store numbers in base two. Hex is the shorthand programmers use for base two because one hex digit equals exactly four bits — a byte is always two hex digits. All three are the same number, written differently. Python has five built-in functions that convert between them."

### Run the demo

**TYPE**
```bash
python3.12 -c "print(hex(72))"
```

**OUTPUT**
```
0x48
```

**EXPLAIN**
> "`hex()` takes an integer and gives you a string with a `0x` prefix. 72 in base 16 is 48 — four times sixteen is 64, plus 8, equals 72."

**TYPE**
```bash
python3.12 -c "print(bin(72))"
```

**OUTPUT**
```
0b1001000
```

**EXPLAIN**
> "Same number in binary. Seven bits — one 64, one 8, zeros everywhere else."

**TYPE**
```bash
python3.12 -c "print(int('0x48', 16))"
```

**OUTPUT**
```
72
```

**EXPLAIN**
> "Round trip. `int(s, 16)` parses a string as base 16. We just took the string `0x48` and got back the integer 72."

**TYPE**
```bash
python3.12 -c "print(ord('H'))"
python3.12 -c "print(chr(72))"
```

**OUTPUT**
```
72
H
```

**EXPLAIN**
> "`ord` gives you the code point of a character; `chr` goes the other way. Capital H is the number 72. And that number is `0x48`. And `0x48` was the first byte of `hello.txt`. Four representations, one number."

**TYPE**
```bash
python3.12 -c "print(chr(int('48', 16)))"
```

**OUTPUT**
```
H
```

**EXPLAIN**
> "Composed two functions in one line. The hex string goes to an integer, the integer goes to a character. This is how you'd decode a file header by hand."

**TYPE**
```bash
python3.12 -c "print(''.join(chr(int(b,16)) for b in ['48','65','6c','6c','6f']))"
```

**OUTPUT**
```
Hello
```

**EXPLAIN**
> "Those five hex values came straight from the `xxd` dump of `hello.txt`. We just rebuilt the word `Hello` from its bytes, with stdlib Python, in one line. That's the whole payoff of the lab."

### Exercise — 255 in three bases

**SPEAK**
> "Your turn. Print the number 255 in decimal, hex, and binary on three separate lines using one Python invocation."

**TYPE**
```bash
python3.12 -c "n=255; print(n); print(hex(n)); print(bin(n))"
```

**OUTPUT**
```
255
0xff
0b11111111
```

**EXPLAIN**
> "255 is `0xff` is `0b11111111` — the largest value a single byte can hold. All three mean the exact same thing."

## Recording notes

- Capture on a real Linux install (bare metal or a VM), not inside a Docker container — `lsblk` inside containers shows the host's disks or nothing useful, and `df` reports `overlay` instead of a real filesystem. WSL2 is acceptable and gives realistic output.
- Ubuntu 22.04 is the baseline assumed in the script; other distros will work identically but may need `dnf install` instead of `apt` in Setup.
- Pause 2 seconds after each OUTPUT block before narrating, so the viewer has time to read.
- Terminal font size 18+ recommended; the `lscpu` Flags line is extremely long — truncate with `(output continues...)` in the recording if it wraps more than three lines.
- Part 4's detour about `pwd` / `cd` / `cd -` is a pacing-sensitive segment. Go slow; these commands are muscle memory for many students but new for others.
- The PNG dump in Part 4 is the "aha" moment. Pause longer there.
- Total captured elapsed time on a full clean run: ~19 minutes. Budget ~22 with pauses.
