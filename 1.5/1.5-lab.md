## Part 1 — The CPU

### What is a CPU?

Every program you write becomes a list of tiny numbered instructions. Something has to execute those instructions — that is the CPU. But "the CPU" is not a single thing. It is a stack of physical layers, and the numbers on the spec sheet describe different layers.

The outermost layer is the **socket** — the physical slot where the chip plugs into the motherboard. A laptop typically has one. A large server might have two. Each socket holds a chip that contains multiple **cores**: independent processing engines that can work on different tasks simultaneously. Each core can further run multiple **threads** using a technique called Simultaneous Multithreading (SMT). The CPU duplicates only the register file for each thread, so when one thread stalls waiting for data from memory, the other thread's instructions use the idle execution units.

The operating system sees each thread as a separate CPU. The formula is:

```
logical CPUs = sockets × cores per socket × threads per core
```

Between the CPU and main memory sit three layers of caches: **L1** (32–64 KB per core, shared by nothing, ~1–4 clock cycles to access), **L2** (256 KB–1 MB per core, ~4–15 cycles), and **L3** (shared across all cores on a socket, ~20–50 cycles). Retrieving a value from RAM takes ~100 cycles. Caches keep recently-used data nearby so the CPU rarely has to wait that long.

### Run the demo

Start by asking the kernel how many CPUs it sees in total.

```bash
nproc
```

**Output:**
```
16
```

That is the total number of logical CPUs. Now read the full topology to understand where that 16 comes from.

```bash
lscpu | grep -E "^CPU\(s\):|Thread|Core|Socket|Model name|Vendor"
```

**Output:**
```
CPU(s):                               16
Vendor ID:                            AuthenticAMD
Model name:                           AMD Ryzen 7 5700G with Radeon Graphics
Thread(s) per core:                   2
Core(s) per socket:                   8
Socket(s):                            1
```

*(Your CPU model and counts will differ — the structure is the same.)*

Check the formula: 1 socket × 8 cores × 2 threads = 16 logical CPUs. `nproc` and `CPU(s)` agree.

Now read the cache hierarchy.

```bash
lscpu -C
```

**Output:**
```
NAME ONE-SIZE ALL-SIZE WAYS TYPE        LEVEL  SETS PHY-LINE COHERENCY-SIZE
L1d       32K     256K    8 Data            1    64        1             64
L1i       32K     256K    8 Instruction     1    64        1             64
L2       512K       4M    8 Unified         2  1024        1             64
L3        16M      16M   16 Unified         3 16384        1             64
```

*(Your cache sizes will differ based on your CPU model.)*

The ONE-SIZE column is per core; ALL-SIZE is the total across all cores. L1d and L1i are separate (data and instructions have their own caches). L2 is unified (holds both). L3's ONE-SIZE and ALL-SIZE match because there is only one L3 instance shared by all cores — notice its COHERENCY-SIZE is still 64, meaning data still moves in 64-byte cache lines even at this level.

> **Why does the OS see 16 CPUs when the chip has only 8 physical cores?**
> <details><summary>Answer</summary>Each core runs two hardware threads simultaneously using Simultaneous Multithreading (SMT). The register file is duplicated so each thread has its own state, but the execution units — the parts that actually do arithmetic — are shared. When thread A stalls on a cache miss, thread B's instructions use the idle execution units. The OS treats each thread as an independent CPU, so 8 cores × 2 threads = 16 logical CPUs.</details>

## Part 2 — Memory

### What is RAM?

RAM (Random Access Memory) is the computer's working scratchpad. Unlike disk, it loses everything when power cuts. Unlike cache, it is large enough to hold an entire running program plus all its data. The kernel manages RAM as a shared resource — each running process gets a slice, the kernel reserves some for itself, and any leftover is used for **page cache**: recently-read file data stored in RAM so repeat reads bypass the disk entirely.

The misleading part: on a healthy Linux system, the "free" number reported by most tools is nearly zero. That is not a problem. Nearly all the apparently "used" memory is page cache, and the kernel will silently evict it the moment any process needs more RAM. The column that matters is **available** — it includes both genuinely free RAM and reclaimable cache. If `available` is low, you have a memory pressure problem. If only `free` is low, you do not.

Physically, each bit of RAM is one capacitor (charge = 1, no charge = 0) gated by one transistor. Capacitors leak — the memory controller must re-read and rewrite every row every ~64 milliseconds to keep the data intact. This constant refresh is why it is called **Dynamic** RAM.

### Run the demo

First, look at the raw counters the kernel maintains.

```bash
cat /proc/meminfo | head -10
```

**Output:**
```
MemTotal:       65178768 kB
MemFree:        28082812 kB
MemAvailable:   57248468 kB
Buffers:         1716728 kB
Cached:         25447016 kB
SwapCached:            0 kB
Active:          7662832 kB
Inactive:       24973704 kB
Active(anon):    5536308 kB
Inactive(anon):     1320 kB
```

*(Your values will differ — structure is the same.)*

All values are in kibibytes (1 kB = 1024 bytes). `MemFree` is raw unoccupied RAM. `MemAvailable` (57 GB) is much higher than `MemFree` (28 GB) because the kernel knows it can reclaim most of `Cached` (25 GB of disk data cached in RAM) if needed.

Now see the same data in human-readable form.

```bash
free -h
```

**Output:**
```
               total        used        free      shared  buff/cache   available
Mem:            62Gi       6.8Gi        26Gi        48Mi        28Gi        54Gi
Swap:             0B          0B          0B
```

*(Your values will differ.)*

The `buff/cache` column combines `Buffers` and `Cached` from `/proc/meminfo`. The `available` column is your real headroom — not `free`. Swap shows 0 used, which is normal when RAM is not under pressure.

Now look at the physical memory address ranges.

```bash
lsmem
```

**Output:**
```
RANGE                                 SIZE  STATE REMOVABLE BLOCK
0x0000000000000000-0x000000107fffffff  66G online       yes  0-32

Memory block size:         2G
Total online memory:      66G
Total offline memory:      0B
```

*(Your range end address and size will differ.)*

This shows the physical address range your RAM occupies. Servers with hardware memory hot-plug can take individual 2 GB blocks offline without rebooting; on a typical laptop or VM all memory appears as one contiguous online range.

> **Why can `available` memory be larger than `free` memory?**
> <details><summary>Answer</summary>The kernel uses idle RAM as a disk cache — recently read files stay in memory so repeated reads are instant. This cached memory shows as "used" but the kernel can evict it immediately when a process requests more RAM. `available` accounts for both genuinely free RAM and this reclaimable cache. `free` only counts RAM that is completely unoccupied right now.</details>

## Part 3 — Files Are Just Bytes

### What is a block device?

A disk — SSD or HDD — is a large sequence of fixed-size locations called **sectors** (usually 4096 bytes each). The kernel exposes each disk as a **block device**: a file in `/dev/` you can read in fixed chunks. `lsblk` shows these devices as a tree — the disk itself on the left, its partitions indented beneath it.

Every file on disk is ultimately a sequence of bytes, each a number from 0 to 255. The filename, the extension, the icon your OS shows — none of that is inside the file. It is metadata. The content of the file is just bytes, and the meaning of those bytes depends entirely on how you interpret them.

`xxd` makes this concrete. It prints a file's raw bytes in **hexadecimal** alongside an ASCII interpretation. Hexadecimal (base 16) is used because exactly two hex digits represent one byte, keeping the display compact. The **magic number** — the first few bytes of a file — is how the OS actually identifies file types. A Linux binary starts with `7f 45 4c 46` (`.ELF`); a PNG starts with `89 50 4e 47` (`.PNG`). Extensions are a hint; magic bytes are the truth.

### Run the demo

First, see your disk layout.

```bash
lsblk
```

**Output:**
```
NAME        MAJ:MIN RM   SIZE RO TYPE MOUNTPOINTS
sda           8:0    0     5G  0 disk 
sdb           8:16   0    10G  0 disk 
sdc           8:32   0     1G  0 disk 
nvme0n1     259:0    0 931.5G  0 disk 
|-nvme0n1p1 259:1    0     1G  0 part 
|-nvme0n1p2 259:2    0     2G  0 part 
`-nvme0n1p3 259:3    0 928.5G  0 part
```

*(Your disk names, sizes, and partition count will differ. Your Ubuntu VM will show MOUNTPOINTS like `/`, `/boot`, and `/boot/efi`.)*

The tree shows `nvme0n1` is the full NVMe SSD (931.5 GB). The three indented entries are partitions carved out of it: a 1 GB boot partition, a 2 GB partition, and the main 928.5 GB partition. `MAJ:MIN` is the major:minor device number the kernel uses internally to route I/O to the right driver.

Now create a text file and look at its raw bytes.

```bash
echo "Hello World" > test.txt
xxd test.txt
```

**Output:**
```
00000000: 4865 6c6c 6f20 576f 726c 640a            Hello World.
```

Each line has three columns: the **offset** (byte position of the first byte on this line, in hex), the **hex bytes** (16 bytes per line, grouped in pairs), and the **ASCII sidebar** (the same bytes decoded as text; non-printable bytes show as `.`).

The offset on this line is `00000000` — byte 0. If there were more bytes, the next line would start at offset `00000010`, which in hex means byte 16 (not byte 10). Verify:

```bash
python3 -c "print(0x10)"
```

**Output:**
```
16
```

Now view the same file one byte at a time to make counting easier.

```bash
xxd -g 1 test.txt
```

**Output:**
```
00000000: 48 65 6c 6c 6f 20 57 6f 72 6c 64 0a              Hello World.
```

Each hex pair is now a single byte. Count them: `48 65 6c 6c 6f 20 57 6f 72 6c 64 0a` — that is 12 bytes. 11 visible characters ("Hello World") plus one invisible `0a` (newline, ASCII 10) at the end that `echo` adds automatically.

Now look at the magic number of a real binary.

```bash
xxd -l 16 /bin/ls
```

**Output (Linux):**
```
00000000: 7f45 4c46 0201 0100 0000 0000 0000 0000  .ELF............
```

The first four bytes are `7f 45 4c 46`. In the ASCII column: `.ELF`. This is the ELF magic number — every compiled Linux binary starts with these exact four bytes. The OS checks them before attempting to execute the file.

> **On macOS:**
> ```bash
> xxd -l 16 /bin/ls
> ```
> ```
> 00000000: cafe babe 0000 0002 0100 0007 0000 0003  ................
> ```
> macOS binaries use Mach-O Universal Binary format. Magic: `ca fe ba be` ("cafébabe" — a deliberate joke by Java engineers who later reused this same magic for Java `.class` files). All bytes past the first four are architecture metadata, hence the dots in the ASCII column.

> **Why does `xxd` offset `00000010` mean byte 16, not byte 10?**
> <details><summary>Answer</summary>The offset column is in hexadecimal. In hex, the digits go 0, 1, 2, … 9, A, B, C, D, E, F, and then 10 — which equals 16 in decimal (1×16 + 0×1). So `00000010` means the 17th byte (byte index 16, counting from 0). You can always verify: `python3 -c "print(0x10)"` prints 16.</details>

## Part 4 — Number Costumes

### What is a number base?

The number 65 is not inherently decimal. It is a quantity — the size of a set of 65 things. Decimal, binary, hexadecimal, and ASCII are all different **costumes** the same quantity can wear:

- **Decimal** (base 10): `65` — what you learned in school, using digits 0–9
- **Binary** (base 2): `01000001` — the actual switch positions in hardware, using only 0 and 1
- **Hexadecimal** (base 16): `41` — compact notation where each character represents 4 bits
- **ASCII**: `'A'` — the agreement that the integer 65 means the letter A

Every number you saw in Parts 1–3 — the cache sizes, the memory counters in kB, the hex bytes in `xxd` — is just an integer in a particular costume.

Python provides built-in functions for every conversion, no imports required.

### Run the demo

See all four representations of the same number at once.

```bash
python3 -c "n=65; print('dec:', n, '| bin:', format(n,'08b'), '| hex:', format(n,'02x'), '| char:', chr(n))"
```

**Output:**
```
dec: 65 | bin: 01000001 | hex: 41 | char: A
```

`format(n, '08b')` formats as binary with 8-digit zero-padding. `format(n, '02x')` formats as lowercase hex with 2-digit zero-padding. `chr(65)` is the ASCII character for integer 65.

Now go the other direction — from characters to numbers.

```bash
python3 -c "
for ch in 'Hi!':
    print(f'{ch!r:4}  dec={ord(ch):3}  hex={ord(ch):02x}  bin={ord(ch):08b}')
"
```

**Output:**
```
'H'   dec= 72  hex=48  bin=01001000
'i'   dec=105  hex=69  bin=01101001
'!'   dec= 33  hex=21  bin=00100001
```

`ord()` takes a character and returns its integer code. Every character is just a number wearing a costume. Notice `'H'` has hex `48` — the same byte `xxd` printed for 'H' in "Hello World".

Now parse numbers written in binary and hex back to decimal.

```bash
python3 -c "print(int('01000001', 2), int('41', 16), int('0b01000001', 0))"
```

**Output:**
```
65 65 65
```

`int('01000001', 2)` parses the string as base 2. `int('41', 16)` parses as base 16. `int('0b01000001', 0)` auto-detects the base from the `0b` prefix. All three give 65.

Now look at how Python stores text as bytes.

```bash
python3 -c "b=b'Hi!'; print(b, b[0], b.hex(), list(b))"
```

**Output:**
```
b'Hi!' 72 486921 [72, 105, 33]
```

`b'Hi!'` is a bytes object — a sequence of integers 0–255. Indexing it (`b[0]`) returns the integer 72, not the character `'H'`. `.hex()` gives the entire sequence as a hex string with no spaces. `list(b)` shows each byte as a decimal integer.

Connect it back to `xxd`. Create a file and read it both ways.

```bash
echo -n "Hi!" > hi.txt
xxd hi.txt
python3 -c "print([format(b,'02x') for b in open('hi.txt','rb').read()])"
```

**Output:**
```
00000000: 4869 21                                  Hi!
['48', '69', '21']
```

`xxd` and Python's bytes object produce identical hex values: `48`, `69`, `21`. The file is the same bytes — `xxd` just formats them for display. Python's `open('hi.txt', 'rb')` opens in binary mode, giving raw integers. `'rb'` means read-binary; without the `b`, Python would decode the bytes as UTF-8 text and you would lose the raw integer view.

> **`ord('0')` returns 48, not 0. Why?**
> <details><summary>Answer</summary>ASCII codes and numeric values are different things. The digit character `'0'` is assigned code 48 in the ASCII table — it is just an arbitrary choice made in 1963 to put the digit characters starting at 48. The integer 0 and the character `'0'` are not the same thing. To get the numeric value of a digit character, use `int('0')` (which is 0) or `ord('0') - ord('0')` (which is also 0). `ord('0')` gives you the ASCII position of the character, which happens to be 48.</details>

### Exercise

Without looking anything up, predict the hex code for uppercase `'B'`. (Hint: `'A'` is `0x41`.) Then verify with Python, and check what happens when you flip bit 5 of `'B'`'s code to convert it to lowercase:

```bash
python3 -c "print(hex(ord('B')))"
python3 -c "print(chr(ord('B') ^ 0x20))"
```

<details><summary>Solution</summary>

`'B'` is one step past `'A'` in the ASCII table, so its code is 66 = `0x42`.

```bash
python3 -c "print(hex(ord('B')))"
```
Output: `0x42`

Flipping bit 5 (XOR with `0x20`):
```bash
python3 -c "print(chr(ord('B') ^ 0x20))"
```
Output: `b`

`0x42` XOR `0x20` = `0x62` = 98 = `'b'`. The difference between uppercase and lowercase ASCII codes is always exactly 32 (`0x20`), which corresponds to bit 5. Flipping that one bit converts any letter's case — this was a deliberate design decision in the original ASCII standard.

</details>

## Putting it together

| Tool | Problem it solves | Key idea | Where it reads from |
|------|-------------------|----------|---------------------|
| `lscpu` | How many CPUs and caches do I have? | Aggregates per-core kernel data into a topology table | `/proc/cpuinfo`, `/sys/devices/system/cpu/` |
| `free -h` | How much RAM is actually usable? | `available` includes reclaimable cache; `free` does not | `/proc/meminfo` |
| `lsmem` | What are the physical memory address ranges? | RAM occupies specific physical address regions; servers can hot-remove blocks | `/sys/devices/system/memory/` |
| `lsblk` | How is my disk divided? | Block devices, partitions, and mount points form a tree | `/sys/block/` (sysfs) |
| `xxd` | What bytes are inside a file? | Every file is integers 0–255; hex is the standard notation for individual bytes | The file itself (raw I/O) |
| `python3 -c` | How do I convert between number bases? | Decimal, binary, hex, and ASCII are the same integer in different costumes | Python builtins: `bin()`, `hex()`, `ord()`, `chr()`, `format()` |

Every output you read in Parts 1–3 — cache sizes in kB, memory in GiB, hex bytes in `xxd` — is an integer. Part 4 is the vocabulary for reading those integers in any costume.
