## Part 1 — Seeing the CPU

### What is the CPU?

The CPU is the chip that runs your programs. Its data sheet has three numbers that matter for this lab: the **architecture** (the instruction set — either `x86_64` on most Intel/AMD chips, or `arm64` / `aarch64` on Apple Silicon and Windows-on-ARM), the **core count** (how many instructions can be in flight at once), and the **cache hierarchy** (tiny fast memory sitting next to the cores).

Lecture 1.4 drew the pyramid: registers → L1 → L2 → L3 → RAM → disk. Each tier is bigger than the one above, and each is slower. L1 is usually 32–128 KiB *per core* and takes a handful of clock cycles to read. L3 is typically a few MiB and shared across the whole chip, taking ten times longer. Your OS knows every one of these numbers and will print them on request.

### Run the demo

**Read the chip model and core counts.**

**🍎 macOS**
```bash
sysctl -n machdep.cpu.brand_string
sysctl -n hw.physicalcpu
sysctl -n hw.logicalcpu
```
**Output:**
```
Apple M1 Pro
8
8
```
*(Your chip name and numbers will differ — the shape is the same.)*

**🐧 Linux / WSL**
```bash
lscpu | grep -E 'Architecture|Model name|^CPU\(s\):'
nproc
```
**Output:**
```
Architecture:                         x86_64
CPU(s):                               16
Model name:                           AMD Ryzen 7 5700G with Radeon Graphics
16
```
*(Your chip and counts will differ.)*

`lscpu` and `sysctl` both tell you what silicon is underneath. On most x86 laptops, *logical cores* is double *physical cores* — that's Hyper-Threading (Intel) or SMT (AMD), one physical core pretending to be two. On Apple Silicon there's no SMT, so the two match.

> **Why does the kernel have to expose this information at all? Why can't your programs just ask the CPU directly?**
> <details><summary>Answer</summary>
> User programs run in an unprivileged mode and can't talk to the hardware directly — that's the OS's job. The kernel reads this information at boot (from CPU instructions like <code>CPUID</code> on x86 or from ACPI/DeviceTree tables) and caches it in kernel memory. Commands like <code>lscpu</code> and <code>sysctl</code> are thin readers over that kernel-maintained cache. Without this layer, every program would need CPU-specific knowledge and privileged access to detect its own environment.
> </details>

## Part 2 — Seeing RAM

### What is RAM?

RAM is the working scratchpad. Variables, open files, and running programs all live there while they're active. The OS owns it and hands pieces out to processes on request.

Students come in thinking of RAM as one number — "I have 16 GB." The real picture is layered. Some is the kernel itself (wired memory, never reclaimed). Some is active — being read or written right now. Some is inactive — in use but cold, ready to be reclaimed if someone needs it. Some is used as a disk cache: the last file you read is still sitting in RAM, so if you read it again it's instant. Free memory, literally untouched, is almost always tiny on a used machine.

### Run the demo

**How much RAM is installed?**

**🍎 macOS**
```bash
sysctl -n hw.memsize
```
**Output:**
```
34359738368
```
*(Bytes. Divide by 1024³ = 1073741824 to get GiB.)*

**🐧 Linux / WSL**
```bash
grep MemTotal /proc/meminfo
```
**Output:**
```
MemTotal:       65178768 kB
```
*(kB = kibibytes. Divide by 1024² = 1048576 to get GiB.)*

Both expose the same fact — total installed RAM — in different units. Your machine has a specific number; it should match (roughly) the spec sheet.

**Look at the live breakdown.**

**🍎 macOS**
```bash
vm_stat
```
**Output (first 7 lines):**
```
Mach Virtual Memory Statistics: (page size of 16384 bytes)
Pages free:                               78745.
Pages active:                            507575.
Pages inactive:                          444596.
Pages speculative:                        82603.
Pages throttled:                              0.
Pages wired down:                        152544.
```
*(`vm_stat` output is in **pages**, not bytes. Multiply any row by 16384 for bytes.)*

**🐧 Linux / WSL**
```bash
free -h
```
**Output:**
```
               total        used        free      shared  buff/cache   available
Mem:            62Gi       6.6Gi        26Gi        48Mi        28Gi        54Gi
Swap:             0B          0B          0B
```

Both tools distinguish "used" from "reclaimable cache." On Linux `free` is tiny but `available` is large — that's because `buff/cache` holds disk blocks the kernel will give up on demand. On macOS `vm_stat`'s `Pages inactive` is the similar concept: in use now, but cold and droppable.

> **Why does the kernel keep so much RAM as disk cache instead of leaving it "free"?**
> <details><summary>Answer</summary>
> Free RAM is wasted RAM. If the kernel has empty pages, it uses them to cache recently-read disk blocks, so if you re-read the same file it comes from RAM (nanoseconds) instead of the disk (milliseconds). The moment an application actually needs memory, the kernel drops the cheapest cache pages and hands them over — no performance cost. That's why <code>MemFree</code> or <code>Pages free</code> is always small on a machine that's been running a while, even with plenty of "available" capacity.
> </details>

## Part 3 — Seeing Disks

### What is a disk?

A disk is the slow, huge, persistent layer at the bottom of the memory hierarchy. Unlike RAM, it keeps its data when the power is off. Unlike RAM, it's almost never used as one unbroken space — it's sliced into **partitions**, each holding a **filesystem**, each **mounted** at some directory in the tree (`/`, `/boot`, `/home`, `/Volumes/SomeDrive`).

The topology a tool like `lsblk` or `diskutil list` prints has three levels: physical device → partitions or volumes → mount points. The command `df -h` answers a slightly different question — for each *mount point*, how big is it and how full?

### Run the demo

**Show the physical disks and their partitions.**

**🍎 macOS**
```bash
diskutil list
```
**Output (first 20 lines):**
```
/dev/disk0 (internal, physical):
   #:                       TYPE NAME                    SIZE       IDENTIFIER
   0:      GUID_partition_scheme                        *500.3 GB   disk0
   1:             Apple_APFS_ISC Container disk1         524.3 MB   disk0s1
   2:                 Apple_APFS Container disk3         494.4 GB   disk0s2
   3:        Apple_APFS_Recovery Container disk2         5.4 GB     disk0s3

/dev/disk3 (synthesized):
   #:                       TYPE NAME                    SIZE       IDENTIFIER
   0:      APFS Container Scheme -                      +494.4 GB   disk3
                                 Physical Store disk0s2
   1:                APFS Volume Macintosh HD - Data     218.0 GB   disk3s1
   2:                APFS Volume Macintosh HD            11.3 GB    disk3s3
   3:              APFS Snapshot com.apple.os.update-... 11.3 GB    disk3s3s1
   4:                APFS Volume Preboot                 8.1 GB     disk3s4
   5:                APFS Volume Recovery                2.0 GB     disk3s5
   6:                APFS Volume VM                      4.3 GB     disk3s6
```

`disk0` is the physical SSD; it's cut into three APFS containers. `disk3` is *not* a second physical disk — it's a "synthesized" view of the big APFS container on `disk0`. That container holds all the APFS volumes (Macintosh HD, Data, Preboot, Recovery, VM), all sharing free space dynamically.

**🐧 Linux / WSL**
```bash
lsblk
```
**Output:**
```
NAME        MAJ:MIN RM   SIZE RO TYPE MOUNTPOINTS
sda           8:0    0     5G  0 disk 
sdb           8:16   0    10G  0 disk 
sdc           8:32   0     1G  0 disk 
nvme0n1     259:0    0 931.5G  0 disk 
|-nvme0n1p1 259:1    0     1G  0 part 
|-nvme0n1p2 259:2    0     2G  0 part 
`-nvme0n1p3 259:3    0 928.5G  0 part 
```
*(Your tree will look different — `nvme0n1` is typical on a laptop SSD; WSL2 usually shows a single `sda` virtual disk. Exact sizes depend on your hardware.)*

Each `disk` row is a physical device. `part` rows nested underneath are its partitions. On this machine the main SSD (`nvme0n1`) is cut into three partitions — partition 3 is the big data partition that holds `/`.

**How full is each mount?**

**🍎 macOS**
```bash
df -h
```
**Output (first 3 lines):**
```
Filesystem        Size    Used   Avail Capacity iused ifree %iused  Mounted on
/dev/disk3s3s1   460Gi    10Gi   233Gi     5%    426k  2.4G    0%   /
devfs            202Ki   202Ki     0Bi   100%     698     0  100%   /dev
```

**🐧 Linux / WSL**
```bash
df -h
```
**Output (first 3 lines):**
```
Filesystem                         Size  Used Avail Use% Mounted on
overlay                            913G   40G  828G   5% /
tmpfs                               64M     0   64M   0% /dev
```

`df -h` works identically on both OSes. The top row on both shows the filesystem mounted at `/` — that's the root of your tree, where everything you navigate with `cd /` starts. Its Size and Avail columns are the answer to "do I have room to download this?"

> **Why does `diskutil list` show a "synthesized" `/dev/disk3` on macOS when you only have one physical SSD?**
> <details><summary>Answer</summary>
> APFS (Apple's filesystem) separates the <em>container</em> from the <em>volumes</em> inside it. The container is one contiguous chunk of the physical SSD. The volumes inside the container share free space — you don't have to decide ahead of time that Macintosh HD gets 200 GB and Data gets 250 GB; they grow and shrink in the same pool. <code>diskutil list</code> shows the physical slice (<code>disk0s2</code>) and then "synthesizes" a second device node (<code>disk3</code>) to represent the container as a browsable thing with its own volumes underneath. It's one real disk; the second <code>/dev/diskN</code> is a logical view.
> </details>

## Part 4 — Seeing a File as Bytes

### What is a file at the byte level?

Lecture 1.1 claimed that everything in a computer is bytes — text, images, programs, all of them. This Part is where you see it. `xxd` reads any file and prints its raw bytes as a three-column hex dump: offset on the left, the bytes in hex in the middle, and those same bytes shown as characters on the right when they happen to be printable ASCII.

Open a text file with `xxd` and you can match each hex pair to a letter. Open a PNG image with `xxd` and you see the file starts with eight specific bytes — the PNG "magic number" — that every image viewer checks to identify the format.

### Run the demo

**Create a tiny text file and dump it.**

**🍎 macOS** and **🐧 Linux / WSL** (identical)
```bash
echo 'Hello!' > hello.txt
xxd hello.txt
```
**Output:**
```
00000000: 4865 6c6c 6f21 0a                        Hello!.
```

Left column: offset in the file (0, in hex). Middle: the 7 bytes of content as hex pairs — `48 65 6c 6c 6f 21 0a`. Right: those bytes shown as ASCII characters — `Hello!` followed by `.` because byte `0a` (newline) isn't a printable character. The file is literally those seven numbers and nothing else.

**See the same bytes as binary bits.**

**🍎 macOS** and **🐧 Linux / WSL** (identical)
```bash
xxd -b hello.txt
```
**Output:**
```
00000000: 01001000 01100101 01101100 01101100 01101111 00100001  Hello!
00000006: 00001010                                               .
```

`-b` swaps hex for binary. Now you see the actual bits for each letter. `H` is `01001000`; `e` is `01100101`. Lecture 1.2 said gates combine into flip-flops that each hold one of these bits — each of those eight zeros and ones in `01001000` is a physical state somewhere on your SSD.

**Dump a PNG and find the magic number.**

**🍎 macOS** and **🐧 Linux / WSL** (identical)
```bash
curl -sO https://www.libpng.org/pub/png/img_png/pnglogo-blk.png
xxd -l 16 pnglogo-blk.png
```
**Output:**
```
00000000: 8950 4e47 0d0a 1a0a 0000 000d 4948 4452  .PNG........IHDR
```

`-l 16` limits the dump to the first 16 bytes. The first 8 bytes — `89 50 4e 47 0d 0a 1a 0a` — are the PNG signature. Every PNG on Earth starts with exactly these bytes. In the ASCII column you can see `PNG` in the middle because bytes 2–4 (`50 4e 47`) are the ASCII codes for `P`, `N`, `G`. The next 4 bytes (`00 00 00 0d`) are a length field, and `49 48 44 52` (`IHDR` in ASCII) is the tag of the first chunk.

Windows, Safari, Preview, Chrome — they all look at the first 8 bytes of a file to decide whether it's a PNG. Not the filename. The bytes.

> **Why does `xxd` show `.` for byte `0a` (newline) in the ASCII column, but the file really does contain that byte?**
> <details><summary>Answer</summary>
> The ASCII column is only for <em>printable</em> characters — code points 0x20 (space) through 0x7E (tilde). Bytes below 0x20 are control characters: newline (0x0a), tab (0x09), null (0x00), carriage return (0x0d). They're real bytes and affect your terminal when printed, but they don't have a visible glyph, so <code>xxd</code> shows <code>.</code> as a placeholder. This keeps the ASCII column a fixed width regardless of content.
> </details>

## Part 5 — Same Bytes, Now in Python

### What does it mean for a number to be "in a base"?

When you write the number eighty, you can spell it any of four ways:

- `80` in decimal (base 10)
- `0x50` in hex (base 16)
- `0b1010000` in binary (base 2)
- `P` via the ASCII table

These are all the *same number*. Only the label is different. Python has built-ins for every conversion between these views, and each conversion is one function call.

You just saw in `xxd` that a PNG starts with `89 50 4e 47`, and the ASCII column showed `PNG`. This Part is where you prove that `50 4e 47` really *is* `PNG`. Once you see `ord('P')` return 80, and `hex(80)` return `'0x50'`, the hex in the dump and the letter in the gutter become the same thing in three costumes.

### Run the demo

**One number, four views.**

**🍎 macOS**
```bash
python3.12 -c "print(hex(80))"
python3.12 -c "print(bin(80))"
python3.12 -c "print(ord('P'))"
python3.12 -c "print(chr(80))"
```
**Output:**
```
0x50
0b1010000
80
P
```

**🐧 Linux / WSL**
```bash
python3 -c "print(hex(80))"
python3 -c "print(bin(80))"
python3 -c "print(ord('P'))"
python3 -c "print(chr(80))"
```
**Output:**
```
0x50
0b1010000
80
P
```

`hex()` and `bin()` take an integer and return a string with a `0x` or `0b` prefix — just like you'd write the literal in Python source code. `ord()` takes a single-character string and returns the integer code point. `chr()` is its inverse. Same 80 in all four calls.

**Round-trip from string back to number.**

**🍎 macOS**
```bash
python3.12 -c "print(int('50', 16))"
python3.12 -c "print(int('01010000', 2))"
```
**Output:**
```
80
80
```

**🐧 Linux / WSL**
```bash
python3 -c "print(int('50', 16))"
python3 -c "print(int('01010000', 2))"
```
**Output:**
```
80
80
```

`int(s, base)` parses a string in any base 2–36 into its integer value. Both give 80 because both strings represent the same number in different bases.

**Format bytes the way xxd does.**

**🍎 macOS**
```bash
python3.12 -c "print(f'{15:02x}')"
python3.12 -c "print(f'{5:08b}')"
```
**Output:**
```
0f
00000101
```

**🐧 Linux / WSL**
```bash
python3 -c "print(f'{15:02x}')"
python3 -c "print(f'{5:08b}')"
```
**Output:**
```
0f
00000101
```

The f-string format spec `:02x` means "hex, minimum 2 digits, zero-padded." That's why `xxd` shows `0f`, not `f`, for byte value 15 — it always uses two hex digits per byte. `:08b` does the same for binary, with 8 digits = one full byte of bits.

**Decode the PNG magic number you saw in Part 4.**

**🍎 macOS**
```bash
python3.12 -c "print(bytes.fromhex('89504e47'))"
```
**Output:**
```
b'\x89PNG'
```

**🐧 Linux / WSL**
```bash
python3 -c "print(bytes.fromhex('89504e47'))"
```
**Output:**
```
b'\x89PNG'
```

`bytes.fromhex()` takes a hex string (no prefix, no spaces) and returns the raw bytes. The output `b'\x89PNG'` means: byte 0x89 (not printable, shown as escape), then `P`, `N`, `G`. This is exactly the first 4 bytes of every PNG on your disk — including the one you dumped in Part 4.

> **Why does Python print `b'\x89PNG'` instead of all four bytes as escape codes, or all four as characters?**
> <details><summary>Answer</summary>
> Python's bytes-repr format is a compromise. Bytes that correspond to printable ASCII characters (0x20–0x7E) are shown as the character itself — that's why you see <code>PNG</code>. Bytes outside that range are shown as <code>\xNN</code> hex escapes — that's why <code>0x89</code> appears as <code>\x89</code>. It's the shortest form that still round-trips: if you copy <code>b'\x89PNG'</code> back into Python, you get the same 4-byte sequence out.
> </details>

### Exercise

Convert your own first initial to decimal, hex, and binary in a single Python one-liner.

**🍎 macOS**
```bash
python3.12 -c "c='J'; print(f'{c!r} -> dec={ord(c)}, hex={ord(c):02x}, bin={ord(c):08b}')"
```
**Output:**
```
'J' -> dec=74, hex=4a, bin=01001010
```

**🐧 Linux / WSL**
```bash
python3 -c "c='J'; print(f'{c!r} -> dec={ord(c)}, hex={ord(c):02x}, bin={ord(c):08b}')"
```
**Output:**
```
'J' -> dec=74, hex=4a, bin=01001010
```

<details><summary>Solution</summary>

For the letter `J`: 74 decimal, `4a` hex, `01001010` binary. `J` is the 10th letter, which is code 65 + 9 = 74. Replace `'J'` with your own initial and run. If you try both an uppercase and a lowercase letter, you'll notice the hex digits are the same except one bit: `A` is `0x41` (binary `01000001`) and `a` is `0x61` (binary `01100001`). The difference is exactly `0x20` (bit 6). That single bit is the case toggle in ASCII.

</details>

## Putting it together

| Concept | What it lets you see | Key tool (🍎 macOS / 🐧 Linux) | Where the data comes from |
|---|---|---|---|
| CPU and caches | Chip name, cores, cache sizes | `sysctl` / `lscpu` | Kernel data from CPUID / ACPI / DeviceTree |
| RAM | Total and live memory usage | `vm_stat` + `sysctl hw.memsize` / `free -h` + `/proc/meminfo` | Kernel memory subsystem |
| Disks | Physical devices, partitions, mounts | `diskutil list` / `lsblk` | Kernel block-device registry |
| Bytes in a file | Hex + ASCII columns, or bits | `xxd` (identical on both) | The file itself |
| Bytes in code | Dec/hex/bin/char conversions | Python built-ins (identical on both) | Python standard library |

The first three rows are different questions you ask the OS about the hardware. The last two rows are different views of the same thing: every file is a sequence of bytes, every byte is a number, every number has four equivalent written forms. When a later lab reads a network socket, parses a binary file, or debugs a memory dump, it will still be the same translations you just did here with a single letter.
