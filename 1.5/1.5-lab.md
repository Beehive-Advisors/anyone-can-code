## Part 1 — Reading CPU Facts

### What is machine architecture?

Your program does not run on a generic box called “a computer.” It runs on a processor that understands a specific instruction set. That instruction set is the machine's vocabulary. If the program and the processor do not agree on that vocabulary, the program cannot run.

That is why names like `arm64` and `x86_64` matter. They tell you which family of machine instructions the processor understands. They also help explain why the same binary may work on one machine and fail on another.

The processor also has structure beyond its architecture name. It has cores, and it has caches of different sizes because the CPU is much faster than main memory. Those caches are there to keep useful data physically closer to the core.

In this part, you are not benchmarking the processor. You are asking the operating system what it already knows about the processor in your machine.

### Run the demo

Start by asking for the broadest identity of the machine: its architecture.

```bash
echo "Architecture: $(uname -m)"
```

**Output:**

```text
Architecture: arm64
```

This tells you the instruction-set family the operating system is running on. On this machine, the answer is `arm64`.

Now ask for the processor model name.

```bash
echo "CPU brand: $(sysctl -n machdep.cpu.brand_string)"
```

**Output:**

```text
CPU brand: Apple M4 Pro
```

This is a human-readable model string. It helps you orient yourself, but it is not the whole story.

Now ask for core counts and cache sizes.

```bash
sysctl hw.physicalcpu hw.logicalcpu hw.l1dcachesize hw.l1icachesize hw.l2cachesize
```

**Output:**

```text
hw.physicalcpu: 14
hw.logicalcpu: 14
hw.l1dcachesize: 65536
hw.l1icachesize: 131072
hw.l2cachesize: 4194304
```

The values are in bytes. `65536` bytes is 64 KiB. `4194304` bytes is 4 MiB. The key point is not the exact number. The key point is that the operating system exposes the machine as layered hardware with caches, not just as a brand name.

If you are on Linux or WSL, the equivalent teaching command is:

```bash
lscpu
```

### Conceptual question

Why is it useful for the operating system to expose cache sizes instead of only exposing the CPU model name?

<details><summary>Answer</summary>Because the model name is only a label, while cache sizes describe part of how the processor is structured and how it behaves. Programs do not only depend on which processor they run on. They also depend on how quickly data can be fed to that processor.</details>

### Exercise

Convert the L1 and L2 cache sizes you saw from bytes into KiB and MiB.

<details><summary>Solution</summary>

```bash
python3 -c "print('L1 data cache in KiB:', 65536 / 1024); print('L2 cache in MiB:', 4194304 / 1024 / 1024)"
```

```text
L1 data cache in KiB: 64.0
L2 cache in MiB: 4.0
```

You are not changing the underlying hardware fact. You are only expressing the same value in units that are easier for humans to read.
</details>

## Part 2 — Reading Memory As Pages

### What is memory accounting?

When people talk casually about RAM, they usually talk about one big number such as 16 GB or 24 GB. Operating systems do not manage memory as one giant blob. They manage it in fixed-size chunks called pages.

That is why memory tools report categories such as free, active, inactive, and wired memory. Those names are the operating system's bookkeeping categories. They describe what the kernel thinks each group of pages is doing right now.

This part separates two different questions. One question is how much memory is installed in the machine. The other question is how the operating system is accounting for that memory at this moment.

### Run the demo

Start by asking for total installed memory in bytes.

```bash
sysctl hw.memsize
```

**Output:**

```text
hw.memsize: 25769803776
```

That number is large because it is raw bytes. It is accurate, but not yet human-friendly.

Convert that raw byte count into GiB.

```bash
python3 - <<'PY'
import subprocess
mem = int(subprocess.check_output(['sysctl', '-n', 'hw.memsize'], text=True).strip())
print(f'Total memory in GiB: {mem / 1024 / 1024 / 1024:.2f}')
PY
```

**Output:**

```text
Total memory in GiB: 24.00
```

Now inspect the operating system's page counters.

```bash
vm_stat | awk 'NR==1 || /Pages free|Pages active|Pages inactive|Pages wired down/'
```

**Output:**

```text
Mach Virtual Memory Statistics: (page size of 16384 bytes)
Pages free:                               55102.
Pages active:                            655574.
Pages inactive:                          652725.
Pages wired down:                        144388.
```

The first line tells you the page size. The next lines tell you how many pages are in several categories.

Now convert the free-page count into MiB.

```bash
python3 - <<'PY'
import subprocess, re
text = subprocess.check_output(['vm_stat'], text=True)
page_size = int(re.search(r'page size of (\d+) bytes', text).group(1))
free_pages = int(re.search(r'Pages free:\s+(\d+)\.', text).group(1))
print(f'Free pages: {free_pages}')
print(f'Page size: {page_size} bytes')
print(f'Free memory in MiB: {free_pages * page_size / 1024 / 1024:.2f}')
PY
```

**Output:**

```text
Free pages: 53333
Page size: 16384 bytes
Free memory in MiB: 833.33
```

Your own free-page count will change over time. The structure is the important part, not the exact number.

If you are on Linux or WSL, the equivalent teaching command is:

```bash
free -h
```

### Conceptual question

Why is page-based output more useful to an operating system than a single top-line “free RAM” number?

<details><summary>Answer</summary>Because the kernel manages memory at the page level. Pages are the units it maps, reclaims, protects, and moves. A single top-line number is useful to humans, but pages are the level where the operating system actually does its work.</details>

### Exercise

Change the Python snippet so it computes wired memory in MiB instead of free memory.

<details><summary>Solution</summary>

```bash
python3 - <<'PY'
import subprocess, re
text = subprocess.check_output(['vm_stat'], text=True)
page_size = int(re.search(r'page size of (\d+) bytes', text).group(1))
wired_pages = int(re.search(r'Pages wired down:\s+(\d+)\.', text).group(1))
print(f'Wired memory in MiB: {wired_pages * page_size / 1024 / 1024:.2f}')
PY
```

```text
Wired memory in MiB: 2221.41
```

The structure is identical. You only changed which page counter you multiplied by the page size.
</details>

## Part 3 — From Disks To Bytes

### What is a storage layout?

Files do not float in empty space. They live inside a file system, which lives on top of volumes, containers, partitions, and physical devices. That stack is the storage layout the operating system sees.

Looking at that layout matters because it turns “my file is on disk” into something more concrete. The operating system can describe disks, partition schemes, APFS containers, and volumes before you ever open a single file.

After you look at the storage layout, the next useful step is to zoom all the way in. A file is still just bytes. A hex dump tool makes those bytes visible.

### Run the demo

Create a tiny file containing the text `Hello` and a newline, then confirm that the file exists.

```bash
mkdir -p 1.5 && cd 1.5 && printf 'Hello\n' > hello.txt && ls -l hello.txt
```

**Output:**

```text
-rw-r--r--@ 1 giacobbe  staff  6 Apr 20 16:11 hello.txt
```

The file is 6 bytes long because each visible character plus the newline takes one byte in this example.

Now inspect the storage layout that file lives on.

```bash
diskutil list | head -12
```

**Output:**

```text
/dev/disk0 (internal, physical):
   #:                       TYPE NAME                    SIZE       IDENTIFIER
   0:      GUID_partition_scheme                        *500.3 GB   disk0
   1:             Apple_APFS_ISC Container disk2         524.3 MB   disk0s1
   2:                 Apple_APFS Container disk3         494.4 GB   disk0s2
   3:        Apple_APFS_Recovery Container disk1         5.4 GB     disk0s3

/dev/disk3 (synthesized):
   #:                       TYPE NAME                    SIZE       IDENTIFIER
   0:      APFS Container Scheme -                      +494.4 GB   disk3
                                 Physical Store disk0s2
   1:                APFS Volume Macintosh HD            12.6 GB    disk3s1
```

This is the operating system's map of the storage stack. It shows a physical disk and higher-level APFS structures built on top of it.

Now look at the file itself as hexadecimal bytes.

```bash
cd 1.5 && xxd -g 1 -l 16 hello.txt
```

**Output:**

```text
00000000: 48 65 6c 6c 6f 0a                                Hello.
```

`48` is the hexadecimal value for uppercase `H`. `0a` is the newline byte at the end.

Now look at the same file in binary.

```bash
cd 1.5 && xxd -b -l 6 hello.txt
```

**Output:**

```text
00000000: 01001000 01100101 01101100 01101100 01101111 00001010  Hello.
```

This is still the same file. The representation changed, not the bytes on disk.

If you are on Linux or WSL, the equivalent storage-layout command is:

```bash
lsblk
```

### Conceptual question

Why is it useful to pair a disk-layout command with a hex dump in the same lab part?

<details><summary>Answer</summary>Because they show two different levels of the same storage story. The disk-layout command shows how the operating system organizes storage devices and volumes. The hex dump shows what the contents of one specific file look like once you zoom all the way down to bytes.</details>

### Exercise

Overwrite `hello.txt` with `Hi\n`, then dump it again. Which bytes changed, and why?

<details><summary>Solution</summary>

```bash
cd 1.5 && printf 'Hi\n' > hello.txt && xxd -g 1 -l 16 hello.txt
```

```text
00000000: 48 69 0a                                         Hi.
```

The bytes changed because the file content changed. You removed several letters, so the stored byte sequence became shorter.
</details>

## Part 4 — Converting Between Number Systems And Text

### What is representation?

A byte does not come with one built-in display format. Humans can write the same value as decimal, binary, or hexadecimal. They can also interpret that value as text if both sides agree on an encoding.

That is why a hex dump is not showing “different data” from the original file. It is showing the same underlying value in a different notation. Binary is explicit. Hex is compact. Decimal is familiar. Text is meaningful to people.

Python is useful here because its built-in functions are small and direct. They let you move between these representations without hiding the mechanics inside a third-party library.

### Run the demo

Start with one decimal value and format it in binary and hexadecimal.

```bash
python3 -c "print('Decimal 72 -> binary:', bin(72)); print('Decimal 72 -> hex:', hex(72))"
```

**Output:**

```text
Decimal 72 -> binary: 0b1001000
Decimal 72 -> hex: 0x48
```

The prefixes `0b` and `0x` tell you which base Python used for the representation.

Now convert between characters and code points, then parse binary and hexadecimal text back into decimal.

```bash
python3 -c "print('Character H -> code point:', ord('H')); print('Code point 72 -> character:', chr(72)); print('Binary 01001000 -> decimal:', int('01001000', 2)); print('Hex 48 -> decimal:', int('48', 16))"
```

**Output:**

```text
Character H -> code point: 72
Code point 72 -> character: H
Binary 01001000 -> decimal: 72
Hex 48 -> decimal: 72
```

This is the same underlying value viewed four ways.

Now recreate `Hello\n` and read the stored bytes directly with Python.

```bash
cd 1.5 && printf 'Hello\n' > hello.txt && python3 -c "import pathlib; data = pathlib.Path('hello.txt').read_bytes(); print('Byte values:', list(data))"
```

**Output:**

```text
Byte values: [72, 101, 108, 108, 111, 10]
```

Those decimal byte values line up with the hex and binary views you already saw with `xxd`.

### Conceptual question

Why can the same underlying byte be shown as text, decimal, binary, or hexadecimal without changing the file itself?

<details><summary>Answer</summary>Because the byte is the underlying stored value. Text, decimal, binary, and hexadecimal are different representations or interpretations of that same value. Changing how you display a byte does not change what is stored on disk.</details>

### Exercise

Use Python to convert the character `e` into its code point, then show that value in hexadecimal.

<details><summary>Solution</summary>

```bash
python3 -c "value = ord('e'); print('Code point:', value); print('Hex:', hex(value))"
```

```text
Code point: 101
Hex: 0x65
```

This works because `ord()` gives you the numeric code point, and `hex()` reformats that same number as hexadecimal text.
</details>

## Putting it together

| Mechanism | What problem it solves | Key idea | Real systems |
| --- | --- | --- | --- |
| CPU architecture reporting | Identifies what kind of machine instructions a computer understands | The OS exposes hardware facts through kernel interfaces | Choosing the correct binary for `arm64` vs `x86_64` |
| Page-based memory accounting | Lets the kernel manage memory precisely | Memory is tracked in fixed-size pages | Virtual memory, page reclaim, cache management |
| Storage layout inspection | Shows how files sit on top of real storage structures | Files live inside a storage hierarchy | Partitioning, APFS volumes, file-system recovery |
| Base and text conversion | Lets humans reason about one stored value in several useful notations | The same byte can be shown as text, decimal, binary, or hex | Debugging file formats, protocols, and encodings |

This lab moves through several layers of the machine. The operating system names the hardware. The kernel tracks memory in pages. Storage tools show the structure beneath files. Representation tools show that file content is still just bytes.
