## Part 1 — Seeing the CPU

### What is a CPU introspection command?

Every program you run becomes instructions executed by the chip inside your Mac. To reason about what the chip is doing, you first need to name it: how many cores it has, what model it is, how much fast on-chip memory (cache) it has. macOS exposes all of this through a dotted namespace called **sysctl**. The kernel already tracks every one of these numbers to do its job — `sysctl` is the front door you use to ask.

### Run the demo

**1. Read the CPU brand string.**

```
sysctl -n machdep.cpu.brand_string
```

**Output:**

```
Apple M1 Pro
```

*(Your chip name will look different — an M2, M3, or Intel model, depending on your Mac.)*

This is the exact model name the kernel knows your CPU by.

**2. Count the logical CPUs.**

```
sysctl -n hw.ncpu
```

**Output:**

```
8
```

*(Your core count will look different — 4 on entry-level Macs, up to 24 on a Studio Ultra.)*

Each logical CPU is one independent stream of instructions the kernel can schedule work onto.

**3. Dump a slice of the full hardware and CPU MIB tree.**

```
sysctl hw machdep.cpu | head -20
```

**Output:**

```
hw.ncpu: 8
hw.byteorder: 1234
hw.memsize: 34359738368
hw.activecpu: 8
hw.perflevel0.physicalcpu: 6
hw.perflevel0.physicalcpu_max: 6
hw.perflevel0.logicalcpu: 6
hw.perflevel0.logicalcpu_max: 6
hw.perflevel0.l1icachesize: 196608
hw.perflevel0.l1dcachesize: 131072
hw.perflevel0.l2cachesize: 12582912
hw.perflevel0.cpusperl2: 3
hw.perflevel0.name: Performance
hw.perflevel1.physicalcpu: 2
hw.perflevel1.physicalcpu_max: 2
hw.perflevel1.logicalcpu: 2
hw.perflevel1.logicalcpu_max: 2
hw.perflevel1.l1icachesize: 131072
hw.perflevel1.l1dcachesize: 65536
hw.perflevel1.l2cachesize: 4194304
```

*(Your values will look different depending on your chip. Apple Silicon Macs expose two performance levels — `perflevel0` is the fast Performance cores, `perflevel1` is the power-efficient Efficiency cores. Intel Macs have only one level.)*

Every line is a single fact the kernel knows about your hardware. `l1dcachesize: 131072` means each Performance core has 128 KiB of L1 data cache — the fastest memory the chip owns.

> **Why does Apple Silicon list two sets of `perflevel` entries, but Intel Macs list only one?**
> <details><summary>Answer</summary>Apple Silicon chips physically contain two kinds of cores on the same die — high-performance "P-cores" and power-efficient "E-cores". They have different clock speeds, different cache sizes, and different scheduling priorities, so the kernel reports them as two separate performance levels. Intel Macs have one uniform core design, so there is nothing to split.</details>

### Exercise

Ask the kernel for the size of your Performance-core L2 cache and print the number of **kilobytes** (the raw value is in bytes). Use `sysctl -n` to read one specific MIB and shell arithmetic (`$(( ... ))`) to divide by 1024.

<details><summary>Solution</summary>

```
echo "$(( $(sysctl -n hw.perflevel0.l2cachesize) / 1024 )) KiB"
```

On an M1 Pro this prints `12288 KiB` (12 MiB). The point: the kernel gives you bytes; unit conversion is always your job.

If your Mac is Intel-based, there is no `hw.perflevel0.l2cachesize`. Use `sysctl -n hw.l2cachesize` instead.

</details>

## Part 2 — Seeing the RAM

### What is memory introspection?

RAM is the finite pool of fast storage your programs live in while they run. Two questions come before any interesting analysis: *how much RAM does this machine have*, and *how much is in use right now*. macOS answers the first with a single `sysctl` entry in bytes, and the second with `vm_stat`, a lower-level Mach tool that counts **pages** of memory.

### Run the demo

**1. Read total RAM in bytes.**

```
sysctl -n hw.memsize
```

**Output:**

```
34359738368
```

*(Your value will look different — total RAM in bytes as a single integer.)*

That is the exact number of bytes of physical RAM the kernel can address.

**2. Convert to gibibytes by hand.**

```
echo "scale=2; $(sysctl -n hw.memsize) / 1073741824" | bc
```

**Output:**

```
32.00
```

*(Your value will look different — e.g. 8.00, 16.00, 64.00.)*

`bc` is a command-line calculator. We divide by 1073741824 (that's 2^30) to convert bytes to gibibytes. This is exactly what a human-readable tool like Activity Monitor does behind the scenes.

**3. Look at live memory usage.**

```
vm_stat | head -12
```

**Output:**

```
Mach Virtual Memory Statistics: (page size of 16384 bytes)
Pages free:                                3803.
Pages active:                            483734.
Pages inactive:                          482071.
Pages speculative:                          947.
Pages throttled:                              0.
Pages wired down:                        160882.
Pages purgeable:                              4.
"Translation faults":               11107401219.
Pages copy-on-write:                  115147619.
Pages zero filled:                   5606101891.
Pages reactivated:                     75600017.
```

*(Your page counts will look different — they depend on what's running right now.)*

`vm_stat` reports counts in **pages**, not bytes. The first line tells you the page size — 16384 bytes on Apple Silicon, 4096 bytes on Intel Macs. "Pages wired down" is memory the kernel itself has locked; "Pages active" is memory currently in use by processes.

> **Why does `vm_stat` report pages instead of bytes?**
> <details><summary>Answer</summary>Memory is allocated in fixed-size chunks called pages because the hardware's memory-management unit works at page granularity, not byte granularity. Reporting in pages is closer to what the kernel actually tracks internally; converting to bytes is a human-readability concern that's left to you.</details>

### Exercise

Compute your machine's **free memory in MiB** by multiplying `vm_stat`'s `Pages free` count by the page size (16384 on Apple Silicon, 4096 on Intel) and dividing by 1048576 (2^20). Do this in one line of shell using `vm_stat`, `grep`, and `awk` (or `bc`).

<details><summary>Solution</summary>

```
echo "$(( $(vm_stat | awk '/Pages free/ {gsub("\\.","",$3); print $3}') * 16384 / 1048576 )) MiB"
```

On an Apple Silicon Mac this prints something like `101 MiB` — a surprisingly small number on a 32 GiB machine. That's because macOS aggressively uses "free" RAM for caches; what matters on modern macOS is the combination of free + inactive + purgeable pages, which Activity Monitor aggregates into its "Memory Pressure" gauge.

</details>

## Part 3 — Seeing the Disk

### What is a disk layout tool?

Your "disk" is a physical SSD, but what you interact with — folders, apps, the `Macintosh HD` icon — lives inside a nested tree of containers, volumes, and filesystems on top of it. Two tools pull the curtain back: `df -h` shows you the mounted filesystems and how full each one is, and `diskutil list` shows the full APFS hierarchy underneath.

### Run the demo

**1. See how full the root filesystem is.**

```
df -h /
```

**Output:**

```
Filesystem        Size    Used   Avail Capacity iused ifree %iused  Mounted on
/dev/disk3s3s1   460Gi    10Gi   232Gi     5%    426k  2.4G    0%   /
```

*(Your sizes, device name, and usage will look different.)*

`df` reads the system mount table. `/` is the root of your filesystem. The device name `/dev/disk3s3s1` tells you it's the first snapshot of the third slice of the third disk — which will make more sense after the next command.

**2. See the full disk tree.**

```
diskutil list
```

**Output:**

```
/dev/disk0 (internal, physical):
   #:                       TYPE NAME                    SIZE       IDENTIFIER
   0:      GUID_partition_scheme                        *500.3 GB   disk0
   1:             Apple_APFS_ISC Container disk1         524.3 MB   disk0s1
   2:                 Apple_APFS Container disk3         494.4 GB   disk0s2
   3:        Apple_APFS_Recovery Container disk2         5.4 GB     disk0s3

/dev/disk3 (synthesized):
   #:                       TYPE NAME                    SIZE       IDENTIFIER
   0:      APFS Container Scheme -                      +494.4 GB   disk3
                                 Physical Store disk0s2
   1:                APFS Volume Macintosh HD - Data     219.4 GB   disk3s1
   2:                APFS Volume Macintosh HD            11.3 GB    disk3s3
   3:              APFS Snapshot com.apple.os.update-... 11.3 GB    disk3s3s1
   4:                APFS Volume Preboot                 8.1 GB     disk3s4
   5:                APFS Volume Recovery                2.0 GB     disk3s5
   6:                APFS Volume VM                      4.3 GB     disk3s6
```

*(Your disk sizes and volume layout will look different, and you may see additional external drives if any are attached.)*

`disk0` is the real SSD inside your Mac. `disk3` is labeled `(synthesized)` — it's not a physical device at all. It's the APFS **container** APFS exports so you can refer to the pool of volumes that share the container's free space dynamically. `Macintosh HD` and `Macintosh HD - Data` are two volumes inside that one container — they do not have pre-sized partitions; they grow and shrink against the same pool.

> **Why does APFS use a synthesized disk instead of giving each volume a real partition?**
> <details><summary>Answer</summary>Traditional partition schemes pre-allocate a fixed size for each volume, so if one fills up and another is mostly empty, you cannot easily move space between them. APFS containers share free space across every volume inside them — the OS volume, the Data volume, and the Recovery volume all draw from one pool — which means you never have to repartition as usage shifts.</details>

### Exercise

Use `df -h` with no argument to list every mounted filesystem, then count how many entries whose mount point starts with `/System/Volumes/`. Use `grep` and `wc -l`.

<details><summary>Solution</summary>

```
df -h | grep "/System/Volumes/" | wc -l
```

On modern macOS this typically prints `7` or `8` — there are separate filesystems for `/System/Volumes/Data`, `/System/Volumes/Preboot`, `/System/Volumes/Update`, `/System/Volumes/VM`, `/System/Volumes/xarts`, `/System/Volumes/Hardware`, and a few others (your exact set may differ slightly by macOS version). The point: what you think of as "one disk" is many mounted volumes stitched together at boot.

</details>

## Part 4 — Seeing Files as Bytes

### What is `xxd`?

Every file on your disk is a sequence of bytes. A byte is a number from 0 to 255. When Preview opens a `.png` it interprets those bytes as pixels; when TextEdit opens a `.txt` it interprets them as characters. The file itself does not know the difference — the interpretation lives in the program. `xxd` is the tool that lets you stop interpreting and just see the numbers.

Before you start, a short detour to make sure you know where you are in the filesystem.

**Check your current working directory.**

```
pwd
```

**Output:**

```
/Users/you/path/to/1.5
```

*(Your path prefix will look different — the important part is that it ends in `/1.5`.)*

`pwd` prints your current working directory. This is where file commands like `xxd` look when you type a relative filename.

**Go home and come back.**

```
cd
pwd
```

**Output:**

```
/Users/you
```

*(Your username will look different.)*

`cd` with no argument is a shortcut for "go to my home directory."

```
cd -
pwd
```

**Output:**

```
/Users/you/path/to/1.5
```

*(Same as the first `pwd` — `cd -` jumps back to the previous directory.)*

Your shell remembers the last place you were. Use this to bounce between two folders without retyping paths. With that mental model in place, the rest of this Part assumes you are back inside `1.5/`.

### Run the demo

**1. Create a small text file and dump it.**

```
echo "Hello, world!" > hello.txt
xxd hello.txt
```

**Output:**

```
00000000: 4865 6c6c 6f2c 2077 6f72 6c64 210a       Hello, world!.
```

Three columns: the byte offset `00000000` (we are at the start of the file), sixteen bytes of hex in the middle (only 14 appear because the file is 14 bytes long), and the ASCII rendering on the right. `0x48` is the letter `H`. `0x0a` at the very end is the line feed your `echo` added.

**2. Look at the first 16 bytes of the shipped PNG.**

```
xxd -l 16 sample.png
```

**Output:**

```
00000000: 8950 4e47 0d0a 1a0a 0000 000d 4948 4452  .PNG........IHDR
```

Wait — look at the right column. That is the PNG **signature**: `89` `50` `4e` `47` `0d` `0a` `1a` `0a`. The ASCII column shows `.PNG` because `0x50 0x4e 0x47` spell `P N G` — the first byte `0x89` and the trailing control bytes are non-printable and render as `.`. This is how your OS recognizes a PNG file: the first 8 bytes are a fixed magic number. The file extension is a hint; the signature is the truth.

**3. Dump the whole PNG.**

```
xxd sample.png | head -6
```

**Output:**

```
00000000: 8950 4e47 0d0a 1a0a 0000 000d 4948 4452  .PNG........IHDR
00000010: 0000 0010 0000 0010 0802 0000 0090 9168  ...............h
00000020: 3600 0000 1749 4441 5478 9c63 f8cf c040  6....IDATx.c...@
00000030: 1222 4df5 a886 510d 434a 0300 90f9 ff01  ."M...Q.CJ......
00000040: f9e1 fa78 0000 0000 4945 4e44 ae42 6082  ...x....IEND.B`.
```

Five rows for an 80-byte file. You can see `IHDR` (the image header chunk), `IDAT` (the compressed pixel data), and `IEND` (the terminator) peeking through the ASCII column — the PNG specification mandates these three chunks in that order, and you are looking at them directly.

> **Why does the ASCII column show `.PNG` but not `\x89`, even though the file starts with `0x89`?**
> <details><summary>Answer</summary>`xxd` renders only printable ASCII bytes (0x20–0x7E) in the right column and replaces everything else with `.`. The byte `0x89` is outside that range, so it renders as a dot. The hex column (middle) is always the truth; the ASCII column is a convenience that only works for roughly half of the 256 possible byte values.</details>

### Exercise

Create a text file containing only the single letter `A` (no newline) and dump it with `xxd`. Then create another file with only the letter `a` and dump it. What's the one-byte difference, and why?

<details><summary>Solution</summary>

```
printf 'A' > a.txt
xxd a.txt
printf 'a' > b.txt
xxd b.txt
rm a.txt b.txt
```

`A` is `0x41`, `a` is `0x61`. The two differ in bit 5 — that's how the ASCII designers encoded case: flipping bit 5 converts between uppercase and lowercase for every letter. You just saw this with your own eyes.

</details>

## Part 5 — Seeing Bytes as Numbers, Letters, and Bits

### What is base conversion?

Humans count in base 10. Computers store numbers in base 2. Hexadecimal (base 16) is the shorthand programmers use for base 2 because every hex digit maps neatly to 4 bits — one byte is always exactly 2 hex digits. All three are the *same number*, written differently. Python ships five built-in functions that convert between them and the ASCII letter each number corresponds to.

### Run the demo

**1. Decimal to hex.**

```
python3.12 -c "print(hex(72))"
```

**Output:**

```
0x48
```

`hex()` returns a string with a `0x` prefix. The number 72 in base 16 is `48` — because `4 × 16 + 8 = 72`.

**2. Decimal to binary.**

```
python3.12 -c "print(bin(72))"
```

**Output:**

```
0b1001000
```

Same number, base 2. Seven bits, because `1 × 64 + 1 × 8 = 72`.

**3. Hex string back to decimal.**

```
python3.12 -c "print(int('0x48', 16))"
```

**Output:**

```
72
```

`int(s, 16)` parses the string `s` as a base-16 number. You just round-tripped 72 → `0x48` → 72.

**4. Letter to number, and number back to letter.**

```
python3.12 -c "print(ord('H'))"
python3.12 -c "print(chr(72))"
```

**Output:**

```
72
H
```

`ord()` returns the numeric code point of a single-character string; `chr()` goes the other way. ASCII capital `H` is the number 72 — which is the number you have now written as `0x48`, `0b1001000`, and `72` with your own hands.

**5. Tie it back to xxd.**

Remember the first byte of `hello.txt` was `48` in the hex column. What letter is that?

```
python3.12 -c "print(chr(int('48', 16)))"
```

**Output:**

```
H
```

Of course it is. The file starts with `Hello, world!` — the first byte has to be `H`. You did not need a decoder; you just composed two built-in functions.

**6. Spell the whole word from its bytes.**

```
python3.12 -c "print(''.join(chr(int(b,16)) for b in ['48','65','6c','6c','6f']))"
```

**Output:**

```
Hello
```

Those five hex values came straight out of the `xxd` output you ran in Part 4. The PNG was bytes you could not read; the text file was bytes that, interpreted as ASCII, spelled a word. Same tool, same command, two entirely different meanings.

> **Why does `bin(72)` start with `0b` and `hex(72)` start with `0x`?**
> <details><summary>Answer</summary>Python prepends the prefix so you can tell from the string alone which base it is in. Without a prefix, `10` could mean ten (decimal), two (binary), or sixteen (hex). The `0b` and `0x` markers are also valid syntax for writing integer literals directly — `0x48` in Python source code is exactly the integer 72.</details>

### Exercise

Write a single `python3.12 -c "..."` invocation that takes the number `255` and prints it in three forms on three lines: decimal, hex, and binary. Use an f-string or three separate `print()` calls.

<details><summary>Solution</summary>

```
python3.12 -c "n=255; print(n); print(hex(n)); print(bin(n))"
```

Prints `255`, `0xff`, `0b11111111`. That's the same number written in three ways — and `0xff` is exactly what `xxd` would show you for a byte whose every bit is set.

</details>

## Part 1 — Seeing the CPU

### What is a CPU introspection command?

Every program you run becomes instructions executed by the chip inside your computer. To reason about what the chip is doing, you first need to name it: how many cores it has, what model it is, how much fast on-chip memory (cache) it has. Linux exposes all of this through a kernel-generated pseudo-file at `/proc/cpuinfo` and a structured tree at `/sys/devices/system/cpu/`. The `lscpu` command parses both and prints an aggregated summary.

### Run the demo

**1. Read the CPU model name.**

```
grep -m1 "model name" /proc/cpuinfo
```

**Output:**

```
model name	: AMD Ryzen 7 5700G with Radeon Graphics
```

*(Your CPU will look different — Intel Core i5/i7/i9, AMD Ryzen, ARM Cortex, depending on your machine.)*

`/proc/cpuinfo` is a pseudo-file that the kernel generates on demand — it does not exist on disk. It lists every logical CPU with a repeated block of key-value lines; we grab just the first `model name` line with `grep -m1`.

**2. Count the logical CPUs.**

```
nproc
```

**Output:**

```
16
```

*(Your core count will look different — 2 on a tiny VM, up to 64+ on a workstation.)*

Each logical CPU is one independent stream of instructions the kernel can schedule work onto.

**3. Get the full aggregated summary.**

```
lscpu
```

**Output:**

```
Architecture:                         x86_64
CPU op-mode(s):                       32-bit, 64-bit
Address sizes:                        48 bits physical, 48 bits virtual
Byte Order:                           Little Endian
CPU(s):                               16
On-line CPU(s) list:                  0-15
Vendor ID:                            AuthenticAMD
Model name:                           AMD Ryzen 7 5700G with Radeon Graphics
CPU family:                           25
Model:                                80
Thread(s) per core:                   2
Core(s) per socket:                   8
Socket(s):                            1
Stepping:                             0
CPU max MHz:                          4673.0000
CPU min MHz:                          400.0000
BogoMIPS:                             7599.86

(output continues...)

L1i cache:                            256 KiB (8 instances)

```

*(Your values will look different — counts, clock speeds, flags, and cache sizes all depend on your exact chip.)*

`lscpu` did the work of reading `/proc/cpuinfo` and `/sys/`, deduplicating across all 16 logical CPUs, and producing one aligned report. The cache lines at the bottom tell you how much fast on-chip memory each level has — L1 is per-core, L2 is per-core or per-cluster, L3 is shared across the whole chip.

> **Why does `lscpu` show "Thread(s) per core: 2" and "Core(s) per socket: 8" but "CPU(s): 16"?**
> <details><summary>Answer</summary>The chip has 8 physical cores, each of which supports 2 hardware threads (Intel calls this Hyper-Threading; AMD calls it SMT). 8 cores × 2 threads = 16 logical CPUs, which is what the kernel schedules work onto. A single-threaded program can only ever use one of them at a time; a multi-threaded program can saturate all 16, though pairs sharing a core compete for the same execution units.</details>

### Exercise

Extract just the L1d (data) cache size from `lscpu` output using `grep` and print it. Don't rely on the exact line format — match on the label `L1d cache:`.

<details><summary>Solution</summary>

```
lscpu | grep "L1d cache"
```

Prints something like `L1d cache: 256 KiB (8 instances)`. On a chip with 8 cores and a per-core L1d of 32 KiB, `lscpu` sums them and reports the total — 256 KiB. The per-core size lives under `/sys/devices/system/cpu/cpu0/cache/index0/size` if you want to drill down.

</details>

## Part 2 — Seeing the RAM

### What is memory introspection?

RAM is the finite pool of fast storage your programs live in while they run. Two questions come before any interesting analysis: *how much RAM does this machine have*, and *how much is in use right now*. Linux answers both through the pseudo-file `/proc/meminfo`. The `free -h` command reads that file, applies human-readable units, and prints a compact table.

### Run the demo

**1. Read total RAM from `/proc/meminfo`.**

```
grep MemTotal /proc/meminfo
```

**Output:**

```
MemTotal:       65178768 kB
```

*(Your value will look different — total RAM in kilobytes, always labeled `kB` even on systems with TiB of memory.)*

Every line in `/proc/meminfo` is labeled with its unit (always `kB`). The kernel picks kilobytes because it is round and small; converting to human-readable is `free`'s job.

**2. Look at live memory usage.**

```
free -h
```

**Output:**

```
               total        used        free      shared  buff/cache   available
Mem:            62Gi       6.7Gi        26Gi        49Mi        28Gi        54Gi
Swap:             0B          0B          0B
```

*(Your values will look different — used, free, and available depend on what's running right now.)*

Six columns. `total` is the same number you got from `/proc/meminfo`, just in gibibytes. `used` is physical memory held by processes. `free` is literally unused. `buff/cache` is memory the kernel is using as a filesystem cache — it looks "used" but can be reclaimed instantly if something else needs it. **`available` is the honest answer to "how much memory could I allocate right now":** it adds `free` plus the reclaimable portion of `buff/cache`.

> **Why is `free` so low (26Gi) but `available` so high (54Gi) on the same machine?**
> <details><summary>Answer</summary>Linux uses "unused" RAM as a filesystem cache — every file you read stays resident so the next read is instant. That's what appears under `buff/cache`. If a program asks for memory, the kernel evicts cache pages to make room. So the RAM is not really "used" — it's opportunistically held. `available` subtracts that reclaimable cache from the used total to give you the real answer.</details>

### Exercise

Use `free` with the `-m` flag (units in MiB) and print just the `Mem:` line's `available` column. Use `grep` and `awk`.

<details><summary>Solution</summary>

```
free -m | awk '/Mem:/ {print $7 " MiB available"}'
```

The `-m` flag forces the units to MiB (raw integers), and `awk` picks the 7th whitespace-separated field on the `Mem:` line, which is `available`. This is the shape of one-liner that monitoring scripts and health checks use in the wild.

</details>

## Part 3 — Seeing the Disk

### What is a disk layout tool?

Your physical disk is usually divided into partitions, and each partition can hold a different filesystem (or be encrypted, or hold swap space). Two tools reveal this: `df -h` shows mounted filesystems with their usage, and `lsblk` shows the full tree of block devices below those mounts.

### Run the demo

**1. See how full the root filesystem is.**

```
df -h /
```

**Output:**

```
Filesystem      Size  Used Avail Use% Mounted on
overlay         913G   40G  828G   5% /
```

*(Your filesystem and device name will look different — typically `/dev/sda2` or `/dev/nvme0n1p2` on a bare-metal machine. `overlay` appears when you're inside a container or on WSL2.)*

`df` reads the system mount table and reports one row per mounted filesystem. `/` is the root of your filesystem. The `Use%` column is the first number to look at when a machine starts misbehaving — a full root filesystem breaks almost everything.

**2. See the full block-device tree.**

```
lsblk
```

**Output:**

```
NAME        MAJ:MIN RM   SIZE RO TYPE MOUNTPOINTS
sda           8:0    0     5G  0 disk 
sdb           8:16   0    10G  0 disk 
sdc           8:32   0     1G  0 disk 
nvme0n1     259:0    0 931.5G  0 disk 
|-nvme0n1p1 259:1    0     1G  0 part 
|-nvme0n1p2 259:2    0     2G  0 part 
`-nvme0n1p3 259:3    0 928.5G  0 part
```

*(Your disk names and partition layout will look different — typically a single `sda` or `nvme0n1` with 2–4 partitions on a laptop.)*

`lsblk` prints a tree. Each top-level entry (`sda`, `nvme0n1`) is a physical disk; indented entries below are partitions. `TYPE` distinguishes `disk` from `part` (partition). `MOUNTPOINTS` shows where each partition is mounted — empty means it is present but not currently mounted.

> **Why does `df -h /` show a device name but `lsblk` is the only place you see the full partition tree?**
> <details><summary>Answer</summary>`df` only knows about filesystems the kernel has mounted — one row per active mount. `lsblk` reads sysfs, which lists every block device the kernel has discovered whether or not it is mounted. Unmounted partitions, swap partitions, and spare disks all appear in `lsblk` and all are invisible to `df`.</details>

### Exercise

Use `lsblk` with the `-f` flag to also show filesystem types and UUIDs. Identify the filesystem type of your root partition.

<details><summary>Solution</summary>

```
lsblk -f
```

The `FSTYPE` column appears. Common values: `ext4` (default on most Linux distros), `xfs` (Red Hat family default), `btrfs` (Fedora), `vfat` (EFI System Partition). Swap partitions show `swap`. This is how installers figure out what's already on a disk before they offer to partition it.

</details>

## Part 4 — Seeing Files as Bytes

### What is `xxd`?

Every file on your disk is a sequence of bytes. A byte is a number from 0 to 255. When your image viewer opens a `.png` it interprets those bytes as pixels; when `cat` prints a `.txt` it interprets them as characters. The file itself does not know the difference — the interpretation lives in the program. `xxd` is the tool that lets you stop interpreting and just see the numbers.

Before you start, a short detour to make sure you know where you are in the filesystem.

**Check your current working directory.**

```
pwd
```

**Output:**

```
/home/you/path/to/1.5
```

*(Your path prefix will look different — the important part is that it ends in `/1.5`.)*

`pwd` prints your current working directory. This is where file commands like `xxd` look when you type a relative filename.

**Go home and come back.**

```
cd
pwd
```

**Output:**

```
/home/you
```

*(Your username will look different.)*

`cd` with no argument is a shortcut for "go to my home directory."

```
cd -
```

**Output:**

```
/home/you/path/to/1.5
```

*(Bash prints the new directory automatically when you use `cd -`. `pwd` will confirm it.)*

Your shell remembers the last place you were. Use this to bounce between two folders without retyping paths. With that mental model in place, the rest of this Part assumes you are back inside `1.5/`.

### Run the demo

**1. Create a small text file and dump it.**

```
echo "Hello, world!" > hello.txt
xxd hello.txt
```

**Output:**

```
00000000: 4865 6c6c 6f2c 2077 6f72 6c64 210a       Hello, world!.
```

Three columns: the byte offset `00000000` (we are at the start of the file), sixteen bytes of hex in the middle (only 14 appear because the file is 14 bytes long), and the ASCII rendering on the right. `0x48` is the letter `H`. `0x0a` at the very end is the line feed your `echo` added.

**2. Look at the first 16 bytes of the shipped PNG.**

```
xxd -l 16 sample.png
```

**Output:**

```
00000000: 8950 4e47 0d0a 1a0a 0000 000d 4948 4452  .PNG........IHDR
```

Wait — look at the right column. That is the PNG **signature**: `89` `50` `4e` `47` `0d` `0a` `1a` `0a`. The ASCII column shows `.PNG` because `0x50 0x4e 0x47` spell `P N G` — the first byte `0x89` and the trailing control bytes are non-printable and render as `.`. This is how your OS recognizes a PNG file: the first 8 bytes are a fixed magic number. The file extension is a hint; the signature is the truth.

**3. Dump the whole PNG.**

```
xxd sample.png | head -6
```

**Output:**

```
00000000: 8950 4e47 0d0a 1a0a 0000 000d 4948 4452  .PNG........IHDR
00000010: 0000 0010 0000 0010 0802 0000 0090 9168  ...............h
00000020: 3600 0000 1749 4441 5478 9c63 f8cf c040  6....IDATx.c...@
00000030: 1222 4df5 a886 510d 434a 0300 90f9 ff01  ."M...Q.CJ......
00000040: f9e1 fa78 0000 0000 4945 4e44 ae42 6082  ...x....IEND.B`.
```

Five rows for an 80-byte file. You can see `IHDR` (the image header chunk), `IDAT` (the compressed pixel data), and `IEND` (the terminator) peeking through the ASCII column — the PNG specification mandates these three chunks in that order, and you are looking at them directly.

> **Why does the ASCII column show `.PNG` but not `\x89`, even though the file starts with `0x89`?**
> <details><summary>Answer</summary>`xxd` renders only printable ASCII bytes (0x20–0x7E) in the right column and replaces everything else with `.`. The byte `0x89` is outside that range, so it renders as a dot. The hex column (middle) is always the truth; the ASCII column is a convenience that only works for roughly half of the 256 possible byte values.</details>

### Exercise

Create a text file containing only the single letter `A` (no newline) and dump it with `xxd`. Then create another file with only the letter `a` and dump it. What's the one-byte difference, and why?

<details><summary>Solution</summary>

```
printf 'A' > a.txt
xxd a.txt
printf 'a' > b.txt
xxd b.txt
rm a.txt b.txt
```

`A` is `0x41`, `a` is `0x61`. The two differ in bit 5 — that's how the ASCII designers encoded case: flipping bit 5 converts between uppercase and lowercase for every letter. You just saw this with your own eyes.

</details>

## Part 5 — Seeing Bytes as Numbers, Letters, and Bits

### What is base conversion?

Humans count in base 10. Computers store numbers in base 2. Hexadecimal (base 16) is the shorthand programmers use for base 2 because every hex digit maps neatly to 4 bits — one byte is always exactly 2 hex digits. All three are the *same number*, written differently. Python ships five built-in functions that convert between them and the ASCII letter each number corresponds to.

### Run the demo

**1. Decimal to hex.**

```
python3.12 -c "print(hex(72))"
```

**Output:**

```
0x48
```

`hex()` returns a string with a `0x` prefix. The number 72 in base 16 is `48` — because `4 × 16 + 8 = 72`.

**2. Decimal to binary.**

```
python3.12 -c "print(bin(72))"
```

**Output:**

```
0b1001000
```

Same number, base 2. Seven bits, because `1 × 64 + 1 × 8 = 72`.

**3. Hex string back to decimal.**

```
python3.12 -c "print(int('0x48', 16))"
```

**Output:**

```
72
```

`int(s, 16)` parses the string `s` as a base-16 number. You just round-tripped 72 → `0x48` → 72.

**4. Letter to number, and number back to letter.**

```
python3.12 -c "print(ord('H'))"
python3.12 -c "print(chr(72))"
```

**Output:**

```
72
H
```

`ord()` returns the numeric code point of a single-character string; `chr()` goes the other way. ASCII capital `H` is the number 72 — which is the number you have now written as `0x48`, `0b1001000`, and `72` with your own hands.

**5. Tie it back to xxd.**

Remember the first byte of `hello.txt` was `48` in the hex column. What letter is that?

```
python3.12 -c "print(chr(int('48', 16)))"
```

**Output:**

```
H
```

Of course it is. The file starts with `Hello, world!` — the first byte has to be `H`. You did not need a decoder; you just composed two built-in functions.

**6. Spell the whole word from its bytes.**

```
python3.12 -c "print(''.join(chr(int(b,16)) for b in ['48','65','6c','6c','6f']))"
```

**Output:**

```
Hello
```

Those five hex values came straight out of the `xxd` output you ran in Part 4. The PNG was bytes you could not read; the text file was bytes that, interpreted as ASCII, spelled a word. Same tool, same command, two entirely different meanings.

> **Why does `bin(72)` start with `0b` and `hex(72)` start with `0x`?**
> <details><summary>Answer</summary>Python prepends the prefix so you can tell from the string alone which base it is in. Without a prefix, `10` could mean ten (decimal), two (binary), or sixteen (hex). The `0b` and `0x` markers are also valid syntax for writing integer literals directly — `0x48` in Python source code is exactly the integer 72.</details>

### Exercise

Write a single `python3.12 -c "..."` invocation that takes the number `255` and prints it in three forms on three lines: decimal, hex, and binary. Use an f-string or three separate `print()` calls.

<details><summary>Solution</summary>

```
python3.12 -c "n=255; print(n); print(hex(n)); print(bin(n))"
```

Prints `255`, `0xff`, `0b11111111`. That's the same number written in three ways — and `0xff` is exactly what `xxd` would show you for a byte whose every bit is set.

</details>

## Putting it together

| Tool | What problem it solves | Key idea | Real uses |
|------|------------------------|----------|-----------|
| `lscpu` | Reading hardware facts from the kernel | Aggregates `/proc/cpuinfo` + `/sys/` | Benchmarking tools, monitoring agents |
| `free -h` | Reading live memory accounting | `MemAvailable` is the honest number | Capacity planning, OOM debugging |
| `lsblk` | Showing physical disks + partitions | Reads sysfs; unmounted disks visible | Installers, backup tools, RAID management |
| `xxd` | Seeing a file's actual bytes | Hex + ASCII side-by-side | File-format reverse engineering, debugging binary I/O |
| `python3.12 -c "..."` | Base conversion in the terminal | `bin`, `hex`, `int`, `ord`, `chr` | Any time you read a protocol spec, a crash dump, or a file header |

Every tool you ran here answered the same question: *what is actually happening inside my machine?* The kernel already knows the answer. Your job is just to learn which door to knock on.
