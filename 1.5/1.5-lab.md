## Part 1 — Seeing the Machine

### What is "the machine" your programs run on?

Every program eventually runs on a piece of silicon. That silicon has three facts worth knowing:

1. An **architecture** — the instruction set it understands. Almost every modern laptop is either `x86_64` (Intel or AMD) or `arm64` / `aarch64` (Apple Silicon, Windows on ARM).
2. A **core count** — how many independent workers the chip has. A 2026-era laptop typically has 4–16 cores.
3. A **cache hierarchy** — tiny pockets of very fast memory sitting right next to the cores so the CPU does not have to pay the cost of reading from main RAM every time. Lecture 1.4 introduced caches as "L1 (tiny, fastest), L2 (larger, slower), L3 (largest, slowest)." This lab shows you those numbers on your actual machine.

RAM sits below the caches — much larger, somewhat slower. Disk (SSD or HDD) sits below RAM — much larger again, much slower again. When the lectures say "the memory hierarchy," those four tiers are what they mean.

Your operating system discovered all of this at boot and stored it in kernel data structures you can read with one command each.

> **Why does `Physical cores` often differ from `Logical cores` on Intel and AMD chips?**
> <details><summary>Answer</summary>
> Intel and AMD include a feature called hyperthreading (Intel) or SMT (AMD) that lets one physical core run two instruction streams at once, sharing its execution units. The OS sees those as two separate "logical" CPUs even though they are backed by one physical core. A 4-core Intel with hyperthreading reports 4 physical / 8 logical. Apple Silicon chips do not have hyperthreading — each physical core is a single logical core — which is why the example output above shows 8 / 8.
> </details>

## Part 2 — Every File is Bytes

### What is a file at the byte level?

Lectures 1.1 and 1.4 made a claim that is easy to nod at but harder to feel: **every file on your disk is a sequence of numbers**. A plain-text file is a sequence of numbers that happen to match letters in an encoding table called ASCII. A PNG image is a sequence of numbers that a viewer program knows how to turn into pixels. A compiled program is a sequence of numbers that the CPU knows how to execute. The file doesn't "know" what it is — a file is just bytes.

`xxd` is the tool that takes a file and shows you those bytes directly. It prints three columns per line:

1. The **offset** on the left — how far into the file we are, counted in hex.
2. The **hex bytes** in the middle — each byte of the file shown as a two-digit hex number. Grouped in pairs for readability.
3. The **ASCII gutter** on the right — each byte as its printable character if one exists, or `.` if the byte is not a printable character.

Run the demo and match the output against the ASCII alphabet.

### Run the demo

```bash
bash bytes_demo.sh
```

**Output (Section A):**
```

═══════════════════════════════════════════════════════════════
SECTION A — Every letter is a byte
═══════════════════════════════════════════════════════════════
  Wrote /Users/giacobbe/src/courses/anyone-can-code/1.5/hello.txt  (14 bytes)

  --- xxd hello.txt ---
00000000: 4865 6c6c 6f2c 2077 6f72 6c64 210a       Hello, world!.

  Read the hex column left to right:
    48 = 'H'   65 = 'e'   6c = 'l'   6c = 'l'   6f = 'o'
    2c = ','   20 = ' '   77 = 'w'   6f = 'o'   72 = 'r'
    6c = 'l'   64 = 'd'   21 = '!'   0a = newline
```
*(Your filename path will differ. The hex bytes and ASCII gutter will be identical — the file content is the same on every machine.)*

The file is 14 bytes long. The offset on the left is `00000000` because we are at byte 0 of the file. The hex pairs in the middle are the actual content of the file — `48 65 6c 6c 6f` is `Hello` — and the ASCII gutter on the right (`Hello, world!.`) is a printable view of the same bytes. The final `0a` is a newline character (line feed); the gutter shows it as `.` because newlines are not printable.

**Output (Section B):**
```

═══════════════════════════════════════════════════════════════
SECTION B — One byte, four ways: char, hex, decimal, binary
═══════════════════════════════════════════════════════════════
  The first byte of hello.txt is 'H'.
    As hex:     0x48
    As decimal: 72
    As binary:  01001000
    As char:    H
```

The byte `0x48` from Section A's hex column is the same number as decimal 72, the same number as binary `01001000`, and (via ASCII) the same symbol as the letter `H`. Four representations, one value.

**Output (Section C — binary view):**
```

═══════════════════════════════════════════════════════════════
SECTION C — See the actual bits with xxd -b
═══════════════════════════════════════════════════════════════
  --- xxd -c 4 -b hello.txt  (4 bytes per line, binary view) ---
00000000: 01001000 01100101 01101100 01101100  Hell
00000004: 01101111 00101100 00100000 01110111  o, w
00000008: 01101111 01110010 01101100 01100100  orld
0000000c: 00100001 00001010                    !.

  The first two groups on line 1 are 01001000 01100101 = 'H' 'e'.
```

`xxd -b` swaps the hex column for binary — the actual bits stored on disk. The first eight bits `01001000` are the letter H. Count them: that is eight voltages (eight flip-flops) set to 0-1-0-0-1-0-0-0. Lecture 1.2's "logic gates combine into flip-flops combine into memory cells" claim is being demonstrated right here.

**Output (Section D — PNG signature):**
```

═══════════════════════════════════════════════════════════════
SECTION D — A non-text file has bytes too (PNG magic bytes)
═══════════════════════════════════════════════════════════════
  Wrote /Users/giacobbe/src/courses/anyone-can-code/1.5/tiny.png  (8 bytes — just the file signature)

  --- xxd tiny.png ---
00000000: 8950 4e47 0d0a 1a0a                      .PNG....

  Byte 0 (0x89) is a sentinel: its high bit makes it not a valid ASCII char.
  Bytes 1–3 spell 'PNG' in the ASCII gutter: 0x50 0x4e 0x47.
  Every real PNG on your disk starts with these exact 8 bytes.
```
*(Your filename path will differ; the hex bytes are fixed by the PNG specification.)*

The 8 bytes `89 50 4e 47 0d 0a 1a 0a` are the PNG "magic number" — a signature every valid PNG begins with. When an image viewer opens a file, it reads the first few bytes and checks them against a known list of signatures to decide what kind of file it is. Most file types have a signature like this.

> **Why does the ASCII gutter show `.` for the byte `0x0a` even though it's a valid byte value?**
> <details><summary>Answer</summary>
> The ASCII gutter is for <em>printable</em> characters — ones that have a visible glyph. Bytes 0x00–0x1f and 0x7f are "control characters": newline (0x0a), tab (0x09), null (0x00), escape (0x1b), etc. They exist and are fully valid bytes, but they don't produce a visible character when printed — a terminal would just execute them (moving to the next line, inserting a tab, ringing a bell). <code>xxd</code> shows <code>.</code> for all non-printable bytes so the column stays aligned.
> </details>

## Part 3 — Same Bytes, Now in Python

### What does it mean for a number to "be" in a base?

When you write the number seventy-two, you can spell it any of four ways:

- `72` in decimal (base 10)
- `0x48` in hex (base 16)
- `0b1001000` in binary (base 2)
- `H` via the ASCII table

These are all the *same number*. Only the label is different. Python has built-ins for every conversion between these views, and each conversion is one function call.

Learning these by hand turns the lecture's "binary, decimal, hex" taxonomy from a slide into a reflex: when the debugger prints `0x41`, you know it's `65`, which is `A`.

### Run the demo

```bash
python3.12 conversions_demo.py
```

**Output (Section A):**
```
======================================================================
SECTION A — one byte, four representations
======================================================================
  Character:     'H'
  ord('H')        = 72            (decimal)
  hex(72)        = '0x48'          (hex string, with '0x' prefix)
  bin(72)        = '0b1001000'     (binary string, with '0b' prefix)
  chr(72)        = 'H'            (back to the character)

  Clean formats via f-strings (no prefix, zero-padded):
  f"{n:02x}"      = '48'          (two hex digits — like xxd)
  f"{n:08b}"      = '01001000'    (eight bits — one full byte)
```

`ord('H')` returns the integer code point of the character `H`. `hex()` and `bin()` turn an integer into a string that spells it in a given base — they add `0x` / `0b` prefixes automatically. `chr(72)` goes back from integer to character.

The f-string forms at the bottom are the ones you want for clean output: `:02x` means "two hex digits, zero-padded" (so `15` becomes `0f`, not `f`). `:08b` means "eight binary digits, zero-padded" (so you always see one full byte of bits). Both forms drop the `0x` / `0b` prefix.

**Output (Section B — round trip):**
```
======================================================================
SECTION B — round trip: 'H' -> 72 -> '48' -> '01001000' -> 72 -> 'H'
======================================================================
  Forward:
    ord('H')                 -> 72
    f'{72:02x}'              -> '48'
    f'{72:08b}'              -> '01001000'
  Backward:
    int('48', 16)           -> 72
    int('01001000', 2)     -> 72
    chr(72)                 -> 'H'
  Round trip holds:          True
```

Going in one direction uses `ord`, `hex`, `bin`. Going the other uses `int(string, base)` and `chr`. If you end up where you started, the conversions are inverses — no information was lost.

**Output (Section C — a whole word):**
```
======================================================================
SECTION C — a whole word, byte by byte
======================================================================
  Word: 'Hello'

  char  decimal   hex   binary    
  ----  -------   ---   ------    
  'H'   72        48    01001000
  'e'   101       65    01100101
  'l'   108       6c    01101100
  'l'   108       6c    01101100
  'o'   111       6f    01101111

  Compare the 'hex' column to what xxd printed for hello.txt.
  The numbers match byte-for-byte. Python and xxd are describing the
  same underlying reality — bytes on your disk — from two directions.
```

Look at the `hex` column: `48 65 6c 6c 6f`. Look at Part 2's `xxd hello.txt` output: the first five bytes were `4865 6c6c 6f`. Same five bytes, produced by Python from a string literal; produced by `xxd` from a file on disk. The match is not a coincidence — it is the definition.

**Output (Section D — bytes round trip):**
```
======================================================================
SECTION D — bytes objects: hex() forward, fromhex() back
======================================================================
  Original bytes:     b'Hello, world!'
  original.hex():     '48656c6c6f2c20776f726c6421'
  bytes.fromhex(...): b'Hello, world!'
  Round trip holds:   True

  The hex string is the exact same 26 chars you see in xxd's middle column.
```

A Python `bytes` object (`b"..."`) is a fixed-length sequence of integers 0–255 — exactly what a file on disk is. `.hex()` flattens it to a hex string; `bytes.fromhex()` rebuilds the bytes. This is the programmatic equivalent of what `xxd` does visually with a file.

> **Why does `bin(72)` return `'0b1001000'` (a string), not the number `1001000`?**
> <details><summary>Answer</summary>
> The number stored inside your computer is the <em>same</em> regardless of base — it's a sequence of bits in RAM. <code>bin()</code> doesn't change that value. What it returns is a new object: a <em>string</em> whose characters happen to spell the number in base 2. The <code>0b</code> prefix is part of the string (matching the syntax you'd type to <em>write</em> a binary literal in Python source code: <code>x = 0b1001000</code>). If <code>bin(72)</code> returned the number 1001000 instead, you'd lose information — in base 10, 1001000 is a million, which is a completely different number than 72. Keep the two ideas separate: <strong>values</strong> live in RAM as bits, <strong>representations</strong> of those values are strings in a chosen base.
> </details>

Write a one-line Python command that converts your own first initial to its decimal, hex, and binary forms, all on one line. Replace `'J'` below with your actual initial:

<details><summary>Solution</summary>

```bash

The `0x20` bit (5th bit, counting from 0) is the case-toggle bit.

</details>

## Putting it together

| Concept | What it lets you see | Key tool | Where the data comes from |
|---|---|---|---|
| CPU / caches | Chip name, cores, cache sizes | `lscpu` (Linux) / `sysctl` (macOS) | Kernel data populated at boot |
| RAM | Total and live memory usage | `free -h` / `vm_stat` | `/proc/meminfo` on Linux; Mach stats on macOS |
| Disks | Devices, partitions, mount points | `lsblk` / `diskutil list` | Kernel block-device registry |
| Bytes in a file | Every byte as hex + ASCII | `xxd` | The file itself |
| Bytes in memory | Conversions between representations | `ord`, `chr`, `hex`, `bin`, f-strings | Python standard library |

The first three rows are different questions you can ask the OS about the hardware underneath. The last two rows are different views of the same thing: a byte is a number, a number has four equivalent representations, and any file is just a sequence of them. When a later lab reads from a network socket, parses a binary file format, or debugs a memory dump, it will still be the same four-way translation you did here with a single letter.
