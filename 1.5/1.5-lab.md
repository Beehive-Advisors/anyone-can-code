## Part 1 - Reading CPU Facts

### What is machine architecture?

Your code does not run on a generic “computer.” It runs on a processor that speaks a specific instruction set. That instruction set is the vocabulary of the machine. If the processor and the program disagree about that vocabulary, the program cannot run.

That is why names like `arm64` and `x86_64` matter. They are not cosmetic labels. They tell you which kind of machine instructions the CPU understands. They also tell you why one binary might run on one machine and fail on another.

The processor also has physical limits and design choices. It has a certain number of cores. It has caches of different sizes. Those caches exist because the CPU is much faster than main memory, so the machine needs smaller, faster storage close to the core.

In this part, you are not benchmarking the processor. You are asking the operating system what it already knows about the processor installed in this machine.

### Run the demo

Start with the broadest question: what architecture does this machine report?

```bash
echo "Architecture: $(uname -m)"
```

**Output:**

```text
Architecture: arm64
```

This tells you the instruction-set family the operating system is running on. On this machine, the answer is `arm64`.

Now ask for the processor model name.

```bash
echo "CPU brand: $(sysctl -n machdep.cpu.brand_string)"
```

**Output:**

```text
CPU brand: Apple M4 Pro
```

This is a human-friendly model string. It is useful for orientation, but it does not explain the whole architecture by itself.

Now ask for a few more detailed facts: physical cores, logical cores, and cache sizes.

```bash
sysctl hw.physicalcpu hw.logicalcpu hw.l1dcachesize hw.l1icachesize hw.l2cachesize
```

**Output:**

```text
hw.physicalcpu: 14
hw.logicalcpu: 14
hw.l1dcachesize: 65536
hw.l1icachesize: 131072
hw.l2cachesize: 4194304
```

The operating system is reporting counts and sizes in bytes. `65536` bytes is 64 KiB. `4194304` bytes is 4 MiB. The important thing is not memorizing those exact numbers. The important thing is seeing that the machine exposes several cache layers because the CPU needs fast local storage.

If you are on Linux or WSL, the equivalent teaching command is:

```bash
lscpu
```

### Conceptual question

Why does the operating system bother exposing cache sizes instead of only exposing the CPU model name?

<details><summary>Answer</summary>Because the model name is a label, but cache sizes describe part of how the processor behaves. Programs do not just depend on “which chip” they run on. They also depend on how quickly data can be fed to the core. Cache sizes are part of the machine's performance structure.</details>

### Exercise

Convert the cache sizes you saw from bytes into KiB or MiB with Python. Start with `65536` and `4194304`.

<details><summary>Solution</summary>

```bash
python3 -c "print('L1 data cache in KiB:', 65536 / 1024); print('L2 cache in MiB:', 4194304 / 1024 / 1024)"
```

You are not changing the underlying value. You are changing the unit humans use to read it more easily.
</details>

## Part 2 - Reading Memory As Pages

### What is memory accounting?

When people talk about RAM casually, they usually picture one big pool of bytes. Operating systems do not manage memory as one giant lump. They manage it in fixed-size pieces called pages.

That is why memory tools often look strange at first. They talk about free pages, active pages, inactive pages, wired pages, and more. Those categories are the kernel's bookkeeping system. They tell the operating system which memory can move, which memory is in active use, and which memory must stay where it is.

A second important idea is that total memory and currently free memory are different questions. The first asks how much RAM is installed. The second asks how much memory is available right now according to the kernel's page counters.

In this part, you will inspect both. First you will read the total installed memory. Then you will inspect page counts. Then you will convert those low-level counts into a human-sized number.

### Run the demo

Start by asking for total installed memory in bytes.

```bash
sysctl hw.memsize
```

**Output:**

```text
hw.memsize: 25769803776
```

That number is large because it is raw bytes. The machine has 25,769,803,776 bytes of RAM, which is much easier to read as 24 GiB.

Convert that raw byte count into GiB.

```bash
python3 - <<'PY'
import subprocess
mem = int(subprocess.check_output(['sysctl', '-n', 'hw.memsize'], text=True).strip())
print(f'Total memory in GiB: {mem / 1024 / 1024 / 1024:.2f}')
PY
```

**Output:**

```text
Total memory in GiB: 24.00
```

Now inspect the kernel's page counters.

```bash
vm_stat | awk 'NR==1 || /Pages free|Pages active|Pages inactive|Pages wired down/'
```

**Output:**

```text
Mach Virtual Memory Statistics: (page size of 16384 bytes)
Pages free:                               58672.
Pages active:                            650907.
Pages inactive:                          658467.
Pages wired down:                        142136.
```

The first line tells you the page size. The lines below it tell you how many pages are in each category right now.

Now convert the free-page count into MiB.

```bash
python3 - <<'PY'
import subprocess, re
text = subprocess.check_output(['vm_stat'], text=True)
page_size = int(re.search(r'page size of (\d+) bytes', text).group(1))
free_pages = int(re.search(r'Pages free:\s+(\d+)\.', text).group(1))
print(f'Free pages: {free_pages}')
print(f'Page size: {page_size} bytes')
print(f'Free memory in MiB: {free_pages * page_size / 1024 / 1024:.2f}')
PY
```

**Output:**

```text
Free pages: 54185
Page size: 16384 bytes
Free memory in MiB: 846.64
```

Your free-page count will change over time because the machine is alive and allocating memory constantly. The structure is the same even when the number changes.

If you are on Linux or WSL, the equivalent teaching command is:

```bash
free -h
```

### Conceptual question

Why is page-based output more useful to an operating system than only showing a single “free RAM” number?

<details><summary>Answer</summary>Because the kernel needs to manage memory at a smaller unit than the whole machine. Pages are the pieces it can map, reclaim, protect, and move. A single top-line number is useful to humans, but pages are the level where the operating system actually does its work.</details>

### Exercise

Change the Python snippet so it computes the amount of wired memory in MiB instead of free memory.

<details><summary>Solution</summary>

```bash
python3 - <<'PY'
import subprocess, re
text = subprocess.check_output(['vm_stat'], text=True)
page_size = int(re.search(r'page size of (\d+) bytes', text).group(1))
wired_pages = int(re.search(r'Pages wired down:\s+(\d+)\.', text).group(1))
print(f'Wired memory in MiB: {wired_pages * page_size / 1024 / 1024:.2f}')
PY
```

The structure is the same. You are only changing which page counter you multiply by the page size.
</details>

## Part 3 - From Disks To Bytes

### What is a disk layout?

Storage is not just “one disk.” Operating systems describe storage as a hierarchy. There are physical devices, partition schemes, containers, volumes, and file systems layered on top of one another.

That hierarchy matters because files do not float in empty space. They live inside a file system, which lives on top of some underlying storage structure. If you do not understand that layering, later topics like booting, partitioning, and file systems stay abstract.

Once you have seen the layout, the next step is to zoom all the way in. A file on disk is just bytes. A hex dump tool makes that visible by printing the byte values directly.

This part uses a tiny file, `hello.txt`, so you can trace every byte by hand.

### Run the demo

Create a six-byte file containing the text `Hello` followed by a newline, then confirm that the file exists.

```bash
cd 1.5 && printf 'Hello\n' > hello.txt && ls -l hello.txt
```

**Output:**

```text
-rw-r--r--@ 1 giacobbe  staff  6 Apr 20 15:47 hello.txt
```

The file is 6 bytes long because `H`, `e`, `l`, `l`, `o`, and the newline each take one byte in this example.

Now inspect the storage layout that file lives on.

```bash
diskutil list | head -12
```

**Output:**

```text
/dev/disk0 (internal, physical):
   #:                       TYPE NAME                    SIZE       IDENTIFIER
   0:      GUID_partition_scheme                        *500.3 GB   disk0
   1:             Apple_APFS_ISC Container disk2         524.3 MB   disk0s1
   2:                 Apple_APFS Container disk3         494.4 GB   disk0s2
   3:        Apple_APFS_Recovery Container disk1         5.4 GB     disk0s3

/dev/disk3 (synthesized):
   #:                       TYPE NAME                    SIZE       IDENTIFIER
   0:      APFS Container Scheme -                      +494.4 GB   disk3
                                 Physical Store disk0s2
   1:                APFS Volume Macintosh HD            12.6 GB    disk3s1
```

This is the operating system's map of the storage stack. It is showing a physical disk, partitions, and APFS structures built on top of that disk.

Now look at the file itself as hexadecimal bytes.

```bash
cd 1.5 && xxd -g 1 -l 16 hello.txt
```

**Output:**

```text
00000000: 48 65 6c 6c 6f 0a                                Hello.
```

`48` is the hexadecimal value for `H`. `0a` is the newline byte at the end.

Now look at the same file in binary form.

```bash
cd 1.5 && xxd -b -l 6 hello.txt
```

**Output:**

```text
00000000: 01001000 01100101 01101100 01101100 01101111 00001010  Hello.
```

This is the same data again. Nothing changed in the file. Only the representation changed.

If you are on Linux or WSL, the equivalent storage-layout command is:

```bash
lsblk
```

### Conceptual question

Why is it useful to show both a disk-layout command and a hex dump in the same lab part?

<details><summary>Answer</summary>Because they show two different levels of the same storage story. The disk-layout command shows how the operating system organizes storage devices and volumes. The hex dump shows what a specific file looks like at the byte level inside that storage world.</details>

### Exercise

Overwrite `hello.txt` with the text `Hi\n`, then run `xxd -g 1 -l 16 hello.txt` again. Which bytes changed, and why?

<details><summary>Solution</summary>

```bash
cd 1.5 && printf 'Hi\n' > hello.txt && xxd -g 1 -l 16 hello.txt
```

The byte for `e` disappears because the file content changed from `Hello` to `Hi`. The file now contains fewer characters, so the byte sequence changes too.
</details>

## Part 4 - Converting Between Number Systems And Text

### What is representation?

A byte does not come with an attached meaning. Humans can look at the same value and write it in decimal, hexadecimal, or binary. Programs can also interpret that same value as part of text if both sides agree on an encoding.

That is why a hex dump is not “different data” from the original file. It is the same underlying value shown in a notation humans use for debugging and reasoning. Hex is compact. Binary is explicit. Decimal is familiar. Text is meaningful to people.

Python is useful here because its built-in functions make the conversion visible without hiding it behind a library. `bin()` and `hex()` format a value. `int(text, base)` parses it. `ord()` and `chr()` cross the boundary between characters and code points.

This is the moment where the file bytes from Part 3 and the number systems from earlier lectures finally meet.

### Run the demo

Start with one decimal value and ask Python to format it in binary and hexadecimal.

```bash
python3 -c "print('Decimal 72 -> binary:', bin(72)); print('Decimal 72 -> hex:', hex(72))"
```

**Output:**

```text
Decimal 72 -> binary: 0b1001000
Decimal 72 -> hex: 0x48
```

The prefixes `0b` and `0x` tell you which base Python used to format the value.

Now cross the boundary between text and numbers, then parse binary and hexadecimal text back into decimal.

```bash
python3 -c "print('Character H -> code point:', ord('H')); print('Code point 72 -> character:', chr(72)); print('Binary 01001000 -> decimal:', int('01001000', 2)); print('Hex 48 -> decimal:', int('48', 16))"
```

**Output:**

```text
Character H -> code point: 72
Code point 72 -> character: H
Binary 01001000 -> decimal: 72
Hex 48 -> decimal: 72
```

This is the same value seen four ways: as a character, as a decimal integer, as a binary string, and as a hexadecimal string.

To connect this back to the file from Part 3, recreate `Hello\n` and read the bytes directly with Python.

```bash
cd 1.5 && printf 'Hello\n' > hello.txt && python3 -c "import pathlib; data = pathlib.Path('hello.txt').read_bytes(); print('Byte values:', list(data))"
```

**Output:**

```text
Byte values: [72, 101, 108, 108, 111, 10]
```

Those decimal byte values line up with the hex and binary views you already saw with `xxd`.

### Conceptual question

Why can the same underlying byte be shown as text, decimal, binary, or hexadecimal without changing the file itself?

<details><summary>Answer</summary>Because the byte is the underlying value. Text, decimal, binary, and hexadecimal are different representations or interpretations of that value. Changing how you display a byte does not change the byte stored on disk.</details>

### Exercise

Use Python to convert the character `e` into its code point, then into hexadecimal.

<details><summary>Solution</summary>

```bash
python3 -c "value = ord('e'); print('Code point:', value); print('Hex:', hex(value))"
```

This works because `ord()` gives you the numeric code point, and `hex()` reformats that same numeric value as hexadecimal text.
</details>

## Putting It Together

| Mechanism | What problem it solves | Key idea | Real systems |
| --- | --- | --- | --- |
| CPU architecture reporting | Tells you what kind of machine instructions your computer understands | The OS exposes hardware facts through kernel interfaces | Choosing the correct binary for `arm64` vs `x86_64` |
| Page-based memory accounting | Lets the OS manage memory precisely instead of as one huge blob | Memory is tracked in fixed-size pages | Virtual memory, page reclaim, cache management |
| Disk and volume inspection | Shows how storage is organized before you even open a file | Files live inside a storage hierarchy | Partitioning, boot volumes, file-system recovery |
| Base and text conversion | Lets humans reason about the same value in multiple useful notations | One byte can be shown as decimal, binary, hex, or text | Debugging protocols, reading file headers, parsing formats |

The machine is not mysterious. It is layered. The operating system names the hardware. The kernel counts memory in pages. Storage tools show the layout beneath your files. Representation tools show that file content is still just bytes.
