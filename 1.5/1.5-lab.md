## Part 1 — Seeing the CPU

### What is the CPU?

The CPU is the chip that runs your programs. Its data sheet has three numbers that matter for this lab: the **architecture** (the instruction set — `arm64` on Apple Silicon, `x86_64` on Intel Macs), the **core count** (how many instructions can be in flight at once), and the **cache hierarchy** (tiny fast memory sitting next to the cores).

Lecture 1.4 drew the pyramid: registers → L1 → L2 → L3 → RAM → disk. Each tier is bigger than the one above, and each is slower. L1 is usually 32–128 KiB *per core* and takes a handful of clock cycles to read. L3 is typically a few MiB and shared across the whole chip, taking ten times longer. Your OS knows every one of these numbers and will print them on request.

### Run the demo

**Read the chip model.**
```bash
sysctl -n machdep.cpu.brand_string
```
**Output:**
```
Apple M1 Pro
```
*(Your chip will differ — the command doesn't.)*

**Count your cores.**
```bash
sysctl -n hw.physicalcpu
sysctl -n hw.logicalcpu
```
**Output:**
```
8
8
```

On Apple Silicon the two numbers always match because there's no simultaneous multithreading. On an Intel Mac with Hyper-Threading you'd see 4 physical and 8 logical — one real core pretending to be two.

**Read the cache sizes.**
```bash
sysctl -n hw.perflevel0.l1dcachesize
sysctl -n hw.perflevel0.l2cachesize
```
**Output:**
```
131072
12582912
```
*(Bytes. Apple Silicon splits cores into Performance (`perflevel0`) and Efficiency (`perflevel1`) clusters.)*

128 KiB of L1 data cache per performance core, 12 MiB of L2 shared across the performance cluster. That's a tiny fraction of your RAM — which is the whole point of the pyramid.

**Look at a full summary.**
```bash
system_profiler SPHardwareDataType | grep -E "Model Name|Chip|Total Number of Cores|Memory"
```
**Output:**
```
      Model Name: MacBook Pro
      Chip: Apple M1 Pro
      Total Number of Cores: 8 (6 performance and 2 efficiency)
      Memory: 32 GB
```
*(Your model and memory will differ. Running without the grep filter also prints your Mac's serial number and hardware UUID — fine at home, careful if screen-sharing.)*

> **Why does the kernel have to expose this information at all? Why can't your programs just ask the CPU directly?**
> <details><summary>Answer</summary>
> User programs run in an unprivileged mode and can't talk to the hardware directly — that's the OS's job. The kernel reads this information at boot (from CPU instructions like <code>CPUID</code> on x86 or from the DeviceTree on Apple Silicon) and caches it in kernel memory. Commands like <code>sysctl</code> are thin readers over that kernel-maintained cache.
> </details>

### Exercise

Compute the ratio of your L1 data cache to your L2 cache. Instead of copying the byte counts by hand, capture them into shell variables first so this runs on your machine with your numbers.

```bash
L1=$(sysctl -n hw.perflevel0.l1dcachesize)
L2=$(sysctl -n hw.perflevel0.l2cachesize)
python3.12 -c "print($L2 / $L1)"
```
**Output:**
```
96.0
```
*(Your ratio will differ by chip.)*

<details><summary>Solution</summary>

`$(...)` runs a command and captures its output into a shell variable. `L1` now holds the byte count for your L1 data cache; `L2` holds your L2 size. We pass both into Python as `$L1` and `$L2` and let it do the division. On my Mac, L2 is 96× L1d. Your ratio will differ but land in the same order of magnitude. That's the pyramid: each tier down multiplies capacity by roughly 10-100×.

</details>

## Part 2 — Seeing RAM

### What is RAM?

RAM is the working scratchpad. Variables, open files, and running programs all live there while they're active. Students come in thinking of RAM as one number — "I have 16 GB." The real picture is layered: some is the kernel (wired memory, never reclaimed), some is active, some is inactive but droppable, and some is used as a disk cache.

### Run the demo

**How much RAM is installed?**
```bash
sysctl -n hw.memsize
```
**Output:**
```
34359738368
```
*(Bytes. Divide by 1024³ to get GiB.)*

**Look at the live breakdown.**
```bash
vm_stat | head -7
```
**Output:**
```
Mach Virtual Memory Statistics: (page size of 16384 bytes)
Pages free:                               78745.
Pages active:                            507575.
Pages inactive:                          444596.
Pages speculative:                        82603.
Pages throttled:                              0.
Pages wired down:                        152544.
```

`vm_stat` reports in *pages*, not bytes. The header line tells you the page size — 16384 bytes on Apple Silicon. Multiply any row by 16384 for bytes. `Pages free` is truly untouched (usually small — that's fine). `Pages active` is in use right now. `Pages inactive` is in use but cold and reclaimable. `Pages wired down` is kernel-pinned.

> **Why does the kernel keep so much RAM as disk cache instead of leaving it "free"?**
> <details><summary>Answer</summary>
> Free RAM is wasted RAM. If the kernel has empty pages, it uses them to cache recently-read disk blocks, so the next time you re-read the same file it comes from RAM (nanoseconds) instead of the disk (milliseconds). The moment an application actually needs memory, the kernel drops the cheapest cache pages and hands them over. That's why <code>Pages free</code> stays small on a machine that's been running a while — not a bug.
> </details>

### Exercise

Convert your `hw.memsize` from bytes into GiB (base-2) and GB (base-10). This is why a "32 GB" laptop reports less than 32 in Finder. Capture your byte count into a shell variable first.

```bash
b=$(sysctl -n hw.memsize)
python3.12 -c "b = $b; print(f'{b / 1024**3:.2f} GiB, {b / 1000**3:.2f} GB')"
```
**Output:**
```
32.00 GiB, 34.36 GB
```
*(Your values will differ by machine; the 7% gap between them will not.)*

<details><summary>Solution</summary>

`$(sysctl -n hw.memsize)` runs sysctl and captures the result into the variable `b`. Python divides `b` by 1024³ for GiB (binary, what the OS reports) and by 1000³ for GB (decimal, what marketing prints). The ratio GB/GiB is always ~1.074 — that's the 7% mismatch that explains a 500 GB SSD showing as 465 GiB.

</details>

## Part 3 — Seeing Disks

### What is a disk?

A disk is the slow, huge, persistent layer at the bottom of the memory hierarchy. Unlike RAM, it keeps its data when power is off. Unlike RAM, it's almost never used as one unbroken space — it's sliced into **partitions**, each holding a **filesystem**, each **mounted** at some directory in the tree.

### Run the demo

**Show the physical disks.**
```bash
diskutil list | head -20
```
**Output:**
```
/dev/disk0 (internal, physical):
   #:                       TYPE NAME                    SIZE       IDENTIFIER
   0:      GUID_partition_scheme                        *500.3 GB   disk0
   1:             Apple_APFS_ISC Container disk1         524.3 MB   disk0s1
   2:                 Apple_APFS Container disk3         494.4 GB   disk0s2
   3:        Apple_APFS_Recovery Container disk2         5.4 GB     disk0s3

/dev/disk3 (synthesized):
   #:                       TYPE NAME                    SIZE       IDENTIFIER
   0:      APFS Container Scheme -                      +494.4 GB   disk3
                                 Physical Store disk0s2
   1:                APFS Volume Macintosh HD - Data     218.0 GB   disk3s1
   2:                APFS Volume Macintosh HD            11.3 GB    disk3s3
   3:              APFS Snapshot com.apple.os.update-... 11.3 GB    disk3s3s1
   4:                APFS Volume Preboot                 8.1 GB     disk3s4
   5:                APFS Volume Recovery                2.0 GB     disk3s5
   6:                APFS Volume VM                      4.3 GB     disk3s6
```

`disk0` is the physical SSD. `disk3` is *not* a second physical disk — it's a "synthesized" view of the big APFS container on disk0s2. The six APFS volumes inside (Macintosh HD, Data, Preboot, Recovery, VM, and a snapshot) all share one 494 GB pool. That dynamic-sharing model is specific to APFS; on Windows or classic Linux each partition has a fixed size.

**How full is each mount?**
```bash
df -h | head -3
```
**Output:**
```
Filesystem        Size    Used   Avail Capacity iused ifree %iused  Mounted on
/dev/disk3s3s1   460Gi    10Gi   233Gi     5%    426k  2.4G    0%   /
devfs            202Ki   202Ki     0Bi   100%     698     0  100%   /dev
```

`df -h` asks each mounted filesystem how big it is and how full. The top row — `/`, backed by `disk3s3s1` — is the one that matters. `Avail` is the column you check when asking 'do I have room for this?'

> **Why does `diskutil list` show a "synthesized" `/dev/disk3` when you only have one physical SSD?**
> <details><summary>Answer</summary>
> APFS separates the <em>container</em> (one contiguous slice of disk0) from the <em>volumes</em> inside it. Volumes share the container's free space — you don't pre-decide how big each one gets. <code>diskutil</code> synthesizes a second device node so you can browse the container's volumes as if they were separate disks. One real SSD, one logical view.
> </details>

### Exercise

Find which partition backs your home directory.

```bash
df -h ~
```
**Output:**
```
Filesystem      Size    Used   Avail Capacity iused ifree %iused  Mounted on
/dev/disk3s3s1  460Gi    10Gi   233Gi     5%    426k  2.4G    0%   /
```

<details><summary>Solution</summary>

`~` expands to your home directory path. `df -h ~` names the single filesystem that actually contains it — on stock macOS it's the root volume. When you save a file, this is where its bytes land.

</details>

## Part 4 — Seeing a File as Bytes

### What is a file at the byte level?

Lecture 1.1 claimed that everything in a computer is bytes — text, images, programs, all of them. This Part is where you see it. `xxd` reads any file and prints its raw bytes as a three-column hex dump: offset on the left, the bytes in hex in the middle, and those same bytes shown as characters on the right when they happen to be printable ASCII.

### Run the demo

**Confirm you're in the lab folder.**

You `cd`'d into `1.5/` back in Setup. Every command in this Part will run from here. Quick aside on `cd` while we're at it: your shell stays in whatever directory you last `cd`'d to — running other commands (`echo`, `xxd`, `python3`) does *not* move you. If you ever lose track of where you are, `pwd` prints your current directory. `cd` with no argument snaps you back to your home folder. And `/tmp` — which you'll see in other tutorials — is the Unix convention for scratch files when you *don't* have a dedicated workspace like this one.

```bash
pwd
```
**Output:**
```
/path/to/course/1.5
```
*(The full path will vary depending on where you put the course materials; the last component should be `1.5`.)*

**Create a tiny text file and dump it.**
```bash
echo 'Hello!' > hello.txt
xxd hello.txt
```
**Output:**
```
00000000: 4865 6c6c 6f21 0a                        Hello!.
```

Left column: offset in the file (0, in hex). Middle: the 7 bytes as hex pairs — `48 65 6c 6c 6f 21 0a`. Right: `Hello!` plus a `.` because byte `0a` (newline) isn't printable. `48` is `H`, `65` is `e`, `21` is `!`. The file is those seven numbers and nothing else.

**See the same bytes as binary bits.**
```bash
xxd -b hello.txt
```
**Output:**
```
00000000: 01001000 01100101 01101100 01101100 01101111 00100001  Hello!
00000006: 00001010                                               .
```

`-b` swaps hex for binary. `01001000` is H. Eight flash memory cells on my SSD are storing that pattern right now. Lecture 1.2's "gates combine into flip-flops holding bits" is exactly this.

**Dump a PNG and find the magic number.**

The `sample.png` file in this folder is a 1×1 red pixel PNG — 69 bytes, a real openable image. We ship it with the lab so you don't have to download anything.

```bash
xxd -l 16 sample.png
```
**Output:**
```
00000000: 8950 4e47 0d0a 1a0a 0000 000d 4948 4452  .PNG........IHDR
```

The first 8 bytes — `89 50 4e 47 0d 0a 1a 0a` — are the PNG signature, identical in every PNG on Earth. `50 4e 47` spells `PNG` in ASCII (visible in the gutter). Every image viewer checks these bytes to identify the format — not the filename.

> **Why does `xxd` show `.` for byte `0a` (newline) in the ASCII column, even though it's a real byte?**
> <details><summary>Answer</summary>
> The ASCII column is only for <em>printable</em> characters — code points 0x20 (space) through 0x7E (tilde). Bytes below 0x20 are control characters: newline (0x0a), tab (0x09), null (0x00), escape (0x1b). They're real bytes and affect your terminal when printed, but they don't have a visible glyph. <code>xxd</code> shows <code>.</code> as a placeholder so the column stays aligned.
> </details>

### Exercise

Hex-dump any file of your own and see if you can identify the file type from its first few bytes.

```bash
xxd -l 32 ~/Downloads/SOME_FILE
```

<details><summary>Solution</summary>

Try a few:
- A Python script starts with `23 21` (`#!`).
- A JPEG starts with `ff d8 ff`.
- A PDF starts with `25 50 44 46` (`%PDF`).
- A compiled macOS executable starts with `cf fa ed fe` (Mach-O).
- A Linux binary starts with `7f 45 4c 46` (ELF).

Nearly every file format announces itself in its first handful of bytes. The [Wikipedia list of file signatures](https://en.wikipedia.org/wiki/List_of_file_signatures) has a big reference table.

</details>

## Part 5 — Same Bytes, Now in Python

### What does it mean for a number to be "in a base"?

When you write the number eighty, you can spell it four ways: `80` in decimal, `0x50` in hex, `0b1010000` in binary, or `P` via ASCII. All the *same number*. Python has one-line built-ins for every conversion.

You just saw `50 4e 47` spell `PNG` in xxd. This Part is where you prove it in Python.

### Run the demo

**One number, four views.**
```bash
python3.12 -c "print(hex(80))"
python3.12 -c "print(bin(80))"
python3.12 -c "print(ord('P'))"
python3.12 -c "print(chr(80))"
```
**Output:**
```
0x50
0b1010000
80
P
```

`hex()` and `bin()` take an integer and return a string with a `0x` or `0b` prefix. `ord()` returns the integer code point of a character; `chr()` is its inverse.

**Round-trip from string back to number.**
```bash
python3.12 -c "print(int('50', 16))"
python3.12 -c "print(int('01010000', 2))"
```
**Output:**
```
80
80
```

`int(s, base)` parses a string in any base 2–36. Both strings represent 80 in different bases.

**Format bytes the way xxd does.**
```bash
python3.12 -c "print(f'{15:02x}')"
python3.12 -c "print(f'{5:08b}')"
```
**Output:**
```
0f
00000101
```

`:02x` means hex, zero-padded to 2 digits — matches xxd's per-byte format. `:08b` means binary, 8 digits — one full byte of bits.

**Decode the PNG magic number you saw in Part 4.**
```bash
python3.12 -c "print(bytes.fromhex('89504e47'))"
```
**Output:**
```
b'\x89PNG'
```

`bytes.fromhex()` takes a hex string and returns the raw bytes. Python shows printable bytes as characters (`PNG`) and non-printable ones as `\xNN` escapes. Same four bytes xxd showed at the start of the PNG.

> **Why does Python print `b'\x89PNG'` instead of all four bytes as escapes, or all four as characters?**
> <details><summary>Answer</summary>
> Python's bytes-repr is a compromise. Printable ASCII bytes (0x20–0x7E) are shown as the character itself — <code>PNG</code>. Bytes outside that range are shown as <code>\xNN</code> hex escapes — <code>\x89</code>. It's the shortest form that still round-trips: copy <code>b'\x89PNG'</code> back into Python and you get the same 4-byte sequence.
> </details>

### Exercise

Convert your own first initial to decimal, hex, and binary in one Python line.

```bash
python3.12 -c "c='J'; print(f'{c!r} -> dec={ord(c)}, hex={ord(c):02x}, bin={ord(c):08b}')"
```
**Output:**
```
'J' -> dec=74, hex=4a, bin=01001010
```

<details><summary>Solution</summary>

`J` is code 74, `4a` in hex, `01001010` in binary. Replace `J` with your initial. Try uppercase and lowercase of the same letter — the difference is always exactly one bit, the `0x20` bit. That single bit is the ASCII case toggle.

</details>

## Part 1 — Seeing the CPU

### What is the CPU?

The CPU is the chip that runs your programs. Its data sheet has three numbers that matter for this lab: the **architecture** (the instruction set — `x86_64` on most Intel/AMD chips, `aarch64` on ARM systems like Raspberry Pi or AWS Graviton), the **core count** (how many instructions can be in flight at once), and the **cache hierarchy** (tiny fast memory sitting next to the cores).

Lecture 1.4 drew the pyramid: registers → L1 → L2 → L3 → RAM → disk. Each tier is bigger than the one above, and each is slower. L1 is usually 32–128 KiB *per core* and takes a handful of clock cycles to read. L3 is typically a few MiB and shared across the whole chip, taking ten times longer. Your OS knows every one of these numbers and will print them on request.

### Run the demo

**Read the chip model and architecture.**
```bash
lscpu | grep -E 'Architecture|Model name|^CPU\(s\):'
```
**Output:**
```
Architecture:                         x86_64
CPU(s):                               16
Model name:                           AMD Ryzen 7 5700G with Radeon Graphics
```
*(Your architecture, CPU count, and chip name will differ.)*

**Count your cores.**
```bash
nproc
```
**Output:**
```
16
```

`nproc` is the no-frills CPU counter — same number `lscpu` reported under `CPU(s):`. Both count *logical* CPUs. On a chip with SMT (AMD) or Hyper-Threading (Intel), logical is usually double physical — each physical core presents as two to the kernel scheduler. 16 logical with 8 physical means this chip is using SMT.

**Read the cache sizes.**
```bash
lscpu -C
```
**Output:**
```
NAME ONE-SIZE ALL-SIZE WAYS TYPE        LEVEL  SETS PHY-LINE COHERENCY-SIZE
L1d       32K     256K    8 Data            1    64        1             64
L1i       32K     256K    8 Instruction     1    64        1             64
L2       512K       4M    8 Unified         2  1024        1             64
L3        16M      16M   16 Unified         3 16384        1             64
```

Four rows for four cache levels. `ONE-SIZE` is per-instance; `ALL-SIZE` is total across instances. L1 splits into data (L1d) and instructions (L1i), 32 KiB each per core. L2 is 512 KiB per core, unified. L3 is 16 MiB shared across the whole chip. That's the pyramid in numbers.

> **Why does the kernel have to expose this information at all? Why can't your programs just ask the CPU directly?**
> <details><summary>Answer</summary>
> User programs run in an unprivileged mode and can't talk to the hardware directly — that's the OS's job. The kernel reads this information at boot (from CPU instructions like <code>CPUID</code> on x86 or from ACPI / DeviceTree on ARM) and caches it in <code>/proc/cpuinfo</code> and <code>/sys/devices/system/cpu/</code>. Commands like <code>lscpu</code> are thin readers over those kernel-maintained virtual files.
> </details>

### Exercise

Compute the ratio of your L1 data cache to your L2 cache. Instead of copying the sizes by hand, capture them into shell variables so this runs on your machine with your numbers.

```bash
L1=$(lscpu -C -B | awk '$1=="L1d"{print $2}')
L2=$(lscpu -C -B | awk '$1=="L2"{print $2}')
python3 -c "print($L2 / $L1)"
```
**Output:**
```
16.0
```
*(Your ratio will differ by chip.)*

<details><summary>Solution</summary>

`lscpu -C -B` prints cache sizes in raw bytes — the `-B` flag turns off the human-readable K/M suffixes that would break integer division. `awk` picks the L1d and L2 rows by name and grabs the second column (ONE-SIZE). `$(...)` captures each into a shell variable, and Python divides. Your ratio will land in the same order of magnitude as the example. That's the pyramid: each tier down multiplies capacity by roughly 10-100×.

</details>

## Part 2 — Seeing RAM

### What is RAM?

RAM is the working scratchpad. Variables, open files, and running programs all live there while they're active. Students come in thinking of RAM as one number — "I have 16 GB." The real picture is layered: some is the kernel, some is active (in use right now), some is held as disk-block cache (reclaimable on demand), and some is truly free.

### Run the demo

**How much RAM is installed?**
```bash
grep MemTotal /proc/meminfo
```
**Output:**
```
MemTotal:       65178768 kB
```
*(Kibibytes — Linux uses `kB` as the label but it means 1024 bytes, not 1000. Divide by 1024² for GiB.)*

**Look at the live breakdown.**
```bash
free -h
```
**Output:**
```
               total        used        free      shared  buff/cache   available
Mem:            62Gi       6.6Gi        26Gi        48Mi        28Gi        54Gi
Swap:             0B          0B          0B
```

Six columns. `used` is committed to running programs. `free` is truly untouched. `buff/cache` is disk blocks the kernel holds in RAM in case you re-read them. `available` is `free + reclaimable cache` — the number you actually care about when asking 'can I open this?' Empty memory is wasted memory, so a healthy Linux system keeps `free` small.

**Peek at the raw kernel counters.**
```bash
head -5 /proc/meminfo
```
**Output:**
```
MemTotal:       65178768 kB
MemFree:        28204856 kB
MemAvailable:   57427248 kB
Buffers:         1717748 kB
Cached:         25486936 kB
```

`/proc/meminfo` is a virtual file the kernel generates on demand. `free` is a small parser over these rows.

> **Why does the kernel keep so much RAM as disk cache instead of leaving it "free"?**
> <details><summary>Answer</summary>
> Free RAM is wasted RAM. If the kernel has empty pages, it uses them to cache recently-read disk blocks, so the next time you re-read the same file it comes from RAM (nanoseconds) instead of the disk (milliseconds). The moment an application actually needs memory, the kernel drops the cheapest cache pages and hands them over. That's why <code>MemFree</code> stays small on a machine that's been running a while — not a bug.
> </details>

### Exercise

Convert your `MemTotal` from kB into GiB (base-2) and GB (base-10). This is why a "64 GB" module reports less than 64 in a file manager. Capture MemTotal into a shell variable first.

```bash
kb=$(awk '/^MemTotal/{print $2}' /proc/meminfo)
python3 -c "kb = $kb; print(f'{kb / 1024**2:.2f} GiB, {kb * 1024 / 1000**3:.2f} GB')"
```
**Output:**
```
62.16 GiB, 66.74 GB
```
*(Your values will differ by machine; the 7% gap between them will not.)*

<details><summary>Solution</summary>

`awk` pulls the MemTotal line from `/proc/meminfo` and prints the second column (the number in kB). `$(...)` captures that into `kb`, which Python divides two ways: by 1024² for GiB (binary, what the OS reports) and by 1000³ after multiplying by 1024 for GB (decimal, what marketing prints). The OS reports slightly less than installed — a few MiB go to kernel and firmware reservations at boot.

</details>

## Part 3 — Seeing Disks

### What is a disk?

A disk is the slow, huge, persistent layer at the bottom of the memory hierarchy. Unlike RAM, it keeps its data when power is off. Unlike RAM, it's almost never used as one unbroken space — it's sliced into **partitions**, each holding a **filesystem**, each **mounted** at some directory in the tree.

### Run the demo

**Show the physical disks and partitions.**
```bash
lsblk
```
**Output:**
```
NAME        MAJ:MIN RM   SIZE RO TYPE MOUNTPOINTS
sda           8:0    0     5G  0 disk 
sdb           8:16   0    10G  0 disk 
sdc           8:32   0     1G  0 disk 
nvme0n1     259:0    0 931.5G  0 disk 
|-nvme0n1p1 259:1    0     1G  0 part 
|-nvme0n1p2 259:2    0     2G  0 part 
`-nvme0n1p3 259:3    0 928.5G  0 part 
```
*(Your tree will look different. On a typical laptop you'd see one `nvme0n1`; on WSL2 you'd see a single `sda` — the virtual disk Microsoft gives the Linux VM.)*

Each row with `TYPE disk` is a physical device. Nested under it are the partitions. `nvme0n1` is this machine's main SSD (931 GiB); its big partition `p3` holds `/`.

**How full is each mount?**
```bash
df -h | head -3
```
**Output:**
```
Filesystem                         Size  Used Avail Use% Mounted on
overlay                            913G   40G  828G   5% /
tmpfs                               64M     0   64M   0% /dev
```

`df -h` asks each mounted filesystem how big it is and how full. The top row is the one that matters — `/` with 913 GB total, 40 used, 828 free. `Avail` is the column you check when asking 'do I have room?' Identical command and columns on macOS and Linux.

> **Why might `lsblk` inside WSL2 show only a single `sda` device, even if the Windows host has multiple drives?**
> <details><summary>Answer</summary>
> WSL2 is a real Linux kernel running inside a lightweight Hyper-V virtual machine. Microsoft gives that VM a single virtual disk (<code>sda</code>) and mounts the Windows C: drive under <code>/mnt/c</code> through a special filesystem driver. <code>lsblk</code> shows block devices visible to the Linux VM, not devices on the Windows host — so you see one <code>sda</code> regardless of how many physical drives your Windows machine has.
> </details>

### Exercise

**Output:**

<details><summary>Solution</summary>

## Part 4 — Seeing a File as Bytes

### What is a file at the byte level?

Lecture 1.1 claimed that everything in a computer is bytes — text, images, programs, all of them. This Part is where you see it. `xxd` reads any file and prints its raw bytes as a three-column hex dump: offset on the left, the bytes in hex in the middle, and those same bytes shown as characters on the right when they happen to be printable ASCII.

### Run the demo

**Confirm you're in the lab folder.**

You `cd`'d into `1.5/` back in Setup. Every command in this Part will run from here. Quick aside on `cd` while we're at it: your shell stays in whatever directory you last `cd`'d to — running other commands (`echo`, `xxd`, `python3`) does *not* move you. If you ever lose track of where you are, `pwd` prints your current directory. `cd` with no argument snaps you back to your home folder. And `/tmp` — which you'll see in other tutorials — is the Unix convention for scratch files when you *don't* have a dedicated workspace like this one.

```bash
pwd
```
**Output:**
```
/path/to/course/1.5
```
*(The full path will vary depending on where you put the course materials; the last component should be `1.5`.)*

**Create a tiny text file and dump it.**
```bash
echo 'Hello!' > hello.txt
xxd hello.txt
```
**Output:**
```
00000000: 4865 6c6c 6f21 0a                        Hello!.
```

Left column: offset in the file (0, in hex). Middle: the 7 bytes as hex pairs — `48 65 6c 6c 6f 21 0a`. Right: `Hello!` plus a `.` because byte `0a` (newline) isn't printable. `48` is `H`, `65` is `e`, `21` is `!`. The file is those seven numbers and nothing else.

**See the same bytes as binary bits.**
```bash
xxd -b hello.txt
```
**Output:**
```
00000000: 01001000 01100101 01101100 01101100 01101111 00100001  Hello!
00000006: 00001010                                               .
```

`-b` swaps hex for binary. `01001000` is H. Eight flash memory cells on my SSD are storing that pattern right now. Lecture 1.2's "gates combine into flip-flops holding bits" is exactly this.

**Dump a PNG and find the magic number.**

The `sample.png` file in this folder is a 1×1 red pixel PNG — 69 bytes, a real openable image. We ship it with the lab so you don't have to download anything.

```bash
xxd -l 16 sample.png
```
**Output:**
```
00000000: 8950 4e47 0d0a 1a0a 0000 000d 4948 4452  .PNG........IHDR
```

The first 8 bytes — `89 50 4e 47 0d 0a 1a 0a` — are the PNG signature, identical in every PNG on Earth. `50 4e 47` spells `PNG` in ASCII (visible in the gutter). Every image viewer checks these bytes to identify the format — not the filename.

> **Why does `xxd` show `.` for byte `0a` (newline) in the ASCII column, even though it's a real byte?**
> <details><summary>Answer</summary>
> The ASCII column is only for <em>printable</em> characters — code points 0x20 (space) through 0x7E (tilde). Bytes below 0x20 are control characters: newline (0x0a), tab (0x09), null (0x00), escape (0x1b). They're real bytes and affect your terminal when printed, but they don't have a visible glyph. <code>xxd</code> shows <code>.</code> as a placeholder so the column stays aligned.
> </details>

### Exercise

Hex-dump any file of your own and see if you can identify the file type from its first few bytes.

<details><summary>Solution</summary>

## Part 5 — Same Bytes, Now in Python

### What does it mean for a number to be "in a base"?

When you write the number eighty, you can spell it four ways: `80` in decimal, `0x50` in hex, `0b1010000` in binary, or `P` via ASCII. All the *same number*. Python has one-line built-ins for every conversion.

You just saw `50 4e 47` spell `PNG` in xxd. This Part is where you prove it in Python.

### Run the demo

**One number, four views.**
```bash
python3 -c "print(hex(80))"
python3 -c "print(bin(80))"
python3 -c "print(ord('P'))"
python3 -c "print(chr(80))"
```
**Output:**
```
0x50
0b1010000
80
P
```

`hex()` and `bin()` take an integer and return a string with a `0x` or `0b` prefix. `ord()` returns the integer code point of a character; `chr()` is its inverse.

**Round-trip from string back to number.**
```bash
python3 -c "print(int('50', 16))"
python3 -c "print(int('01010000', 2))"
```
**Output:**
```
80
80
```

`int(s, base)` parses a string in any base 2–36. Both strings represent 80 in different bases.

**Format bytes the way xxd does.**
```bash
python3 -c "print(f'{15:02x}')"
python3 -c "print(f'{5:08b}')"
```
**Output:**
```
0f
00000101
```

`:02x` means hex, zero-padded to 2 digits — matches xxd's per-byte format. `:08b` means binary, 8 digits — one full byte of bits.

**Decode the PNG magic number you saw in Part 4.**
```bash
python3 -c "print(bytes.fromhex('89504e47'))"
```
**Output:**
```
b'\x89PNG'
```

`bytes.fromhex()` takes a hex string and returns the raw bytes. Python shows printable bytes as characters (`PNG`) and non-printable ones as `\xNN` escapes. Same four bytes xxd showed at the start of the PNG.

> **Why does Python print `b'\x89PNG'` instead of all four bytes as escapes, or all four as characters?**
> <details><summary>Answer</summary>
> Python's bytes-repr is a compromise. Printable ASCII bytes (0x20–0x7E) are shown as the character itself — <code>PNG</code>. Bytes outside that range are shown as <code>\xNN</code> hex escapes — <code>\x89</code>. It's the shortest form that still round-trips: copy <code>b'\x89PNG'</code> back into Python and you get the same 4-byte sequence.
> </details>

### Exercise

Convert your own first initial to decimal, hex, and binary in one Python line.

```bash
python3 -c "c='J'; print(f'{c!r} -> dec={ord(c)}, hex={ord(c):02x}, bin={ord(c):08b}')"
```
**Output:**
```
'J' -> dec=74, hex=4a, bin=01001010
```

<details><summary>Solution</summary>

`J` is code 74, `4a` in hex, `01001010` in binary. Replace `J` with your initial. Try uppercase and lowercase of the same letter — the difference is always exactly one bit, the `0x20` bit. That single bit is the ASCII case toggle.

</details>

## Putting it together (Linux / WSL)

| Concept | What it lets you see | Key tool | Where the data comes from |
|---|---|---|---|
| CPU and caches | Chip, cores, cache sizes | `lscpu`, `nproc` | `/proc/cpuinfo`, `/sys/devices/system/cpu/` |
| RAM | Total and live memory usage | `free -h`, `/proc/meminfo` | Kernel memory subsystem |
| Disks | Physical devices, partitions, mounts | `lsblk`, `df -h` | `/sys/block/`, kernel block-device registry |
| Bytes in a file | Hex + ASCII columns, or bits | `xxd` | The file itself |
| Bytes in code | Dec/hex/bin/char conversions | Python built-ins | Python standard library |

The first three rows are different questions you ask the OS about the hardware. The last two rows are different views of the same thing: every file is a sequence of bytes, every byte is a number, every number has four equivalent written forms.
