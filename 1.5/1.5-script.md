## PART 1 — THE CPU

### What is a CPU?

**SPEAK**
> "Every program you run is a list of instructions. The CPU — Central Processing Unit — is the thing that executes them. But 'the CPU' is actually a stack of layers. The outermost layer is the socket — the physical chip slot on the motherboard. Inside the chip are multiple cores: independent processing engines. And each core can run two threads simultaneously using a technique called Simultaneous Multithreading. The OS sees each thread as a separate CPU. So when you hear '8-core processor', you might actually have 16 logical CPUs. Let's see this on our machine."

### Run the demo

**TYPE**
```bash
nproc
```

**OUTPUT**
```
16
```

**EXPLAIN**
> "16 logical CPUs. That is what the OS sees — 16 independent workers it can schedule tasks on. Now let's see where that number comes from."

**TYPE**
```bash
lscpu | grep -E "^CPU\(s\):|Thread|Core|Socket|Model name|Vendor"
```

**OUTPUT**
```
CPU(s):                               16
Vendor ID:                            AuthenticAMD
Model name:                           AMD Ryzen 7 5700G with Radeon Graphics
Thread(s) per core:                   2
Core(s) per socket:                   8
Socket(s):                            1
```

*(Your CPU model and counts will differ — structure is the same.)*

**EXPLAIN**
> "There is the math: 1 socket, times 8 cores per socket, times 2 threads per core — equals 16. nproc and CPU(s) agree. The 'Thread(s) per core: 2' line is the SMT — Simultaneous Multithreading. One physical core is pretending to be two by keeping two register files while sharing the execution units."

**SPEAK**
> "Now let's look at the cache hierarchy — the fast memory inside the chip that keeps the CPU from waiting on RAM."

**TYPE**
```bash
lscpu -C
```

**OUTPUT**
```
NAME ONE-SIZE ALL-SIZE WAYS TYPE        LEVEL  SETS PHY-LINE COHERENCY-SIZE
L1d       32K     256K    8 Data            1    64        1             64
L1i       32K     256K    8 Instruction     1    64        1             64
L2       512K       4M    8 Unified         2  1024        1             64
L3        16M      16M   16 Unified         3 16384        1             64
```

*(Your sizes will differ based on CPU model.)*

**EXPLAIN**
> "Four rows, three cache levels. L1d is 32 kilobytes per core — dedicated to data. L1i is 32 kilobytes per core — dedicated to instructions. L2 is 512 kilobytes per core — unified, handles both. L3 is 16 megabytes and the ONE-SIZE equals ALL-SIZE — because there is only one L3, shared across all 8 cores. The COHERENCY-SIZE column, 64 bytes on every row, is the cache line size: data always moves in 64-byte chunks, even if you only touch one byte. That is how the cache exploits spatial locality — you asked for byte 0, so the chip fetches bytes 0 through 63 all at once."

### Exercise — Cache Math

**SPEAK**
> "Quick exercise: convert 32 kilobytes to bytes using Python. This connects the lscpu output to actual numbers you can reason about."

**TYPE**
```bash
python3 -c "print(32 * 1024, 'bytes of L1d per core')"
```

**OUTPUT**
```
32768 bytes of L1d per core
```

**EXPLAIN**
> "32,768 bytes. For comparison, a typical university lecture note in plain text is a few thousand bytes. The L1 cache on one core is large enough to hold the entire thing with room to spare — and it is accessed in 1 to 4 clock cycles instead of the 60 to 100 cycles a trip to RAM would cost."

## PART 2 — MEMORY

### What is RAM?

**SPEAK**
> "RAM is the scratchpad — bigger than cache but still volatile, meaning it forgets everything when power cuts. The kernel manages RAM as a shared resource. Every running process gets a slice. Any RAM left over gets used as a disk cache — recently read file data stays in RAM so repeated reads are instant instead of going back to disk. Here is the twist: on a Linux system, the 'free' column in most memory tools is almost always tiny, and that looks alarming, but it usually means nothing is wrong. Nearly all that apparently used memory is disk cache, which the kernel will reclaim instantly the moment any process needs more space. The column that actually matters is 'available' — not 'free'."

### Run the demo

**SPEAK**
> "Let's start with the raw kernel numbers."

**TYPE**
```bash
cat /proc/meminfo | head -10
```

**OUTPUT**
```
MemTotal:       65178768 kB
MemFree:        28082812 kB
MemAvailable:   57248468 kB
Buffers:         1716728 kB
Cached:         25447016 kB
SwapCached:            0 kB
Active:          7662832 kB
Inactive:       24973704 kB
Active(anon):    5536308 kB
Inactive(anon):     1320 kB
```

*(Your values will differ.)*

**EXPLAIN**
> "This is the file that every memory tool on Linux reads from. MemTotal is physical RAM — 65 gigabytes here. MemFree is completely unoccupied — 28 GB. But MemAvailable is 57 GB — way bigger than MemFree. That difference, about 25 GB, is the Cached line — disk data sitting in RAM. The kernel will hand that back immediately when needed. This machine has 57 gigabytes the OS can give to a new program without touching swap. That is the number that matters."

**TYPE**
```bash
free -h
```

**OUTPUT**
```
               total        used        free      shared  buff/cache   available
Mem:            62Gi       6.8Gi        26Gi        48Mi        28Gi        54Gi
Swap:             0B          0B          0B
```

*(Your values will differ.)*

**EXPLAIN**
> "Same data, easier to read. The buff/cache column combines Buffers and Cached from /proc/meminfo — 28 gigabytes of disk data living in RAM. Swap: 0 bytes used, which is normal on a machine with adequate RAM. If you see any swap usage on a system with 62 gigabytes of RAM, something is wrong."

**SPEAK**
> "Now let's look at the physical memory address ranges."

**TYPE**
```bash
lsmem
```

**OUTPUT**
```
RANGE                                 SIZE  STATE REMOVABLE BLOCK
0x0000000000000000-0x000000107fffffff  66G online       yes  0-32

Memory block size:         2G
Total online memory:      66G
Total offline memory:      0B
```

*(Your address range and size will differ.)*

**EXPLAIN**
> "lsmem shows the physical address range your RAM occupies. This machine has 66 gigabytes starting at address zero and running to 0x107fffffff. The 'REMOVABLE: yes' means the memory controller on this hardware supports taking individual 2-gigabyte blocks offline without rebooting — a server feature for replacing faulty DIMMs on live systems. On a laptop you would still see this column, but it is rarely used. Total online: 66 gigabytes. Total offline: zero."

```bash
python3 -c "
```

## PART 3 — FILES ARE JUST BYTES

### What is a block device?

**SPEAK**
> "A disk is a large sequence of fixed-size locations called sectors. The kernel exposes each disk as a block device — a file in /dev/ that you can read in fixed chunks. lsblk shows these as a tree: the whole disk on the left, its partitions indented beneath it. Then we are going to use xxd to open a real file and see the individual bytes inside it. And I want to show you something interesting about the first few bytes of any file — they tell you what kind of file it is, independent of the filename."

### Run the demo

**TYPE**
```bash
lsblk
```

**OUTPUT**
```
NAME        MAJ:MIN RM   SIZE RO TYPE MOUNTPOINTS
sda           8:0    0     5G  0 disk 
sdb           8:16   0    10G  0 disk 
sdc           8:32   0     1G  0 disk 
nvme0n1     259:0    0 931.5G  0 disk 
|-nvme0n1p1 259:1    0     1G  0 part 
|-nvme0n1p2 259:2    0     2G  0 part 
`-nvme0n1p3 259:3    0 928.5G  0 part
```

*(Your disk names, sizes, and MOUNTPOINTS will differ. Your Ubuntu VM will show `/`, `/boot`, and `/boot/efi` in the MOUNTPOINTS column.)*

**EXPLAIN**
> "The tree structure shows nvme0n1 — an NVMe SSD, 931.5 gigabytes — with three partitions inside it. nvme0n1p1 is 1 gigabyte, probably the EFI system partition. nvme0n1p2 is 2 gigabytes, probably /boot. nvme0n1p3 is 928.5 gigabytes — the main root filesystem. The MAJ:MIN column is the major and minor device number: a pair of integers the kernel uses to route I/O requests to the correct driver. The sda, sdb, sdc disks on the left are additional block devices visible in this environment — they have no partitions because they have not been formatted yet."

**SPEAK**
> "Now let's open a file and see its bytes."

**TYPE**
```bash
echo "Hello World" > test.txt
xxd test.txt
```

**OUTPUT**
```
00000000: 4865 6c6c 6f20 576f 726c 640a            Hello World.
```

**EXPLAIN**
> "Three columns. The leftmost is the offset — the byte position of the first byte on this line, in hexadecimal. The middle is 16 bytes displayed as hex pairs. The right side is those same bytes decoded as ASCII. 'H' is 48 in hex, 'e' is 65 in hex, 'l' is 6c, 'o' is 6f, space is 20. The dot at the end of the ASCII column — that is the newline character, hex 0a, which echo automatically appends. Non-printable characters always show as a dot."

**SPEAK**
> "I want to call out one thing that trips everyone up. If this line were longer, the second line would start at offset 00000010. Students always read that as 'byte 10' but it is not."

**TYPE**
```bash
python3 -c "print(0x10)"
```

**OUTPUT**
```
16
```

**EXPLAIN**
> "Offset 00000010 is hex for 16. Hex 10 is one-sixteen plus zero-ones. The offset column counts bytes, but it writes them in hexadecimal. Always verify an unfamiliar hex offset by evaluating it in Python."

**TYPE**
```bash
xxd -g 1 test.txt
```

**OUTPUT**
```
00000000: 48 65 6c 6c 6f 20 57 6f 72 6c 64 0a              Hello World.
```

**EXPLAIN**
> "The -g 1 flag changes the group size to one byte. Now each hex pair stands alone with a space before the next one. This is the easiest format for counting: 48, 65, 6c, 6c, 6f, 20, 57, 6f, 72, 6c, 64, 0a — twelve bytes. Eleven printable characters plus the newline."

**SPEAK**
> "Now let's look at the magic number of a real binary. Every Linux program starts with the same four bytes."

**TYPE**
```bash
xxd -l 16 /bin/ls
```

**OUTPUT**
```
00000000: 7f45 4c46 0201 0100 0000 0000 0000 0000  .ELF............
```

**EXPLAIN**
> "First four bytes: 7f, 45, 4c, 46. Look at the ASCII column: .ELF. The dot is hex 7f, which is not printable. ELF — Executable and Linkable Format — is the file format for Linux programs. The OS checks these exact four bytes before attempting to execute any file. If a file's first bytes are not 7f 45 4c 46, the kernel refuses to run it, regardless of what the filename says. Extensions are a human convention; magic bytes are what the OS trusts."

### Exercise — Name File Bytes

**TYPE**

```bash
echo -n "Alice" > name.txt

python3 -c "print(hex(ord('A')))"
```

**OUTPUT**
```

```

**EXPLAIN**
> "The -n flag suppresses the newline, so the file is exactly 5 bytes — just the letters. First byte: 41. Python's hex(ord('A')) also gives 0x41. Same byte, two views. xxd and Python are both reading the same raw integer: 65 decimal, 41 hex, 01000001 binary, 'A' in ASCII."

## PART 4 — NUMBER COSTUMES

### What is a number base?

**SPEAK**
> "The number 65 is a quantity. It means 65 things. But you can write it many ways. Decimal uses digits 0 through 9. Binary uses only 0 and 1 — matching the physical switches in the hardware. Hexadecimal uses 0 through F, so two hex digits cover exactly one byte. And ASCII is just an agreement: the integer 65 means the letter A. These are not different types of data — they are different costumes for the same integer. Python makes it trivial to switch between them."

### Run the demo

**TYPE**
```bash
python3 -c "n=65; print('dec:', n, '| bin:', format(n,'08b'), '| hex:', format(n,'02x'), '| char:', chr(n))"
```

**OUTPUT**
```
dec: 65 | bin: 01000001 | hex: 41 | char: A
```

**EXPLAIN**
> "All four representations of the same integer. format(n, '08b') formats as binary with 8-digit zero padding — that is one full byte of bits. format(n, '02x') formats as two-digit lowercase hex — one full byte. chr(65) applies the ASCII agreement: 65 means A. And the decimal is just plain 65. Same quantity, four costumes."

**TYPE**
```bash
python3 -c "
for ch in 'Hi!':
    print(f'{ch!r:4}  dec={ord(ch):3}  hex={ord(ch):02x}  bin={ord(ch):08b}')
"
```

**OUTPUT**
```
'H'   dec= 72  hex=48  bin=01001000
'i'   dec=105  hex=69  bin=01101001
'!'   dec= 33  hex=21  bin=00100001
```

**EXPLAIN**
> "ord() takes a character and gives you its integer code. Notice 'H' has hex code 48 — the same 48 xxd printed in the last part when we looked at 'Hello World'. That is not a coincidence. The xxd hex column and ord() are both reading the same byte from the file — 72 in decimal, 48 in hex, 01001000 in binary, 'H' in ASCII. Four costumes, same integer."

**SPEAK**
> "Now let's go the other direction — parse binary and hex strings back to decimal."

**TYPE**
```bash
python3 -c "print(int('01000001', 2), int('41', 16), int('0b01000001', 0))"
```

**OUTPUT**
```
65 65 65
```

**EXPLAIN**
> "int() with a second argument is the base — 2 for binary, 16 for hex. The third form passes 0 as the base, which tells Python to auto-detect from the prefix: 0b means binary, 0x means hex. All three give 65."

**SPEAK**
> "Now I want to show you the bytes object — Python's way of holding raw binary data."

**TYPE**
```bash
python3 -c "b=b'Hi!'; print(b, b[0], b.hex(), list(b))"
```

**OUTPUT**
```
b'Hi!' 72 486921 [72, 105, 33]
```

**EXPLAIN**
> "b'Hi!' is a bytes object. It looks like a string but it is not. When you index it — b[0] — you get the integer 72, not the character H. .hex() gives the entire sequence as a hex string with no spaces: 48, 69, 21. list(b) shows each byte as a decimal integer: 72, 105, 33. This is the raw binary view of the text Hi! — the same values xxd would print."

**SPEAK**
> "Let's do the final connection — xxd and Python reading the same file."

**TYPE**
```bash
echo -n "Hi!" > hi.txt
xxd hi.txt
python3 -c "print([format(b,'02x') for b in open('hi.txt','rb').read()])"
```

**OUTPUT**
```
00000000: 4869 21                                  Hi!
['48', '69', '21']
```

**EXPLAIN**
> "xxd shows 48, 69, 21. Python's list comprehension formats each byte as two-digit hex and shows the same values: '48', '69', '21'. The open call uses 'rb' — read binary — which gives raw integers instead of decoded text. Both tools are reading the same three bytes from the same file. There is nothing magic about xxd — it is just a hex formatter. Python can do the same thing."

### Exercise — Case Bit Trick

**SPEAK**
> "Here is a puzzle. What is the hex code for uppercase B? Make a prediction, then verify with Python. Then flip bit 5 with XOR and see what happens."

**TYPE**
```bash
python3 -c "print(hex(ord('B')))"
python3 -c "print(chr(ord('B') ^ 0x20))"
```

**OUTPUT**
```
0x42
b
```

**EXPLAIN**
> "A is 0x41, B is 0x42 — one more. Now chr(ord('B') XOR 0x20): XOR flips the bits that are set in the mask. 0x20 is bit 5. 0x42 is 01000010. Flip bit 5: 01100010 = 0x62 = 98 = lowercase b. This works for any letter. The ASCII designers placed uppercase A-Z at codes 65-90 and lowercase a-z at codes 97-122, with a gap of exactly 32. 32 is 0x20, which is bit 5. Flipping one bit converts case — a deliberate decision made in 1963."

## PUTTING IT TOGETHER

**SPEAK**
> "Everything you inspected today is the same idea in different costumes. lscpu reads the kernel's CPU topology and gives you the sockets-cores-threads structure behind the logical CPU count. free reads /proc/meminfo and the key insight is available, not free. lsblk reads sysfs and shows the block device tree. xxd reads raw file bytes and shows them as hex. And Python's builtins let you convert the same integer among decimal, binary, hex, and ASCII with one function call. The hex code xxd prints for the letter H in a text file is the same 0x48 that ord('H') returns in Python — because a file is just integers, and integers wear costumes."
