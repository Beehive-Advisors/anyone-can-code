## PART 1 — SEEING THE MACHINE

### What is "the machine"?

**SPEAK**
> "Before we run anything, let's name what we're going to see. Your program eventually runs on a physical chip. That chip has an architecture — `x86_64` or `arm64` — a number of cores, and a cache hierarchy: small, fast memory sitting next to the cores."
>
> "Below the caches is RAM — much bigger, somewhat slower. Below RAM is disk — much bigger still, much slower still. Those four tiers are the 'memory hierarchy' from lecture 1.4. This part of the lab shows you those four tiers with real numbers from the machine you're sitting at."

### Run the demo

```bash
bash hardware_demo.sh
```

> "This is the CPU section. `Brand` is the chip — on my machine, Apple M1 Pro. Yours could be an Intel Core i7, an AMD Ryzen, an Apple M3 — whatever you have."

>
> "Cache line is 128 bytes — that's how much data moves between cache levels at once. And those five cache lines below show the size of L1 and L2 on the performance cores and efficiency cores. On Apple Silicon, the efficiency cores have their own smaller caches. The 'per cluster' note means the L2 is shared between the cores in that cluster, not per core."

```
    L1 cache:  ~128 KiB   per core         (fastest, smallest)
```

## PART 2 — EVERY FILE IS BYTES

### What does it mean that a file is bytes?

**SPEAK**
> "Lecture 1.1 made the claim that everything in a computer is bytes. Text, images, programs — all of it. This part proves that claim by opening a plain-text file and showing you the exact numeric value of every byte in it. The tool is called `xxd` — hex dump. It's already on your machine."

### Run the demo

**TYPE**
```bash
bash bytes_demo.sh
```

**OUTPUT**
```

═══════════════════════════════════════════════════════════════
SECTION A — Every letter is a byte
═══════════════════════════════════════════════════════════════
  Wrote /Users/giacobbe/src/courses/anyone-can-code/1.5/hello.txt  (14 bytes)

  --- xxd hello.txt ---
00000000: 4865 6c6c 6f2c 2077 6f72 6c64 210a       Hello, world!.
```
*(Your path will differ; the hex bytes are identical on every machine.)*

**EXPLAIN**
> "The script wrote a 14-byte file containing `Hello, world!` plus a newline. Then ran `xxd` on it."
>
> "Three columns. The offset on the left — `00000000` — is where in the file we are, in hex. We're at byte 0. The middle column is the hex content: `48 65 6c 6c 6f 2c 20 77 6f 72 6c 64 21 0a`. The right column is the ASCII gutter — each byte as its printable character if one exists: `Hello, world!`, then a dot for the newline because newlines aren't printable."
>
> "The hex matches the text letter for letter. 0x48 is H. 0x65 is e. Every lowercase L is 0x6c. 0x20 is a space. 0x21 is an exclamation point. 0x0a is the newline. That's it. That's what the file is on your disk."

**OUTPUT**
```

═══════════════════════════════════════════════════════════════
SECTION B — One byte, four ways: char, hex, decimal, binary
═══════════════════════════════════════════════════════════════
  The first byte of hello.txt is 'H'.
    As hex:     0x48
    As decimal: 72
    As binary:  01001000
    As char:    H
```

**EXPLAIN**
> "Zoom in on one byte. 0x48, the first byte of the file, is also the number 72 in decimal, also the pattern `01001000` in binary, also the letter H in ASCII. All four rows are the same value, just dressed differently."

**OUTPUT**
```

═══════════════════════════════════════════════════════════════
SECTION C — See the actual bits with xxd -b
═══════════════════════════════════════════════════════════════
  --- xxd -c 4 -b hello.txt  (4 bytes per line, binary view) ---
00000000: 01001000 01100101 01101100 01101100  Hell
00000004: 01101111 00101100 00100000 01110111  o, w
00000008: 01101111 01110010 01101100 01100100  orld
0000000c: 00100001 00001010                    !.
```

**EXPLAIN**
> "Same file, binary view. `-b` for binary, `-c 4` for four bytes per line. The first group on line 1 is `01001000` — eight bits, the bits for H. Second group `01100101` is the bits for e. Third and fourth are the bits for l and l."
>
> "That's the content of your file at the bit level. Physically, those ones and zeros are the state of eight memory cells on your SSD."

**OUTPUT**
```

═══════════════════════════════════════════════════════════════
SECTION D — A non-text file has bytes too (PNG magic bytes)
═══════════════════════════════════════════════════════════════
  Wrote /Users/giacobbe/src/courses/anyone-can-code/1.5/tiny.png  (8 bytes — just the file signature)

  --- xxd tiny.png ---
00000000: 8950 4e47 0d0a 1a0a                      .PNG....
```

**EXPLAIN**
> "Those eight bytes — `89 50 4e 47 0d 0a 1a 0a` — are the PNG signature. Every real PNG on your disk starts with them. The 0x89 at the start is chosen deliberately to not be valid ASCII — it signals 'this is binary, not text.' Then P, N, G in the gutter. Then some control bytes to detect corrupt transfers."
>
> "The point: file formats are conventions about byte patterns. When your image viewer opens a file, it reads the first few bytes and checks them against a list of known signatures. That's how it knows what it's looking at."

## PART 3 — SAME BYTES, NOW IN PYTHON

### What does it mean for a number to be "in a base"?

**SPEAK**
> "The last part of the lab closes the loop. What xxd showed you is the exact numeric value of each byte. Python has built-ins for every base conversion, and we can use them to reproduce xxd's output from a Python string — proving that the bytes on disk and the numbers in Python are the same thing."

### Run the demo

**TYPE**
```bash
python3.12 conversions_demo.py
```

**OUTPUT**
```
======================================================================
SECTION A — one byte, four representations
======================================================================
  Character:     'H'
  ord('H')        = 72            (decimal)
  hex(72)        = '0x48'          (hex string, with '0x' prefix)
  bin(72)        = '0b1001000'     (binary string, with '0b' prefix)
  chr(72)        = 'H'            (back to the character)

  Clean formats via f-strings (no prefix, zero-padded):
  f"{n:02x}"      = '48'          (two hex digits — like xxd)
  f"{n:08b}"      = '01001000'    (eight bits — one full byte)
```

**EXPLAIN**
> "`ord('H')` returns 72 — the integer code point of the character. `hex(72)` returns the string `'0x48'`. `bin(72)` returns the string `'0b1001000'`. `chr(72)` returns back to `'H'`."
>
> "The `0x` and `0b` prefixes are built in so you can tell by looking what base the string is in. If you want clean output without the prefix — which is what xxd does — use f-strings with a format spec. `:02x` means 'two hex digits, zero-padded'; `:08b` means 'eight bits, zero-padded'."

**OUTPUT**
```
======================================================================
SECTION B — round trip: 'H' -> 72 -> '48' -> '01001000' -> 72 -> 'H'
======================================================================
  Forward:
    ord('H')                 -> 72
    f'{72:02x}'              -> '48'
    f'{72:08b}'              -> '01001000'
  Backward:
    int('48', 16)           -> 72
    int('01001000', 2)     -> 72
    chr(72)                 -> 'H'
  Round trip holds:          True
```

**EXPLAIN**
> "Forward and backward. `ord` goes char-to-number. `hex` and `bin` go number-to-string. `int(string, base)` parses a string in any base back to a number. `chr` goes number-to-char. Every function has an inverse, and the final boolean shows we end where we started — no information is lost in any conversion."

**OUTPUT**
```
======================================================================
SECTION C — a whole word, byte by byte
======================================================================
  Word: 'Hello'

  char  decimal   hex   binary    
  ----  -------   ---   ------    
  'H'   72        48    01001000
  'e'   101       65    01100101
  'l'   108       6c    01101100
  'l'   108       6c    01101100
  'o'   111       6f    01101111
```

**EXPLAIN**
> "The loop. Each row is one character of the word 'Hello'. Character, decimal code, two-digit hex, eight-bit binary."
>
> "Look at the hex column: `48 65 6c 6c 6f`. Now remember what xxd said for hello.txt back in Part 2: `4865 6c6c 6f2c 2077 6f72 6c64 210a` — the first five bytes were `48 65 6c 6c 6f`. Same five bytes. Python and xxd are describing the exact same thing from two directions: xxd reads bytes out of a file on disk; Python reads characters from a string in memory. The numbers match because they're the same bytes."

**OUTPUT**
```
======================================================================
SECTION D — bytes objects: hex() forward, fromhex() back
======================================================================
  Original bytes:     b'Hello, world!'
  original.hex():     '48656c6c6f2c20776f726c6421'
  bytes.fromhex(...): b'Hello, world!'
  Round trip holds:   True
```

**EXPLAIN**
> "A Python bytes object — the `b'...'` prefix — is what you get when you read bytes from a file or a network socket. `.hex()` gives you the exact string xxd prints in its middle column. `bytes.fromhex()` reverses it. Later labs that work with binary data will use these two methods constantly."


**EXPLAIN**
> "J is the 10th letter of the alphabet. A is 65, so J is 65 + 9 = 74. In hex that's 0x4a. In binary it's 01001010. Change the `'J'` to your own initial and run it. If you try a lowercase letter, you'll notice it's always 32 more than the uppercase — that's one bit different. Case is literally one bit."
