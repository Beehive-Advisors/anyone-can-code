# Research Report: Lab 4.13 — Speaking HTTP by Hand

## Overview

HTTP is plain ASCII text sent over a TCP connection. Every web request you've ever made — from loading a website to calling an API — boils down to your computer opening a TCP socket, sending a few lines of text, and reading the response. This lab strips away every abstraction (browser, library, framework) and shows students the raw protocol.

The two tools are `curl -v` (which shows the full HTTP conversation as it happens) and `nc` (netcat, which opens a raw TCP socket and lets you type the request yourself). Together they prove that HTTP is not magic — it is structured text that any tool capable of writing to a socket can speak.

---

## 1. HTTP Wire Format

### Background

HTTP was designed to be human-readable. RFC 2616 (1999) and its successors RFC 9110 and RFC 9112 (2022) specify the protocol as ASCII text over TCP. There are no binary fields, no custom encoding, no handshake beyond the TCP handshake itself (for HTTP, not HTTPS).

An HTTP/1.1 message has exactly four parts, always in this order:

1. **Start line** — the request line (client → server) or status line (server → client)
2. **Headers** — zero or more `Name: Value` lines
3. **Blank line** — a line containing only CRLF (`\r\n`) that marks the end of headers
4. **Body** — optional; present for POST/PUT requests and for responses with content

### How it works

**Request line format:** `METHOD PATH HTTP/VERSION\r\n`
```
GET / HTTP/1.1\r\n
POST /api/users HTTP/1.1\r\n
```

**Header format:** `Name: Value\r\n` (names are case-insensitive; order doesn't matter)
```
Host: example.com\r\n
Content-Type: application/json\r\n
Content-Length: 43\r\n
```

**The blank line:** Exactly `\r\n` — a line with nothing on it. This is how the server knows headers are done and the body begins. Without it, the server waits forever.

**Status line format:** `HTTP/VERSION STATUS_CODE REASON\r\n`
```
HTTP/1.1 200 OK\r\n
HTTP/1.1 301 MOVED PERMANENTLY\r\n
HTTP/1.1 404 NOT FOUND\r\n
```

**A complete minimal GET request as bytes:**
```
GET / HTTP/1.1\r\n
Host: example.com\r\n
\r\n
```

**A complete POST request with JSON body:**
```
POST /api/users HTTP/1.1\r\n
Host: api.example.com\r\n
Content-Type: application/json\r\n
Content-Length: 43\r\n
\r\n
{"name":"Alice","course":"Anyone Can Code"}
```

**Key headers students will see:**

| Header | Direction | Meaning |
|--------|-----------|---------|
| `Host: example.com` | Request | Required in HTTP/1.1; identifies which server on a shared IP |
| `User-Agent: curl/8.7.1` | Request | Identifies the client software |
| `Accept: */*` | Request | What content types the client accepts |
| `Content-Type: application/json` | Request & Response | Format of the body |
| `Content-Length: 43` | Request & Response | Byte count of the body |
| `Location: /redirect/1` | Response | Where to go on a redirect |
| `Set-Cookie: name=value; Path=/` | Response | Server is setting a cookie |
| `Cookie: name=value` | Request | Client sending a stored cookie back |
| `Connection: keep-alive` | Both | HTTP/1.1 default — TCP connection stays open |
| `Connection: close` | Both | Close TCP after this response |

### HTTP/1.0 vs HTTP/1.1 for nc

HTTP/1.1 defaults to persistent connections. After the server sends its response, it keeps the TCP socket open waiting for more requests. If you type a request by hand in nc and use HTTP/1.1, the server responds but nc hangs — it keeps waiting for more input. HTTP/1.0 closes the connection after each response, so nc exits cleanly. For the manual nc exercise, always use `HTTP/1.0`.

### Tools

**curl** — pre-installed on macOS and most Linux systems. All flags work identically on both. No install required.

**nc (netcat)** — pre-installed on macOS (BSD variant at `/usr/bin/nc`). On Ubuntu/WSL: `apt install netcat-openbsd`. The key platform difference:
- macOS: `nc -c hostname port` converts LF (Enter) to CRLF automatically
- Linux: use `printf "GET / HTTP/1.0\r\nHost: example.com\r\n\r\n" | nc -q 3 hostname port`

**python3 -m http.server** — Python stdlib, no install. Used as a local server for the nc exercise. Shows the request in Terminal 1 when it arrives.

- [RFC 9110 — HTTP Semantics](https://httpwg.org/specs/rfc9110.html) — status codes, methods, headers

## 2. curl -v

### Background

`curl -v` doesn't do anything different from a normal curl request — it just shows you what's happening. It displays the full HTTP conversation: the DNS lookup, TCP connection, every header sent and received, and metadata about the transfer.

### How it works

curl sends verbose output to stderr. The `2>&1` redirect merges stderr into stdout so both appear together in the terminal.

**Line prefixes:**
- `*` — curl's own informational messages (DNS, TCP state, TLS, connection reuse)
- `>` — bytes curl **sent** to the server (the request)
- `<` — bytes the server **sent back** (the response headers)
- Lines with no prefix — the response body

**Key flags:**

| Flag | Effect |
|------|--------|
| `-v` / `--verbose` | Show full `*`/`>`/`<` conversation |
| `-s` / `--silent` | Suppress progress bar (use with -v) |
| `-i` | Include response headers in stdout output |
| `-I` / `--head` | Send HEAD request (headers only, no body) |
| `-X POST` | Set the HTTP method |
| `-H "Name: Value"` | Add a request header |
| `-d "body"` | Set the request body (implies POST) |
| `-b "name=value"` | Send a Cookie header |
| `-b cookiefile` | Send cookies from a saved file |
| `-c cookiefile` | Save received Set-Cookie headers to a file |
| `-L` | Follow Location redirects |
| `-o /dev/null` | Discard body |
| `-w "%{http_code}"` | Print status code after transfer |
| `-w "%{url_effective}"` | Print final URL after redirects |

**Sources:**
- [curl HTTP scripting guide](https://curl.se/docs/httpscripting.html)
- [everything.curl.dev — verbose](https://ec.haxx.se/usingcurl/verbose/)
- [everything.curl.dev — write-out](https://everything.curl.dev/usingcurl/verbose/writeout.html)
- [curl man page](https://curl.se/docs/manpage.html)

## 3. nc (netcat) for Raw HTTP

### Background

netcat opens a raw TCP socket and bidirectionally relays stdin to the socket and the socket to stdout. It knows nothing about HTTP — it just moves bytes. This makes it the perfect tool for proving that HTTP is just text: you type the request yourself, byte by byte.

### How it works

```bash
# macOS — -c converts LF to CRLF on send
nc -c example.com 80
# Type:
# GET / HTTP/1.0
# Host: example.com
# [Enter]
```

```bash
# Scripted version (works on macOS and Linux)
(printf "GET / HTTP/1.0\r\nHost: example.com\r\n\r\n"; sleep 3) | nc example.com 80
```

The `sleep 3` is required on macOS: BSD nc closes the socket almost immediately when stdin hits EOF. Keeping stdin open 3 seconds gives the server time to respond.

**CRLF requirement:** HTTP requires `\r\n` line endings. On macOS, `nc -c` handles this by converting every LF (Enter key) to CRLF before sending. On Linux, use `printf` with explicit `\r\n`.

**Sources:**
- [OpenBSD nc man page](https://man.openbsd.org/nc.1)
- [Debian netcat-openbsd man page](https://manpages.debian.org/testing/netcat-openbsd/nc.1.en.html)
- [ss64 macOS nc reference](https://ss64.com/mac/nc.html)

## 4. Cookies

### Background

A cookie is just text that the server asks the client to remember and send back on future requests. The entire mechanism is two HTTP headers: `Set-Cookie` (server → client) and `Cookie` (client → server).

### How it works

**Server sets a cookie:**
```
HTTP/1.1 302 FOUND
Set-Cookie: flavor=chocolate; Path=/
Location: /cookies
```

**Client sends it back:**
```
GET /cookies HTTP/1.1
Cookie: flavor=chocolate
```

`Set-Cookie` attributes:
- `Path=/` — send on all paths
- `Expires=date` / `Max-Age=seconds` — when it expires (omit for session cookie)
- `HttpOnly` — JavaScript cannot read this cookie
- `Secure` — only send over HTTPS
- `SameSite=Strict|Lax|None` — cross-site request protection

curl saves cookies in Netscape cookie file format (tab-separated: domain, flag, path, secure, expiry, name, value).

**Sources:**
- [MDN: Set-Cookie header](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Set-Cookie)
- [everything.curl.dev — cookies](https://everything.curl.dev/libcurl-http/cookies.html)
- [httpbin.org /cookies endpoints](https://httpbin.org/legacy)

## Lab Design Decisions

### Shell-only (curl + nc), no Python libraries

This is a networking/protocol lab. The concepts being taught are HTTP message structure, header semantics, and raw TCP communication — not programming. Shell tools get students closest to the wire with zero abstraction. curl is pre-installed everywhere; nc is pre-installed on macOS and available on Linux with one apt command. No virtual environment, no pip install, no language setup.

The one exception: `python3 -m http.server` for the nc bonus exercise. This uses stdlib only (no install) and gives students the experience of seeing both ends of an HTTP conversation simultaneously — their request appearing as a log line in Terminal 1 while the response arrives in Terminal 2.

### httpbin.org over a custom local server

httpbin.org is a purpose-built HTTP testing service that echoes request details back as JSON. Students can see exactly what headers and body the server received — without writing any server code. It handles POST, cookies, redirects, and status codes cleanly. It runs on HTTP port 80 (no TLS complexity). All endpoints used in this lab were verified working on April 14, 2026.

### example.com for nc exercise

example.com is maintained by IANA specifically for use in documentation and examples. It will reliably exist, stay at the same IP range, and return a simple response. No risk of the demo URL going away between course versions.
