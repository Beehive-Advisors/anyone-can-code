# Research Report: Lab 4.13 — Speaking HTTP by Hand

## Overview

HTTP is the protocol that powers the web — every time a browser loads a page, an API
call returns data, or a form is submitted, an HTTP message is traveling over the network.
What most developers don't realize until they look closely is that these messages are plain,
human-readable text. There is no binary encoding, no compiled format — just a short sequence
of text lines that both sides have agreed to interpret in a specific way.

This lab teaches students to read and write that text directly. Using `curl -v` to reveal
the raw headers behind every request, and `nc` (netcat) to send hand-crafted HTTP messages
to a real server, students see that the protocol they use every day is as simple as typing
a few lines into a terminal. The goal is demystification: once you've typed an HTTP request
by hand and watched the server respond, the protocol stops being magic.

The lab covers four concepts: the structure of HTTP messages (request line, headers, body),
the `curl -v` annotation format for reading exchanges, POST requests and Content-Type headers,
and HTTP status codes as the server's verdict system.

## 1. HTTP as Structured Text

### Background

When computers need to talk to each other, they need an agreed-upon format — like a formal
letter has a specific layout (address, salutation, body, signature). HTTP (HyperText Transfer
Protocol) is that agreed format for the web.

The surprising thing about HTTP/1.1 is that the format is plain ASCII text. Every character
you see in an HTTP message is readable as ordinary text. When Tim Berners-Lee designed HTTP
at CERN in 1991, he deliberately made it human-readable so that developers could debug
connections manually — which is exactly what this lab does.

### How It Works

An HTTP/1.1 request has three parts, separated by specific line endings:

```
GET /path HTTP/1.1          ← request line: method, path, version
Host: example.com           ← headers (one per line, name: value)
Connection: close           ← another header
                            ← blank line (CRLF only — signals end of headers)
[optional body]
```

The line endings matter: HTTP requires `\r\n` (carriage return + newline, 0x0D 0x0A) after
each line, not just `\n`. Most servers accept bare `\n` out of leniency, but the spec
(RFC 9112) requires CRLF.

The **blank line** is the critical separator. It tells the server "headers are done, body
follows." Without it, the server waits indefinitely for more headers — the connection hangs.

An HTTP response follows the same structure but leads with a status line:

```
HTTP/1.1 200 OK             ← status line: version, code, reason phrase
Content-Type: text/html     ← response headers
Content-Length: 646
Connection: close

<html>...</html>            ← body
```

The three-digit status code is the machine-readable verdict. The reason phrase ("OK",
"Not Found") is cosmetic — clients use the number, not the text.

**Required headers**: HTTP/1.1 requires the `Host` header in every request. Without it,
servers return 400 Bad Request. This wasn't required in HTTP/1.0 — it was added in 1.1
to support virtual hosting (multiple domains on one IP address).

### Demo

```bash
# Send a raw HTTP request using nc as a TCP pipe
printf 'GET / HTTP/1.1\r\nHost: info.cern.ch\r\nConnection: close\r\n\r\n' \
  | nc -w 5 info.cern.ch 80
```

- [RFC 9110 — HTTP Semantics](https://www.rfc-editor.org/rfc/rfc9110.html) — method definitions, status codes

## 2. curl -v: Reading the Conversation

### Background

`curl` normally hides everything it does. You see the response body and nothing else.
`curl -v` (verbose mode) reveals the full exchange: DNS resolution, TCP connection,
every header sent, every header received.

The output uses three prefixes to separate different kinds of events:

- `*` — connection-level events (DNS, TCP, TLS)
- `>` — lines your computer sent to the server (the request)
- `<` — lines the server sent back (the response)

### How It Works

When you run `curl -v -s -o /dev/null https://example.com`, you see:

```
* Host example.com was resolved.
*   Trying 192.0.2.1:80...
* Connected to example.com port 80
> GET / HTTP/1.1         ← exact bytes sent
> Host: example.com
> User-Agent: curl/8.7.1
> Accept: */*
>                        ← blank line (the mandatory blank line)
< HTTP/1.1 200 OK        ← exact bytes received
< Content-Type: text/html
< Content-Length: 1256
<                        ← blank line (end of response headers)
```

The blank `>` line corresponds to the HTTP blank-line separator — it's the moment your
request is complete. The blank `<` line marks the end of response headers.

Important technical note: `curl -v` writes its annotation output to **stderr**, not stdout.
The response body goes to stdout. To capture both together for piping or redirection:
`curl -v ... 2>&1`.

### Key Flags

| Flag | Meaning |
|------|---------|
| `-v` | Verbose — show `*`, `>`, `<` annotation |
| `-s` | Silent — suppress the progress bar |
| `-o /dev/null` | Discard the response body |
| `-I` | HEAD request — get headers only, no body |
| `-w "%{http_code}"` | Print the HTTP status code after the transfer |
| `-L` | Follow redirects |
| `--fail` | Exit non-zero on 4xx/5xx responses |

**Sources:**
- [Verbose — everything curl](https://ec.haxx.se/usingcurl/verbose/index.html) — comprehensive curl verbose guide
- [curl man page](https://curl.se/docs/manpage.html) — complete flag reference
- [curl: The Art of Scripting HTTP](https://curl.se/docs/httpscripting.html) — official scripting guide

## 3. POST Requests and Content-Type

### Background

GET requests ask for data by putting parameters in the URL. POST requests send data by
putting it in the request body — the bytes after the blank line. This is why login forms
use POST: your password travels in the body, not the URL (which gets logged, cached, and
stored in browser history).

The server needs to know how to parse the body. That's what the `Content-Type` header
says: "the body is JSON" or "the body is form-encoded key=value pairs."

### How It Works

A POST request looks like this:

```
POST /api/users HTTP/1.1
Host: api.example.com
Content-Type: application/json
Content-Length: 28

{"name": "alice", "age": 30}
```

The `Content-Length` tells the server exactly how many bytes to read. curl calculates
this automatically from the data you pass with `-d`.

**Two common Content-Type values:**

| Content-Type | Body format | When used |
|---|---|---|
| `application/json` | `{"key":"value"}` | REST APIs |
| `application/x-www-form-urlencoded` | `key=value&key2=value2` | HTML forms |

curl defaults to `application/x-www-form-urlencoded` when you use `-d`. For JSON, you
must always set `-H "Content-Type: application/json"` explicitly.

### Key Flag Behavior

- `-d "data"` — sets the request body AND implies POST (you don't need `-X POST`)
- `-X POST` — changes the method to POST but sends no body by itself
- `-H "Header: Value"` — adds or overrides a header
- `-d @filename` — reads body from a file (the `@` prefix is required)

### Demo

```bash
# JSON POST — Content-Type must be set manually
curl -s https://httpbin.org/post \
  -H "Content-Type: application/json" \
  -d '{"name": "alice", "age": 30}'

# Form POST — curl defaults to application/x-www-form-urlencoded
curl -s https://httpbin.org/post \
  -d "username=alice&city=boston"
```

`httpbin.org/post` echoes the full request back as JSON, making it ideal for
verifying that headers and body were sent correctly.

**Sources:**

- [POST with curl — everything curl](https://everything.curl.dev/http/post/simple) — curl POST variants explained
- [MDN: HTTP Methods](https://developer.mozilla.org/en-US/docs/Web/HTTP/Methods) — GET vs POST semantics

## 4. HTTP Status Codes

### Background

Every HTTP response starts with a three-digit code. The server sends this before anything
else so the client can immediately decide how to handle the response. A well-designed client
never ignores the status code.

The first digit is the category. Once you know the five categories, you can make an educated
guess about any status code you haven't seen before.

### How It Works

| Class | Range | Meaning |
|-------|-------|---------|
| Informational | 1xx | Server received request, still processing |
| Success | 2xx | Request fulfilled |
| Redirect | 3xx | Client must take further action (follow Location header) |
| Client Error | 4xx | Request is wrong — your fault |
| Server Error | 5xx | Server failed — server's fault |

**Most important codes:**

| Code | Name | When you see it |
|------|------|----------------|
| 200 | OK | Standard success |
| 201 | Created | POST that created a resource |
| 301 | Moved Permanently | URL has permanently moved; update bookmarks |
| 302 | Found | Temporary redirect |
| 400 | Bad Request | Server can't parse your request |
| 401 | Unauthorized | You need to authenticate first |
| 403 | Forbidden | Server knows who you are, refuses anyway |
| 404 | Not Found | Nothing at this URL |
| 500 | Internal Server Error | Server crashed |

**401 vs 403**: 401 means "I don't know who you are." 403 means "I know who you are and
no." A server may also return 403 instead of 404 to avoid revealing that a resource exists.

**curl's exit code is not the HTTP status code.** curl exits 0 if the network transfer
completed, regardless of the HTTP response code. To exit non-zero on 4xx/5xx, use `--fail`.
To extract the status code in a script: `curl -s -o /dev/null -w "%{http_code}" URL`.

**Sources:**
- [HTTP status codes — MDN Web Docs](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Status) — complete reference
- [List of HTTP status codes — Wikipedia](https://en.wikipedia.org/wiki/List_of_HTTP_status_codes) — with notes on each
- [Write-out http_code — everything curl](https://everything.curl.dev/usingcurl/verbose/writeout.html) — extracting status in scripts

## 5. nc (netcat) for Raw TCP

### Background

`nc` (netcat) opens a raw TCP connection. That's it. Whatever you write to nc's stdin
gets sent byte-for-byte to the server's port. Whatever the server sends back appears on
your stdout.

No HTTP library. No header generation. No status code parsing. Just a pipe between your
shell and a socket. If you want the server to understand you, you have to speak HTTP
yourself — every header, every colon, every carriage return. This is what makes nc the
perfect tool for proving that HTTP is just text.

### How It Works

```bash
# nc as HTTP client — send a hand-crafted request
printf 'GET / HTTP/1.1\r\nHost: info.cern.ch\r\nConnection: close\r\n\r\n' \
  | nc -w 5 info.cern.ch 80
```

`printf` is used instead of `echo -e` because `printf` interprets `\r\n` consistently
on all platforms (macOS, Linux, WSL). `echo -e` behavior varies by shell.

**Critical rules:**
- Every line must end with `\r\n` (CRLF) — not just `\n`
- The blank line (just `\r\n`) after headers is mandatory
- Add `Connection: close` to prevent nc from hanging on HTTP/1.1 keep-alive
- Add `-w 5` to nc as a timeout safety net

**Platform difference for server mode:**
```bash
# macOS (BSD nc):
nc -l 8080

# Linux (GNU nc):
nc -l -p 8080
```

For client mode (`nc hostname port`), the syntax is identical on both platforms.

**Sources:**
- [OpenBSD nc man page](https://man.openbsd.org/nc.1) — BSD nc reference (macOS)
- [Linux nc man page](https://linux.die.net/man/1/nc) — GNU nc reference
- [Using nc to make an HTTP request — osric.com](https://osric.com/chris/accidental-developer/2018/01/using-nc-netcat-to-make-an-http-request/) — practical walkthrough

## Lab Design Decisions

### Shell before Python

This lab covers HTTP as a protocol — not a Python library. `curl` and `nc` are the direct
tools: they expose the raw protocol without any abstraction layer. Adding Python would add
indirection that obscures the "HTTP is just text" insight.

### info.cern.ch for the nc demo

CERN's original WWW server at info.cern.ch always serves plain HTTP on port 80 and returns
200 OK with stable content. Unlike example.com (now Cloudflare-fronted, which silently
drops nc connections without User-Agent headers) or httpbin.org (requires HTTPS on port
443), info.cern.ch is a reliable plain-HTTP target for the raw nc demo. The historical
context is also a bonus: students are literally connecting to the server where the web
was invented.

### httpbin.org for POST and status code demos

`httpbin.org` is purpose-built for HTTP testing. `/post` echoes the full request back as
JSON so students can verify Content-Type and body were sent correctly. `/status/N` returns
any status code on demand. These make the POST and status code demos self-contained and
verifiable without a local server.

### No local server

A local server would remove the internet dependency but also remove the impact. When a
student connects to info.cern.ch with nc and sees CERN's server respond to their hand-typed
HTTP request, the "HTTP is just text over a TCP connection" lesson is visceral in a way
a localhost demo isn't.
