# Research Report: Lab 4.13 — Speaking HTTP by Hand

## Overview

Every time you click a link or submit a form, your browser sends a small chunk of text to a server and reads another chunk back. HTTP — HyperText Transfer Protocol — is nothing more than an agreed-upon format for that text exchange. Yet because browsers and libraries handle this invisibly, most developers never see it directly. This lab changes that: students use `curl -v` and `nc` (netcat) to put HTTP on full display, making the invisible conversation visible and the abstraction concrete.

The lab covers four interlocking skills. First, `curl -v` reveals the full request/response exchange — the exact headers your machine sends and receives — while `curl` without `-v` shows only the final result (as a browser does). Second, POST requests demonstrate that HTTP can carry data in both directions: a `Content-Type` header labels the format, and the server behaves completely differently depending on whether that label says `application/x-www-form-urlencoded` or `application/json`. Third, status codes and cookies expose the mechanics behind two things students encounter daily but rarely inspect: three-digit response codes that tell clients whether to retry, redirect, or give up, and the cookie-based trick that makes stateless HTTP behave like a stateful session. Finally, `nc` removes every abstraction and lets students type a raw HTTP request by hand — proving that the protocol is just text over a TCP socket.

By the end of the lab, students have a permanent mental model: when something goes wrong in an API call or a web form submission, they know what to look for, how to inspect it, and what questions to ask.

## 1. curl -v: Reading the HTTP Conversation

### Background

Curl without flags is like reading the last paragraph of an email thread — you see the outcome but not the exchange. The `-v` flag (verbose) reveals the whole conversation. When curl connects to a server, it exchanges messages: it sends a request (your machine's side) and receives a response (the server's side). Each message has two parts: headers (metadata about the message) and a body (the actual content). The `-v` flag shows all the headers in both directions, plus curl's own internal notes about what it's doing.

### How it works

Curl `-v` prefixes output with three symbols:

| Prefix | Meaning |
|--------|---------|
| `*` | Curl's own status messages — DNS resolution, TCP connection, TLS handshake, connection close |
| `>` | Data your machine **sent** to the server (the outgoing request line and headers) |
| `<` | Data the server **sent back** (the response status line and headers) |

The response body appears without any prefix, or as `{ [N bytes data] }` when discarded with `-o /dev/null`.

A critical gotcha: the `*`, `>`, and `<` lines go to **stderr**, not stdout. The response body goes to stdout. This means piping `curl -v ... | grep Host` finds nothing — grep only sees stdout. The fix is `2>&1` to merge the streams before piping.

The verbose output begins with `*` lines tracing DNS resolution and TCP connection. For HTTPS URLs, this includes the TLS handshake — the negotiation of encryption parameters before any HTTP bytes travel. The `>` lines follow: first the request line (`GET /path HTTP/version`), then headers (`Host:`, `User-Agent:`, `Accept:`), then a blank line marking the end of headers. The `<` lines mirror this structure on the response side.

### Implementation: curl

Key flags:

| Flag | Purpose |
|------|---------|
| `-v` | Verbose: show `*`, `>`, `<` lines |
| `-s` | Silent: suppress progress meter (use alongside `-v`) |
| `-o /dev/null` | Discard response body (keeps terminal clean for header inspection) |
| `-I` | HEAD request: ask server for headers only, no body |
| `2>&1` | Merge stderr into stdout so you can pipe or grep the full output |
| `-w "%{http_code}"` | Write-out: print just the status code after the response |

```bash
# Show full conversation, body discarded:
curl -v -s -o /dev/null https://httpbin.org/get 2>&1

# Extract just the Host header curl sent:
curl -v -s -o /dev/null https://httpbin.org/get 2>&1 | grep 'Host'
```

## 2. POST Requests: Sending a Body

### Background

A GET request says "give me something." A POST request says "here is something — do something with it." The difference at the protocol level is that POST carries a **body**: a chunk of data attached after the headers. The server needs to know what format that data is in before it can read it, so every POST includes a `Content-Type` header. Two formats matter most:

- **`application/x-www-form-urlencoded`** — what browsers send when you submit an HTML form. Looks like `username=alice&city=Portland`. This is curl's default when you use `-d`.
- **`application/json`** — what modern APIs expect. Looks like `{"username": "alice", "city": "Portland"}`. Requires `-H "Content-Type: application/json"`.

### How it works

The wire-level difference between GET and POST is minimal: POST adds a body separated from the headers by a blank line. The request-line changes from `GET` to `POST`, and two new headers appear: `Content-Type` (the format of the body) and `Content-Length` (the byte count of the body, so the server knows when it ends).

```
POST /post HTTP/2
Host: httpbin.org
Content-Type: application/x-www-form-urlencoded
Content-Length: 28

username=alice&city=Portland
```

Swap `Content-Type` to `application/json` and change the body format:

```
POST /post HTTP/2
Host: httpbin.org
Content-Type: application/json
Content-Length: 41

{"username": "alice", "city": "Portland"}
```

From the TCP connection's perspective, both are the same: bytes going over the socket. The `Content-Type` header is what instructs the server to parse the body differently. Httpbin.org makes this visible: it echoes form-encoded data back under the `"form"` key in its JSON response, and JSON data under the `"json"` key.

### Implementation: curl

| Flag | Purpose |
|------|---------|
| `-d "key=value"` | Set a form-urlencoded body; implies POST method automatically |
| `-d '{"k":"v"}'` | Set a JSON body (must add `-H "Content-Type: application/json"`) |
| `-H "Content-Type: TYPE"` | Set or override any request header |
| `-X POST` | Explicitly set the method (usually redundant when using `-d`) |
| `--data-urlencode "key=val with spaces"` | URL-encode the value automatically |

```bash
# Form POST:
curl -v -s https://httpbin.org/post -d "username=alice&city=Portland"

# JSON POST (only the Content-Type header changes):
curl -v -s https://httpbin.org/post \
  -H "Content-Type: application/json" \
  -d '{"username": "alice", "city": "Portland"}'
```

## 3. Status Codes and Cookies

### Background

**Status codes** are the server's first word in every response. Before you read any headers or body, a three-digit number tells you how the request went. The first digit gives the category: 2xx means success, 3xx means redirect, 4xx means the client did something wrong, 5xx means the server had a problem. A client can decide whether to retry, redirect, or give up based on the first digit alone.

**Cookies** are the solution to a fundamental problem with HTTP: it is stateless. Each request is independent; the server has no memory of previous requests. This works for fetching static pages but breaks any feature requiring continuity — login sessions, shopping carts, preferences. The fix is a convention: after a successful login, the server adds a `Set-Cookie` header to its response. The client stores this text. On every subsequent request to that server, the client adds a `Cookie` header with the stored text. The server reads it and knows who you are.

### How it works

**Status code structure:**

```
HTTP/1.1 200 OK
```

Format: `HTTP-version SP status-code SP reason-phrase`. The reason phrase ("OK", "Not Found") is human-readable only; the numeric code is what clients act on.

| Code | Meaning | What happened |
|------|---------|---------------|
| 200 | OK | Request succeeded; body has the resource |
| 201 | Created | POST succeeded; new resource was created |
| 204 | No Content | Success, but no body (common for DELETE) |
| 301 | Moved Permanently | Resource lives at a new URL forever |
| 302 | Found | Resource is temporarily at another URL |
| 400 | Bad Request | Malformed request — fix the client |
| 401 | Unauthorized | Authentication required |
| 403 | Forbidden | Authenticated but not permitted |
| 404 | Not Found | This URL doesn't exist on this server |
| 500 | Internal Server Error | The server crashed or has a bug |

**Cookie wire format:**

Server response setting a cookie:
```
HTTP/1.1 200 OK
Set-Cookie: user=alice; Path=/; HttpOnly
```

Client's next request sending it back:
```
GET /profile HTTP/1.1
Host: example.com
Cookie: user=alice
```

Curl cookie flags:
- `-c FILE` — write cookies received from server to a file (the "cookie jar")
- `-b FILE` — read cookies from file and send them with the request
- `-L` — follow redirects (required for httpbin's `/cookies/set` which responds with a 302)

The cookie jar file uses Netscape format — tab-separated with seven fields per cookie: domain, subdomain-flag, path, secure, expiration, name, value.

### Implementation: curl

```bash
# Check any status code instantly:
curl -s -o /dev/null -w '%{http_code}' https://httpbin.org/status/404

# See a redirect without following it:
curl -v -s -o /dev/null https://httpbin.org/redirect/1

# Follow the redirect:
curl -v -s -o /dev/null -L https://httpbin.org/redirect/1

# Cookie lifecycle:
curl -s -L -c cookies.txt "https://httpbin.org/cookies/set?user=alice"   # set
curl -s -b cookies.txt https://httpbin.org/cookies                        # send back
```

## 4. netcat: HTTP is Just Text

### Background

`nc` (netcat) is often called the "Swiss Army knife" of networking tools, but its core function is simple: open a TCP connection to a host and port, then relay whatever you type to that connection, and whatever the connection sends back to your screen. It has no HTTP awareness. It does not know about methods, headers, or status codes. It is just a bidirectional pipe to a socket.

This makes nc the perfect tool for the most important insight in the lab: HTTP is just text. A browser sending a request is doing exactly the same thing as a student who types `GET / HTTP/1.0` into an nc session — formatting that text and sending it over TCP. Once students have typed a raw HTTP request and received a server response, the abstraction is permanently demystified.

### How it works

The minimum valid HTTP/1.0 request:
```
GET / HTTP/1.0\r\n
\r\n
```

Two lines. The request-line (`METHOD path version`), then a blank line to signal that headers are finished. HTTP requires `\r\n` (CRLF — carriage return then line feed). The `printf` command handles this correctly; typing in a terminal sends only `\n`.

HTTP/1.1 adds one mandatory header:
```
GET / HTTP/1.1\r\n
Host: httpforever.com\r\n
Connection: close\r\n
\r\n
```

The `Host` header is required because one server IP can host many domain names. `Connection: close` tells the server to close after responding — otherwise HTTP/1.1's persistent connections keep the socket open, and nc appears to hang.

**nc as a server:**
```bash
nc -l 8080
```
This listens on port 8080. Whatever TCP client connects, nc prints what that client sends (the raw request) and passes anything from stdin back to the client. Running this in one terminal and `curl http://localhost:8080/` in another reveals the exact bytes curl sends to every server.

**BSD vs GNU nc syntax:**

| Task | macOS (BSD nc) | Linux (may vary) |
|------|---------------|-----------------|
| Listen on port | `nc -l 8080` | `nc -l -p 8080` or `nc -l 8080` |
| Client connect | `nc host port` | `nc host port` |
| Timeout | `-w SECS` | `-w SECS` |

Ubuntu ships with `netcat-openbsd` by default, which uses the BSD syntax (`nc -l 8080`). If `nc -l 8080` fails on Linux, try `nc -l -p 8080`.

### Implementation: nc

```bash
# HTTP/1.0 by hand (printf handles CRLF):
printf "GET / HTTP/1.0\r\n\r\n" | nc -w 5 httpforever.com 80

# HTTP/1.1 by hand:
printf "GET / HTTP/1.1\r\nHost: httpforever.com\r\nConnection: close\r\n\r\n" \
  | nc -w 5 httpforever.com 80

# nc as a server — see what curl sends:
# Terminal A:
nc -l 8080
# Terminal B:
curl http://localhost:8080/
```

## Lab Design Decisions

### Shell-only: no installs required
Both `curl` and `nc` are pre-installed on macOS and standard Linux distributions. Students spend zero time on environment setup and 100% of time on the concepts. This is the right call for a lab about protocol-level mechanics — the tools should be invisible.

### httpbin.org for curl demos
Httpbin.org is purpose-built for HTTP education: it echoes back exactly what it received, has endpoints for every status code and redirect pattern, and works with standard HTTPS. Used for curl demos in Parts 1–3.

### httpforever.com for nc demos
Httpbin.org redirects HTTP to HTTPS, which nc cannot speak (no TLS). `httpforever.com` (maintained specifically to always serve plain HTTP on port 80) is the right target for nc demos. It returns a clean 200 with a full set of response headers.

### localhost for nc server demos
The "nc as server, curl as client" exercise uses localhost port 18080 to avoid firewall complications and ensure it works identically on every student's machine. No network dependency.

### httpbin.org /cookies/set requires -L
The `/cookies/set` endpoint responds with a 302 redirect. Without `-L`, curl doesn't follow the redirect and the cookie is never received. Always use `-L -c jar.txt` for this endpoint. This is intentional — it shows students the redirect pattern they'll encounter in real login flows.
