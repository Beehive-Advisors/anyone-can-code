## PART 1 — What HTTP Actually Looks Like

---

### What is HTTP?

**SPEAK**
> "Before we run anything, I want to make sure we're on the same page about what HTTP actually is at the bytes level."

> "HTTP is a protocol — a set of rules about how to format text. When you make a request, you open a TCP connection to port 80, write a few lines of text, and close or wait. When the server responds, it writes a few lines of text back. Every request and every response has the same four-part structure, always in this order."

> "Part one: the start line. For a request this is the method, the path, and the HTTP version — GET / HTTP/1.1. For a response it's the version, the status code, and a reason phrase — HTTP/1.1 200 OK."

> "Part two: headers. One per line, in the format Name colon space Value. You've seen these — Content-Type, Host, Authorization. They're metadata about the message."

> "Part three: a blank line. Just a line with nothing on it. This is the separator between headers and body. Without it, the server keeps reading, waiting for more headers that never come. This blank line is so important I'm going to say it twice: the server will not process your request until it sees the blank line."

> "Part four: the body. Optional. GET requests don't have bodies. POST requests do — that's where the data you're sending lives."

> "Here's what a real request looks like as raw text:"

> "GET / HTTP/1.1 — that's the start line. Host: example.com — required in HTTP/1.1. User-Agent — who's making the request. Accept: star/star — we'll take any response format. Then a blank line. Done."

> "curl -v shows you exactly this. Let's run it."

---

**TYPE**

**OUTPUT**

>

< Date: Tue, 14 Apr 2026 20:04:41 GMT

*(Your Date, IP addresses, and CF-RAY values will differ — the structure is always the same.)*

**EXPLAIN**
> "Three line prefixes. Star lines are curl talking to itself — DNS resolved, TCP connected, connection intact. These aren't HTTP, they're curl metadata."

> "Greater-than lines are what WE sent. Start at 'GET / HTTP/1.1' — that's the start line. Then Host, User-Agent, Accept. Then that greater-than on a line by itself with nothing after it — that's the blank line. That's the separator. The server didn't start processing until it saw that."

> "Less-than lines are what the SERVER sent back. HTTP/1.1 200 OK — the status line. Then the response headers: Date, Content-Type, Transfer-Encoding. Then a less-than on its own — that's the blank line on the response side. After that comes the body."

> "The 2>&1 at the end merges curl's stderr into stdout — curl sends verbose output to stderr by default, so without that redirect you wouldn't see the star and greater-than lines in a pipe."

**TYPE**
```bash
curl -s -o /dev/null -w "HTTP Status: %{http_code}\n" http://example.com/
```

**OUTPUT**
```
HTTP Status: 200
```

**EXPLAIN**
> "-o /dev/null throws away the response body — we don't need to see the HTML. -w is 'write-out': after the transfer completes, print this format string. %{http_code} is replaced with the actual three-digit status code. Useful for scripting: curl silently, just give me the number."

## PART 2 — Status Codes: What the Server Is Saying

---

### What is a status code?

**SPEAK**
> "Every HTTP response starts with a three-digit number. This is the status code, and it tells the client what happened. You've seen 404 before. Let me give you the full picture."

> "The codes are grouped into ranges. 2xx means success — 200 is the most common, 201 means something was created. 3xx means redirect — the thing you asked for is somewhere else. 4xx means client error — you asked for something wrong. 404 means it doesn't exist, 401 means you're not authenticated, 403 means you're authenticated but not allowed. 5xx means server error — the server itself broke."

> "The most interesting range for understanding HTTP is 3xx, because a redirect shows you something you can't see in the browser: the Location header."

---

### Run the demo

**TYPE**
```bash
curl -v -s -o /dev/null http://httpbin.org/status/301 2>&1 \
  | grep -E "^[*<>]"
```

**OUTPUT**
```
* Host httpbin.org:80 was resolved.
*   Trying ...
* Connected to httpbin.org ... port 80
> GET /status/301 HTTP/1.1
> Host: httpbin.org
> User-Agent: curl/8.7.1
> Accept: */*
> 
* Mark bundle as not supporting multiuse
< HTTP/1.1 301 MOVED PERMANENTLY
< Content-Length: 0
< location: /redirect/1
< 
* Connection #0 to host httpbin.org left intact
```

**EXPLAIN**
> "301 MOVED PERMANENTLY in the status line. Content-Length: 0 — there's no body on this redirect. And then the key header: location: /redirect/1. That's the server saying: the thing you want has permanently moved to this URL. Without -L, curl stops here and shows you the redirect. A browser would follow it automatically — curl is giving us the raw view."

**TYPE**
```bash
curl -s -o /dev/null \
  -w "Final status: %{http_code}  →  final URL: %{url_effective}\n" \
  -L http://httpbin.org/status/301
```

**OUTPUT**
```
Final status: 200  →  final URL: http://httpbin.org/get
```

**EXPLAIN**
> "Add -L and curl follows the chain. It made a request to /status/301, got the 301 and the location header, made another request to /redirect/1, and ended up at a 200. %{url_effective} is the final URL after all redirects. This is how a browser behaves — you type one URL, you land on another."

**TYPE**
```bash
curl -s -o /dev/null -w "Status: %{http_code}\n" http://httpbin.org/status/404
```

**OUTPUT**
```
Status: 404
```

**EXPLAIN**
> "404. httpbin.org/status/N is a testing endpoint — it returns whatever status code you ask for. Useful for building clients that need to handle error cases."

## PART 3 — POST: Sending Data to the Server

---

### What makes POST different?

**SPEAK**
> "GET requests ask for a resource. POST requests send data. The difference shows up in three places."

> "First: the method. GET becomes POST in the start line."

> "Second: the body. POST requests have content after the blank line — the data you're sending. GET requests have no body."

> "Third: two new headers that GET requests don't need. Content-Type tells the server what format the body is in — is it JSON? URL-encoded form data? Plain text? Content-Length tells the server how many bytes the body contains. Without Content-Length, the server reads and reads, waiting for the body to end."

> "Two formats you'll see constantly. application/json looks like {'name':'Alice'} — the standard format for API calls. application/x-www-form-urlencoded looks like name=Alice&email=alice%40example.com — that's what HTML forms have used since the 90s. Same data, different encoding."

---

### Run the demo

**SPEAK**
> "Watch the greater-than section. You're going to see Content-Type and Content-Length headers that weren't in the GET request."

**TYPE**
```bash
curl -v -s \
  -X POST http://httpbin.org/post \
  -H "Content-Type: application/json" \
  -d '{"name":"Alice","course":"Anyone Can Code"}' \
  2>&1
```

**OUTPUT**
```
> POST /post HTTP/1.1
> Host: httpbin.org
> User-Agent: curl/8.7.1
> Accept: */*
> Content-Type: application/json
> Content-Length: 43
> 
} [43 bytes data]
* upload completely sent off: 43 bytes
< HTTP/1.1 200 OK
< Content-Type: application/json
< Content-Length: 493
< 
{
  "data": "{\"name\":\"Alice\",\"course\":\"Anyone Can Code\"}", 
  "headers": {
    "Content-Length": "43", 
    "Content-Type": "application/json", 
    ...
  }, 
  "json": {
    "course": "Anyone Can Code", 
    "name": "Alice"
  }
}
```

**EXPLAIN**
> "The start line changed: POST /post HTTP/1.1. Now we have Content-Type: application/json and Content-Length: 43. curl counted the bytes in our -d string — 43 characters — and added that header automatically."

> "The curly brace line — '} [43 bytes data]' — is curl showing the upload direction. The body is going up to the server."

> "httpbin.org /post is a testing endpoint that echoes everything back. Look at the response body: 'data' is the raw body as a string, 'headers' is what headers the server received — it confirms it got Content-Type and Content-Length — and 'json' is the parsed body. The server correctly identified our JSON, parsed it, and shows us the fields. That's how you verify a POST is working."

**TYPE**
```bash
curl -s \
  -X POST http://httpbin.org/post \
  -d "username=alice&message=Hello+World" \
  | python3 -c "import sys,json; d=json.load(sys.stdin); print('Content-Type sent:', d['headers']['Content-Type']); print('Form data received:', json.dumps(d['form'], indent=2))"
```

**OUTPUT**
```
Content-Type sent: application/x-www-form-urlencoded
Form data received: {
  "message": "Hello World",
  "username": "alice"
}
```

**EXPLAIN**
> "Without -H, curl defaults to Content-Type: application/x-www-form-urlencoded when you use -d. The data format is key=value pairs separated by ampersands — same as a URL query string. The plus sign in 'Hello+World' decoded to a space. That's URL-encoding: spaces become plus signs. The server received the same key-value pairs but decoded."

## PART 4 — Cookies: How Servers Remember You

---

### What is a cookie?

**SPEAK**
> "HTTP is stateless. Every request arrives with no memory of previous ones. The server can't tell whether request number 5,000 and request number 5,001 came from the same person. Cookies solve this."

> "The mechanism is exactly two HTTP headers. When a server wants you to remember something, it puts Set-Cookie in a response header — Set-Cookie: flavor=chocolate. Your browser stores that value. On every future request to that domain, your browser adds a Cookie request header — Cookie: flavor=chocolate. The server reads it and knows who you are."

> "That's it. No magic. No special protocol. Just two headers. The browser is doing the bookkeeping, but the mechanism is entirely visible in the HTTP conversation."

> "Let's trace the full round-trip."

---

### Run the demo

**SPEAK**
> "Step 1: watch the Set-Cookie header arrive."

**TYPE**
```bash
curl -v -s -o /dev/null http://httpbin.org/cookies/set?flavor=chocolate 2>&1 \
  | grep -E "^[<>]"
```

**OUTPUT**
```
> GET /cookies/set?flavor=chocolate HTTP/1.1
> Host: httpbin.org
> User-Agent: curl/8.7.1
> Accept: */*
> 
< HTTP/1.1 302 FOUND
< Location: /cookies
< Set-Cookie: flavor=chocolate; Path=/
< 
```

**EXPLAIN**
> "There it is. The server sent us a 302 FOUND redirect — it wants us to go to /cookies. And in the same response: Set-Cookie: flavor=chocolate; Path=/. That's the server saying 'remember this, and send it back on all paths for this domain.' Without -L, curl shows us the 302 and stops. A browser would follow the redirect automatically, sending the cookie on the way. Let's do that."

**SPEAK**
> "Step 2: save the cookie to a file and follow the redirect."

**TYPE**
```bash
curl -s -c /tmp/my_cookies.txt \
  "http://httpbin.org/cookies/set?flavor=chocolate" \
  -L -o /dev/null

cat /tmp/my_cookies.txt
```

**OUTPUT**
```
# Netscape HTTP Cookie File
# https://curl.se/docs/http-cookies.html
# This file was generated by libcurl! Edit at your own risk.

httpbin.org	FALSE	/	FALSE	0	flavor	chocolate
```

**EXPLAIN**
> "-c tells curl to save received Set-Cookie headers to a file. The format is tab-separated: domain, subdomain flag, path, secure-only flag, expiry in Unix time (0 means session cookie), name, value. Everything the browser would store internally, written to a plain text file. The cookie jar."

**SPEAK**
> "Step 3: send the cookie back on the next request."

**TYPE**
```bash
curl -v -s -b /tmp/my_cookies.txt "http://httpbin.org/cookies" 2>&1
```

**OUTPUT**
```
> GET /cookies HTTP/1.1
> Host: httpbin.org
> User-Agent: curl/8.7.1
> Accept: */*
> Cookie: flavor=chocolate
> 
< HTTP/1.1 200 OK
< Content-Type: application/json
< 
{
  "cookies": {
    "flavor": "chocolate"
  }
}
```

**EXPLAIN**
> "-b tells curl to read cookies from the file. It found httpbin.org in the file and automatically added Cookie: flavor=chocolate to the request. Look at the greater-than section: Cookie header is there, right alongside Host and User-Agent."

> "The response body confirms it: the server received the cookie and echoes it back. This is the entire cookie mechanism. Server sends Set-Cookie. Client stores it. Client sends Cookie on future requests. Server reads it. Session state implemented entirely in HTTP headers."

## PART 5 — Raw HTTP with nc: No Library, Just Text

---

### What is nc?

**SPEAK**
> "Everything so far has been curl. curl knows about HTTP — it adds headers, handles redirects, manages cookies. Now we're going to go one level lower: netcat."

> "nc is a TCP utility. It opens a raw TCP connection to a host and port, and then it does exactly two things: it sends whatever you type to the remote end, and it prints whatever the remote end sends back. That's all. It knows nothing about HTTP. It doesn't add any headers. It doesn't format anything. It just moves bytes."

> "This is the proof. If you can type an HTTP request into nc and get a valid HTTP response back, then HTTP is just text. No special software required. Any tool that can write to a TCP socket can speak HTTP."

---

### The request

**SPEAK**
> "Here's what we're going to type. Three lines."

> "GET / HTTP/1.0 — that's the start line. Notice: HTTP/1.0, not HTTP/1.1. This is important. HTTP/1.1 defaults to persistent connections — after the server responds, it keeps the TCP socket open waiting for more requests. nc would sit there forever. HTTP/1.0 tells the server: close the connection after this one response. nc exits when the connection closes."

> "Host: example.com — we need this even in HTTP/1.0 because example.com could resolve to a server hosting multiple websites."

> "Then a blank line. Press Enter twice: once to end the Host line, once more to send the blank separator. The server will not respond until it sees that blank line."

---

### Run it

**SPEAK**
> "On macOS, nc -c converts your Enter key — which sends a single newline character — into the CRLF that HTTP requires. HTTP spec says carriage return plus line feed. Most servers are lenient, but -c is correct."

**TYPE**
```bash
nc -c example.com 80
```

*(Type each line, pressing Enter after each)*

```
GET / HTTP/1.0
Host: example.com

```

**OUTPUT**
```
HTTP/1.1 200 OK
Date: Tue, 14 Apr 2026 20:04:44 GMT
Content-Type: text/html
Connection: close
Server: cloudflare

<!doctype html><html lang="en"><head><title>Example Domain</title>...
```

*(Your Date will differ.)*

**EXPLAIN**
> "The server responded. And notice: Connection: close. That's the server confirming it's going to close the TCP connection after sending this response, because we asked for HTTP/1.0. nc exits when the connection closes."

> "You just spoke HTTP. No curl. No library. No framework. You opened a TCP socket, typed six lines including a blank one, and a server in a data center somewhere sent you an HTML page. That is literally what curl does — curl just automates those six lines."

---

### The scripted version

**SPEAK**
> "The nc exercise in the lab is interactive — you type it yourself. But for the demo script we use a scripted version that sends the same bytes non-interactively."

**TYPE**
```bash
(printf "GET / HTTP/1.0\r\nHost: example.com\r\n\r\n"; sleep 3) | nc example.com 80
```

**OUTPUT**
```
HTTP/1.1 200 OK
Date: Tue, 14 Apr 2026 20:04:44 GMT
Content-Type: text/html
Connection: close
Server: cloudflare

<!doctype html><html lang="en"><head><title>Example Domain</title>
<meta charset="utf-8" />
...
```

**EXPLAIN**
> "printf sends the exact bytes — the explicit \\r\\n is the CRLF that HTTP requires. The sleep 3 keeps stdin open so the response has time to arrive before nc closes — macOS BSD netcat has a quirk where it shuts down the socket quickly when stdin hits EOF. Same request, same response, no typing required."

**SPEAK**
> "Let's zoom out. You've now seen HTTP from five angles."

> "curl -v showed you that every HTTP conversation is start line, headers, blank line, body. The blank line is the separator — server won't process without it. The greater-than prefix is what you sent, less-than is what came back."

> "Status codes are the server's first word in every response. 200 means fine. 301 means it moved — here's the Location. 404 means it doesn't exist. 500 means the server broke."

> "POST adds a body to the request. When you add a body you need Content-Type — what format is this — and Content-Length — how many bytes. curl sets Content-Length automatically. Frameworks do too. But now you know what they're doing."

> "Cookies are two headers. Set-Cookie flows from server to client. Cookie flows from client to server. Everything stored in plain text on both ends. The browser is just a cookie jar that sends them back automatically."

> "And nc: HTTP is text over TCP. If you can write to a socket, you can speak HTTP. curl, your browser, every HTTP library you will ever use — they are all doing what you did with nc. They just automate the typing."

## Recording Notes

- **Terminal state at start:** fresh terminal, dark theme, font 16+
- **Time budget:** ~30–35 minutes. Spend the most time on Part 1 (HTTP anatomy) — it's the foundation. Parts 2–4 go faster once the structure is clear.
- **Verbosity level:** Speak every curl flag the first time it appears. Students will copy-paste these commands — they should understand what each flag does.
- **nc section:** Type the request slowly and deliberately. Pause after `Host: example.com` before pressing Enter the second time — say out loud "this next Enter is the blank line" before pressing it.
- **Variables that will differ:** Date headers, IP addresses, CF-RAY values, Transfer-Encoding values, Content-Length values. These are noted in the OUTPUT blocks. Tell students their output will look structurally identical but with different timestamps and IPs.
- **httpbin.org dependency:** All sections B–D use httpbin.org. If httpbin.org is down during recording, substitute `httpbin.org` with `postman-echo.com` — it supports the same POST and header echo endpoints.
- **Platform note:** The nc interactive exercise uses `-c` for macOS. If recording on Linux, mention that Linux users run `printf "GET / HTTP/1.0\r\nHost: example.com\r\n\r\n" | nc -q 3 example.com 80` instead. The -c flag is macOS BSD netcat only.
- **Cut point between parts:** The script runs all five sections sequentially when called as `bash http_demo.sh`. For the recording, run each section command individually so the output is visible one section at a time. Add `echo "---"` separators in the terminal by hand if needed.
