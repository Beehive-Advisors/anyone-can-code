## PART 1 — READING THE HTTP CONVERSATION WITH CURL -V

### What is `curl -v`?

**SPEAK**
> "When you use curl without any flags, it shows you the response body — the content. The same thing a browser shows you. But a browser is hiding a conversation that happened before you saw that content."

> "The -v flag — verbose — makes that conversation visible. Every line curl sends gets printed with a > prefix. Every line the server sends back gets a < prefix. And curl's own status notes get a * prefix. These are the actual bytes going over the TCP connection, decoded into readable text."

> "There's one gotcha you need to know first. The verbose output — the >, <, and * lines — goes to stderr, not stdout. So if you try to pipe the output or grep it, you get nothing. The fix is 2>&1, which merges stderr into stdout before the pipe."

### Run the demo

**TYPE**
```bash
bash http_curl_demo.sh
```

**EXPLAIN**
> "The * lines at the top are curl's notes. DNS resolved httpbin.org to an IP. Then the TLS handshake — the encrypted channel being set up. This is HTTPS, so before any HTTP text goes over the wire, the two sides negotiate encryption. The SSL line tells you which cipher suite they agreed on."

> "Then the > lines: your request. GET /get HTTP/2 is the request-line — method, path, version. Then the headers: Host, User-Agent, Accept. A blank > line signals the end of headers."

> "Then the < lines: the server's response. HTTP/2 200 is the status line. Then response headers: content-type, content-length, server. A blank < line signals end of headers. The body — 254 bytes — went to /dev/null so we could read the headers without the JSON getting in the way."

**OUTPUT**
```
═══════════════════════════════════════════════════════════════
SECTION B — The stderr gotcha: why piping -v output fails
═══════════════════════════════════════════════════════════════

  --- Fails: piping without 2>&1 ---
  Command: curl -v -s https://httpbin.org/get | grep 'Host'
  Result:
  (nothing found — the Host line is on stderr, not stdout)

  --- Works: merge stderr into stdout first ---
  Command: curl -v -s -o /dev/null https://httpbin.org/get 2>&1 | grep 'Host'
  Result:
* Host httpbin.org:443 was resolved.
> Host: httpbin.org
```

**EXPLAIN**
> "The pipe | only connects stdout. The -v lines go to stderr. So grep sees an empty stream and finds nothing. Adding 2>&1 before the pipe merges stderr into stdout, and now grep can see the > and < lines. You'll use this constantly when debugging API calls."

### Exercise — Custom Header

**SPEAK**
> "Try it yourself. Add a custom header to a request and find it in the > block. Use 2>&1 | grep to extract just that line."

**TYPE**
```bash
curl -v -s -o /dev/null https://httpbin.org/headers \
  -H "X-My-Name: Alice" 2>&1 | grep "X-My-Name"
```

**OUTPUT**
```
> X-My-Name: Alice
```

**EXPLAIN**
> "The -H flag injects any header you choose into the outgoing request. The > prefix confirms it was in YOUR request. Httpbin's /headers endpoint echoes all headers back in its JSON response — try it without -o /dev/null to see the full echo."

## PART 2 — SENDING DATA WITH POST

### What is a POST request?

**SPEAK**
> "A GET request says 'give me something.' It carries no data. When you click a link, your browser sends a GET."

> "A POST request says 'here is something — process it.' When you submit a form or call an API, you send a POST. The difference at the protocol level: POST has a body. A chunk of data after the headers. The server needs to know the format before it can read it — that's the Content-Type header."

> "The http_curl_demo.sh script we ran includes two POST sections. Sections C and D. Let's look at them."

### Exercise — JSON POST

**SPEAK**
> "Your exercise: send a POST to httpbin.org/post with a JSON body of your choice. Use -H to set the Content-Type and -d for the body. Confirm httpbin echoes your data under the 'json' key, not 'form'."

**TYPE**
```bash
curl -s https://httpbin.org/post \
  -H "Content-Type: application/json" \
  -d '{"name": "Alice", "score": 42}'
```

**OUTPUT**
```json
{
  "form": {},
  "json": {
    "name": "Alice",
    "score": 42
  },
  ...
}
```

**EXPLAIN**
> "The 'form' key is empty because we labeled the body as JSON. 'json' has our data. This is exactly what every REST API call does — send JSON, label it correctly, and the server parses it. Wrong Content-Type and it ends up in the wrong place, or as a 400 error."

## PART 3 — STATUS CODES AND COOKIES

### What are status codes and cookies?

**SPEAK**
> "Every response starts with a three-digit number. Before you read a single byte of content, this number tells you how the request went. 2xx is success. 3xx is redirect. 4xx is a client error — you did something wrong. 5xx is a server error — their problem, not yours."

> "Cookies are the solution to a problem: HTTP has no memory. Each request is independent. But you need to stay logged in across multiple requests. The trick: after login, the server puts a 'Set-Cookie' header in the response with a session ID. Your client stores this and adds it to every future request as a 'Cookie' header. Text going back and forth in headers. That's all a session is."

### Run the demo

**TYPE**
```bash
bash http_status_cookies_demo.sh
```

**OUTPUT**
```
═══════════════════════════════════════════════════════════════

  The first digit tells you the category — no body needed.
```

**EXPLAIN**
> "The -w flag with %{http_code} prints just the status code as a number. This is useful in scripts: you can check whether a URL is up, what it returned, and branch on that — without downloading any content."

> "This is a session in miniature. Two independent HTTP requests. The second one 'knows' who you are because you sent back the token the first one gave you."

## PART 4 — HTTP IS JUST TEXT

### What is netcat?

**SPEAK**
> "Everything you've done with curl involves curl formatting the request text for you. You gave it flags, it formatted the headers, it opened the socket. What if we remove curl entirely?"

> "nc — netcat — is a tool with one job: open a TCP connection and relay bytes. Your keyboard to the socket; the socket to your screen. No HTTP awareness. No header formatting. Just bytes."

> "If HTTP is just text over TCP, then nc connected to a web server should be able to speak HTTP. All we have to do is type the request in the right format."

### Run the demo

**TYPE**
```bash
bash http_nc_demo.sh
```

> "HTTP/1.1 requires the Host header. Without it the server returns 400. ## PUTTING IT TOGETHER

**SPEAK**
> "Curl -v shows you what was always happening invisibly. The > and < lines are the request and response your browser hides. POST adds a body to the request, labeled with Content-Type so the server knows how to parse it. Status codes give the client category information in three digits. Cookies carry state across stateless requests. And nc proves that all of it is just text over a TCP socket — something you can type yourself."

> "When something breaks in an API call and you don't know why: curl -v is your first tool. When you want to know whether a server is up and what it returned: -s -o /dev/null -w %{http_code}. When you're debugging a session or a cookie flow: -c and -b with a jar file. These aren't tricks — they're the underlying protocol, visible."
