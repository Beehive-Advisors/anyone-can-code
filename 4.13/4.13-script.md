## PART 1 — HTTP Is Just Text

---

### What does an HTTP request look like?

**SPEAK**
> "Before we run anything, let me show you the actual format of an HTTP request.
>
> Every HTTP/1.1 request is three parts. First: a request line — the method, the path,
> and the version. GET means 'give me this resource.' The path is what comes after the
> domain in the URL. HTTP/1.1 is the version.
>
> Second: headers. One per line. Name, colon, value. The Host header is mandatory —
> it tells the server which domain you're asking for, because one server can host
> many domains.
>
> Third: a blank line. Just an empty line. This is the signal that headers are done.
> Without it, the server keeps reading, waiting for the next header. The connection hangs.
>
> That's the entire protocol. Let's prove it."

---

### Run Section A

**TYPE**
```bash
bash http_demo.sh
```

**OUTPUT** (Section A)
```
═══════════════════════════════════════════════════════════════════
SECTION A — HTTP Is Just Text
═══════════════════════════════════════════════════════════════════

An HTTP/1.1 GET request is plain, human-readable text.
Every request has exactly three parts:

  GET /path HTTP/1.1           request line  (method, path, version)
  Host: example.com            headers       (one per line, colon-separated)
  Connection: close
                               blank line    (signals: headers are done)

The blank line is required by the HTTP spec (RFC 9112).
Without it, the server keeps waiting for more headers — the connection hangs.
```

**EXPLAIN**
> "Look at what just happened. We typed three lines of text, piped them to nc — which
> just opened a TCP connection to port 80 — and CERN's server responded with a full
> HTTP response.
>
> Look at the response structure. The very first line is the status line: `HTTP/1.1 200 OK`.
> Version, code, reason phrase. 200 means success.
>
> Then the response headers — Server, Last-Modified, Content-Length, Content-Type. Each
> one is a name-colon-value line. Just like the request headers we sent.
>
> Then the blank line — you can see it between `Content-Type: text/html` and the `<html>`
> tag. End of headers.
>
> Then the body — the HTML page. `http://info.cern.ch — home of the first website.`
>
> We just connected to CERN's server and typed our HTTP request character by character.
> This is the same request your browser sends. The only difference is your browser
> generates it automatically. We just proved it's text."

## PART 2 — Annotated HTTP: curl -v

---

### What curl -v shows

**SPEAK**
> "nc gives you the raw bytes. curl -v gives you the annotated version. Same protocol,
> same data — but curl labels every line so you know what's a request header, what's a
> response header, and what's a connection event.
>
> Three prefixes. Asterisk for connection events — DNS resolution, TCP handshake. Right
> angle bracket for lines your computer sent. Left angle bracket for lines the server
> sent back. That's all of it."

**OUTPUT** (Section B — continues from running http_demo.sh)
```
═══════════════════════════════════════════════════════════════════
SECTION B — Annotated HTTP: curl -v
═══════════════════════════════════════════════════════════════════

curl -v annotates the conversation line by line:

  *  connection events  (DNS lookup, TCP handshake)
  >  text YOUR computer sent  (the HTTP request)
  <  text the SERVER sent back (the HTTP response headers)

Running: curl -v http://info.cern.ch ...

* Host info.cern.ch:80 was resolved.
* IPv6: 2001:1458:201:a4::100:1a0
* IPv4: 188.184.67.127
*   Trying [2001:1458:201:a4::100:1a0]:80...
* Connected to info.cern.ch (2001:1458:201:a4::100:1a0) port 80
> GET / HTTP/1.1
> Host: info.cern.ch
> User-Agent: curl/8.7.1
> Accept: */*
>
* Request completely sent off
< HTTP/1.1 200 OK
< Date: Tue, 14 Apr 2026 21:49:33 GMT
< Server: Apache
< Last-Modified: Wed, 05 Feb 2014 16:00:31 GMT
< ETag: "286-4f1aadb3105c0"
< Accept-Ranges: bytes
< Content-Length: 646
< Connection: close
< Content-Type: text/html
<
{ [646 bytes data]
* Closing connection

The > block above IS the HTTP request — the same text nc sent in Section A.
The < block IS the HTTP response headers — the same text nc received.
```

*(Your Date will differ.)*

**EXPLAIN**
> "Walk through this with me.
>
> The asterisk lines are connection events. DNS resolved info.cern.ch to an IPv4 and IPv6
> address. TCP connected on port 80. Those happen before any HTTP.
>
> Now the right-angle-bracket block — the request. Look at `GET / HTTP/1.1` — that's
> the request line. Then `Host: info.cern.ch`. Then notice two extra headers: `User-Agent`
> and `Accept`. We didn't add those. curl adds them automatically as defaults.
>
> In Part 1, we only sent three headers with nc. curl sends five. The server handled
> both the same way — the User-Agent and Accept headers are optional, just polite.
>
> Then the blank `>` line — that's the blank line. Headers are done. Request sent.
>
> Now the left-angle-bracket block — the response. `HTTP/1.1 200 OK`. Then all the
> response headers. Then the blank `<` line — end of response headers.
>
> `{ [646 bytes data] }` — that's the body. We said `-o /dev/null` to discard it,
> so curl counted the bytes but threw them away.
>
> The critical thing: the `>` block and the `<` block are exactly what nc printed in
> Section A. Same server, same port, same protocol. curl just labels the lines."

## PART 3 — POST: Sending Data to the Server

---

### GET vs POST

**SPEAK**
> "GET puts parameters in the URL. You've seen this — `/search?q=python+tutorial`. The
> whole question is in the URL, which gets logged, cached, and stored in browser history.
>
> POST puts parameters in the body — the bytes after the blank line. The body doesn't
> appear in URLs. This is why login forms use POST. Your password travels in the body,
> invisible in the URL.
>
> The server needs to know how to parse the body. That's the Content-Type header. Two
> common values: `application/json` for REST APIs, and `application/x-www-form-urlencoded`
> for HTML forms.
>
> curl does NOT automatically set Content-Type to JSON when you pass JSON data. If you
> forget the header, curl defaults to form encoding and the server will misparse your JSON."

**OUTPUT** (Section C — continues)
```
═══════════════════════════════════════════════════════════════════
SECTION C — POST: Sending Data to the Server
═══════════════════════════════════════════════════════════════════

GET fetches data. POST sends it.
The data travels in the body — the bytes after the blank line.

httpbin.org/post echoes the full request back as JSON.
Look at the 'headers' field — it shows what curl sent.

--- JSON POST ---
(must set Content-Type manually — curl does not default to JSON)
```

**OUTPUT** (JSON POST response)
```
{
  "args": {},
  "data": "{\"name\": \"alice\", \"age\": 30}",
  "files": {},
  "form": {},
  "headers": {
    "Accept": "*/*",
    "Content-Length": "28",
    "Content-Type": "application/json",
    "Host": "httpbin.org",
    "User-Agent": "curl/8.7.1",
    "X-Amzn-Trace-Id": "Root=1-..."
  },
  "json": {
    "age": 30,
    "name": "alice"
  },
  "origin": "...",
  "url": "https://httpbin.org/post"
}
```

*(X-Amzn-Trace-Id and origin will differ — they're assigned by AWS per request.)*

**EXPLAIN**
> "httpbin.org echoes our request back. Look at the 'headers' field. `Content-Type:
> application/json` — that's what we set with `-H`. `Content-Length: 28` — curl
> calculated that automatically from the length of our JSON string. We didn't specify it.
>
> Now look at 'json' — `{'age': 30, 'name': 'alice'}`. The server parsed our body as
> JSON successfully because we declared the right Content-Type.
>
> And 'form' is empty. If we'd sent this as form data, those names would be there instead."

**OUTPUT** (Form POST response)
```
--- Form data POST (what HTML forms submit) ---
(no -H needed: curl defaults to application/x-www-form-urlencoded)

{
  "form": {
    "city": "boston",
    "username": "alice"
  },
  "headers": {
    "Content-Type": "application/x-www-form-urlencoded",
    ...
  },
  "json": null,
  ...
}
```

**EXPLAIN**
> "No `-H` flag this time. curl defaulted to `application/x-www-form-urlencoded`. Same
> flag, different format. The data ended up in 'form' instead of 'json'. `json` is null.
>
> The body format and the Content-Type must match. If they don't, the server parses
> garbage or returns 400."

## PART 4 — Status Codes: The Server's Verdict

---

### The five classes

**SPEAK**
> "Every HTTP response starts with a three-digit code. The first digit is the category.
> 2xx is success. 3xx is redirect — go look somewhere else. 4xx is client error — you
> made a mistake. 5xx is server error — the server has a problem.
>
> Once you know the five categories, you can guess what any code means. 429? 4xx class,
> so it's your fault — probably rate limiting. 503? 5xx class, so the server is down.
> You don't need to memorize every code."

**OUTPUT** (Section D)
```
═══════════════════════════════════════════════════════════════════
SECTION D — Status Codes: The Server's Verdict
═══════════════════════════════════════════════════════════════════

Every HTTP response begins with a three-digit status code.
The first digit is the category:

  2xx = success            3xx = redirect (go somewhere else)
  4xx = client error       5xx = server error

httpbin.org/status/N returns exactly status code N. Trying several:

  Requested 200  →  received 200
  Requested 201  →  received 201
  Requested 301  →  received 301
  Requested 400  →  received 400
  Requested 404  →  received 404
  Requested 500  →  received 500
```

**EXPLAIN**
> "httpbin.org returns exactly the code you ask for. Every row requested a code and
> received it. This confirms: the first digit is the category, and the server controls
> what it returns. You can't fake or override this on the client side."

**OUTPUT** (Exit code section)
```
--- curl exit code vs HTTP status ---

curl exits 0 even when the server returns 404 or 500.
The exit code means: did the network transfer complete?
It does NOT mean: did the server accept the request?

curl exit code after a 404 response: 0

Add --fail to exit non-zero on 4xx/5xx:

curl exit code with --fail on 404: 56  (non-zero = server returned error)
```

**EXPLAIN**
> "This is a subtle but critical point. curl returns exit code 0 after a 404 response.
> Not because the request succeeded. Because the transfer succeeded — curl connected,
> sent the request, received a response. From curl's perspective, that's a completed job.
>
> The HTTP status code is in the response data. curl's exit code is about the network
> transport. These are different things.
>
> Without `--fail`, a shell script checking `$?` will never know the server returned an
> error. You'd silently process a 404 response as if it were a 200.
>
> Add `--fail` and curl checks the HTTP layer too. Non-zero exit on 4xx or 5xx."

### Exercise 4

**SPEAK**
> "Write a one-liner that checks whether a URL returns 200 and prints OK or the error code."

**TYPE**
```bash
code=$(curl -s -o /dev/null -w "%{http_code}" https://httpbin.org/status/418)
[ "$code" = "200" ] && echo "OK" || echo "FAIL $code"
```

**OUTPUT**
```
FAIL 418
```

**EXPLAIN**
> "418 is 'I'm a teapot' — an April Fools RFC that stuck around. The pattern here is
> what matters: extract the status code with `-w '%{http_code}'`, suppress the body
> with `-o /dev/null`, check it with a shell test.
>
> This is the standard way to do HTTP health checks in shell scripts."

## PUTTING IT TOGETHER

**SPEAK**
> "Let's connect the four parts.
>
> nc showed you HTTP at the lowest level: text over a TCP socket. No library. You wrote
> it yourself.
>
> curl -v showed you the same exchange annotated: asterisks for connection events,
> right brackets for what you sent, left brackets for what the server sent back.
>
> curl -d added a body and a Content-Type header. POST puts data after the blank line.
> The server parses it according to the Content-Type you declared.
>
> Status codes gave you the server's verdict as a three-digit number, before the body.
> First digit = category. `--fail` makes curl treat 4xx/5xx as errors.
>
> Every HTTP client — Python's requests library, JavaScript's fetch, your browser —
> is doing exactly what nc did. Composing text in this format and sending it over a
> TCP connection. The libraries hide the format. Now you know what's underneath."
