## Concept: What HTTP Actually Is

### The wire

**SPEAK**
> "When your browser loads a web page, this is what physically happens: your computer opens a TCP connection — a reliable two-way channel — to port 80 on the server. Then your computer sends a few lines of text. The server reads those lines and sends a few lines of text back. That's it. HTTP is a convention about how to format that text."

> "There's no binary encoding. No special handshake beyond TCP. No custom protocol. If you can write lines of text to a TCP socket, you can speak HTTP. That's what we'll prove in Part 5 of the lab."

### The four-part structure

**SPEAK**
> "Every HTTP message — request or response — has exactly four parts, always in this order."

> "Part one: the start line. For a request, it's the method, the path, and the version. GET / HTTP/1.1 — that's the whole thing. For a response, it's the version, the status code, and a short description. HTTP/1.1 200 OK."

> "Part two: headers. One per line, format is Name colon space Value. These are metadata. Host tells the server which website you're requesting — important because multiple websites can share one IP address. Content-Type tells the server or client what format the body is in. There are dozens of standard headers."

> "Part three: the blank line. A line containing nothing. This is the separator between headers and body. It is the single most important structural element in the protocol. Without it, the server waits forever — it's still reading headers, waiting for the next one, and your request is never processed. We'll see this concretely when we use nc."

> "Part four: the body. Optional. GET requests have no body. POST requests have a body — it's the data you're sending. Responses usually have a body — it's the content you asked for."

### Status codes

**SPEAK**
> "Before we get to the code: status codes. Every response starts with a three-digit number. It's grouped into ranges. 2xx means success — 200 is 'here's your content'. 3xx means redirect — the thing you want is somewhere else; the Location header tells you where. 4xx means you made an error — 404 is 'that path doesn't exist on this server'. 5xx means the server broke."

> "The most interesting for understanding HTTP is 301. When a server sends a 301, it also sends a Location header with the new address. Without the -L flag, curl shows you the 301 and stops — you can see the Location header explicitly. With -L, curl follows the chain automatically. We'll use this in Section B."

### Cookies: two headers

**SPEAK**
> "One more concept before the code: cookies. HTTP is stateless — every request arrives with no memory of previous ones. Cookies add memory. The mechanism is two headers."

> "The server sets a cookie by putting Set-Cookie in a response header: `Set-Cookie: flavor=chocolate`. The client stores that. On every future request to that domain, the client adds `Cookie: flavor=chocolate` to the request headers. The server reads it."

> "That's it. No special protocol. No magic. Two HTTP headers. The browser is just a cookie jar that stores Set-Cookie values and sends them back automatically. We'll see the raw headers in Section D."

## http_demo.sh

code http_demo.sh

### Header

**EXPLAIN**
> "Line 1: `#!/usr/bin/env bash` — the shebang. Tells the OS to run this file with bash. Without it, the file wouldn't be executable as a script."

> "Line 11: `set -euo pipefail`. Three options. `-e` means exit immediately if any command fails — we won't silently continue past errors. `-u` means treat unset variables as errors. `-o pipefail` means a pipeline fails if any command in it fails, not just the last one. These three together make the script behave predictably."

### Section A — curl -v

**EXPLAIN**
> "Lines 27: `curl -v -s http://example.com/ 2>&1`. Four parts."

> "`-v` is verbose — shows all headers sent and received, with prefixes: `*` for curl metadata, `>` for what we sent, `<` for what the server sent back."

> "`-s` is silent — suppresses the progress bar that curl normally shows. Without it, the progress bar and the verbose output interleave and look messy."

> "`2>&1` — curl sends verbose output to stderr. This redirect merges stderr into stdout so everything appears in the same stream. Without it, if you pipe the output, you'd lose the headers."

> "Line 33: `curl -s -o /dev/null -w \"HTTP Status: %{http_code}\\n\" http://example.com/`. Three new flags."

> "`-o /dev/null` — throw away the response body. We don't want to see the HTML, just the status code."

> "`-w \"...\"` — write-out. After the transfer completes, print this format string. `%{http_code}` is replaced with the actual three-digit status code."

### Section B — Status codes

**EXPLAIN**
> "Lines 53–55: the 301 demo. `-o /dev/null` discards the body. `grep -E \"^[*<>]\"` filters output to only lines starting with `*`, `<`, or `>` — so we see just the HTTP conversation, not the response body."

> "Lines 64–66: the same URL but with `-L`. `-L` means follow Location redirects. curl makes the initial request, reads the 301 and Location header, makes a second request to the redirect target, and continues until it gets a non-3xx response."

> "`-w \"%{url_effective}\"` — prints the final URL after all redirects. Useful for scripts that need to know where they actually ended up."

### Section C — POST

**EXPLAIN**
> "Lines 86–91: POST with JSON body. Four new flags here."

> "`-X POST` — sets the HTTP method. Without this, curl defaults to GET."

> "`-H \"Content-Type: application/json\"` — adds a request header. The `-H` flag takes any `Name: Value` string. You can add multiple headers with multiple `-H` flags."

> "`-d '...'` — sets the request body. curl measures the byte length of the body string and automatically adds `Content-Length: 43` to the headers. You don't have to compute it manually."

> "Lines 98–102: form data. No `-H` needed — when you use `-d` without a Content-Type header, curl defaults to `application/x-www-form-urlencoded`. The body format is `key=value&key=value`, same as a URL query string. The `+` in `Hello+World` is URL-encoding for a space."

> "The one-liner at line 102: `python3 -c \"import sys,json; ...\"` — this takes the JSON response from httpbin and pretty-prints just the relevant fields. A way to extract specific values from JSON in a shell pipeline without a tool like jq."

### Section D — Cookies

**EXPLAIN**
> "Line 117: the first cookie request. `grep -E \"^[<>]\"` shows only sent and received headers — no curl metadata. We want to see the raw Set-Cookie header."

> "Lines 127–129: `-c /tmp/lab413_cookies.txt` saves received Set-Cookie headers to a file in Netscape cookie format — tab-separated fields: domain, subdomain flag, path, secure-only flag, expiry timestamp, name, value. `-L` follows the redirect. `-o /dev/null` discards the body."

> "Line 141: `-b /tmp/lab413_cookies.txt` reads cookies from the file and adds the Cookie header automatically. curl looks up the domain, finds matching cookies, and adds `Cookie: flavor=chocolate` to the request. This is what the browser does — stores cookies keyed by domain, sends them back automatically."

> "Line 147: `-b \"flavor=chocolate\"` — inline cookie, no file. Useful when you know the cookie value and don't need persistence."

> "Line 151: `rm -f /tmp/lab413_cookies.txt` — cleanup. The `-f` flag means 'don't fail if the file doesn't exist'."

### Section E — nc

**EXPLAIN**
> "Line 175: `(printf \"GET / HTTP/1.0\\r\\nHost: example.com\\r\\n\\r\\n\"; sleep 3) | nc example.com 80`"

> "Starting from inside: `printf \"...\\r\\n\"` sends the exact bytes we want. `\\r\\n` is carriage return plus line feed — that's what HTTP requires as a line ending. Enter in a terminal sends only `\\n`. Most servers are lenient, but we're using the correct format."

> "We send three lines: the request line (`GET / HTTP/1.0`), the Host header, and a blank line (just `\\r\\n` with nothing before it). That blank line is the separator — the server won't respond until it sees it."

> "`HTTP/1.0` not `HTTP/1.1` — we'll explain this in the lab. Short version: HTTP/1.1 keeps connections open after responding. nc would hang. HTTP/1.0 closes after one response."

> "The `sleep 3` — macOS's nc closes the socket very quickly when stdin hits EOF. We keep stdin open for 3 seconds to give the server time to respond before nc disconnects. On Linux you'd use `-q 3` flag instead."

> "The parentheses create a subshell — both commands run in the same shell context, and stdout from both goes into the pipe."

> "Lines 185–195: the interactive version instructions. This is what students type manually. `nc -c example.com 80` — the `-c` flag on macOS converts each Enter (LF) to CRLF before sending, handling the line ending requirement automatically."

## Recording notes

- **Conceptual sections first** — spend 8–10 minutes on the HTTP concept before opening the file. Students who understand the four-part structure will recognize each flag in context.
- **Scroll slowly in VS Code** — the script is 207 lines. Zoom in on the command blocks; don't rush through comments.
- **Pause on `set -euo pipefail`** — explain each flag. Students writing their own shell scripts should always start with this.
- **Pause on the `2>&1` redirect** — this confuses people. Draw out: curl writes verbose to stderr, `2>&1` means "send stderr (2) to wherever stdout (1) is going".
- **Pause on `sleep 3` in Section E** — explain the macOS nc EOF behavior. Students on Linux will use a different approach.
- **Connect the blank line** — every time you see a blank line in the output during the lab, connect it back to the four-part structure concept: "that's the separator you heard about in the walkthrough."
