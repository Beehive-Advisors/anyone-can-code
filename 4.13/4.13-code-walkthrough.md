## Concept: HTTP is a Text Conversation

**SPEAK**
> "Let's start with the core idea before we look at any code."

> "When you visit a website, your browser sends a request and receives a response. Both are text — a short block of formatted text that travels over a TCP connection. The browser hides this from you. It shows you a rendered webpage. But the actual exchange looks more like two people typing messages to each other."

> "HTTP defines the format of those messages. The request has three parts: a request-line (method, path, version), then any headers (key-value pairs), then a blank line to signal 'I'm done with headers.' The response has the same structure: a status line, then headers, then the body."

> "Everything in these three demo files is showing you that exchange in different ways. curl -v labels the lines. nc shows you the raw bytes. The demos exist to make this text visible."

## http_curl_demo.sh — Section A: Without -v vs With -v

**TYPE**
```bash
code 4.13/http_curl_demo.sh
```

**EXPLAIN** (lines 1–10 — file header)
```bash
#!/usr/bin/env bash
# http_curl_demo.sh — Reading raw HTTP with curl -v and sending POST requests
#
# What this shows:
#   - What curl -v reveals about the HTTP conversation (>, <, * lines)
#   - How POST sends a body with a Content-Type label
#   - How the same bytes with a different Content-Type are parsed differently
#   - The stdout vs stderr distinction in curl -v output
#
# Run:
#   bash http_curl_demo.sh
```
> "Lines 1–10 are the standard header for every demo in this course: what the file shows and how to run it. The shebang on line 1 tells the OS to use bash."

**EXPLAIN** (lines 25–26 — body-only curl)
```bash
# -s suppresses the progress meter; we only see the JSON response
curl -s https://httpbin.org/get | head -5
```
> "Line 26 runs curl without -v. The -s flag suppresses the download progress bar so we see just the response body. We pipe it to head -5 to show only the first five lines — the JSON response that httpbin sends back. This is what your browser equivalent would show: the content, with no conversation visible."

**EXPLAIN** (lines 36–39 — verbose curl)
```bash
# -s suppresses progress meter
# -o /dev/null discards the body so we see only the header conversation
# 2>&1 merges stderr (where -v output lives) into stdout
curl -v -s -o /dev/null https://httpbin.org/get 2>&1
```
> "Line 39 adds three flags. -v turns on verbose — this is what reveals the >, <, and * lines. -o /dev/null throws away the response body so the terminal shows only the header conversation. And 2>&1 is the critical piece: the -v output goes to stderr by default. Without 2>&1, we see nothing useful. With it, we merge stderr into stdout so both streams appear in the terminal together."

## http_curl_demo.sh — Section B: The stderr Gotcha

**EXPLAIN** (lines 53–54 — the failing pipe)
```bash
# This intentionally produces no output — that's the demo point
curl -v -s -o /dev/null https://httpbin.org/get 2>/dev/null | grep 'Host' || echo "  (nothing found — the Host line is on stderr, not stdout)"
```
> "Lines 53–54 demonstrate the gotcha explicitly. We pipe the curl output to grep looking for 'Host'. But we redirect stderr to /dev/null instead of stdout — so grep sees nothing. The || echo at the end catches the case where grep finds no matches and prints an explanation. This is intentional: students need to experience the failure before they see the fix."

**EXPLAIN** (line 60 — the fix)
```bash
curl -v -s -o /dev/null https://httpbin.org/get 2>&1 | grep 'Host'
```
> "Line 60 is the fix: 2>&1 merges stderr into stdout before the pipe. Now grep can see the verbose lines. Students will use this pattern constantly when debugging API calls."

## http_curl_demo.sh — Sections C and D: POST Requests

**EXPLAIN** (lines 77–80 — form POST)
```bash
# -v shows the full conversation
# The response JSON shows what httpbin received under "form"
curl -v -s https://httpbin.org/post \
  -d "username=alice&city=Portland" 2>&1
```
> "Lines 77–80: -d sets the request body to 'username=alice&city=Portland'. When curl sees -d, it automatically changes the method to POST and sets Content-Type to application/x-www-form-urlencoded. We're not writing a library — we're telling curl what to send, and it handles the formatting. The 2>&1 is there again so we see the full conversation. Watch for the > lines in the output: Content-Length and Content-Type will appear there because they describe the body curl is about to send."

**EXPLAIN** (lines 96–100 — JSON POST)
```bash
# -H overrides the Content-Type header
# Single quotes protect the JSON from shell interpretation
curl -v -s https://httpbin.org/post \
  -H "Content-Type: application/json" \
  -d '{"username": "alice", "city": "Portland"}' 2>&1
```
> "Lines 96–100: structurally identical to the form POST. The only differences are: -H sets Content-Type to application/json, overriding curl's default, and the body is now JSON syntax instead of key=value syntax. Single quotes around the -d value prevent the shell from interpreting the curly braces. Watch the httpbin response: it puts the data under 'json' instead of 'form' because of that one header change."

## http_status_cookies_demo.sh — Section A: Status Gallery

**TYPE**
```bash
code 4.13/http_status_cookies_demo.sh
```

**EXPLAIN** (lines 31–48 — status code gallery)
```bash
echo "  200 (Success):"
echo "  $(curl -s -o /dev/null -w '%{http_code}' https://httpbin.org/status/200) — 2xx means the request succeeded"
```
> "Lines 31–48 use command substitution — the $() syntax — to embed curl's output directly in an echo statement. The -w flag (write-out) is powerful: it prints a formatted string after the request completes. '%{http_code}' is a curl variable that expands to the numeric status code. -s suppresses the progress bar and -o /dev/null discards the body. The result: just the status code number, nothing else. Students will use this pattern in shell scripts to check whether a URL is up."

## http_status_cookies_demo.sh — Section C: Cookie Lifecycle

**EXPLAIN** (lines 16–17 — cookie jar setup)
```bash
COOKIE_JAR="$(dirname "$0")/cookies.txt"
rm -f "$COOKIE_JAR"
```
> "Lines 16–17 set up a cookie jar file path relative to the script's location, and delete any leftover file from a previous run. A cookie jar is just a text file where curl stores cookies between requests."

**EXPLAIN** (lines 83–85 — setting the cookie)
```bash
# /cookies/set returns a 302 redirect; -L follows it
# -c writes the cookie jar (Netscape format) to a file
curl -s -L -c "$COOKIE_JAR" "https://httpbin.org/cookies/set?user=alice&role=student"
```
> "Line 85 makes the first request. The -L flag follows redirects — httpbin's /cookies/set endpoint responds with a 302 and a Set-Cookie header, then redirects to /cookies. Without -L, curl stops at the 302 and the cookie is never saved. The -c flag tells curl to write any Set-Cookie headers it receives into the cookie jar file."

**EXPLAIN** (lines 96–97 — sending the cookie back)
```bash
# -b reads from the cookie jar and adds a Cookie: header to the request
curl -s -b "$COOKIE_JAR" https://httpbin.org/cookies
```
> "Line 97 makes the second request — to a different endpoint, /cookies. The -b flag reads the cookie jar file and adds a Cookie: header to the outgoing request. Httpbin echoes back whatever cookies it received. This is the stateless-HTTP-plus-cookie pattern in its simplest form: two independent requests that feel connected because the client carries state between them."

## http_nc_demo.sh — Sections A and B: nc by Hand

**TYPE**
```bash
code 4.13/http_nc_demo.sh
```

**EXPLAIN** (lines 33–37 — HTTP/1.0 request)
```bash
# printf correctly handles \r\n (CRLF) which HTTP requires
# nc connects, sends the request, and prints whatever the server sends back
# -w 5 times out after 5 seconds so the script does not hang
# head -20 shows headers + start of body (full body is HTML, not relevant here)
printf "GET / HTTP/1.0\r\n\r\n" | nc -w 5 httpforever.com 80 | head -20
```
> "Line 37: printf handles the \\r\\n escapes. If you just typed 'GET / HTTP/1.0' and pressed Enter in a terminal, you'd send \\n, not \\r\\n. Many servers accept either, but printf guarantees the spec-correct CRLF. The pipe to nc sends those bytes to a TCP connection on port 80. Nc returns whatever the server sends. Head -20 shows the first 20 lines so we don't dump the full HTML body. There's no magic here — printf makes a string, nc opens a socket and sends it."

**EXPLAIN** (lines 54–57 — HTTP/1.1 request)
```bash
printf "GET / HTTP/1.1\r\nHost: httpforever.com\r\nConnection: close\r\n\r\n" \
  | nc -w 5 httpforever.com 80 | head -15
```
> "Lines 54–57 add two headers. Host is mandatory in HTTP/1.1 — without it, a compliant server returns 400. Connection: close tells the server to close the socket after responding. Without it, HTTP/1.1 keeps the connection open waiting for another request, and nc would appear to hang. Compare the output of this section with Section A: the response has more headers and the Connection value changed."

## http_nc_demo.sh — Section C: nc as Server

**EXPLAIN** (lines 77–93 — nc server capturing curl's request)
```bash
RAW_REQUEST_FILE="$(mktemp)"

nc -l 18080 > "$RAW_REQUEST_FILE" &
NC_PID=$!

sleep 0.3

curl -s -o /dev/null --max-time 2 \
  http://localhost:18080/index.html \
  -H "User-Agent: curl-lab-demo" \
  -H "X-Lab: speaking-http-by-hand" \
  2>/dev/null || true
```
> "Lines 77–93 automate the two-terminal pattern for the demo. Line 77 creates a temporary file. Line 80 starts nc listening on port 18080 in the background — the & sends it to the background — and redirects its output to the temp file. That output is whatever the client sends. NC_PID captures the process ID so we can kill it later. Line 84 waits 0.3 seconds for nc to start. Lines 89–93 run curl with --max-time 2 so it doesn't hang waiting for a response that nc will never send. The || true means 'if curl fails, that's fine, keep going.' We don't care about the response here — we care about what nc received."

**EXPLAIN** (lines 100–102 — showing the captured request)
```bash
echo "  --- Raw HTTP request that curl sent to nc ---"
cat "$RAW_REQUEST_FILE"
rm -f "$RAW_REQUEST_FILE"
```
> "Lines 100–102 print the file nc wrote and clean it up. This is the payoff: the raw HTTP text that curl sent. Students see that curl, their browser, and every HTTP client formats the same kind of text. The tools are different; the protocol is the same."
