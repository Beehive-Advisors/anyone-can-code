## Concept: HTTP Is a Text Protocol

**SPEAK**
> "When you type a URL and press Enter, your browser sends a message to a server. What does
> that message look like? Most people assume it's some kind of binary format — compiled
> instructions, compressed data, something a machine has to interpret.
>
> It's not. An HTTP/1.1 request is plain text. Every character is human-readable ASCII.
> You could, in principle, type one yourself. This lab proves it by doing exactly that.
>
> Here's the structure. Every HTTP request has three parts. First: the request line. Three
> things on one line — the method, the path, and the version. GET means 'give me this
> resource.' The path is the thing after the domain in the URL. HTTP/1.1 is the version.
>
> Second: the headers. Each header is one line. Name, colon, space, value. The Host header
> is mandatory in HTTP/1.1 — the server uses it to know which virtual host you're asking for.
> Connection: close tells the server to close the TCP connection after responding. Without
> it, HTTP/1.1 keeps the connection open — and nc, which has no HTTP logic, will just sit
> there waiting.
>
> Third: the blank line. This is the most important structural rule in HTTP. The blank line
> is how the server knows your headers are done. Not a special byte. Not a length counter.
> Just an empty line — CRLF with nothing before it. Without it, the server keeps reading,
> waiting for the next header. The connection hangs.
>
> The response follows the same structure. A status line first — version, three-digit code,
> reason phrase. Then headers. Then a blank line. Then the body.
>
> The three-digit code is the server's verdict before you read a single byte of the body.
> 200 means success. 404 means not found. 500 means the server crashed. The first digit
> is the category — 2xx success, 3xx redirect, 4xx client mistake, 5xx server error.
> That pattern holds for every code you'll ever encounter."

## http_demo.sh — Section A: HTTP Is Just Text

**TYPE**
```bash
code 4.13/http_demo.sh
```

**EXPLAIN**
> "The file opens with a comment block. Every demo script in this course has this — what
> the file is, how to run it, and what tools it needs. On macOS, curl and nc are built in.
> On Linux, you'd install them with apt.
>
> Now Section A. The first part just echoes the request structure — it's documentation
> for the student, showing the three-part format with inline labels.
>
> Then: the actual command. Two parts joined by a pipe.
>
> On the left: `printf 'GET / HTTP/1.1\r\nHost: info.cern.ch\r\nConnection: close\r\n\r\n'`
>
> printf is used instead of echo because printf interprets escape sequences consistently
> on every platform. The `\r\n` sequences are literal carriage-return plus newline — ASCII
> codes 0x0D and 0x0A. The HTTP spec requires both. Most servers accept bare `\n` out of
> leniency, but the spec says `\r\n`. We send the correct thing.
>
> The last `\r\n\r\n` at the end is two CRLF sequences — that's the blank line. One `\r\n`
> ends the last header. The second `\r\n` is the blank line itself. That's when the server
> starts parsing.
>
> On the right: `nc -w 5 info.cern.ch 80`
>
> nc opens a TCP connection to info.cern.ch port 80. Port 80 is the default port for plain
> HTTP. nc takes whatever comes from printf's output — our hand-crafted request — and sends
> it byte for byte over that TCP socket. Whatever the server writes back, nc prints to
> stdout. The `-w 5` flag is a timeout: if 5 seconds pass with no data, nc exits. This
> prevents the demo from hanging if something goes wrong.
>
> info.cern.ch is CERN's original WWW server — the server where the web was invented in
> 1991. It still runs on port 80. We use it because it serves plain HTTP reliably, unlike
> many modern servers that are behind Cloudflare or require a full User-Agent header."

## http_demo.sh — Section B: Annotated HTTP with curl -v

**EXPLAIN**
> "Section B does the same thing as Section A — sends an HTTP GET to the same server —
> but uses curl instead of nc. The difference is annotation.
>
> `curl -v -s -o /dev/null http://info.cern.ch 2>&1`
>
> `-v` is verbose mode. It adds three prefixes to every line of output: asterisk for
> connection events, right-angle-bracket for lines your computer sent, left-angle-bracket
> for lines the server sent back.
>
> `-s` suppresses the progress bar. Without it, you'd see a spinning progress indicator
> mixed in with the headers.
>
> `-o /dev/null` discards the response body. We're studying headers, not content.
>
> `2>&1` is important. curl writes its verbose output to stderr, not stdout. That's
> intentional — it keeps the annotation separate from the body, so you can pipe the body
> to another command without the annotation mixed in. We merge them here so everything
> appears on screen in order."

## http_demo.sh — Section C: POST Requests

**EXPLAIN**
> "Section C demonstrates POST. Two commands back to back.
>
> The first sends JSON. The flags:
> `-H 'Content-Type: application/json'` adds a header that tells the server 'the body
> is JSON.' This is not optional — curl defaults to application/x-www-form-urlencoded,
> which is the HTML form format. If you send JSON without declaring it, the server will
> try to parse `{\"name\": \"alice\"}` as key=value pairs and fail.
>
> `-d '{\"name\": \"alice\", \"age\": 30}'` sets the request body. The -d flag also
> implies POST — you don't need `-X POST` separately. curl calculates Content-Length
> automatically from the data length and adds it.
>
> httpbin.org/post is a purpose-built echo service — it receives your request and sends
> back a JSON object showing exactly what it received. You'll see your headers, your
> Content-Type, and your parsed JSON body. Perfect for verifying you sent what you meant.
>
> The second command sends form data with no `-H` flag. curl defaults to
> application/x-www-form-urlencoded. The body format is `key=value&key2=value2` —
> the same format an HTML form submits. The response will show the data under 'form'
> instead of 'json'."

## http_demo.sh — Section D: Status Codes

**EXPLAIN**
> "Section D loops through a list of status codes and requests each one from
> httpbin.org/status/N. For each code, it extracts just the number using `-w '%{http_code}'`.
>
> `-w` stands for write-out. It prints a formatted string after the transfer completes.
> `%{http_code}` is replaced by the three-digit status code. `-o /dev/null` discards
> the body — we only care about the code.
>
> The for loop runs curl six times, one per code, and prints what was requested vs what
> was received.
>
> Then: the exit code demo. This is a subtle but important point.
>
> `curl -s -o /dev/null https://httpbin.org/status/404` returns exit code 0. Not because
> the request succeeded — the server returned 404. But because the transfer itself
> succeeded. curl connected to the server, sent the request, received the response, and
> delivered it. From curl's perspective, that's success.
>
> `--fail` changes this. With `--fail`, curl checks the HTTP status code and exits
> non-zero if it's 4xx or 5xx. This is what you add to shell scripts when you want to
> treat server errors as failures."
