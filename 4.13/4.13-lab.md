## Part 1 — What HTTP Actually Looks Like

### Before you run: make a prediction

HTTP has a four-part structure: start line → headers → blank line → body. The blank line is the separator between headers and body.

**Prediction:** When curl shows you headers with `>` and `<` prefixes, where do you expect to see the blank line — before the body? After all headers? At the very end?

<details>
<summary>Answer</summary>

After all headers, before the body. You'll see a `>` line by itself (nothing after the `>`) in the request, and a `<` line by itself in the response. That's the blank line — the CRLF separator. The server processes the request the moment it receives it.

</details>

### Run the demo

```bash
curl -v -s http://example.com/ 2>&1
```

**Output (Section A):**

```
* Host example.com:80 was resolved.
* IPv6: 2606:4700:10::6814:179a, 2606:4700:10::ac42:93f3
* IPv4: 172.66.147.243, 104.20.23.154
*   Trying [2606:4700:10::6814:179a]:80...
* Connected to example.com (2606:4700:10::6814:179a) port 80
> GET / HTTP/1.1
> Host: example.com
> User-Agent: curl/8.7.1
> Accept: */*
> 
* Request completely sent off
< HTTP/1.1 200 OK
< Date: Tue, 14 Apr 2026 20:04:41 GMT
< Content-Type: text/html
< Transfer-Encoding: chunked
< Connection: keep-alive
< Server: cloudflare
< 
{ [540 bytes data]
* Connection #0 to host example.com left intact
<!doctype html>...
```

*(Your Date, IP addresses will differ — structure is always the same.)*

Walk through the prefixes:
- `*` lines — curl's own metadata: DNS resolution, TCP connection, connection reuse. Not HTTP.
- `>` lines — bytes curl sent. The blank `>` is the separator line.
- `<` lines — bytes the server sent back. The blank `<` is the separator on the response side.

### Fill in the blank

What flag suppresses curl's progress bar?

```bash
curl _____ -v http://example.com/ 2>&1
```

<details>
<summary>Answer</summary>

```bash
curl -s -v http://example.com/ 2>&1
```

`-s` is `--silent`. Without it, curl prints a download progress bar that mixes with the verbose output and makes it unreadable.

</details>

### Exercise

Get just the HTTP status code — no headers, no body:

```bash
curl -s -o /dev/null -w "Status: %{http_code}\n" http://example.com/
```

**Short answer:** What does `-o /dev/null` do, and why do we need it here?

<details>
<summary>Answer</summary>

`-o /dev/null` throws away the response body. `/dev/null` is a special file on Unix systems that discards everything written to it. We use it because we only care about the status code — without it, the HTML body would print to the screen and clutter the output.

`-w "%{http_code}"` prints specific fields from the transfer after it completes. `%{http_code}` is replaced with the actual three-digit status code.

</details>

**Try it:** Run the same command against `http://httpbin.org/status/404`. What status code do you get? Is that surprising?

## Part 2 — Status Codes: What the Server Is Saying

### Before you run: make a prediction

**Prediction:** When you request `http://httpbin.org/status/301` without the `-L` flag, curl will show a 301 response. What header in that response tells the client where to go next?

<details>
<summary>Answer</summary>

The `Location` header. Every 3xx response includes a `Location` header containing the URL the client should request instead.

</details>

### Run the demo

```bash
# Without -L: stop at the redirect, show me the raw headers
curl -v -s -o /dev/null http://httpbin.org/status/301 2>&1 \
  | grep -E "^[*<>]"
```

**Output (Section B — 301 without -L):**

```
> GET /status/301 HTTP/1.1
> Host: httpbin.org
> User-Agent: curl/8.7.1
> Accept: */*
> 
< HTTP/1.1 301 MOVED PERMANENTLY
< Content-Length: 0
< location: /redirect/1
< 
* Connection #0 to host httpbin.org left intact
```

Find the `location: /redirect/1` header. That's the server telling the client: the thing you want is at this path.

```bash
# With -L: follow all redirects, print final status and URL
curl -s -o /dev/null \
  -w "Final status: %{http_code}  →  final URL: %{url_effective}\n" \
  -L http://httpbin.org/status/301
```

**Output (Section B — 301 with -L):**

```
Final status: 200  →  final URL: http://httpbin.org/get
```

### Fill in the blank

Complete this command to follow redirects AND print the final URL:

```bash
curl -s -o /dev/null -w "%{url_effective}\n" _____ http://httpbin.org/status/301
```

<details>
<summary>Answer</summary>

```bash
curl -s -o /dev/null -w "%{url_effective}\n" -L http://httpbin.org/status/301
```

`-L` is the flag that makes curl follow `Location` headers automatically. Without it, curl stops at each redirect.

</details>

### Exercise

Try a URL that redirects three times before reaching the destination:

```bash
curl -s -o /dev/null -w "Final: %{url_effective}\n" -L http://httpbin.org/redirect/3
```

**Short answer:** How many HTTP requests did curl actually make to get that `200`? How do you know?

<details>
<summary>Answer</summary>

4 requests: one for the original URL, and one for each of the 3 redirects. You know because the path is `/redirect/3` — httpbin issues exactly 3 redirects before landing on `/get`. You can verify by adding `-v` and counting the `> GET` lines.

</details>

## Part 3 — POST: Sending Data to the Server

### Before you run: make a prediction

A GET request has a start line and headers. A POST request has a body.

**Prediction:** When you add a body to an HTTP request, two extra headers appear that GET requests don't have. What are they called, and what do they describe?

<details>
<summary>Answer</summary>

`Content-Type` — the format of the body (e.g., `application/json`, `application/x-www-form-urlencoded`).

`Content-Length` — the number of bytes in the body. Without it, the server doesn't know when your body ends and would wait forever.

</details>

### Run the demo

Watch the `>` section — you're about to see those two headers for the first time:

```bash
curl -v -s \
  -X POST http://httpbin.org/post \
  -H "Content-Type: application/json" \
  -d '{"name":"Alice","course":"Anyone Can Code"}' \
  2>&1
```

**Output (Section C — POST headers):**

```
> POST /post HTTP/1.1
> Host: httpbin.org
> User-Agent: curl/8.7.1
> Accept: */*
> Content-Type: application/json
> Content-Length: 43
> 
} [43 bytes data]
* upload completely sent off: 43 bytes
< HTTP/1.1 200 OK
< Content-Type: application/json
< Content-Length: 493
< 
{
  "data": "{\"name\":\"Alice\",\"course\":\"Anyone Can Code\"}", 
  "headers": {
    "Content-Length": "43", 
    "Content-Type": "application/json", 
    ...
  }, 
  "json": {
    "course": "Anyone Can Code", 
    "name": "Alice"
  }
}
```

httpbin.org echoes everything back as JSON — you can verify the server received your headers and body exactly as sent. The `json` field shows the parsed body.

**Output (Section C — form data):**

```
Content-Type sent: application/x-www-form-urlencoded
Form data received: {
  "message": "Hello World",
  "username": "alice"
}
```

The `+` in `Hello+World` decoded to a space — that's URL-encoding.

### Fill in the blank

Complete this command to set the Content-Type to JSON:

```bash
curl -s -X POST http://httpbin.org/post \
  _____ "Content-Type: application/json" \
  -d '{"lang":"Python"}'
```

<details>
<summary>Answer</summary>

```bash
curl -s -X POST http://httpbin.org/post \
  -H "Content-Type: application/json" \
  -d '{"lang":"Python"}'
```

`-H` adds any request header in `Name: Value` format. You can use it multiple times to add multiple headers.

</details>

### Exercise

Send a POST request with your name and favorite language as JSON. Verify the server received it by checking the `json` field in the response.

<details>
<summary>Solution</summary>

```bash
curl -s \
  -X POST http://httpbin.org/post \
  -H "Content-Type: application/json" \
  -d '{"name":"Your Name","language":"Python"}' \
  | python3 -m json.tool
```

`python3 -m json.tool` pretty-prints the JSON response. Look for your data in the `json` field.

</details>

## Part 4 — Cookies: How Servers Remember You

### Before you run: make a prediction

HTTP is stateless — every request arrives with no memory of previous ones. Cookies are how servers ask clients to remember something.

**Prediction:** The server uses a response header to set a cookie. The client uses a request header to send it back. What are the names of those two headers?

<details>
<summary>Answer</summary>

`Set-Cookie` — server to client (the server is asking the client to store a value).

`Cookie` — client to server (the client is sending back the value it was asked to store).

Two headers. That's the entire mechanism.

</details>

### Run the demo

**Step 1 — See the Set-Cookie header:**

```bash
curl -v -s -o /dev/null http://httpbin.org/cookies/set?flavor=chocolate 2>&1 \
  | grep -E "^[<>]"
```

**Output (Section D — Step 1):**

```
> GET /cookies/set?flavor=chocolate HTTP/1.1
> Host: httpbin.org
> User-Agent: curl/8.7.1
> Accept: */*
> 
< HTTP/1.1 302 FOUND
< Location: /cookies
< Set-Cookie: flavor=chocolate; Path=/
< 
```

**Short answer:** Why is the response a `302 FOUND` instead of `200 OK`?

<details>
<summary>Answer</summary>

`/cookies/set` is a convenience endpoint: it sets the cookie AND immediately redirects you to `/cookies` where you can verify what was set. The Set-Cookie header travels with the redirect response — the browser would store the cookie and follow the redirect in one step.

</details>

**Step 2 — Save and follow:**

```bash
curl -s -c /tmp/my_cookies.txt \
  "http://httpbin.org/cookies/set?flavor=chocolate" \
  -L -o /dev/null

cat /tmp/my_cookies.txt
```

**Output (cookie file):**

```
# Netscape HTTP Cookie File
# https://curl.se/docs/http-cookies.html
# This file was generated by libcurl! Edit at your own risk.

httpbin.org	FALSE	/	FALSE	0	flavor	chocolate
```

Tab-separated: domain, subdomain-flag, path, secure-only, expiry (0 = session), name, value.

**Step 3 — Send it back:**

```bash
curl -v -s -b /tmp/my_cookies.txt "http://httpbin.org/cookies" 2>&1
```

**Output (Cookie in request):**

```
> GET /cookies HTTP/1.1
> Host: httpbin.org
> User-Agent: curl/8.7.1
> Accept: */*
> Cookie: flavor=chocolate
> 
< HTTP/1.1 200 OK
< 
{
  "cookies": {
    "flavor": "chocolate"
  }
}
```

### Fill in the blank

Which flag saves received cookies to a file, and which flag sends them from a file?

```bash
curl -s _____ /tmp/my_cookies.txt "http://httpbin.org/cookies/set?color=blue" -L -o /dev/null
curl -s _____ /tmp/my_cookies.txt "http://httpbin.org/cookies"
```

<details>
<summary>Answer</summary>

```bash
curl -s -c /tmp/my_cookies.txt "http://httpbin.org/cookies/set?color=blue" -L -o /dev/null
curl -s -b /tmp/my_cookies.txt "http://httpbin.org/cookies"
```

`-c` = **c**ookie jar write (save incoming Set-Cookie headers to file).  
`-b` = **b**rowser cookies (send cookies from file or inline string).

</details>

### Exercise

Set **two** cookies in one request and send them both back. httpbin.org accepts multiple query parameters:

```bash
http://httpbin.org/cookies/set?flavor=chocolate&size=large
```

<details>
<summary>Solution</summary>

```bash
# Set two cookies
curl -s -c /tmp/my_cookies.txt \
  "http://httpbin.org/cookies/set?flavor=chocolate&size=large" \
  -L -o /dev/null

# Send both back
curl -s -b /tmp/my_cookies.txt "http://httpbin.org/cookies"
```

Expected response:
```json
{
  "cookies": {
    "flavor": "chocolate",
    "size": "large"
  }
}
```

The `Cookie:` request header contains both: `Cookie: flavor=chocolate; size=large`. Multiple cookies are separated by `;` — not `&` (that's URL parameters).

</details>

## Part 5 — Raw HTTP with nc: No Library, Just Text

### Before you run: make a prediction

nc is a raw TCP utility — it knows nothing about HTTP. You type the request yourself.

**Prediction:** HTTP/1.1 defaults to persistent connections — the server keeps the TCP socket open after responding. Why does this make nc hang instead of exiting?

<details>
<summary>Answer</summary>

After the server sends its response, it keeps the connection open waiting for more requests. nc is still connected, stdin is still open, and nc has no reason to exit. It just sits there. Using HTTP/1.0 instead tells the server to close the connection after one response — then nc exits when the connection closes.

</details>

### Run it

**macOS:**
```bash
nc -c example.com 80
```

Then type exactly this (press Enter after each line — the second blank Enter sends the separator):

```
GET / HTTP/1.0
Host: example.com

```

**Linux / WSL:**
```bash
printf "GET / HTTP/1.0\r\nHost: example.com\r\n\r\n" | nc -q 3 example.com 80
```

**What you'll see:**

```
HTTP/1.1 200 OK
Date: Tue, 14 Apr 2026 20:04:44 GMT
Content-Type: text/html
Connection: close
Server: cloudflare

<!doctype html><html lang="en"><head><title>Example Domain</title>...
```

`Connection: close` — the server confirms it's closing the TCP connection. nc exits.

### Short answer

You typed three lines: the request line, one header, and a blank line. Which of those three does the server actually wait for before responding — and why?

<details>
<summary>Answer</summary>

The blank line. The server reads headers until it sees a blank line (CRLF with nothing before it) — that's the signal that headers are done and it should process the request. Without the blank line, the server keeps reading, waiting for more headers, and never responds. This is why the blank line is the most important structural element in the whole protocol.

</details>

### Exercise — Two terminals

See both sides of an HTTP connection simultaneously.

**Terminal 1** (start the server):
```bash
cd ~/src/courses/anyone-can-code/4.13
python3 -m http.server 8080
```

**Terminal 2** (send the request):

**macOS:**
```bash
nc -c 127.0.0.1 8080
```

**Linux / WSL:**
```bash
nc 127.0.0.1 8080
```

Then type:
```
GET / HTTP/1.0
Host: 127.0.0.1

```

Watch Terminal 1.

<details>
<summary>Solution</summary>

Terminal 1 (server) prints:
```
Serving HTTP on :: port 8080 (http://[::]:8080/) ...
127.0.0.1 - - [14/Apr/2026 20:04:44] "GET / HTTP/1.0" 200 -
```

Terminal 2 receives an HTTP response listing the files in the `4.13/` directory.

The log line format: `[client_ip] - - [timestamp] "METHOD PATH VERSION" status_code bytes_sent`

**What just happened:** The text you typed in Terminal 2 — six lines including the blank — is what appeared as the log entry in Terminal 1. There is no hidden layer. Your keystrokes became the HTTP request.

</details>

## Putting it together

| Tool | What it proves | Key idea |
|------|---------------|----------|
| `curl -v` | HTTP has visible request and response headers you can read | `>` = sent, `<` = received, blank line = separator |
| Status codes | Servers communicate outcome in the first line of every response | 2xx success, 3xx redirect (+ Location), 4xx client error |
| `curl -X POST -d` | POST adds a body to the request | `Content-Type` + `Content-Length` describe the body |
| Cookies | Session state is just text in HTTP headers | `Set-Cookie` (server → client), `Cookie` (client → server) |
| `nc` | HTTP is just structured text over TCP | Any tool that can write to a socket can speak HTTP |

A browser does all of this automatically: follows redirects, stores cookies, sets `Content-Type` on form submissions, and shows you only the rendered page. What you did in this lab is what happens underneath every click.
