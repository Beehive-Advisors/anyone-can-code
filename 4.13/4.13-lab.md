## Part 1 — What HTTP Actually Looks Like

HTTP is plain text sent over a TCP connection. A request has exactly four parts, always in this order:

1. **Start line** — `GET / HTTP/1.1`
2. **Headers** — one `Name: Value` pair per line
3. **Blank line** — marks the end of headers
4. **Body** — present for POST/responses with content; absent for GET

### Run the demo

```bash
curl -v -s http://example.com/ 2>&1
```

**Output:**

```
* Connected to example.com (2606:4700:10::6814:179a) port 80
> GET / HTTP/1.1
> Host: example.com
> User-Agent: curl/8.7.1
> Accept: */*
> 
* Request completely sent off
< HTTP/1.1 200 OK
< Date: Tue, 14 Apr 2026 20:04:41 GMT
< Content-Type: text/html
< Transfer-Encoding: chunked
< Connection: keep-alive
< Server: cloudflare
< 
{ [540 bytes data]
* Connection #0 to host example.com left intact
```

Three prefixes: `*` = curl metadata (DNS, TCP). `>` = what we sent. `<` = what the server sent back. The bare `>` and `<` with nothing after them are the blank lines.

**Question:** The server doesn't process your request until it sees the blank line. Why not just process it when you send the last header?

<details>
<summary>Answer</summary>

The server has no way to know whether a header is the last one. Headers keep coming until the blank line — that's the protocol's signal that you're done sending headers. Without a defined terminator, the server would have to guess, and guessing wrong would break everything. The blank line is the contract.

</details>

## Part 2 — Status Codes

Every HTTP response starts with a three-digit number. The ranges:

| Range | Meaning |
|-------|---------|
| 2xx | Success |
| 3xx | Redirect — go here instead (Location header tells you where) |
| 4xx | Client error — you asked for something wrong |
| 5xx | Server error — the server broke |

### Run the demo

```bash
# Stop at the redirect — don't follow it
curl -v -s -o /dev/null http://httpbin.org/status/301 2>&1 | grep -E "^[*<>]"
```

**Output:**

```
> GET /status/301 HTTP/1.1
> Host: httpbin.org
> User-Agent: curl/8.7.1
> Accept: */*
> 
< HTTP/1.1 301 MOVED PERMANENTLY
< Content-Length: 0
< location: /redirect/1
< 
* Connection #0 to host httpbin.org left intact
```

```bash
# Follow the redirect chain to the end
curl -s -o /dev/null \
  -w "Final status: %{http_code}  →  final URL: %{url_effective}\n" \
  -L http://httpbin.org/status/301
```

**Output:**

```
Final status: 200  →  final URL: http://httpbin.org/get
```

**Question:** A 301 redirect includes a `Location` header. Why does the server need to tell the client where to go — why can't the client just figure it out?

<details>
<summary>Answer</summary>

Because the server is the only one who knows where the resource moved. The client has no way to find the new location on its own — the whole point of the redirect is that the server is saying "the address you used is wrong, use this one instead." The client has to be told.

</details>

### Exercise

```bash
curl -s -o /dev/null -w "Final: %{url_effective}\n" -L http://httpbin.org/redirect/3
```

How many HTTP requests did curl actually make? How do you know without counting manually?

<details>
<summary>Solution</summary>

4 requests: one for the original URL, then one per redirect (3 redirects). You know from the path: `/redirect/3` means httpbin will issue exactly 3 redirects before resolving. Add `-v` to see each `> GET` line if you want to confirm.

</details>

## Part 3 — POST: Sending Data

GET asks for a resource. POST sends data to the server. The difference in the HTTP message: a POST has a body after the blank line, and two extra headers that GET requests don't have:

- `Content-Type` — what format is the body (`application/json`, `application/x-www-form-urlencoded`, etc.)
- `Content-Length` — how many bytes the body contains

### Run the demo

```bash
curl -v -s \
  -X POST http://httpbin.org/post \
  -H "Content-Type: application/json" \
  -d '{"name":"Alice","course":"Anyone Can Code"}' \
  2>&1
```

**Output (headers):**

```
> POST /post HTTP/1.1
> Host: httpbin.org
> User-Agent: curl/8.7.1
> Accept: */*
> Content-Type: application/json
> Content-Length: 43
> 
} [43 bytes data]
```

**Response body (httpbin echoes back what it received):**

```json
{
  "json": {
    "course": "Anyone Can Code",
    "name": "Alice"
  },
  "headers": {
    "Content-Length": "43",
    "Content-Type": "application/json"
  }
}
```

**Question:** The server needs `Content-Length` to know when the body ends. What would happen if you sent a POST request without it?

<details>
<summary>Answer</summary>

The server would keep reading, waiting for more body data that never comes. It can't process the request because it doesn't know if you've finished sending. Most servers eventually time out. This is why `Content-Length` is required when you have a body — it tells the server exactly where the message ends.

</details>

### Exercise

Send a POST with your name and a programming language as JSON. Verify the `json` field in the response contains what you sent.

<details>
<summary>Solution</summary>

```bash
curl -s \
  -X POST http://httpbin.org/post \
  -H "Content-Type: application/json" \
  -d '{"name":"Your Name","language":"Python"}' \
  | python3 -m json.tool
```

Look for `"json": { "language": "Python", "name": "Your Name" }` in the response.

</details>

## Part 4 — Cookies: How Servers Remember You

HTTP is stateless — every request arrives with no memory of previous ones. Cookies add memory with two headers:

- **`Set-Cookie`** — server sends this to ask the client to remember a value
- **`Cookie`** — client sends this on future requests, returning the stored value

That's the entire mechanism.

### Run the demo

**Step 1 — see the `Set-Cookie` header:**

```bash
curl -v -s -o /dev/null http://httpbin.org/cookies/set?flavor=chocolate 2>&1 \
  | grep -E "^[<>]"
```

**Output:**

```
> GET /cookies/set?flavor=chocolate HTTP/1.1
> Host: httpbin.org
> User-Agent: curl/8.7.1
> Accept: */*
> 
< HTTP/1.1 302 FOUND
< Location: /cookies
< Set-Cookie: flavor=chocolate; Path=/
< 
```

**Step 2 — save the cookie and follow the redirect:**

```bash
curl -s -c /tmp/my_cookies.txt \
  "http://httpbin.org/cookies/set?flavor=chocolate" \
  -L -o /dev/null

cat /tmp/my_cookies.txt
```

**Output:**

```
# Netscape HTTP Cookie File
httpbin.org	FALSE	/	FALSE	0	flavor	chocolate
```

Tab-separated: domain, subdomain-flag, path, secure-only, expiry (0 = session), name, value.

**Step 3 — send the cookie back:**

```bash
curl -v -s -b /tmp/my_cookies.txt "http://httpbin.org/cookies" 2>&1
```

**Output:**

```
> GET /cookies HTTP/1.1
> Host: httpbin.org
> Cookie: flavor=chocolate
> 
< HTTP/1.1 200 OK
< 
{ "cookies": { "flavor": "chocolate" } }
```

**Question:** The cookie file stores which domain each cookie belongs to. Why does the browser only send a cookie back to the domain that set it?

<details>
<summary>Answer</summary>

Because cookies often carry sensitive information — session tokens, user IDs. If any website could read cookies set by any other website, every site you visited could steal your login sessions from every other site. The domain restriction means a cookie set by bank.com can never be sent to attacker.com. It's a fundamental security boundary.

</details>

### Exercise

Set two cookies in one request and verify both come back:

```bash
curl -s -c /tmp/my_cookies.txt \
  "http://httpbin.org/cookies/set?flavor=chocolate&size=large" \
  -L -o /dev/null

curl -s -b /tmp/my_cookies.txt "http://httpbin.org/cookies"
```

<details>
<summary>Solution</summary>

Response:
```json
{ "cookies": { "flavor": "chocolate", "size": "large" } }
```

Both cookies travel in one `Cookie:` header, separated by `;`: `Cookie: flavor=chocolate; size=large`. Multiple cookies in one header — not multiple headers.

</details>

## Part 5 — Raw HTTP with nc

`nc` opens a raw TCP socket. It knows nothing about HTTP. You type the request yourself.

This proves the point: HTTP is just text. Any tool that can write to a TCP socket can speak HTTP.

### The request you'll type

```
GET / HTTP/1.0
Host: example.com
[blank line]
```

Two things to understand before you run:

**Why HTTP/1.0?** HTTP/1.1 keeps connections open after responding — the server sends a response and waits for more requests. nc would hang. HTTP/1.0 closes after one response.

**Why does Enter work on macOS?** HTTP requires `\r\n` line endings. macOS `nc -c` converts each Enter to `\r\n` before sending.

### Run it

**macOS:**
```bash
nc -c example.com 80
```

Type (press Enter after each line — the second blank Enter is the separator):
```
GET / HTTP/1.0
Host: example.com

```

**Linux / WSL:**
```bash
printf "GET / HTTP/1.0\r\nHost: example.com\r\n\r\n" | nc -q 3 example.com 80
```

**Output:**

```
HTTP/1.1 200 OK
Date: Tue, 14 Apr 2026 20:04:44 GMT
Content-Type: text/html
Connection: close

<!doctype html><html lang="en">...
```

`Connection: close` — the server confirms it's closing after this response. nc exits.

**Question:** You sent exactly three things: a request line, one header, and a blank line. The server sent back a full HTML page. What was the blank line's role in making that happen?

<details>
<summary>Answer</summary>

The blank line told the server you were done sending headers. Without it, the server keeps reading, waiting for more headers that never come, and never processes the request. The blank line is what triggers the server to act on what you sent.

</details>

## Putting it together

| Tool | What it proves |
|------|---------------|
| `curl -v` | HTTP messages are readable text: start line, headers, blank line, body |
| Status codes | Servers communicate outcome in the first response line; 3xx always comes with Location |
| POST | Adding a body requires Content-Type + Content-Length to tell the server what it's receiving |
| Cookies | Session state is two headers: Set-Cookie from server, Cookie from client |
| `nc` | HTTP is structured text over TCP — curl automates what you just typed by hand |
