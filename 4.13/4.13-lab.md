## Part 1 — Reading the HTTP Conversation

### What is `curl -v`?

When you visit a website, your browser sends a request and receives a response. What you see is the end result — a rendered webpage. What your browser actually did was send a short block of text to a server and read another block of text back. The webpage is the content of that response, but it's surrounded by metadata — headers — that tell the browser how to interpret it.

`curl` is a command-line HTTP client. Without flags, it shows you only the response body (like a browser showing only the page). With `-v` (verbose), it reveals the entire conversation:

- `>` lines — the text **your machine sent** to the server
- `<` lines — the text **the server sent back** (headers only)
- `*` lines — curl's own notes (connection status, TLS handshake, DNS result)

This is not an approximation or a pretty-print. These are the actual bytes going over the TCP connection — decoded into readable text.

> **Why does curl send verbose output to stderr instead of stdout?**
> <details><summary>Answer</summary>
> Curl's verbose output (the <code>*</code>, <code>&gt;</code>, <code>&lt;</code> lines) is debugging information — noise that belongs out-of-band. By sending it to stderr, curl lets you pipe the response body (stdout) to other tools like <code>jq</code> or <code>grep</code> without the debug lines getting mixed in. If they shared stdout, <code>curl -v https://example.com | jq '.key'</code> would fail because jq would try to parse the verbose lines as JSON.
> </details>

</details>

## Part 2 — Sending Data with POST

### What is a POST request?

A GET request says "give me something." It carries no data to the server. When you click a link, your browser sends a GET.

A POST request says "here is something — process it." When you submit a login form, your browser sends a POST containing your username and password. The difference at the protocol level is that a POST has a **body**: a chunk of data attached after the headers.

Two things define a POST body:

1. **The data itself** — the bytes you're sending
2. **The `Content-Type` header** — a label telling the server what format those bytes are in

The server cannot read the body without knowing the format. Two formats matter most for this lab:

- `application/x-www-form-urlencoded` — what HTML forms produce: `username=alice&city=Portland`
- `application/json` — what modern APIs expect: `{"username": "alice", "city": "Portland"}`

Curl sets `Content-Type: application/x-www-form-urlencoded` automatically when you use `-d`. You must set `application/json` manually with `-H`.

> **Why does changing `Content-Type` from `application/x-www-form-urlencoded` to `application/json` change where httpbin puts the data in its response?**
> <details><summary>Answer</summary>
> The server reads the <code>Content-Type</code> header to decide how to parse the body bytes. With <code>application/x-www-form-urlencoded</code>, it splits on <code>&</code> and <code>=</code> to extract key-value pairs. With <code>application/json</code>, it hands the bytes to a JSON parser. The body bytes are different too — <code>username=alice&city=Portland</code> vs <code>{"username": "alice", "city": "Portland"}</code> — but even if the bytes happened to be identical, the server would still parse them differently based solely on the <code>Content-Type</code> label.
> </details>

### Exercise

</details>

## Part 3 — Status Codes and Cookies

### What are status codes?

Every HTTP response starts with a three-digit number. This is the server's first word — it tells you how the request went before you read a single byte of content.

The first digit gives the category:

| First digit | Category | Meaning |
|-------------|----------|---------|
| 2xx | Success | Something worked |
| 3xx | Redirect | The resource is elsewhere — follow the `Location` header |
| 4xx | Client error | You did something wrong |
| 5xx | Server error | The server had a problem |

A client can decide whether to retry, redirect, or give up based on the first digit alone, without reading the reason phrase ("OK", "Not Found") at all.

### What are cookies?

HTTP is stateless by design. Each request is completely independent — the server has no memory of who sent it or what they did before. This works for fetching documents but breaks any feature requiring continuity, like staying logged in.

Cookies are the workaround. After you log in, the server adds a `Set-Cookie` header to its response: `Set-Cookie: sessionid=abc123`. Your client stores this text. On every subsequent request to that server, your client adds a `Cookie: sessionid=abc123` header. The server reads it and looks up who `abc123` is.

Cookies are just text in headers — not magic, not encryption (unless you add that separately). They are a convention that both sides agreed to follow.

> **Why would a 404 response mean the connection was still working, while "connection refused" means something completely different?**
> <details><summary>Answer</summary>
> A 404 response is an HTTP message sent by a running server. To send a 404, the server had to: accept your TCP connection, receive your request, understand it, and decide that the requested resource doesn't exist. The connection worked fine — the server just doesn't have what you asked for. "Connection refused" means the TCP connection itself failed — nothing was listening on that port, so the operating system rejected the connection attempt before any HTTP was exchanged. A 404 is a successful conversation with a bad outcome. "Connection refused" means there was no conversation at all.
> </details>

## Part 4 — HTTP is Just Text

### What is netcat?

Netcat (`nc`) is a tool whose only job is: open a TCP connection to a host and port, then relay everything from your keyboard to that connection, and everything from the connection back to your screen. It has no HTTP awareness. It does not know about methods, headers, or status codes. It is just a pipe to a socket.

This makes `nc` the perfect tool for the most important insight in this lab. When `curl` sends an HTTP request, it is doing exactly this: formatting text and sending it over a TCP connection. The server does not care whether the client is Firefox, curl, or a human typing. It reads the bytes, recognizes the format, and responds.

An HTTP/1.0 request in its minimal form:

```
GET / HTTP/1.0\r\n
\r\n
```

Two lines. The first is the **request-line**: method, path, HTTP version. The second is a **blank line** that signals the end of the headers. HTTP requires `\r\n` (carriage return then line feed) at the end of each line — pressing Enter in a terminal sends only `\n`, so we use `printf` to send the correct bytes.

An HTTP/1.1 request requires one additional header:

```
GET / HTTP/1.1\r\n
Host: httpforever.com\r\n
Connection: close\r\n
\r\n
```

`Host` is mandatory in HTTP/1.1 because one server can host thousands of domains. Without it, the server does not know which site you want. `Connection: close` tells the server to close the connection after responding — without it, HTTP/1.1's persistent connection keeps the socket open and `nc` appears to hang.

> **Why would the server wait forever if you type only one line and press Enter, but respond immediately when you press Enter a second time on a blank line?**
> <details><summary>Answer</summary>
> The HTTP specification (RFC 7230) defines an HTTP request as: the request-line, then zero or more header lines, then a blank line. The blank line — an empty <code>\r\n</code> with nothing before it — is the signal that headers have ended and the server can start processing. Until it sees that blank line, the server keeps reading, expecting more headers. It has no way to know you are done sending headers. Pressing Enter once sends one line. Pressing Enter again on an empty line sends the blank-line terminator. The server reads it, recognizes the pattern, and sends its response.
> </details>

### Exercise — Interactive HTTP (type it yourself)

This exercise uses **two terminal windows**. This is the nc-as-server exercise from the demo in automated form — now you type the request yourself.

**Terminal A — start nc listening:**
```bash
nc -l 8080
```

**Terminal B — curl sends a request:**
```bash
curl http://localhost:8080/ -H "X-Student: your-name"
```

Terminal A will show the raw request curl sent. The connection closes when curl times out (or press Ctrl-C in Terminal A).

Now restart Terminal A with `nc -l 8080` and this time, **you** play the role of the server. After curl's request appears in Terminal A, type this response (press Enter after each line, and press Enter once more on the empty line at the end):

```
HTTP/1.0 200 OK
Content-Type: text/plain

Hello from my hand-built server!
```

Then press Ctrl-D to send EOF. Watch Terminal B — curl should print the response body.

<details><summary>What you should see</summary>

Terminal A (nc) receives curl's request:
```
GET / HTTP/1.1
Host: localhost:8080
User-Agent: curl/8.7.1
Accept: */*
X-Student: your-name

```

After you type the HTTP response and press Ctrl-D, Terminal B (curl) prints:
```
Hello from my hand-built server!
```

You just implemented a web server. Curl is the client. Nc is the server. The "protocol" was text you typed. HTTP is nothing more than a text format that both sides agreed to use.

For Linux/WSL users: if `nc -l 8080` fails with "Address already in use," try `nc -l -p 8080`. If `nc -l` keeps the session open after the first connection, try `nc -lk 8080` on macOS or `nc -l -p 8080` in a loop on Linux.
</details>

## Putting it together

| Mechanism | What problem it solves | Key idea | Real systems |
|-----------|----------------------|----------|--------------|
| `curl -v` | Invisible HTTP made visible | `>` = sent, `<` = received, `*` = status | API debugging, network troubleshooting |
| POST + Content-Type | Sending structured data to a server | Label the body format; server parses accordingly | Login forms, REST APIs, webhooks |
| Status codes | Client knows what happened without reading the body | First digit = category; 2xx/3xx/4xx/5xx | Load balancers, monitoring scripts, error handling |
| Cookies | State in a stateless protocol | Server sets text; client echoes it back on every request | Login sessions, shopping carts, A/B testing |
| `nc` | HTTP is just text over TCP | Strip away all tools; it is still just formatted text | Protocol debugging, custom clients, security testing |

Every tool you'll use for the rest of your career — browsers, API clients, frameworks, load balancers — is doing exactly what you did in this lab: formatting text, sending it over a TCP connection, reading the response. The abstraction is now transparent.
