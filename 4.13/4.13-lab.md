## Part 1 — HTTP Is Just Text

### What is HTTP?

Think about how you ask someone for directions. You follow an agreed-upon format: "Excuse
me, how do I get to the library?" You don't just grunt at them. HTTP is that agreed-upon
format — but between your computer and a web server.

What most people don't realize is that the format is **plain text you can read**. There's
no binary encoding, no compiled format. An HTTP request is a short sequence of text lines,
like a letter with a specific layout.

Every HTTP/1.1 request has exactly three parts:

```
GET /path HTTP/1.1           ← request line: verb, path, protocol version
Host: info.cern.ch           ← headers (one per line, name: value)
Connection: close
                             ← blank line (signals: headers are done)
```

And every HTTP response follows the same pattern:

```
HTTP/1.1 200 OK              ← status line: version, three-digit code, reason
Server: Apache               ← response headers
Content-Length: 646
Connection: close

<html>...</html>             ← body (the actual content)
```

### The blank line is non-negotiable

The blank line after the headers is required by the HTTP specification (RFC 9112). It
signals "headers are done — body follows." Without it, the server keeps waiting for more
headers. The connection hangs silently.

### nc: a raw TCP pipe

`nc` (netcat) opens a raw TCP connection between your terminal and a remote port. Whatever
you write to nc's input gets sent byte-for-byte to the server. Whatever the server writes
back appears in your terminal.

There's no HTTP library involved. No header generation. No magic. If you want the server
to understand you, you have to write correct HTTP yourself.

### Run the demo

What you just saw: the server's raw HTTP response. Line 1 is the status line. Lines 2–8

the HTML page.

**Conceptual question:**

> **Why would the server wait forever if you don't send the blank line?**
>
> <details><summary>Answer</summary>
>
> The server reads headers one line at a time, looking for the blank line that marks the
> end. If you never send it, the server never knows you're done sending headers — so it
> keeps reading, waiting for the next header line (or the blank line) to arrive. From the
> server's perspective, your connection is still in progress. It has no way to know you
> stopped typing.
>
> This is why the blank line is in the spec: the protocol needs an unambiguous signal for
> "headers are finished."
>
> </details>

## Part 2 — Reading HTTP with curl -v

### What curl -v shows

`curl` normally hides its work. You see the response body and nothing else. The `-v` flag
reveals every line of the conversation, annotated:

- `*` lines — connection events (DNS lookup, TCP handshake)
- `>` lines — text **your computer sent** (the HTTP request)
- `<` lines — text **the server sent back** (the HTTP response headers)

This is the same text nc showed you in Part 1 — just annotated.

### Run the demo (Section B)

The demo runs `curl -v -s -o /dev/null http://info.cern.ch`.

- `-s` suppresses the progress bar
- `-o /dev/null` discards the response body (we're studying the headers)
- `2>&1` redirects curl's verbose output from stderr to stdout so it appears in order

**Output (Section B):**

```
* Host info.cern.ch:80 was resolved.
* IPv6: 2001:1458:201:a4::100:1a0
* IPv4: 188.184.67.127
*   Trying [2001:1458:201:a4::100:1a0]:80...
* Connected to info.cern.ch (2001:1458:201:a4::100:1a0) port 80
> GET / HTTP/1.1
> Host: info.cern.ch
> User-Agent: curl/8.7.1
> Accept: */*
>
* Request completely sent off
< HTTP/1.1 200 OK
< Date: Tue, 14 Apr 2026 21:49:33 GMT
< Server: Apache
< Last-Modified: Wed, 05 Feb 2014 16:00:31 GMT
< ETag: "286-4f1aadb3105c0"
< Accept-Ranges: bytes
< Content-Length: 646
< Connection: close
< Content-Type: text/html
<
{ [646 bytes data]
* Closing connection
```

*(Your Date will differ. The structure is the same.)*

Walk through the output:
- The `*` lines show nc-level events — DNS resolved `info.cern.ch`, TCP connected to port 80
- The `>` block is the HTTP request curl sent — notice curl added `User-Agent` and `Accept` headers automatically
- The blank `>` line is the mandatory blank line — "request headers are done"
- The `<` block is the HTTP response headers the server sent back
- The blank `<` line marks the end of response headers
- `{ [646 bytes data] }` — curl received the body but discarded it (`-o /dev/null`)

Notice that curl added `User-Agent: curl/8.7.1` and `Accept: */*` even though you didn't
specify them. curl adds these defaults automatically. In Part 1, you sent just three lines
with nc — this is why some Cloudflare-protected servers don't respond to plain nc requests:
they look for a User-Agent header.

**Conceptual question:**

> **Why does `-v` output go to stderr instead of stdout?**
>
> <details><summary>Answer</summary>
>
> The response body goes to stdout. The verbose annotation is separate information about
> the transfer — not part of the response itself. If both went to stdout, they'd be
> mixed together, making it impossible to pipe just the body to another command.
>
> By keeping them separate (body on stdout, annotation on stderr), you can do:
> `curl -v https://api.example.com/data 2>headers.log | jq .` — the body goes to jq,
> the annotation goes to a log file.
>
> </details>

### Exercise 2

Run curl -v against two URLs — one that works and one that doesn't — and compare the
`<` lines:

```bash
curl -v -s -o /dev/null https://httpbin.org/get 2>&1 | grep "^[<>*]"
curl -v -s -o /dev/null https://httpbin.org/status/404 2>&1 | grep "^[<>*]"
```

What is the difference in the first `<` line between the two responses?

<details>
<summary>Solution</summary>

The first `<` line for the working request is:
```
< HTTP/1.1 200 OK
```

For the 404:
```
< HTTP/1.1 404 NOT FOUND
```

The three-digit code changes. Everything else about the exchange structure is identical —
the request is the same, the headers format is the same. Only the server's verdict changes.

The `grep "^[<>*]"` filter shows only the annotation lines, stripping the body and the
curl progress info.

</details>

## Part 3 — POST: Sending Data to the Server

### GET vs POST

GET requests ask for data. The parameters go in the URL:
```
GET /search?q=python+tutorial HTTP/1.1
```

POST requests send data. The parameters go in the **body** — the bytes after the blank line:
```
POST /api/users HTTP/1.1
Host: api.example.com
Content-Type: application/json
Content-Length: 28

{"name": "alice", "age": 30}
```

POST is used for logins, form submissions, and API calls that create or modify data. The
body is separate from the URL — it doesn't appear in browser history or server access logs
the same way URL parameters do.

### Content-Type: telling the server how to parse the body

The body is just bytes. The server needs to know how to interpret those bytes. That's what
`Content-Type` says:

| Content-Type | Body format | When used |
|---|---|---|
| `application/json` | `{"key":"value"}` | REST APIs |
| `application/x-www-form-urlencoded` | `key=value&key2=value2` | HTML forms |

**curl does not automatically set Content-Type to JSON.** If you send JSON with `-d` but
forget `-H "Content-Type: application/json"`, curl sends `application/x-www-form-urlencoded`
by default. Many APIs will reject or misparse the body.

### Run the demo (Section C)

**Output (JSON POST):**

```
{
  "args": {},
  "data": "{\"name\": \"alice\", \"age\": 30}",
  "files": {},
  "form": {},
  "headers": {
    "Accept": "*/*",
    "Content-Length": "28",
    "Content-Type": "application/json",
    "Host": "httpbin.org",
    "User-Agent": "curl/8.7.1",
    "X-Amzn-Trace-Id": "Root=1-..."
  },
  "json": {
    "age": 30,
    "name": "alice"
  },
  "origin": "...",
  "url": "https://httpbin.org/post"
}
```

`httpbin.org/post` echoes the request back as JSON. Look at:
- `"headers"` — the request headers curl sent, including `Content-Type: application/json`
- `"json"` — the parsed JSON body: `{"age": 30, "name": "alice"}`
- `"Content-Length": "28"` — curl calculated this automatically from the `-d` argument

**Output (Form POST):**

```
{
  "form": {
    "city": "boston",
    "username": "alice"
  },
  "headers": {
    "Content-Type": "application/x-www-form-urlencoded",
    ...
  },
  "json": null,
  ...
}
```

With no `-H` flag, curl defaulted to `application/x-www-form-urlencoded`. The body was
parsed into the `"form"` field instead of `"json"`.

**Conceptual question:**

> **The JSON POST and form POST send similar data. Why does the server care which Content-Type you use?**
>
> <details><summary>Answer</summary>
>
> The server doesn't look at the body until it knows what format to expect. Content-Type
> tells the parser which algorithm to use.
>
> For `application/json`, the parser expects: `{"key": "value"}` with quoted keys and
> typed values (strings, numbers, arrays, etc.).
>
> For `application/x-www-form-urlencoded`, the parser expects: `key=value&key2=value2`
> — no quotes, all values are strings, special characters are percent-encoded.
>
> If you send JSON but declare form encoding, the parser tries to split on `=` and `&`
> characters — which will fail or produce garbage. The body content and the declared
> format must match.
>
> </details>

### Exercise 3

Send a POST request to httpbin.org with a **custom header** and verify it appears in the
response. Add your own `X-Student-Name` header:

```bash
curl -s https://httpbin.org/post \
  -H "Content-Type: application/json" \
  -H "X-Student-Name: YourName" \
  -d '{"message": "hello"}'
```

Look at the `"headers"` field in the response. Does `X-Student-Name` appear?

<details>
<summary>Solution</summary>

Yes — it appears in `"headers"`:
```json
"headers": {
    "Content-Type": "application/json",
    "X-Student-Name": "YourName",
    ...
}
```

Every `-H` flag adds one header to the request. The `X-` prefix convention marks
non-standard headers. Servers pass them through without interpreting them — useful for
tracing requests, A/B testing, or adding metadata.

</details>

## Part 4 — Status Codes: The Server's Verdict

### Five categories

Every HTTP response starts with a three-digit code. The first digit is the category:

| Class | Range | Meaning |
|-------|-------|---------|
| Informational | 1xx | Server received request, still processing |
| **Success** | **2xx** | **Request fulfilled** |
| Redirect | 3xx | Client must follow Location header |
| **Client Error** | **4xx** | **Your request is wrong** |
| **Server Error** | **5xx** | **Server has a problem** |

The codes you'll encounter most often:

| Code | Name | What happened |
|------|------|--------------|
| 200 | OK | Standard success |
| 201 | Created | POST created a new resource |
| 301 | Moved Permanently | This URL has moved; update your bookmarks |
| 400 | Bad Request | Server couldn't parse your request |
| 401 | Unauthorized | You need to log in first |
| 403 | Forbidden | You're logged in, but not allowed |
| 404 | Not Found | Nothing at this URL |
| 500 | Internal Server Error | Server crashed |

**401 vs 403**: 401 means "I don't know who you are." 403 means "I know who you are, and
the answer is no." A server may return 403 instead of 404 to hide that a resource even exists.

### Run the demo (Section D)

**Output:**

```
  Requested 200  →  received 200
  Requested 201  →  received 201
  Requested 301  →  received 301
  Requested 400  →  received 400
  Requested 404  →  received 404
  Requested 500  →  received 500

curl exit code after a 404 response: 0

curl exit code with --fail on 404: 56  (non-zero = server returned error)
```

Notice: curl exits 0 even after receiving a 404. curl's exit code means "did the network
transfer complete?" — not "did the server accept the request?" Without `--fail`, your
script will never know the server returned an error.

**Conceptual question:**

> **Why does curl exit 0 on a 404, but an OS command like `ls` exits non-zero on failure?**
>
> <details><summary>Answer</summary>
>
> `ls` either succeeds or fails — its job is to list files, and if it can't, it's an error.
>
> curl's job is to complete a network transfer. From curl's perspective, a 404 IS a
> successful transfer: curl connected to the server, sent the request, received a response,
> and delivered that response to you. The transfer completed. curl has no opinion about
> whether the server liked your request — that's a higher-level concern.
>
> The HTTP status code is in the response data, not in the network transport layer.
> curl's exit code reflects the transport (`--fail` bridges the gap by checking the
> HTTP layer too).
>
> </details>

### Exercise 4

Write a one-liner that checks whether a URL returns 200 and prints "OK" or "FAIL N" where
N is the actual status code:

<details>
<summary>Solution</summary>

```bash
code=$(curl -s -o /dev/null -w "%{http_code}" https://httpbin.org/status/418)
[ "$code" = "200" ] && echo "OK" || echo "FAIL $code"
```

Output: `FAIL 418`

The `-w "%{http_code}"` flag extracts just the three-digit code from the response.
The shell test `[ "$code" = "200" ]` checks for exact equality. This pattern is
the standard way to check HTTP status in shell scripts.

</details>

## Putting it together

| Tool | What it does | Key idea |
|------|-------------|----------|
| nc | Opens a raw TCP connection | HTTP is just text — you can write it yourself |
| curl -v | Shows the annotated HTTP conversation | `>` = sent, `<` = received |
| curl -d | Sends data in the request body | POST separates parameters from the URL |
| HTTP status codes | Server's verdict on your request | First digit = category; check with `-w "%{http_code}"` |

These four tools expose the same underlying protocol at different levels of abstraction.
nc shows the raw bytes. curl -v shows the annotated exchange. curl -d handles content-type
and body framing. Status codes let you act on the server's response programmatically.

When you call an API in Python, JavaScript, or any other language, the library is doing
exactly what you did here — composing text in the HTTP format and sending it over a TCP
connection. The code walkthrough covers how to read what http_demo.sh is doing; the
behavior is the same protocol you now know by hand.
