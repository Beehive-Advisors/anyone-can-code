## Part 1 — What HTTP actually looks like

### HTTP is just text over a TCP connection

You've used curl before to make HTTP requests. But what is curl actually doing? It's opening a TCP socket to port 80, writing a few lines of text, and reading the response. That's it. No encryption (for HTTP), no binary encoding, no special handshake beyond TCP's.

An HTTP message is made of exactly four parts, in this order:

1. **Start line** — what you're asking for (or what the server is saying back)
2. **Headers** — metadata about the request or response, one per line
3. **Blank line** — a line with nothing on it (this is the separator)
4. **Body** — the actual content (optional; present for POST, responses with data)

Here is what a GET request looks like as raw text:

```
GET / HTTP/1.1
Host: example.com
User-Agent: curl/8.7.1
Accept: */*
[blank line]
```

And a response:

```
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 648
[blank line]
<!doctype html>...
```

The blank line is the most important structural element in the whole protocol. The server will wait forever if you don't send it. When you type a request by hand with nc (Section 5), you will literally press Enter twice to send that blank line.

Walk through this line by line:

- `* Connected to example.com ... port 80` — TCP connection is established. HTTP hasn't started yet.
- `> GET / HTTP/1.1` — the request line: method (`GET`), path (`/`), version (`HTTP/1.1`)
- `> Host: example.com` — the only required header in HTTP/1.1. Servers use it to distinguish between multiple websites on the same IP.
- `>` (blank line) — this is the CRLF separator. The server processes the request when it sees this.
- `< HTTP/1.1 200 OK` — the response status line: version, status code (`200`), reason phrase (`OK`)
- `< Content-Type: text/html` — the server is telling us the body is HTML
- `<` (blank line) — response headers are done; body follows
- `{ [540 bytes data]` — curl noting that 540 bytes of body are incoming
- The HTML page

## Part 2 — Status codes: what the server is saying

### What is a status code?

Every HTTP response starts with a three-digit number. That number tells the client what happened. You've seen `404 Not Found` when a web page doesn't exist. These codes are grouped into ranges:

| Range | Meaning | Common examples |
|-------|---------|-----------------|
| 2xx | Success | `200 OK`, `201 Created` |
| 3xx | Redirect | `301 Moved Permanently`, `302 Found` |
| 4xx | Client error | `400 Bad Request`, `401 Unauthorized`, `404 Not Found` |
| 5xx | Server error | `500 Internal Server Error`, `503 Service Unavailable` |

### Redirects: the `Location` header

When a server returns a 3xx response, it includes a `Location` header telling the client where to go instead. curl's default is to show you the redirect and stop. Add `-L` and curl follows the chain automatically.

**Output (Section B — 301 without -L):**

```
> GET /status/301 HTTP/1.1
> Host: httpbin.org
> User-Agent: curl/8.7.1
> Accept: */*
> 
< HTTP/1.1 301 MOVED PERMANENTLY
< Content-Length: 0
< location: /redirect/1
< 
* Connection #0 to host httpbin.org left intact
```

See the `location: /redirect/1` header? That's the server saying: the thing you want is over there. Without `-L`, curl stops here and shows you the redirect. With `-L`, curl automatically makes a second request to `/redirect/1`.

**Output (Section B — 301 with -L):**

```
Final status: 200  →  final URL: http://httpbin.org/get
```

curl followed the chain of redirects until it reached a `200 OK`.

### Exercise

Find the `Location` header on a redirect and extract just the final URL:

```bash
# Without -L: stop at the redirect, show me the Location header
curl -v -s -o /dev/null http://httpbin.org/status/301 2>&1 | grep "location"

# With -L: follow redirects, print the final URL we ended up at
curl -s -o /dev/null -w "%{url_effective}\n" -L http://httpbin.org/status/301
```

Then try this URL — it redirects twice before reaching the final page:
```bash
curl -s -o /dev/null -w "Final: %{url_effective}\n" -L http://httpbin.org/redirect/3
```

<details>
<summary>Solution</summary>

```bash
# grep shows the location header value
curl -v -s -o /dev/null http://httpbin.org/status/301 2>&1 | grep "location"
# Output: < location: /redirect/1

# -w %{url_effective} shows the final URL after all redirects
curl -s -o /dev/null -w "Final: %{url_effective}\n" -L http://httpbin.org/redirect/3
# Output: Final: http://httpbin.org/get
# (httpbin redirects 3 times then lands on /get)
```

</details>

## Part 3 — POST: sending data to the server

### What makes POST different from GET

GET requests say "give me this resource." POST requests say "here is some data — do something with it." The difference shows up in two ways:

1. The method changes from `GET` to `POST` in the request line
2. The request now has a **body** (the data you're sending)

When you add a body, you need two extra headers:
- `Content-Type`: tells the server what format the data is in
- `Content-Length`: tells the server how many bytes the body contains

Without `Content-Length`, the server doesn't know when your body ends and would wait forever.

Two common body formats:
- `application/json` — for APIs: `{"name":"Alice"}`
- `application/x-www-form-urlencoded` — for HTML forms: `name=Alice&email=alice%40example.com`

### Run the demo

Watch the `>` section — it now has `Content-Type` and `Content-Length` that GET requests don't:

```bash
curl -v -s \
  -X POST http://httpbin.org/post \
  -H "Content-Type: application/json" \
  -d '{"name":"Alice","course":"Anyone Can Code"}' \
  2>&1
```

**Output (Section C — POST headers):**

```
> POST /post HTTP/1.1
> Host: httpbin.org
> User-Agent: curl/8.7.1
> Accept: */*
> Content-Type: application/json
> Content-Length: 43
> 
} [43 bytes data]
* upload completely sent off: 43 bytes
< HTTP/1.1 200 OK
< Content-Type: application/json
< Content-Length: 493
< 
{
  "data": "{\"name\":\"Alice\",\"course\":\"Anyone Can Code\"}", 
  "headers": {
    "Content-Length": "43", 
    "Content-Type": "application/json", 
    ...
  }, 
  "json": {
    "course": "Anyone Can Code", 
    "name": "Alice"
  }
}
```

httpbin.org echoes everything back as JSON — you can see that the server received your `Content-Type`, `Content-Length`, and the body exactly as sent. The `json` field shows the parsed body.

The `}` line before the body (`} [43 bytes data]`) is curl showing you the upload direction — the body is going from your machine to the server.

### Form data vs. JSON

The format you choose depends on what the server expects. HTML forms use URL-encoding; APIs usually use JSON. curl defaults to `application/x-www-form-urlencoded` when you use `-d` without setting `-H Content-Type`:

**Output (Section C — form data):**

```
Content-Type sent: application/x-www-form-urlencoded
Form data received: {
  "message": "Hello World",
  "username": "alice"
}
```

The `+` in `Hello+World` was decoded to a space by the server — that's URL-encoding.

### Exercise

## Part 4 — Cookies: how servers remember you

### What is a cookie?

A server can't tell one request from another by default — HTTP is stateless. Every request arrives with no memory of previous ones. Cookies solve this: the server sends a `Set-Cookie` header with a value it wants you to remember, and your browser (or curl with `-c`) stores it. On every subsequent request to that server, you send the cookie back in a `Cookie` header. The server reads it and knows who you are.

That's the entire mechanism. Two HTTP headers. No magic.

### The round-trip

1. You make a request (no cookie)
2. Server responds with `Set-Cookie: flavor=chocolate; Path=/`
3. Your client stores the cookie for that domain
4. You make another request; client adds `Cookie: flavor=chocolate` automatically
5. Server reads the cookie and knows "this is the same client that was here before"

### Run the demo

**Step 1 — See the Set-Cookie header arrive:**

```bash
curl -v -s -o /dev/null http://httpbin.org/cookies/set?flavor=chocolate 2>&1 \
  | grep -E "^[<>]"
```

**Output (Section D — Step 1):**

```
> GET /cookies/set?flavor=chocolate HTTP/1.1
> Host: httpbin.org
> User-Agent: curl/8.7.1
> Accept: */*
> 
< HTTP/1.1 302 FOUND
< Location: /cookies
< Set-Cookie: flavor=chocolate; Path=/
< 
```

The server sent `Set-Cookie: flavor=chocolate; Path=/` and then a `302 Found` redirect to `/cookies`. A browser would follow the redirect automatically and send the cookie on the way. Let's do that explicitly with curl.

**Step 2 — Save cookie, follow the redirect:**

```bash
curl -s -c /tmp/my_cookies.txt \
  "http://httpbin.org/cookies/set?flavor=chocolate" \
  -L -o /dev/null

cat /tmp/my_cookies.txt
```

**Output (Section D — cookie file):**

```
# Netscape HTTP Cookie File
# https://curl.se/docs/http-cookies.html
# This file was generated by libcurl! Edit at your own risk.

httpbin.org	FALSE	/	FALSE	0	flavor	chocolate
```

Tab-separated fields: domain, subdomain-flag, path, secure-only, expiry (0 = session), name, value.

**Step 3 — Send the cookie back:**

```bash
curl -v -s -b /tmp/my_cookies.txt "http://httpbin.org/cookies" 2>&1
```

**Output (Section D — Cookie in request):**

```
> GET /cookies HTTP/1.1
> Host: httpbin.org
> User-Agent: curl/8.7.1
> Accept: */*
> Cookie: flavor=chocolate
> 
< HTTP/1.1 200 OK
< 
{
  "cookies": {
    "flavor": "chocolate"
  }
}
```

curl read the cookie file and automatically added `Cookie: flavor=chocolate` to the request. httpbin.org confirms it received the cookie in the response body.

### Exercise

Set **two** cookies in one request and send them both back. httpbin.org accepts multiple query parameters:

```bash
http://httpbin.org/cookies/set?flavor=chocolate&size=large
```

<details>
<summary>Solution</summary>

```bash
# Set two cookies and save them
curl -s -c /tmp/my_cookies.txt \
  "http://httpbin.org/cookies/set?flavor=chocolate&size=large" \
  -L -o /dev/null

# Send both back in one request
curl -s -b /tmp/my_cookies.txt "http://httpbin.org/cookies"
```

Expected response body:
```json
{
  "cookies": {
    "flavor": "chocolate",
    "size": "large"
  }
}
```

The `Cookie:` request header will contain both: `Cookie: flavor=chocolate; size=large`. Multiple cookies are separated by `;` — not `&` (that's URL parameters). Different syntaxes for different things.

</details>

## Part 5 — Raw HTTP with nc: no library, just text

### What is nc?

`nc` (netcat) opens a raw TCP connection to a host and port, then does exactly two things: it sends whatever you type to the server, and it prints whatever the server sends back to your screen. It doesn't know anything about HTTP. It doesn't add headers or format your request. It just moves bytes.

This is the tool for proving that HTTP is just text. If nc works — and you typed the request by hand — then you know that HTTP is not a special protocol requiring special software. It's a convention about how to format text.

### The request you'll type

```
GET / HTTP/1.0
Host: example.com
[blank line — press Enter one more time]
```

Three things to notice:
1. `HTTP/1.0` not `HTTP/1.1` — HTTP/1.0 closes the connection after responding, so nc exits cleanly. HTTP/1.1 keeps the connection open and nc hangs.
2. `Host: example.com` — even HTTP/1.0 needs to know which server you're talking to.
3. The blank line — you press Enter twice after the last header. The first Enter ends the `Host:` line. The second Enter sends the blank line separator. Without it, the server won't respond.

### CRLF: why pressing Enter is enough on macOS

HTTP requires line endings of `\r\n` (carriage return + line feed, two bytes). When you press Enter in a terminal, it sends only `\n` (one byte). Technically, servers are supposed to reject requests with just `\n`, but most are lenient. On macOS, `nc -c` takes care of this: it converts every `\n` to `\r\n` before sending. Use the `-c` flag and don't think about it.

### Run it

```bash
nc -c example.com 80
```

Then type exactly this (press Enter after each line):

```
GET / HTTP/1.0
Host: example.com

```

The second blank Enter is the blank line. The server will respond and then close the connection.

**What you'll see:**

```
HTTP/1.1 200 OK
Date: Tue, 14 Apr 2026 20:04:44 GMT
Content-Type: text/html
Connection: close
Server: cloudflare

<!doctype html><html lang="en"><head><title>Example Domain</title>...
```

*(Your Date and CF-RAY will differ.)*

Note `Connection: close` — that's the server confirming it will close the TCP connection after this response. nc exits when the connection closes.

### What just happened

You and the server just had this conversation over TCP:

| You | Server |
|-----|--------|
| TCP SYN | |
| | TCP SYN-ACK |
| TCP ACK | |
| `GET / HTTP/1.0\r\n` | |
| `Host: example.com\r\n` | |
| `\r\n` (blank line) | |
| | `HTTP/1.1 200 OK\r\n` |
| | `Content-Type: text/html\r\n` |
| | `\r\n` |
| | `<!doctype html>...` |
| | *closes TCP connection* |

curl does steps 4–6 and 7–9 automatically when you run `curl http://example.com/`. Now you know it's just those lines of text.

## Putting it together

| Tool | What it proves | Key idea |
|------|---------------|----------|
| `curl -v` | HTTP has visible request and response headers you can read | `>` = sent, `<` = received, blank line = separator |
| Status codes | Servers communicate outcome in the first line of every response | 2xx success, 3xx redirect, 4xx client error, 5xx server error |
| `curl -X POST -d` | POST adds a body to the request | `Content-Type` + `Content-Length` describe the body |
| Cookies | Session state is just text in HTTP headers | `Set-Cookie` (server → client), `Cookie` (client → server) |
| `nc` | HTTP is just structured text over TCP | Any tool that can write to a socket can speak HTTP |

A browser does all of this automatically: it follows redirects, stores cookies, sets `Content-Type` on form submissions, and shows you only the rendered page — not the wire. What you did in this lab is what happens underneath every click.
