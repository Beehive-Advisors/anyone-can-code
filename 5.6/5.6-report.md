# Research Report: Lab 5.6 — Auth Mechanics

## Overview

Authentication is the mechanism by which a server decides "I know who you are." Almost every web application needs it, yet the underlying mechanics — how passwords are stored safely, how a server proves identity without re-asking for a password on every request, and how one service grants another limited access on a user's behalf — are rarely explained from first principles.

This lab covers three independent but complementary mechanisms. bcrypt is the answer to "how do you store a password so that even if your database is stolen, attackers can't easily recover the original passwords?" JWT is the answer to "how does a server issue a token that any other server can verify without calling back to a central authority?" OAuth 2.0 is the answer to "how does a user let one app access their account on another app, without giving that app their password?"

All three mechanisms are implemented in a few dozen lines of Python code. This lab runs them locally so students can read the actual bytes, decode them by hand, and watch what happens when you tamper with them.

## 1. Password Hashing with bcrypt

### Background

Storing passwords in plain text is an obvious disaster — a database breach gives an attacker every user's password immediately. The obvious improvement is to store a hash of the password instead. When the user logs in, hash what they typed and compare it to the stored hash.

The problem with naive hashing (MD5, SHA-1, SHA-256) is speed. These algorithms were designed to be fast — SHA-256 can compute about 10 billion hashes per second on a modern GPU. An attacker who steals a hashed password database can try every word in a dictionary, every common password, every combination up to 8 characters, in a matter of hours or days.

bcrypt was designed in 1999 specifically to solve this. It has a tunable cost factor that controls how many iterations the algorithm performs. At cost 12 (the current safe default), hashing one password takes about 250ms. That's not a problem for a legitimate login — the user waits a quarter of a second. But for an attacker trying billions of guesses, it reduces throughput from billions-per-second to about 4-per-second.

### How it works

**The cost factor:** The cost is an exponent. `rounds=12` means 2^12 = 4,096 iterations of the key schedule. Doubling the cost adds one to the exponent and doubles the compute time. As hardware gets faster, you raise the cost to keep pace.

**The salt:** Every bcrypt hash includes a randomly generated 22-character salt mixed into the input before hashing. The salt is stored as part of the hash string — the server doesn't need to store it separately. Because every hash has a unique salt, two users with the same password get different hashes. This defeats precomputed "rainbow table" attacks.

**The hash string format:**
```
$2b$12$SNvBvPD4.MyCTd2hmVWOn.7MqxWPXQavjiGBrtqZwKINuxq30xora
 ^^^ ^^ ^^^^^^^^^^^^^^^^^^^^^^ ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
  |   |         salt                    actual hash (31 chars)
  |   cost factor (2^12)
  version
```

Everything the server needs to verify the password is embedded in this one string. The database stores only this string — no separate salt column, no cost column.

**Verification:** `bcrypt.checkpw()` extracts the salt and cost from the stored hash, re-hashes the candidate password with the same parameters, and does a constant-time comparison. "Constant-time" means it doesn't short-circuit on the first mismatched byte — this prevents timing attacks.

### Python library: bcrypt

```python
import bcrypt

# Hash a password (input must be bytes)
hashed = bcrypt.hashpw(b"my_password", bcrypt.gensalt(rounds=12))

# Verify a password
is_valid = bcrypt.checkpw(b"my_password", hashed)  # True
is_valid = bcrypt.checkpw(b"wrong", hashed)         # False
```

Key constraints:
- Inputs must be `bytes`, not `str`. `b"password"`, not `"password"`.
- bcrypt truncates inputs at 72 bytes — passwords longer than 72 bytes are treated identically after truncation.
- `gensalt()` generates a new random salt each time; `gensalt(rounds=12)` sets the cost.

## 2. JWT (JSON Web Tokens)

### Background

HTTP is stateless — every request arrives with no memory of prior ones. One solution is session cookies: the server stores a mapping from a session ID to user data, and the browser sends the session ID as a cookie on every request. This works but requires server-side storage that scales horizontally with user count.

JWTs offer a stateless alternative. The server encodes the user's identity and permissions directly into a token, signs it with a secret key, and sends it to the client. On future requests, the client sends the token back. The server verifies the signature and reads the claims directly from the token — no database lookup needed.

The trade-off: JWTs cannot be revoked before they expire without additional infrastructure (a blocklist). For short-lived tokens (1 hour) this is often acceptable.

### How it works

A JWT is three base64url-encoded JSON strings joined by dots:

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9  .  eyJzdWIiOiJ1c2VyXzQy...  .  Rr55Pwn...
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^    ^^^^^^^^^^^^^^^^^^^^^^^^^^^   ^^^^^^^^^^^^
        header (JSON)                            payload (JSON)           signature
```

**Header:** `{"alg": "HS256", "typ": "JWT"}` — tells the recipient which algorithm was used to sign.

**Payload:** The claims. Standard ones: `sub` (subject — who this is), `iat` (issued at — Unix timestamp), `exp` (expiry — Unix timestamp). Custom ones: `role`, `email`, anything you want. **The payload is base64url-encoded, not encrypted — anyone can read it without the secret.**

**Signature:** HMAC-SHA256 of `base64url(header) + "." + base64url(payload)` using the secret key. Anyone can verify the signature if they have the secret; nobody can forge a valid signature without it.

**base64url vs base64:** Standard base64 uses `+` and `/` which are unsafe in URLs. base64url replaces them with `-` and `_`. JWT also strips the `=` padding. The `b64url_decode` function in the demo re-adds the padding and swaps the characters before calling Python's `base64.b64decode`.

**Tampering:** If you change even one character of the payload and keep the original signature, `jwt.decode()` will raise `InvalidSignatureError`. The signature mathematically covers the exact bytes of header + payload.

### Python library: PyJWT

```python
import jwt

SECRET = "this-is-exactly-32-bytes-long!!!"  # HS256 requires ≥32 bytes

# Encode
token = jwt.encode({"sub": "user_42", "role": "admin"}, SECRET, algorithm="HS256")

# Decode + verify
payload = jwt.decode(token, SECRET, algorithms=["HS256"])

# Decode without verification (to inspect claims — do NOT use in production auth)
payload = jwt.decode(token, options={"verify_signature": False})
```

Key constraints:
- HS256 key must be at least 256 bits (32 bytes). A shorter key raises `InvalidKeyError`.
- Always specify `algorithms=["HS256"]` in `decode()` — accepting any algorithm allows the "alg:none" attack.
- `exp` claim is automatically validated on decode; an expired token raises `ExpiredSignatureError`.

## 3. OAuth 2.0 Authorization Code Flow

### Background

OAuth 2.0 solves a specific problem: you want to let App B (say, a calendar app) access your data on App A (say, your Google account), but you don't want to give App B your Google password.

The solution is delegated authorization. Instead of a password, Google issues a short-lived "authorization code" that App B can exchange for an access token. The access token is scoped — it can only do specific things (read your calendar, not change your password). You can revoke it at any time without changing your password.

### How it works

The authorization code flow has four actors: the user, the browser, the authorization server (Google), and the client app (the calendar app).

```
User         Browser      Auth Server     Client App
 |             |               |               |
 | clicks login|               |               |
 |------------>|               |               |
 |             | GET /authorize?client_id=...  |
 |             |-------------->|               |
 |             | 302 + Location: /callback?code=XYZ
 |             |<--------------|               |
 |             | GET /callback?code=XYZ        |
 |             |------------------------------>|
 |             |        POST /token            |
 |             |        code=XYZ              |
 |             |        client_secret=...     |
 |             |               |<-------------|
 |             |               | access_token  |
 |             |               |------------->|
 |             |     resource requests         |
 |             |      Authorization: Bearer... |
 |             |------------------------------>|
```

**Step 1 — Authorization request:** The client app redirects the user's browser to the auth server's `/authorize` endpoint with `client_id`, `redirect_uri`, `response_type=code`, and a `state` value (CSRF protection).

**Step 2 — Authorization code:** The auth server verifies the client_id and issues a short-lived, single-use code. It redirects the browser to `redirect_uri?code=XYZ&state=...`. The code is visible in the URL — but it's useless without the client_secret.

**Step 3 — Token exchange:** The client app's back-end server (not the browser) POSTs the code + client_secret to `/token`. This happens server-to-server, not through the browser, so the secret never touches the user's machine. The auth server validates the code and returns an access token.

**Step 4 — Resource access:** The client app uses the access token as a Bearer token in the Authorization header to call protected APIs.

Key properties:
- The authorization code is **single-use** — replaying it returns `invalid_grant`.
- The code is **short-lived** (60 seconds in the demo, typically 5–10 minutes in production).
- The access token is **opaque** from the resource server's perspective — in the demo, it's a random string, not a JWT.
- The `state` parameter prevents CSRF attacks — the client verifies it matches what it sent.

### Python/shell implementation

The demo uses Python's stdlib `http.server` for the auth server (no install needed) and bash + curl for the client flow. This choice is intentional: it makes every HTTP request visible and keeps the focus on the protocol, not the framework.

## Lab Design Decisions

### bcrypt inputs as bytes, not strings

Python's bcrypt library requires `bytes` input. This is non-negotiable — passing a string raises a `TypeError`. The reason: bcrypt operates on raw bytes. Python strings are Unicode; the library refuses to guess at encoding. Students see `b"password"` and learn why: it's an explicit declaration that this is a byte string, not Unicode text.

### JWT secret is exactly 32 bytes

RFC 7518 §3.2 requires HMAC-SHA256 keys to be at least 256 bits (32 bytes). PyJWT enforces this. Using a shorter key raises `InvalidKeyError`. The demo constant `"this-is-exactly-32-bytes-long!!!"` is intentionally crafted so students can count the characters and understand the constraint.

### OAuth server uses Python stdlib only

`http.server.BaseHTTPRequestHandler` requires no third-party install. This keeps the demo self-contained and lets students read the server code without needing framework knowledge. The server is ~100 lines; a production OAuth server would use a framework, but the protocol logic is identical.

### Authorization code flow, not implicit flow

The implicit flow (which skips the code exchange and returns a token directly in the URL) was deprecated in RFC 9700 due to token leakage via browser history and referrer headers. The authorization code flow is the current recommended approach and the one students will encounter in the wild.
