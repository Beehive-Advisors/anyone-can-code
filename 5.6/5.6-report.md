## 1. Password Hashing with bcrypt

### Background

When users create passwords, applications must never store the plaintext. Historically, developers used fast hashes like MD5 or SHA-256, but these are dangerous for passwords because:

- GPUs can compute **billions** of SHA-256 hashes per second
- Precomputed rainbow tables map common passwords to their hashes
- Two users with the same password get the same hash, making bulk cracking easy

bcrypt was designed in 1999 by Niels Provos and David Mazières specifically to defeat these attacks.

### How bcrypt works

bcrypt uses the Blowfish cipher's expensive key scheduling phase as its core operation. Its key properties:

**Salting**: A cryptographically random 128-bit (16-byte) salt is generated for each password. The salt is encoded in 22 base-64 characters and prepended to the hash output. This means:
- No two hashes are the same even for identical passwords
- Rainbow tables are defeated: an attacker must crack each hash individually

**Cost factor (work factor)**: A parameter `rounds` controls how many iterations are performed (specifically, 2^rounds Blowfish key expansions). The default is 12, meaning 4,096 iterations.

| rounds | iterations | time (M1 Mac) |
|--------|-----------|---------------|
| 4      | 16        | ~0.001s       |
| 8      | 256       | ~0.015s       |
| 10     | 1,024     | ~0.06s        |
| 12     | 4,096     | ~0.24s        |

At rounds=12, an attacker can attempt only ~4 guesses/second. Compare this to SHA-256 which allows billions of guesses per second on the same hardware.

**Hash format**: The output is a 60-character string:
```
$2b$12$SaltSaltSaltSaltSaltSaHashHashHashHashHashHashHashHashHashHa
```
- `$2b` — bcrypt version identifier
- `$12` — cost factor
- Next 22 chars — the salt (base-64 encoded)
- Remaining 31 chars — the hash output (base-64 encoded)

### The `checkpw` operation

To verify a password, the library extracts the salt from the stored hash and re-hashes the candidate password with it, then compares using a constant-time comparison function. This prevents timing attacks.

### Python library: `bcrypt`

The canonical Python bcrypt library is `pyca/bcrypt` (PyPI: `bcrypt`). Key functions:

```python
import bcrypt

# Hash
hashed = bcrypt.hashpw(b"password", bcrypt.gensalt(rounds=12))

# Verify
bcrypt.checkpw(b"password", hashed)   # True
bcrypt.checkpw(b"wrong",    hashed)   # False
```

## 2. JSON Web Tokens (JWT)

### Background

Traditional session-based auth stores a session ID in a database and a cookie in the browser. The server must look up the session on every request — this doesn't scale well across multiple servers or microservices.

JWTs are a self-contained credential format defined in [RFC 7519](https://www.rfc-editor.org/rfc/rfc7519). The server can verify a JWT *without* any database lookup because the token carries its own claims AND a cryptographic signature.

### Structure

A JWT is three base64url-encoded JSON objects joined by dots:

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiJ1c2VyXzQyIn0.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
      HEADER                                    PAYLOAD               SIGNATURE
```

**Header**: Algorithm and token type
```json
{"alg": "HS256", "typ": "JWT"}
```

**Payload**: Claims — registered, public, or private
```json
{
  "sub": "user_42",
  "name": "Alice",
  "role": "admin",
  "iat": 1700000000,
  "exp": 1700003600
}
```

Common registered claims: `sub` (subject), `iss` (issuer), `exp` (expiry), `iat` (issued at), `aud` (audience).

**Signature**: HMAC-SHA256 (for HS256) over `base64url(header) + "." + base64url(payload)` using a secret key. This is what prevents tampering.

### base64url vs base64

JWTs use base64url encoding (RFC 4648 §5) which replaces `+` → `-` and `/` → `_`, and omits `=` padding. When decoding on the command line you must reverse these transformations:

```bash
# Decode a JWT payload on macOS/Linux
echo "$TOKEN" | cut -d'.' -f2 | tr '_-' '/+' | base64 -d | python3 -m json.tool
```

The `tr '_-' '/+'` step swaps the url-safe chars back. A padding step (`s += "=" * (-len(s) % 4)`) handles missing `=` in Python.

### Security properties

- **Not encrypted** by default — the payload is readable by anyone. Use JWE (JSON Web Encryption) if you need confidentiality.
- **Tamper-evident** — changing any byte in the payload invalidates the signature.
- **Stateless** — the server needs only the signing secret, not a session database.
- **Expirable** — the `exp` claim allows automatic expiry.

### Algorithms

| Algorithm | Type         | Notes |
|-----------|-------------|-------|
| HS256     | Symmetric HMAC | Same secret for sign and verify. Simple but secret must be shared. |
| RS256     | Asymmetric RSA | Private key signs, public key verifies. Better for multi-service. |
| ES256     | Asymmetric ECDSA | Smaller signatures, faster than RSA. |

### Python library: PyJWT

```python
import jwt
from datetime import datetime, timedelta, timezone

# Encode
token = jwt.encode(
    {"sub": "user_42", "exp": datetime.now(timezone.utc) + timedelta(hours=1)},
    "secret-key-at-least-32-bytes!!!",
    algorithm="HS256"
)

# Decode + verify
payload = jwt.decode(token, "secret-key-at-least-32-bytes!!!", algorithms=["HS256"])
```

## 3. OAuth 2.0 Authorization Code Flow

### Background

OAuth 2.0 (RFC 6749) solves a specific problem: **how to let a third-party app access a user's resources without giving it the user's password**. 

Classic example: "Sign in with GitHub" — your app needs to read a user's GitHub profile, but you don't want them to type their GitHub password into your app.

### The four parties

| Party | Description |
|-------|-------------|
| Resource Owner | The user |
| Client | Your application |
| Authorization Server | Issues tokens (e.g., GitHub's auth server) |
| Resource Server | The API being accessed (e.g., GitHub's API) |

### Authorization Code Flow — step by step

This is the most common flow for web apps. It uses a two-step process to keep the access token out of the browser's URL bar (which gets logged and stored).

```
User         Browser         Your Server        Auth Server
 |                |                |                  |
 |  clicks login  |                |                  |
 |--------------->|                |                  |
 |                | GET /authorize?client_id=...&state=... |
 |                |-------------------------------------->|
 |  login + consent page           |                  |
 |<-----------------------------------                  |
 |  user approves                  |                  |
 |                | 302 → /callback?code=AUTH_CODE     |
 |                |<------------------------------------|
 |                | POST /token (code + client_secret) |
 |                |--------------->|                   |
 |                |                |------------------>|
 |                |                | {access_token}    |
 |                |                |<------------------|
 |                |  GET /api (Authorization: Bearer TOKEN) |
 |                |                |------------------>|
```

**Step 1: Authorization Request**
Client redirects user's browser to the auth server's `/authorize` endpoint with `client_id`, `redirect_uri`, `response_type=code`, and `state` (CSRF protection):
```
GET /authorize?client_id=abc&redirect_uri=https://myapp.com/callback
               &response_type=code&scope=read:profile&state=xyzrandom
```

**Step 2: User authenticates and consents**
The auth server shows a login/consent page. User approves.

**Step 3: Authorization Code Redirect**
Auth server redirects the browser back to `redirect_uri` with a short-lived `code` (valid seconds to minutes):
```
302 Location: https://myapp.com/callback?code=AUTH_CODE&state=xyzrandom
```

**Step 4: Token Exchange (back-channel)**
Your server's back end exchanges the code for an access token. This happens server-to-server — the `client_secret` never touches the browser:
```
POST /token
grant_type=authorization_code&code=AUTH_CODE
&client_id=abc&client_secret=SECRET&redirect_uri=...
```
Response: `{"access_token": "...", "token_type": "Bearer", "expires_in": 3600}`

**Step 5: API Calls**
Use the access token as a Bearer token:
```
GET /api/userinfo
Authorization: Bearer ACCESS_TOKEN
```

### Why two steps (code → token)?

If the token were returned in Step 3's redirect, it would appear in the browser URL, in logs, in Referer headers, and in browser history. The authorization code is worthless without the `client_secret`, which only your server knows.

### State parameter (CSRF protection)

The `state` value is a random nonce generated by your app before the redirect. After the callback, your app verifies that the returned `state` matches what it sent. This prevents a malicious site from tricking your app into accepting their authorization code.

### PKCE (Proof Key for Code Exchange)

For public clients (SPAs, mobile apps) that can't keep a `client_secret`, PKCE (RFC 7636) replaces the client secret with a code_verifier/code_challenge pair derived from randomness. This is now recommended for all clients.

## Lab Design Decisions

### OAuth: local demo server vs. real provider

The lab uses a local OAuth 2.0 server (`oauth_server.py`) rather than a real provider like GitHub for the following reasons:

1. **No account/app registration required** — no friction for students
2. **Fully observable** — all HTTP traffic is on localhost, all server logs are visible
3. **Deterministic** — no network issues, no external service downtime
4. **Pedagogical focus** — students see the exact JSON payloads; nothing is hidden

The local server implements the full Authorization Code Flow with proper code expiry and single-use enforcement. Students who want to wire this to a real provider (GitHub, Google) can do so after understanding the mechanics.

### bcrypt cost factor

The lab uses `rounds=12` (the production default) for the main demo. Section D shows the timing comparison from rounds=4 through rounds=12 to make the key point viscerally clear: the slowness is the feature.

### JWT signing key length

PyJWT 2.x warns when the HMAC key is under 32 bytes (per RFC 7518 §3.2). The demo key `"this-is-exactly-32-bytes-long!!"` is exactly 32 bytes to avoid the warning while keeping it readable.
