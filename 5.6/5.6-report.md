# Research Report: Lab 5.6 — Auth Mechanics

## Overview

This lab covers the three mechanisms that underpin almost every user authentication system in production:
bcrypt for storing passwords safely, JWTs for stateless session tokens, and OAuth 2.0 for delegated
authorization. Each concept solves a distinct problem. bcrypt protects passwords if a database is stolen.
JWTs let servers recognize authenticated users without a database lookup on every request. OAuth lets
a third-party app access resources on a user's behalf without ever seeing that user's credentials.

The lab demonstrates all three with working Python code and curl commands against a local server, so
students see every HTTP request and response. No external accounts, no secrets to register, no live
APIs that might be down during the session.

## 1. bcrypt Password Hashing

### Background

Every website that has user accounts stores something in a database that lets it verify your password
at login. The naive approach — storing the password directly — is catastrophic when the database is
stolen (which happens regularly). An attacker who gets the database immediately has every user's
password, which they'll use to log into email, banks, and every other service where the user reused
that password.

The first improvement: store a hash of the password instead of the password itself. A hash function
takes any input and produces a fixed-length output. The function is one-way — you can't reverse it.
At login, the server hashes what the user typed and compares the result against the stored hash. If
they match, the password was correct. The server never had to store the original.

The problem with fast hash functions (like SHA-256): a modern GPU can compute billions of them per
second. An attacker who steals a SHA-256 password database can try every word in the dictionary,
every common password, every combination of letters and numbers — at billions of guesses per second.
Most passwords crack within hours.

bcrypt solves this by being deliberately slow. It's designed to take significant CPU time — tens of
milliseconds per hash. That time is controlled by a cost factor: each step up doubles the time. An
attacker who could try billions of SHA-256 hashes per second can only try a few thousand bcrypt
hashes per second. A database that would crack in minutes with SHA-256 would take decades with bcrypt.

bcrypt also solves rainbow tables. A rainbow table is a precomputed lookup: for every common password,
what is its SHA-256 hash? bcrypt defeats this by mixing a random 16-byte "salt" into every hash. Even
if two users have the same password, their bcrypt hashes are completely different. There's no point
precomputing anything because every hash has a unique salt.

### How It Works

bcrypt is based on the Blowfish block cipher, modified into a variant called Eksblowfish
("expensive key schedule Blowfish"). The algorithm has two phases:

**Key setup:** Blowfish initializes 18 subkeys and four 256-entry lookup tables from its input key.
bcrypt's modification is to repeat this initialization `2^cost` rounds, alternating between the
password and the salt. At cost 12, that's 4,096 rounds of setup instead of one. Each round costs
as much as setting up the cipher from scratch.

**Encryption:** The initialized cipher state encrypts the fixed ASCII string
`OrpheanBeholderScryDoubt` exactly 64 times. The resulting ciphertext is the hash.

**Output format:** A 60-character string with embedded metadata:

```
$2b$12$SaltSaltSaltSaltSaltSaHashHashHashHashHashHashHashHashHas
^   ^  ^                    ^
ver cost  22-char salt (base64)  31-char hash (base64)
```

The salt is stored inside the hash string. `bcrypt.checkpw()` extracts it automatically —
you never store the salt separately.

### Python Library: bcrypt

```python
import bcrypt

# Hash a password (always bytes — not str)
password = b"hunter2"
hashed = bcrypt.hashpw(password, bcrypt.gensalt(rounds=12))
# hashed is bytes: b'$2b$12$...'

# Verify at login
is_correct = bcrypt.checkpw(password, hashed)  # True
is_wrong = bcrypt.checkpw(b"wrong", hashed)    # False
```

- **PyPI:** `bcrypt` — current version 5.0.0
- **Install:** `uv pip install bcrypt`
- **Critical:** password and salt must both be `bytes`. Passing `str` raises `TypeError`.
- **72-byte limit:** Passwords longer than 72 bytes raise `ValueError` in v5.0.0+.
- **`checkpw` is timing-safe.** It uses constant-time comparison internally — don't replace it with `==`.

**Sources:**
- [bcrypt — PyPI](https://pypi.org/project/bcrypt/) — package page, install, version
- [pyca/bcrypt — GitHub](https://github.com/pyca/bcrypt) — source, README, full API reference

## 2. JWT — JSON Web Tokens

### Background

After a user logs in, every subsequent HTTP request arrives blank. HTTP has no memory. The server
needs some way to recognize "this is the same person who logged in five minutes ago."

The traditional solution is session storage: the server assigns the user an opaque random ID (stored
in a cookie), and keeps a record of that ID in a database. On every request, the server looks up the
ID to find the user's data. This works but costs a database query on every single request.

A JWT offers a different tradeoff: instead of a pointer to data stored on the server, the token is
the data. The user's ID, roles, and expiration time are packed directly into the token. On every
request, the server reads the token directly — no database lookup.

The critical question: how does the server know the token wasn't forged? The payload is signed with
a secret key using HMAC-SHA256. If anyone alters the payload, the signature no longer matches. The
server can detect tampering without a round-trip to the database. But there's an important caveat:
the payload is only *signed*, not encrypted. Anyone who intercepts a JWT can read its contents.
Never put sensitive data in the payload.

### How It Works

A JWT is three base64url-encoded JSON objects concatenated with dots:

```
HEADER.PAYLOAD.SIGNATURE
```

**Header:** Specifies the token type and signing algorithm.
```json
{"alg": "HS256", "typ": "JWT"}
```

**Payload (claims):** The data the server wants to embed. RFC 7519 defines standard claim names:
`sub` (subject/user ID), `exp` (expiration timestamp), `iat` (issued at), `iss` (issuer), and
`aud` (audience). Custom claims can be added freely.

**Signature:** For HS256 (HMAC-SHA256):
```
HMAC-SHA256(base64url(header) + "." + base64url(payload), secret_key)
```
The signature is then base64url-encoded. The server verifies by recomputing the HMAC and comparing.

**base64url vs base64:** Standard base64 uses `+`, `/`, and `=` which aren't URL-safe. base64url
substitutes `+`→`-`, `/`→`_`, and omits the `=` padding. This affects manual decoding (see gotchas).

### Python Library: PyJWT

```python
import jwt

# Encode (returns str in PyJWT 2.x)
token = jwt.encode({"sub": "alice", "role": "admin"}, "secret", algorithm="HS256")

# Decode and verify — algorithms must be a list
payload = jwt.decode(token, "secret", algorithms=["HS256"])

# Read header without verification
header = jwt.get_unverified_header(token)

# Read payload without verifying the signature (inspection only)
payload = jwt.decode(token, options={"verify_signature": False}, algorithms=["HS256"])
```

- **PyPI:** `PyJWT` — current version 2.12.1
- **Install:** `uv pip install PyJWT`
- **`algorithms=` must be a list.** Passing a string (`algorithms="HS256"`) raises `DecodeError`.
- **PyJWT 2.x returns `str` from `encode`.** Old tutorials calling `.decode('utf-8')` on the result will fail.
- **`exp` is validated automatically.** Expired tokens raise `jwt.exceptions.ExpiredSignatureError`.

**Sources:**
- [JWT Introduction — jwt.io](https://jwt.io/introduction) — canonical reference; anatomy diagrams
- [RFC 7519 — JSON Web Token](https://datatracker.ietf.org/doc/html/rfc7519) — formal spec; registered claims, base64url definition
- [PyJWT — PyPI](https://pypi.org/project/PyJWT/) — package page, version
- [How to Handle JWTs in Python — Auth0 Blog](https://auth0.com/blog/how-to-handle-jwt-in-python/) — PyJWT 2.x examples, algorithms list requirement

## 3. OAuth 2.0 Authorization Code Flow

### Background

Suppose you build an app that wants to read a user's Google Calendar. The obvious approach — asking
for their Google password — is unacceptable. The user would be trusting your app with credentials
that work on their email, their photos, their entire Google account. They can't limit what you do.
They can't revoke your access without changing their password entirely.

OAuth solves this with delegated authorization. The user grants permission directly to Google, not
to your app. Google gives your app a temporary access token that works only for the requested scope
(reading the calendar) and which the user can revoke from their Google account settings at any time.
Your app never sees the user's password.

The authorization code flow is the specific protocol for how this happens. The key design is the
separation of two channels:

- **Front channel:** The user's browser — visible, logged, possibly intercepted. The authorization
  code travels here, but it's short-lived and single-use.
- **Back channel:** A direct server-to-server HTTPS POST — private. The actual access token travels
  only here, where it cannot be captured by browser history or network monitoring.

### How It Works

Three parties: **client** (your app), **authorization server** (Google, GitHub), **resource server** (the API).

```
User                Client            Auth Server         Resource Server
 |                    |                    |                      |
 |---visits app------>|                    |                      |
 |                    |--/authorize GET--->|                      |
 |<--redirect---------|                    |                      |
 |----login+consent---+-------------------->|                     |
 |<--redirect with code (front channel)---|                      |
 |---redirect to client callback---------->|                      |
 |                    |---POST /token----->|  (back channel)      |
 |                    |<--access token-----|                      |
 |                    |--GET /resource-----+-------------------->|
 |                    |                                          |
 |                    |<--protected data-------------------------|
```

**Step 1 — Authorization URL:**
```
GET /authorize?response_type=code&client_id=my-app&redirect_uri=...&scope=read_calendar&state=random-token
```
`state` is a random value the client generates and echoes back to detect CSRF attacks.

**Step 2 — Authorization server redirects back with a code:**
```
302 Location: https://myapp.com/callback?code=SplxlOBeZQ&state=random-token
```
The code is short-lived (10 minutes), single-use.

**Step 3 — Client's backend exchanges the code for a token (back channel):**
```
POST /token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code&code=SplxlOBeZQ&client_id=my-app&client_secret=secret&redirect_uri=...
```

**Step 4 — Token response:**
```json
{"access_token": "...", "token_type": "Bearer", "expires_in": 3600}
```

**Step 5 — Call the resource:**
```
GET /calendar/events
Authorization: Bearer <access_token>
```

## Lab Design Decisions

### Why a local server, not GitHub/Google OAuth?

Real OAuth providers require HTTPS (localhost HTTP won't work), a registered application with a
valid redirect URI, and real credentials. For a beginner lab, those setup steps are failure-prone
and obscure the protocol. A local Flask server that students can read line-by-line shows every
decision: what gets validated, why codes are single-use, how the server stores issued tokens. The
protocol is demystified because there's nothing hidden.

### Why Python for bcrypt and JWT, but curl for OAuth HTTP steps?

bcrypt and PyJWT are what's being taught — the library IS the lesson. There's no meaningful shell
alternative for bcrypt (Blowfish is not in any common CLI tool), and PyJWT's four-line interface
makes the encoding steps visible in a way curl can't.

The OAuth flow is the opposite: the lesson is the HTTP protocol — the redirect, the Location header,
the form-encoded POST body. `curl -v` shows all of this in raw headers. Python's `requests` library
would hide the HTTP details that are exactly what students need to see.

### JWT: manual construction before PyJWT

Section A of `jwt_demo.py` builds a JWT by hand using only stdlib (`base64`, `hmac`, `hashlib`).
Students see that a JWT is not magic — it's four steps of standard cryptographic operations.
Section B then shows PyJWT doing exactly the same thing in one function call. This "open the hood
first, then use the tool" approach has proven effective for demystifying opaque-looking tokens.

### macOS port conflict

Flask defaults to port 5000, which macOS 12+ reserves for AirPlay Receiver. The demo server runs
on port 5001 to avoid this conflict without requiring students to change system settings.
