## Concept: Why Slow Hashing

**SPEAK**
> "Every app that has user accounts stores something in a database that lets it verify your
> password. The naive thing to store is the password itself — plain text. That's catastrophic.
> If the database is ever stolen, the attacker has every user's password immediately."
>
> "The obvious fix: don't store the password. Store a scrambled version — a hash. A hash
> function takes any input and produces a fixed-length fingerprint. It's one-way: you can't
> reverse it. At login, hash what the user typed, compare fingerprints. If they match, the
> password was correct. You never had to know the original."
>
> "Here's the problem. Most hash functions — SHA-256, MD5 — are designed to be fast. A
> modern GPU can run billions of SHA-256 operations per second. An attacker who steals a
> SHA-256 password database can try every word in the dictionary, every common password,
> every combination — billions of times per second. Most passwords crack in minutes."
>
> "bcrypt solves this by being deliberately, purposefully slow. It's built on the Blowfish
> cipher and is designed to take around 250 milliseconds per hash at cost factor 12. That
> slowness is the entire point. The same GPU that could try billions of SHA-256 hashes per
> second can only try a few thousand bcrypt hashes per second."
>
> "The second trick is the salt. Even if two users have the same password, bcrypt mixes in a
> random 16 bytes unique to each user before hashing. The result is completely different every
> time. An attacker can't precompute anything — every hash is unique."

## hash_demo.py — Walk Through the Code

**TYPE**
```bash
code hash_demo.py
```

**EXPLAIN**
> "Start at the top. We import bcrypt, hashlib, and time. hashlib is stdlib — it has SHA-256.
> bcrypt is the library we installed. We'll use time to measure how long things take."

**EXPLAIN** (Section A — the problem with fast hashing)
> "Section A sets up the contrast. We hash the same password 100,000 times with SHA-256 and
> measure how long it takes. Then we hash it once with bcrypt and measure that."
>
> "The key formula is here: `speedup = bcrypt_elapsed / (sha256_elapsed / 100_000)`. We're
> asking: if SHA-256 takes this many microseconds per hash, and bcrypt takes this many
> milliseconds per hash, how many times faster is SHA-256? The answer is usually 500,000 to
> 700,000 times faster depending on the hardware."
>
> "That number is what motivates every other bcrypt design decision. If your hashing function
> is 600,000 times faster than bcrypt, an attacker with a GPU gets 600,000 times more guesses
> per second. That's the threat bcrypt was designed to stop."

**EXPLAIN** (Section B — anatomy of the hash string)
> "Section B decodes the hash string. Look at this line: `hashed.decode('utf-8')`. The bcrypt
> library returns bytes, not a string — we decode it to examine it."
>
> "Then we slice it into four fields. Characters 0–3 are the version prefix — `$2b$`. That
> tells any code that reads this hash which bcrypt variant was used. Characters 4–5 are the
> cost factor as a two-digit string. Characters 7 through 28 are the 22-character base64url
> encoding of the 16-byte random salt. Everything after position 29 is the hash itself."
>
> "The crucial thing: the salt is embedded in the hash string. When someone logs in, you pass
> the stored hash to `checkpw()` and it reads the salt out automatically. You don't store the
> salt anywhere else. The hash string is self-contained."

**EXPLAIN** (Section C — salt uniqueness)
> "Section C hashes the same password three times and prints all three results. They will
> always be different because `gensalt()` generates 16 new random bytes each time."
>
> "Then we call `checkpw()` on each of those three different hashes. All three return True.
> How? `checkpw()` extracts the embedded salt from each hash, recomputes the hash using that
> salt and the candidate password, and compares. Different salt → different hash for the same
> password → still verifiable."
>
> "This defeats rainbow tables. A rainbow table precomputes every common password's SHA-256
> hash. But with bcrypt, there's no hash to look up — every hash of 'password123' looks
> completely different from every other hash of 'password123'. Precomputing is pointless."

**EXPLAIN** (Section D — cost factor timing)
> "Section D measures the same password at cost 10, 12, and 14. Each step up doubles the
> work. Cost 14 takes four times as long as cost 12."
>
> "The attacker's guesses per second scales inversely: if cost 12 gives them four guesses per
> second on your hardware, cost 14 gives them one. OWASP recommends cost 12 as today's
> minimum. The right way to tune it: time bcrypt on your production server hardware and pick
> the cost that takes 100–300 milliseconds."

## Concept: What a JWT Is

**SPEAK**
> "After a user logs in, every request comes in blank. HTTP has no memory. The server needs
> to recognize the same person across many requests."
>
> "The traditional approach: the server assigns the user an opaque random session ID and keeps
> a record of it in a database. On every request, the server looks up that ID. Works fine —
> but every request costs a database query just to answer 'who is this?'"
>
> "A JWT is a different approach: instead of storing data on the server and handing the user
> a pointer to it, you put the data directly in the token and hand the user the token. The
> user's ID, their role, when the session expires — all packed into one string."
>
> "The question is: how does the server know the user didn't modify the token? The answer is
> the signature. The token is signed with a secret key only the server knows. If anyone alters
> the payload — changing 'role: user' to 'role: admin', say — the signature no longer matches.
> The server detects the tampering and rejects the token."
>
> "One critical caveat: the JWT payload is only signed, not encrypted. Anyone who reads the
> token can read the payload. Never put passwords, credit card numbers, or secrets in a JWT.
> Use JWE — JSON Web Encryption — if you need encrypted tokens."

## jwt_demo.py — Walk Through the Code

**TYPE**
```bash
code jwt_demo.py
```

**EXPLAIN** (Section A — manual JWT construction)
> "Section A is the most important section. It builds a JWT by hand using only standard library
> modules. This shows exactly what PyJWT does — there's no magic."
>
> "Step 1: We create the header as a Python dict, serialize it to JSON, and base64url-encode
> it. Look at the json.dumps call — `separators=(',', ':')` removes all whitespace from the
> JSON. Compact JSON produces a shorter token. Then `base64.urlsafe_b64encode()` encodes it
> and we strip the trailing `=` padding with `.rstrip(b'=')`."
>
> "Step 2: Same process for the payload. The payload is a dict with the user's sub claim,
> a role, and an iat timestamp. We serialize and encode it the same way."
>
> "Step 3: We compute the HMAC-SHA256 signature. The input to the HMAC is the base64url
> header, a dot, and the base64url payload — joined together. The key is the server's secret.
> We call `hmac.new(secret, signing_input, hashlib.sha256)` and get the digest, then
> base64url-encode that too."
>
> "Step 4: Concatenate all three parts with dots. That's a JWT."
>
> "Notice that PyJWT produces a token with identical structure. In Section B, the output of
> `jwt.encode()` follows this exact same format. PyJWT just packages these four steps into
> one function call."

**EXPLAIN** (Section B — PyJWT encode/decode)
> "Section B uses PyJWT. `jwt.encode()` takes the payload dict, the secret key, and the
> algorithm name. It returns a string in PyJWT 2.x — in older versions it returned bytes, so
> old tutorials may call `.decode('utf-8')` on the result. That will fail here."
>
> "The decode call — this is the most common gotcha with PyJWT: `algorithms=` must be a list.
> Not a string. `algorithms='HS256'` raises a DecodeError. It must be `algorithms=['HS256']`.
> The parameter is plural and it accepts a list because you might want to allow multiple
> algorithms during a key rotation."
>
> "Notice that the `exp` claim is validated automatically. If the token has expired, PyJWT
> raises `ExpiredSignatureError`. You don't check it yourself."

**EXPLAIN** (Section C — payload visibility)
> "Section C shows two ways to read a JWT without verifying it."
>
> "`jwt.get_unverified_header()` reads the header. `jwt.decode()` with
> `options={'verify_signature': False}` reads the payload. No key needed for either."
>
> "Then we do it manually: split on dots, take the second part, add back the padding,
> `base64.urlsafe_b64decode()`. The result is raw JSON. This is exactly what jwt.io does when
> you paste a token into it."
>
> "The takeaway: a JWT is not secret. It's a signed claim, not a secret envelope."

**EXPLAIN** (Section D — tamper detection)
> "Section D builds a valid user token, then manually replaces the payload section with a new
> one that says `'role': 'admin'` — keeping the original signature."
>
> "Then `jwt.decode()` with the correct key rejects it immediately. The HMAC was computed over
> the original payload. The new payload produces a different HMAC. They don't match."
>
> "This is the security guarantee: you can read a JWT, but you cannot forge a new one without
> the server's secret. Role escalation by editing the token is not possible."

## oauth_server.py — Walk Through the Code

**TYPE**
```bash
code oauth_server.py
```

**SPEAK**
> "The OAuth server is a Flask app that plays three roles in one process: the authorization
> server, the token server, and the resource server. This isn't how production systems work —
> in production, these are separate services. But combining them here means we can see the
> entire flow in one file."

**EXPLAIN** (Data structures)
> "At the top, we have two Python dicts: `issued_codes` and `issued_tokens`. These are our
> in-memory databases. A real system would use a proper database — these reset every time the
> server restarts. For a demo, that's fine."
>
> "`REGISTERED_CLIENTS` is the registry of apps allowed to use this OAuth server. In production
> this is also a database — you register your app, get a client ID and secret, and register
> the redirect URIs it's allowed to use."

**EXPLAIN** (/authorize route)
> "The `/authorize` route is Step 1 of the flow. It reads four parameters from the URL query
> string: `response_type`, `client_id`, `redirect_uri`, and `state`."
>
> "It validates the client ID against `REGISTERED_CLIENTS`. If unknown, it returns 400.
> It validates the redirect URI against the registered list — this is a critical security
> check. An attacker could craft an authorization URL with `redirect_uri=attacker.com` and
> try to steal the code. The server rejects any URI not on the list."
>
> "Then it generates a random authorization code with `secrets.token_urlsafe(16)`, stores it
> in `issued_codes`, and redirects to the callback URI with the code and state attached."
>
> "In a real server, this route would show a login page and consent screen. We skip that and
> auto-approve — but that's the only simplification."

**EXPLAIN** (/token route)
> "The `/token` route is Step 2 — the back-channel exchange. Notice: `request.form`, not
> `request.json`. The token endpoint requires `application/x-www-form-urlencoded`, not JSON.
> This is defined in RFC 6749 and is a common mistake when people try to call it with a JSON
> body."
>
> "This route validates six things: the client credentials, that the code exists in our store,
> that the code hasn't been used before, that it hasn't expired, that the client ID matches
> what issued the code, and that the redirect URI matches."
>
> "The single-use enforcement is here: `issued_codes[code]['used'] = True`. If the same code
> comes in a second time, the `code_data['used']` check returns 400."
>
> "If everything passes, it generates an access token with `secrets.token_urlsafe(32)` and
> returns it in a JSON response."

**EXPLAIN** (/protected route)
> "The `/protected` route is the resource server. It reads the Authorization header and checks
> that it starts with 'Bearer '. It strips that prefix and looks up the token in `issued_tokens`."
>
> "If the token is unknown or expired, it returns 401. If it's valid, it returns the protected
> resource — in this demo, a fake user profile."
