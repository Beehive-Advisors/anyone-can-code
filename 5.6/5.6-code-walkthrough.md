### The imports and setup

**EXPLAIN**
> "Two imports: bcrypt is the library, time is stdlib for measuring how long things take. No other dependencies."

> "Notice there's no `SECRET`, no key, no configuration at the top. bcrypt is self-contained. The salt and cost get generated fresh on every call."

---

### Section A — Hashing a password

**EXPLAIN**
> "Line 18: `password = b\"correct-horse-battery-staple\"`. See the lowercase b before the quote? That makes this a bytes literal, not a string. bcrypt works on raw bytes — it refuses a regular Python string and raises TypeError. This is intentional: bytes are unambiguous, strings have encodings. You'll always write `b\"your password\"` when working with bcrypt."

> "Line 21: `bcrypt.hashpw(password, bcrypt.gensalt(rounds=12))`. Two arguments. The first is the password as bytes. The second is the salt — but we're not generating the salt ourselves, we're calling `gensalt()` and letting bcrypt generate a cryptographically random one. `rounds=12` sets the cost factor."

> "Lines 28–34: the anatomy block. This is printing the internals of the hash string by splitting it on dollar signs. The hash string is a self-describing format — everything needed to verify the password is embedded in it. We split on `$` to pull out the three fields: version, cost, and the salt+hash combined."

> "That `2**int(parts[2])` on line 32 — that's computing 2 to the power of 12, which is 4,096. The cost factor is an exponent, not a count. Cost 12 means 4,096 rounds of the key schedule."

---

### Section B — Verifying passwords

**EXPLAIN**
> "Lines 46–47: two calls to `bcrypt.checkpw()`. First argument: the candidate password as bytes. Second argument: the stored hash. Returns True or False."

> "The comment on lines 52–53 is the key insight: `checkpw()` doesn't need the salt as a separate argument. It extracts the salt from the stored hash string, re-hashes the candidate with the same parameters, and compares. The salt travels with the hash — that's why the hash string is self-describing."

> "'Constant-time comparison' at the end of the comment — this means checkpw doesn't short-circuit the comparison if the first byte doesn't match. A naive `==` would return faster on a wrong password than a right one, leaking timing information. Constant-time comparison always takes the same amount of time regardless of where the mismatch is."

---

### Section C — Salt uniqueness

**EXPLAIN**
> "Lines 65–67: we hash the same password three times. Each call to `gensalt()` generates a new random 22-character salt. So even though the input password is identical all three times, the three hash strings will be completely different."

> "Lines 75–76: all three verify correctly against `b\"password123\"`. bcrypt doesn't need to know which salt was used — it reads the salt from each hash string individually. You can store a million hashes of the same password and every one will verify correctly and look different."

> "Lines 78–80: the comment explains why this matters. Without salts, an attacker could precompute a table of common passwords and their hashes — one computation, unlimited lookups. Salts break this: every hash requires its own computation."

---

### Section D — Cost factor and timing

**EXPLAIN**
> "Lines 94–98: a loop over cost factors 4, 8, 10, 12. For each, it hashes the same short password and times it with `perf_counter`. This is the whole point of bcrypt."

> "Look at the pattern: cost 4 is near-instant, cost 8 is about 15ms, cost 12 is about 250ms. Each step roughly doubles the time — because the exponent increases by 1, so the iteration count doubles. Cost 4 is 16 iterations, cost 8 is 256, cost 10 is 1024, cost 12 is 4096."

> "The comment at the end: at cost 12, a legitimate login takes 250ms — imperceptible to the user. An attacker trying to brute-force can only do about 4 guesses per second instead of billions. MD5 and SHA-256 are not designed for this use case — they're designed to be fast. Fast is wrong for passwords."

> "That's hash_demo.py. Four sections: hash a password, verify it, see that salts make every hash unique, and measure the cost factor. Now let's read jwt_demo.py."

### The imports and the SECRET

**EXPLAIN**
> "Lines 7–9: imports. `jwt` is PyJWT. `json` and `base64` are stdlib — we use them in Section B to decode the token manually without the library. `datetime`, `timedelta`, `timezone` are for the expiry timestamps."

> "Line 11: `SECRET = \"this-is-exactly-32-bytes-long!!!\"`". Count the characters if you want — it's exactly 32. Why 32? RFC 7518 section 3.2 requires the HMAC-SHA256 key to be at least 256 bits, which is 32 bytes. PyJWT enforces this. If your secret is 20 characters, it raises `InvalidKeyError`. The name of the constant is documentation — it tells the next developer why this string is this length."

---

### Section A — Creating a JWT

**EXPLAIN**
> "Lines 22–30: building the payload dictionary. `sub` is 'subject' — who this token is about. `iat` is 'issued at' — when it was created, as a Unix timestamp. `exp` is expiry — one hour from now. `name`, `email`, `role` are custom claims. Any JSON-serializable key-value pairs are valid claims."

> "Line 32: `jwt.encode(payload, SECRET, algorithm=\"HS256\")`. Three arguments: the payload dict, the secret key, and the algorithm. HS256 means HMAC-SHA256. This returns a string — the three-part dot-separated token."

> "Lines 34–35: printing the character count and the number of parts. This is so students can see that splitting on `.` always gives three parts: header, payload, signature."

---

### Section B — Manual decode

**EXPLAIN**
> "Line 47: `header_b64, payload_b64, sig_b64 = token.split('.')`. We're splitting the token on dots to get the three raw base64url strings before decoding anything."

> "Lines 50–56: the `b64url_decode` function. This is the interesting one. Standard base64 uses `+` and `/`. base64url — the variant JWT uses — replaces them with `-` and `_` so the token is safe in URLs. So we swap them back: `-` becomes `+`, `_` becomes `/`."

> "Line 55: `s += '=' * (-len(s) % 4)`. Standard base64 requires the length to be a multiple of 4, padded with `=`. JWT strips the padding to keep tokens short. We re-add it here: `-len(s) % 4` computes how many `=` characters we need to add."

> "Line 56: `json.loads(base64.b64decode(s))`. Now we can decode it as standard base64 and parse the JSON. The function takes a base64url string and returns a Python dict."

> "Lines 59–69: we call `b64url_decode` on the header and payload, then print both the raw base64url string and the decoded JSON. The signature is NOT decoded with this function — the signature is raw bytes, not JSON, so we just print it as-is."

---

### Section C — Library verification

**EXPLAIN**
> "Line 81: `jwt.decode(token, SECRET, algorithms=[\"HS256\"])`. Note the plural `algorithms=` — it takes a list. This is intentional: it forces you to be explicit about which algorithms you'll accept. If you accepted any algorithm, an attacker could forge a token with `alg: none` — a real attack from the early days of JWT. Always specify the list."

> "Lines 84–87: we try to decode the token with the wrong secret. `jwt.decode` raises `InvalidSignatureError`. The server has no way to verify a token it didn't sign."

---

### Section D — Tampering

**EXPLAIN**
> "Lines 100–105: we build a forged payload. We copy the original payload, change `role` from `admin` to `superadmin`, serialize it to JSON, and base64url-encode it — following the exact same process the library uses."

> "Line 108: `tampered_token = f\"{header_b64}.{forged_b64}.{sig_b64}\"`. We reassemble the token with the forged payload but keep the original signature from the legitimate token. We can do this because the signature is just a string — we know it from the original token."

> "Lines 114–119: `jwt.decode` on the tampered token raises `InvalidSignatureError`. Why? The signature was computed over the original header + original payload. The forged payload produces different bytes, so the HMAC of `header + \".\" + forged_payload` doesn't match the stored signature. You cannot change the payload without knowing the secret, because recomputing the signature requires the secret."

> "That's the security guarantee: the payload is readable by anyone — it's not encrypted. But it can't be modified without the secret. The server trusts the claims in the payload only because the signature is valid."

> "That's jwt_demo.py. Now you know exactly what each line does. Let's run the lab."

## Recording notes

- **Pacing:** Spend the most time on Section B of jwt_demo.py — the b64url_decode function is the densest part. Read each line of it slowly and explain the padding calculation.
- **Tone:** This is a read-along, not a live-coding session. Keep the code visible and talk students through it. Don't rush.
- **For hash_demo.py:** pause on `b"password"` (bytes vs str) and the cost factor loop — these are the two things students always get wrong.
- **For jwt_demo.py:** pause on the SECRET length (32 bytes), the `algorithms=` list (not a string), and the tamper section. These are the three gotchas that cause bugs.
- **Reference:** At the end of each file, connect back to the concept — "now you know what bcrypt is actually doing when you call hashpw" / "now you know why a JWT can't be forged without the secret."
