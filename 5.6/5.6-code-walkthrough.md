## Concept: Why Slow Hashing

**SPEAK**
> "Every app that has user accounts stores something in a database that lets it verify your
> password. The naive thing to store is the password itself — plain text. That's catastrophic.
> If the database is ever stolen, the attacker has every user's password immediately."
>
> "The obvious fix: don't store the password. Store a scrambled version — a hash. A hash
> function takes any input and produces a fixed-length fingerprint. It's one-way: you can't
> reverse it. At login, hash what the user typed, compare fingerprints. If they match, the
> password was correct. You never had to know the original."
>
> "Here's the problem. Most hash functions — SHA-256, MD5 — are designed to be fast. A
> modern GPU can run billions of SHA-256 operations per second. An attacker who steals a
> SHA-256 password database can try every word in the dictionary, every common password,
> every combination — billions of times per second. Most passwords crack in minutes."
>
> "bcrypt solves this by being deliberately, purposefully slow. It's built on the Blowfish
> cipher and is designed to take around 250 milliseconds per hash at cost factor 12. That
> slowness is the entire point. The same GPU that could try billions of SHA-256 hashes per
> second can only try a few thousand bcrypt hashes per second."
>
> "The second trick is the salt. Even if two users have the same password, bcrypt mixes in a
> random 16 bytes unique to each user before hashing. The result is completely different every
> time. An attacker can't precompute anything — every hash is unique."

**TYPE**

**EXPLAIN** (lines 16–18 — imports)
```python
import bcrypt
import hashlib
import time
```
> "Lines 16–18: three imports. `hashlib` is Python's standard library — it has SHA-256 and
> other fast hash functions. `bcrypt` is what we installed. `time` lets us measure how long
> each hash takes. We need all three to set up the contrast in Section A."

**EXPLAIN** (lines 25–31 — SHA-256 timing loop)
```python
password = b"hunter2"

# Time SHA-256 for 100,000 hashes
start = time.perf_counter()
for _ in range(100_000):
    hashlib.sha256(password).hexdigest()
sha256_elapsed = time.perf_counter() - start
```
> "Line 25: the password is a bytes literal — `b'hunter2'`. The `b` prefix is required;
> bcrypt refuses string input and raises `TypeError`. Lines 28–31 run SHA-256 100,000 times
> and record the total wall-clock time with `time.perf_counter()`. We use 100,000 iterations
> to get a stable measurement — one SHA-256 hash is too fast to time reliably."

**EXPLAIN** (lines 39–49 — bcrypt timing and speedup formula)
```python
# Time one bcrypt hash at cost 12
start = time.perf_counter()
hashed = bcrypt.hashpw(password, bcrypt.gensalt(rounds=12))
bcrypt_elapsed = time.perf_counter() - start

speedup = bcrypt_elapsed / (sha256_elapsed / 100_000)
print(f"  Speed ratio: SHA-256 is {speedup:,.0f}x faster than bcrypt")
```
> "Lines 39–42 time a single bcrypt hash. Line 41 is the actual call: `bcrypt.hashpw()`
> takes the password bytes and a salt. The salt comes from `bcrypt.gensalt(rounds=12)` —
> 16 random bytes, embedded alongside the cost factor."
>
> "Line 49 is the speedup formula: we divide bcrypt's time-per-hash by SHA-256's
> time-per-hash. On this hardware the answer is around 600,000 — SHA-256 is 600,000 times
> faster. That gap is what makes SHA-256 dangerous for passwords and bcrypt appropriate."

**EXPLAIN** (lines 71–81 — hash anatomy)
```python
h = hashed.decode("utf-8")
print("  Decoded fields:")
print(f"    Version:     {h[0:4]!r}   — bcrypt algorithm version")
print(f"    Cost factor: {h[4:6]!r}     — 2^12 = 4,096 Blowfish setup rounds")
print(f"    Salt:        {h[7:29]!r} — 22 chars (16 random bytes, base64-encoded)")
print(f"    Hash:        {h[29:]!r} — 31 chars (24 bytes of ciphertext)")
```
> "Line 71 decodes the bytes to a string so we can slice it. Lines 73–76 slice the 60-character
> hash string into four labeled fields. Characters 0–3 are the version prefix `$2b$`. Characters
> 4–5 are the cost: `12`. Characters 7–28 are the 22-character base64url-encoded salt — that's
> 16 random bytes. Everything from 29 onward is the 31-character ciphertext."
>
> "Lines 80–81 state the key point: the salt is embedded. When someone logs in, you pass the
> entire 60-character string to `checkpw()`. It reads the salt out of characters 7–28 itself.
> You never store the salt separately."

**EXPLAIN** (lines 94–106 — salt uniqueness and checkpw loop)
```python
hashes = []
for i in range(3):
    h = bcrypt.hashpw(password, bcrypt.gensalt(rounds=10))
    hashes.append(h)
    print(f"  Hash {i + 1}: {h.decode()}")

for i, h in enumerate(hashes):
    result = bcrypt.checkpw(password, h)
    print(f"    Hash {i + 1}: checkpw = {result}")
```
> "Lines 94–98 hash the same password three times. Each iteration calls `bcrypt.gensalt()`
> fresh, so each hash gets a different random salt — and produces completely different output."
>
> "Lines 104–106 call `checkpw()` on each of those three hashes. Line 105 is where the
> comparison happens. `checkpw()` re-extracts the embedded salt from whichever hash you pass
> it, recomputes the hash with that salt and the candidate password, and compares the result.
> Different salt, different hash — but `checkpw()` still returns `True` for all three."

**EXPLAIN** (lines 127–132 — cost factor timing loop)
```python
for cost in [10, 12, 14]:
    start = time.perf_counter()
    bcrypt.hashpw(password, bcrypt.gensalt(rounds=cost))
    elapsed = time.perf_counter() - start
    guesses = 1 / elapsed
    print(f"  Cost {cost:2d}: {elapsed:.3f}s  (~{guesses:,.0f} attacker guesses/sec)")
```
> "Lines 127–132 hash the same password at cost 10, 12, and 14. The only thing that changes
> between iterations is `rounds=cost` on line 129. Line 131 computes `1 / elapsed` — that's
> the attacker's guesses per second on this hardware. Each step doubles the rounds, which
> roughly doubles the time."

## Concept: What a JWT Is

**SPEAK**
> "After a user logs in, every request comes in blank. HTTP has no memory. The server needs
> to recognize the same person across many requests."
>
> "The traditional approach: the server assigns the user an opaque random session ID and keeps
> a record of it in a database. On every request, the server looks up that ID. Works fine —
> but every request costs a database query just to answer 'who is this?'"
>
> "A JWT is a different approach: instead of storing data on the server and handing the user
> a pointer to it, you put the data directly in the token and hand the user the token. The
> user's ID, their role, when the session expires — all packed into one string."
>
> "The question is: how does the server know the user didn't modify the token? The answer is
> the signature. The token is signed with a secret key only the server knows. If anyone alters
> the payload — changing 'role: user' to 'role: admin', say — the signature no longer matches.
> The server detects the tampering and rejects the token."
>
> "One critical caveat: the JWT payload is only signed, not encrypted. Anyone who reads the
> token can read the payload. Never put passwords, credit card numbers, or secrets in a JWT.
> Use JWE — JSON Web Encryption — if you need encrypted tokens."

## jwt_demo.py — Walk Through the Code

**TYPE**
```bash
code jwt_demo.py
```

**EXPLAIN** (line 24 — secret key)
```python
SECRET = "this-is-a-32-byte-secret-key-ok!"
```
> "Line 24: the server's secret key. Count the characters — exactly 32. HS256 needs at least
> 32 bytes. This is the value only the server holds; it's what makes the signature unforgeable.
> Anyone who knows this secret can create valid tokens, so in production it lives in an
> environment variable or secrets vault — never in source code."

**EXPLAIN** (lines 32–62 — Section A: manual JWT construction, all four steps)
```python
# Step 1: Build the header
header = {"alg": "HS256", "typ": "JWT"}
header_json = json.dumps(header, separators=(",", ":"))
header_b64 = base64.urlsafe_b64encode(header_json.encode()).rstrip(b"=")

# Step 2: Build the payload
payload = {"sub": "alice", "role": "admin", "iat": int(time.time())}
payload_json = json.dumps(payload, separators=(",", ":"))
payload_b64 = base64.urlsafe_b64encode(payload_json.encode()).rstrip(b"=")

# Step 3: Compute HMAC-SHA256 signature
signing_input = header_b64 + b"." + payload_b64
sig_bytes = hmac.new(SECRET.encode(), signing_input, hashlib.sha256).digest()
sig_b64 = base64.urlsafe_b64encode(sig_bytes).rstrip(b"=")

# Step 4: Assemble
manual_token = signing_input.decode() + "." + sig_b64.decode()
```
> "This whole block is four steps. Step 1 on lines 33–35: create the header dict, serialize
> it to compact JSON with `separators=(',', ':')` — the spaces are removed to keep the token
> short — then base64url-encode it and strip the `=` padding. JWTs never include the padding
> character."
>
> "Step 2 on lines 40–43: same process for the payload. Line 41 is the payload dict — `sub`
> is the user identifier, `role` is a custom claim, `iat` is the issued-at timestamp."
>
> "Step 3 on lines 48–51: the signature. Line 49 joins the encoded header and payload with
> a dot — that dot-separated string is the input to the HMAC. Line 50 calls `hmac.new()` with
> the secret, the signing input, and SHA-256 as the digest function. `.digest()` gives raw
> bytes. Line 51 base64url-encodes those bytes."
>
> "Step 4 on line 57: assemble the final token by joining all three base64url strings with
> dots. That's the complete JWT. PyJWT does exactly these four steps — it just packages them
> into one function call."

**EXPLAIN** (lines 76–91 — Section B: PyJWT encode and decode)
```python
token = jwt.encode(payload, SECRET, algorithm="HS256")

decoded = jwt.decode(token, SECRET, algorithms=["HS256"])
```
> "Line 76: `jwt.encode()` replaces all of Section A with one call. It returns a string in
> PyJWT 2.x — in older versions it returned bytes, so old tutorials call `.decode('utf-8')`
> on the result. That raises `AttributeError` now."
>
> "Line 87: `jwt.decode()`. Notice `algorithms=` takes a list — `['HS256']`, not `'HS256'`.
> Passing a string raises `DecodeError`. The parameter is plural because during a key rotation
> you might accept two algorithms simultaneously. PyJWT validates the `exp` claim
> automatically — expired tokens raise `ExpiredSignatureError`."

**EXPLAIN** (lines 101–115 — Section C: reading the payload without a key)
```python
header_unverified = jwt.get_unverified_header(token)
payload_unverified = jwt.decode(
    token,
    options={"verify_signature": False},
    algorithms=["HS256"],
)
raw_part = token.split(".")[1]
padded = raw_part + "=" * (-len(raw_part) % 4)
raw_json = base64.urlsafe_b64decode(padded).decode()
```
> "Line 101: `get_unverified_header()` reads the header without any key. Lines 105–109 decode
> the payload with `verify_signature: False` — again, no key needed."
>
> "Lines 113–115 show the stdlib version. Line 113 splits on `.` and takes index 1 — the
> payload section. Line 114 adds `=` padding back: `-len(raw_part) % 4` computes exactly
> how many equals signs are needed for base64 to accept it. Line 115 decodes it."
>
> "The takeaway: the payload is readable by anyone. A JWT is a signed claim, not a secret."

**EXPLAIN** (lines 138–157 — Section D: tamper detection)
```python
# Tamper: rebuild the payload with role=admin, keep the original signature
parts = user_token.split(".")
bad_payload = {"sub": "bob", "role": "admin", "exp": int(time.time()) + 3600}
bad_json = json.dumps(bad_payload, separators=(",", ":"))
bad_b64 = base64.urlsafe_b64encode(bad_json.encode()).rstrip(b"=").decode()
tampered = parts[0] + "." + bad_b64 + "." + parts[2]  # same signature, new payload

try:
    jwt.decode(tampered, SECRET, algorithms=["HS256"])
except jwt.exceptions.InvalidSignatureError as e:
    print(f"  Verification: REJECTED — {type(e).__name__}")
```
> "Line 139 splits the original user token into three parts. Lines 140–142 build a new payload
> dict with `role: admin` and base64url-encode it. Line 143 reassembles a token using the
> original header and signature — `parts[0] + '.' + bad_b64 + '.' + parts[2]` — but puts
> the tampered payload in the middle."
>
> "Line 149 tries to verify the tampered token with `jwt.decode()`. Line 151 catches
> `InvalidSignatureError`. The HMAC on line 50 was computed over the original payload string.
> A different payload produces a different HMAC. The old signature doesn't match. You cannot
> upgrade your role by editing the token without the server's secret."

## oauth_server.py — Walk Through the Code

**TYPE**
```bash
code oauth_server.py
```

**SPEAK**
> "This Flask app plays three OAuth roles in one process: authorization server, token server,
> and resource server. In production these are separate services. Combining them here means
> we can read every decision in one file."

**EXPLAIN** (lines 28–42 — registered clients and in-memory stores)
```python
REGISTERED_CLIENTS = {
    "demo-client": {
        "secret": "demo-secret",
        "redirect_uris": ["http://localhost:5001/callback"],
    }
}

issued_codes: dict = {}
issued_tokens: dict = {}
```
> "Lines 30–35: `REGISTERED_CLIENTS`. In production this is a database — you register your
> app, get a client ID and secret, and register the redirect URIs it's allowed to use. Here
> it's a dict with one entry. The `redirect_uris` list is the pre-approved list of callback
> URLs — this is central to the security of the flow."
>
> "Lines 39 and 42: the two in-memory stores. `issued_codes` holds authorization codes that
> have been issued but not yet exchanged. `issued_tokens` holds valid access tokens. Both
> reset when the server restarts."

**EXPLAIN** (lines 63–90 — /authorize: validation and redirect)
```python
if client_id not in REGISTERED_CLIENTS:
    return jsonify({"error": "unknown_client"}), 400

if redirect_uri not in REGISTERED_CLIENTS[client_id]["redirect_uris"]:
    return jsonify({"error": "invalid_redirect_uri"}), 400

code = secrets.token_urlsafe(16)
issued_codes[code] = {
    "client_id": client_id,
    "redirect_uri": redirect_uri,
    "used": False,
    "issued_at": time.time(),
}

return redirect(location, code=302)
```
> "Lines 64–68: two validation checks. Line 64 verifies the client ID is registered. Line 67
> is critical: the redirect URI must be in the pre-registered list. An attacker could craft an
> authorization URL with `redirect_uri=attacker.com` to steal the code. This line blocks any
> URI that isn't registered."
>
> "Lines 75–81 generate and store the authorization code. Line 75 uses
> `secrets.token_urlsafe(16)` — 16 bytes of cryptographic randomness. The code is stored with
> `used: False` and the current timestamp."
>
> "Line 90 performs the redirect — this is the front-channel response. The code travels in
> the URL of this 302 redirect."

**EXPLAIN** (lines 99–136 — /token: form data, validation, single-use enforcement)
```python
grant_type = request.form.get("grant_type", "")
code = request.form.get("code", "")

if code_data["used"]:
    return jsonify({"error": "invalid_grant", "detail": "code already used"}), 400

# Mark code as used — single-use enforcement
issued_codes[code]["used"] = True
```
> "Line 99: `request.form.get()`, not `request.json`. The OAuth spec requires
> `application/x-www-form-urlencoded` for the token endpoint — not JSON. Sending a JSON body
> gets rejected. This is the most common mistake when calling real OAuth providers with curl."
>
> "Line 123: `if code_data['used']` — the first check once we find the code in our store.
> Line 136 is the single-use enforcement: immediately after all checks pass, we flip `used`
> to `True`. The next time this same code arrives, line 123 catches it and returns 400.
> Students test this in Step 5a of the flow."

**EXPLAIN** (lines 162–177 — /protected: bearer token check)
```python
auth_header = request.headers.get("Authorization", "")

if not auth_header.startswith("Bearer "):
    return jsonify({"error": "unauthorized", "detail": "missing Bearer token"}), 401

token_value = auth_header[7:]  # strip "Bearer "

if token_value not in issued_tokens:
    return jsonify({"error": "invalid_token", "detail": "unknown token"}), 401

token_data = issued_tokens[token_value]
if time.time() > token_data["expires_at"]:
    return jsonify({"error": "invalid_token", "detail": "token expired"}), 401
```
> "Line 162 reads the Authorization header. Line 167 checks it starts with `'Bearer '` — seven
> characters including the space. If not, 401 immediately. Line 170 strips those seven characters
> with `auth_header[7:]` to get just the token value."
>
> "Lines 172–177: look up the token and check expiry. Line 172 checks the token exists in
> `issued_tokens`. Line 176 checks `time.time() > token_data['expires_at']`. Both must pass."
