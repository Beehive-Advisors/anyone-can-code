## Part 1 — bcrypt: The Concept

### Why hashing passwords is hard to get right

**SPEAK**
> "Here's the problem. You need to store user passwords in a database. You obviously can't store them in plain text — if your database gets stolen, every user's password is immediately exposed. So you store a hash instead."

> "A hash function takes any input and produces a fixed-length output. It's one-way: you can go from 'password123' to its hash, but not from the hash back to 'password123'. It's deterministic: the same input always gives the same output. So at login, you hash what the user typed and compare it to what you stored. Match? They're in."

> "The problem with using a general-purpose hash function — MD5, SHA-1, SHA-256 — is that they were designed to be fast. A modern GPU can compute 10 billion SHA-256 hashes per second. An attacker who steals your database can try every common password, every dictionary word, every 8-character combination, in minutes."

> "bcrypt was designed in 1999 specifically to solve this. It has one unusual property: it's deliberately slow. And you control how slow."

### What a salt is

**SPEAK**
> "Before I show you how bcrypt controls speed, let me explain what a salt is, because the code uses it immediately."

> "Imagine two users both have the password 'hunter2'. Without extra measures, they'd both have the exact same hash in the database. If an attacker cracks one, they've cracked both. Worse: an attacker could precompute a table of common passwords and their hashes — called a rainbow table — and look up any hash instantly."

> "A salt is a random string generated uniquely for each user, mixed into the password before hashing. Even if two users have the same password, their salts are different, so their hashes are different. The salt is stored alongside the hash — the server needs it to re-hash at login time."

> "bcrypt goes further: the salt is embedded directly inside the hash string. There's no separate column. The 60-character hash string contains everything needed to verify the password. We'll see this in the code."

### The cost factor

**SPEAK**
> "The cost factor is an exponent. Cost 12 means the algorithm runs 2 to the power of 12 — 4,096 iterations — before producing a result. That makes one hash take about 250 milliseconds on modern hardware."

> "250ms is nothing for a legitimate user logging in — they won't notice. But for an attacker trying billions of guesses? They're limited to about 4 per second. SHA-256 would give them billions per second on the same machine. The slowness is not a bug. It's the entire point."

> "And as hardware gets faster, you raise the cost. Cost 13 takes twice as long as 12. You can tune it over time without changing your existing stored hashes."

### Imports and setup

**EXPLAIN**
> "Two imports: bcrypt is the library, time is stdlib for measuring how long things take. No configuration at the top — no key, no secret. bcrypt is self-contained. The salt and cost are generated per-call."

### Section A — Hashing a password

**EXPLAIN**
> "Line 18: `password = b\"correct-horse-battery-staple\"`. The lowercase b makes this a bytes literal, not a string. bcrypt works on raw bytes — it refuses a regular Python string and raises TypeError. This is intentional: bytes are unambiguous about encoding, strings are not. You will always write `b\"your password\"` when working with bcrypt."

> "Line 21: `bcrypt.hashpw(password, bcrypt.gensalt(rounds=12))`. Two arguments. First: the password as bytes. Second: the salt — we're calling `gensalt()` to generate a cryptographically random 22-character salt. `rounds=12` is the cost factor we just talked about."

> "Lines 28–34: this block prints the anatomy of the hash string. We're splitting on dollar signs to expose the internal structure. The hash string is self-describing — everything needed to verify the password is baked into it. No separate salt column. No separate cost column. One string."

> "Line 32: `2**int(parts[2])` — that's computing 2 to the power of 12, giving 4,096. The cost is an exponent stored as a small number, not the full iteration count."

### Section B — Verifying passwords

**EXPLAIN**
> "Lines 46–47: `bcrypt.checkpw()` takes two arguments — the candidate password as bytes, and the stored hash. It extracts the salt from the hash string, re-hashes the candidate with the same cost and salt, and does a constant-time comparison."

> "Constant-time comparison: it doesn't short-circuit on the first mismatched byte. A naive string comparison would return faster on wrong passwords than right ones, leaking information about how similar the guess was to the real password. Constant-time prevents that timing attack."

> "Notice: checkpw doesn't need the salt as a separate argument. It reads the salt out of the stored hash. The hash is self-describing."

### Section C — Salt uniqueness

**EXPLAIN**
> "Lines 65–67: same password hashed three times. Each call to `gensalt()` generates a new random salt, so all three hashes are completely different strings. But lines 75–76 show all three still verify correctly. The server doesn't care which salt was used — it reads it from each hash string individually."

> "This defeats rainbow table attacks. A rainbow table maps common passwords to their hashes and lets attackers look up any hash instantly. But rainbow tables assume a fixed hash function with no randomness. With salts, there's no single hash to look up — every entry in your database has a unique salt, so the table would need to be recomputed per-user."

### Section D — Cost factor and timing

**EXPLAIN**
> "Lines 94–98: the key demo. We hash the same password at cost 4, 8, 10, and 12, and time each one. Watch how the time roughly doubles at each step — because the iteration count doubles. Cost 4 is 16 iterations, cost 12 is 4,096."

> "The comment at the bottom frames the stakes: at cost 12, an attacker can try 4 passwords per second. At MD5 or SHA-256, they can try billions. That difference is why bcrypt exists."

## Part 2 — JWT: The Concept

### The stateless session problem

**SPEAK**
> "HTTP is stateless. Every request arrives with no memory of who made the previous one. Session cookies solve this by storing a session ID on the server — user logs in, server creates a record like 'session abc123 = user 42', stores it in a database, sends 'abc123' to the browser as a cookie. Every future request carries that cookie; the server looks up the session."

> "This works, but every request hits the database. And as you add more servers, you need to share session state between them, or route each user to the same server every time. That's complexity."

> "JWTs offer an alternative: instead of storing state on the server, the server encodes the user's identity directly into a token, signs it, and sends it to the client. On future requests, the client sends the token back. The server verifies the signature and reads the claims directly from the token — no database lookup."

### What base64url is

**SPEAK**
> "Before the code, one more concept: base64url encoding. JWTs are transmitted as strings — in HTTP headers, in URLs. Raw binary data doesn't survive that. base64 encoding converts binary to printable ASCII. Standard base64 uses the characters plus and slash, but those are special characters in URLs. base64url replaces plus with dash and slash with underscore."

> "base64 also pads the output with equals signs to make the length a multiple of 4. JWT strips the padding to keep tokens shorter. The code manually re-adds it when decoding. You'll see this in the b64url_decode function."

### What a signature is

**SPEAK**
> "The signature is how the server proves the token wasn't tampered with. It's an HMAC — a keyed hash. HMAC takes a message and a secret key and produces a fixed-length authentication tag. Only someone who knows the secret key can produce a valid HMAC. Anyone with the key can verify it."

> "For a JWT, the HMAC covers the header and payload. If you change even one character of the payload and don't have the secret, you can't produce a valid signature. The server rejects it. We'll demonstrate this with the tampering section."

### Imports and the SECRET constant

**EXPLAIN**
> "Imports: jwt is PyJWT. json and base64 are stdlib — we use them in Section B to decode the token manually without the library. datetime, timedelta, timezone for the expiry timestamps."

> "Line 11: `SECRET = \"this-is-exactly-32-bytes-long!!!\"`". Count it — 32 characters. Why 32? RFC 7518 section 3.2 requires the HS256 key to be at least 256 bits, which is 32 bytes. PyJWT enforces this at runtime — a shorter key raises InvalidKeyError. The constant name is documentation for the next developer."

### Section A — Creating a JWT

**EXPLAIN**
> "Lines 22–30: the payload dictionary. `sub` is 'subject' — who this token is about. `iat` is 'issued at' as a Unix timestamp. `exp` is 'expiry' — one hour from now, also a Unix timestamp. `name`, `email`, `role` are custom claims. Any JSON-serializable key-value pairs are valid."

> "Line 32: `jwt.encode(payload, SECRET, algorithm=\"HS256\")`. Three arguments: payload dict, secret key, algorithm. HS256 is HMAC-SHA256. Returns a string — the three dot-separated parts."

### Section B — b64url_decode

**EXPLAIN**
> "Line 47: `token.split('.')` — splits the token on dots to get the three raw base64url-encoded strings."

> "Lines 50–56: `b64url_decode`. This is the interesting one — it manually reverses the base64url encoding."

> "Line 53: `s.replace('-', '+').replace('_', '/')` — swap the url-safe characters back to standard base64. base64url replaced `+` with `-` and `/` with `_`; we undo that."

> "Line 55: `s += '=' * (-len(s) % 4)` — re-add the padding that JWT stripped. Standard base64 requires length divisible by 4. This formula computes the minimum number of `=` characters needed. If the length is already divisible by 4, `-len(s) % 4` is 0 and nothing is added."

> "Line 56: `json.loads(base64.b64decode(s))` — now it's valid standard base64. Decode it, parse the JSON, return a dict. This is everything jwt.io does when you paste a token into it."

### Section C — Verification

**EXPLAIN**
> "Line 81: `jwt.decode(token, SECRET, algorithms=[\"HS256\"])`. Note the plural `algorithms=` — it takes a list. This is intentional security design: if you accepted any algorithm, an attacker could submit a token with `alg: none` — a real attack from early JWT implementations. Always pass an explicit list."

> "Lines 84–87: wrong secret raises InvalidSignatureError. The library computed the HMAC using the wrong key. The result didn't match the signature in the token."

### Section D — Tampering

**EXPLAIN**
> "Lines 100–105: we build a forged payload. Copy the original, change `role` to `superadmin`, JSON-serialize it, base64url-encode it — exactly what the library does internally. We now have a new base64url-encoded payload."

> "Line 108: reassemble the token with the original header, the forged payload, and the original signature. We kept the original signature because we don't know the secret — we can't compute a new one."

> "Lines 114–119: `jwt.decode` on the tampered token raises InvalidSignatureError. The server recomputed the HMAC over the new `header.forged_payload` bytes and got a different value than the signature. You cannot change the payload without either knowing the secret or accepting that verification will fail."

> "The payload is readable by anyone — it's encoding, not encryption. But it's tamper-evident. That's the JWT security model."

## Recording notes

- **Conceptual sections first, code second** — spend 5–7 minutes on concepts before opening any file. Students who understand the concept will read the code naturally; students who skip the concept will be confused by the code.
- **Pause on b64url_decode** — the padding formula `-len(s) % 4` is the densest line. Write it out: if len is 10, `-(10) % 4 = 2`, so we add two `=` signs. Let that sit.
- **For hash_demo.py:** pause on `b"password"` (bytes literal) and the cost factor table — these are the two things students always get wrong in their own code.
- **VS Code tips:** use VS Code's line highlighting to point at specific lines while explaining. Zoom in on the b64url_decode function — it's only 6 lines but dense.
- **Connect back:** at the end of each file, say "now you know what [hashpw/jwt.encode] is doing when you call it" — students should feel ownership of the output they're about to see.
