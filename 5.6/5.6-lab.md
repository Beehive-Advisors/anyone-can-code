## Part 1 — Password Hashing with bcrypt

### What is a hash function?

A hash function takes any input — a word, a file, an entire database — and produces a fixed-length string of characters. Two rules define it:

1. **One-way**: you can go from a password to its hash, but not from the hash back to the password. There is no reverse.
2. **Deterministic**: the same input always produces the same output.

This is how websites store your password. They don't store `"password123"` in a column. They store the hash of it. When you log in, they hash what you typed and compare it to what they stored. Same hash? You're in.

So why not just use SHA-256? SHA-256 can compute about 10 billion hashes per second on a modern GPU. An attacker who steals your user database can try every common password in existence in seconds. bcrypt was built specifically to prevent this.

### What bcrypt adds

bcrypt has two properties that SHA-256 doesn't:

**Salt:** a random 22-character string generated fresh for every hash, mixed into the input before hashing. Two users with the same password get completely different hash strings. The salt is stored inside the hash string — no separate column needed.

**Cost factor:** an exponent that controls how many iterations the algorithm runs. Cost 12 means 2^12 = 4,096 iterations. Doubling the cost adds one to the exponent and doubles the time. At cost 12, hashing takes ~250ms — imperceptible to a user logging in, but it limits an attacker to ~4 guesses per second.

### Exercise

Hash `b"hunter2"` at cost 10 and verify it with two candidates: `b"hunter2"` (correct) and `b"password"` (wrong). Print the hash and both verification results.

<details>
<summary>Solution</summary>

```python
import bcrypt

password = b"hunter2"
hashed = bcrypt.hashpw(password, bcrypt.gensalt(rounds=10))

print(f"Hash: {hashed.decode()}")
print(f"Correct:  {bcrypt.checkpw(b'hunter2', hashed)}")
print(f"Wrong:    {bcrypt.checkpw(b'password', hashed)}")
```

The hash will look different each run because `gensalt()` generates a new random salt every time. Both verification calls use the salt embedded in `hashed` — no extra argument needed.

</details>

## Part 2 — JWT: Signed Identity Tokens

### What is a JWT?

A JWT (JSON Web Token) is a way for a server to issue a tamper-proof bundle of claims to a client. Instead of storing session state on the server, the server encodes the user's identity directly into a token, signs it, and sends it to the client. On every future request, the client sends the token back, and the server verifies the signature and reads the claims — no database lookup needed.

A JWT is three base64url-encoded JSON strings joined by dots:

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9 . eyJzdWIiOiJ1c2VyXzQy... . Rr55Pwn...
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^   ^^^^^^^^^^^^^^^^^^^^^^^   ^^^^^^^^^^^
        header                               payload (claims)        signature
```

The payload is **readable by anyone** — it's encoded, not encrypted. But it cannot be modified without the secret key, because the signature covers the exact bytes of header + payload. Change one character and the signature breaks.

### How signing works

The signature is HMAC-SHA256 of `base64url(header) + "." + base64url(payload)` using the server's secret key. To verify a token, the server re-computes the HMAC and compares it to the signature. Any tampering produces a different HMAC.

### Shell decode exercise

You can decode any JWT with two shell commands — no library needed:

```bash
TOKEN=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiJ1c2VyXzQyIiwibmFtZSI6IkFsaWNlIERlbW8iLCJlbWFpbCI6ImFsaWNlQGV4YW1wbGUuY29tIiwicm9sZSI6ImFkbWluIiwiaWF0IjoxNzc2MTk5Nzc2LCJleHAiOjE3NzYyMDMzNzZ9.Rr55PwnCO5z4DpNz11KEKWsRhI8bwXrspIMzDdJgY7k

# Decode the header (part 1)
echo $TOKEN | cut -d. -f1 | base64 -d 2>/dev/null | python3 -m json.tool

# Decode the payload (part 2)
echo $TOKEN | cut -d. -f2 | base64 -d 2>/dev/null | python3 -m json.tool
```

`cut -d. -f1` splits on `.` and takes the first field. `base64 -d` decodes it. `python3 -m json.tool` pretty-prints. The `2>/dev/null` suppresses the "invalid padding" warning that macOS base64 sometimes produces.

### Exercise

Create a JWT with a payload containing `user_id`, `username`, and `exp` set to 24 hours from now. Verify it with `jwt.decode()`. Then try to decode it with a different secret and confirm you get `InvalidSignatureError`.

<details>
<summary>Solution</summary>

```python
import jwt
from datetime import datetime, timedelta, timezone

SECRET = "this-is-exactly-32-bytes-long!!!"

payload = {
    "user_id": 99,
    "username": "bob",
    "exp": datetime.now(timezone.utc) + timedelta(hours=24),
}

token = jwt.encode(payload, SECRET, algorithm="HS256")
print("Token:", token)

# Verify with correct secret
decoded = jwt.decode(token, SECRET, algorithms=["HS256"])
print("Decoded:", decoded)

# Try wrong secret
try:
    jwt.decode(token, "wrong-secret-32-bytes-xxxxxxxxxxx", algorithms=["HS256"])
except jwt.exceptions.InvalidSignatureError:
    print("Wrong secret rejected — as expected")
```

`exp` can be a `datetime` object — PyJWT converts it to a Unix timestamp automatically. The 32-byte constraint applies to the wrong secret too: use at least 32 characters.

</details>

## Part 3 — OAuth 2.0: Delegated Access

### What is OAuth?

OAuth solves a specific problem: you want App B (a calendar app) to access your data on App A (Google), but you don't want to give App B your Google password.

The solution is delegated authorization. Instead of a password, Google issues a short-lived authorization code that App B can exchange for a scoped access token. The token can only do specific things (read your calendar, not change your password). You can revoke it without changing your password.

### The four-step flow

1. **Authorization request** — App B redirects your browser to Google's `/authorize` endpoint. Google responds with a `302` redirect back to App B's callback URL, with an authorization code in the query string.
2. **Code extraction** — App B's callback handler reads the `code` from the URL.
3. **Token exchange** — App B's back-end (not the browser) POSTs the code + client secret to `/token`. Google returns an access token. This happens server-to-server — the secret never touches the user's browser.
4. **Resource access** — App B uses the access token as a Bearer token in the `Authorization` header.

## Putting it together

| Mechanism | Problem it solves | Key idea | Where you see it |
|-----------|------------------|----------|------------------|
| bcrypt | Storing passwords safely | Cost factor makes brute-force slow; salt prevents precomputed tables | `users` table in every web app with accounts |
| JWT | Stateless session tokens | Server signs claims; any server with the secret can verify | API authentication, mobile app sessions |
| OAuth 2.0 | Third-party access without sharing passwords | Delegated authorization via short-lived codes; access tokens are scoped | "Sign in with Google", GitHub OAuth apps |

These three mechanisms often appear together in the same system. bcrypt stores the password in the users table. After login, the server issues a JWT as the session token. If the app also supports Google login, OAuth handles the "prove who you are to Google" step and the app issues its own JWT after the flow completes.
