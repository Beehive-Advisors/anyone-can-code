## Part 1 — Password Hashing with bcrypt

### What is hashing?

A hash function takes any input — a word, a file, a whole database — and produces a fixed-length string of characters. Two important properties:

1. **One-way**: You can go from `password123` → `abc123def456...`, but you can't go backwards. There's no "unhash" function.
2. **Deterministic**: The same input always produces the same output.

This is how websites store your password. They don't store `"password123"` — they store the hash of it. When you log in, they hash what you typed and compare it to the stored hash. If they match, you're in.

### Why not just use SHA-256?

SHA-256 is a popular hash function used everywhere — verifying file downloads, signing git commits, TLS certificates. It was built to be *fast*.

Fast is the problem for passwords.

A modern GPU can compute **10 billion SHA-256 hashes per second**. That means an attacker who steals your user database can try every common password — and billions of variations — in seconds.

bcrypt was designed in 1999 specifically for passwords. It has two features that fix this problem.

### Feature 1: Salt

Every time you hash a password with bcrypt, it first generates a **salt** — a random string of characters that gets mixed into the input before hashing. 

This means two users with the same password (`password123`) will get two completely different hashes, because each one was mixed with a different random salt.

Why does this matter? Imagine an attacker builds a lookup table: every common password hashed → its SHA-256 value. They steal your database, look up each hash, and instantly know every weak password. This is called a **rainbow table** — a precomputed dictionary of `password → hash` pairs.

Salt defeats this completely. With a unique random salt per user, the attacker can't look anything up. They'd need to build a separate table for every possible salt — which is computationally impossible.

### Feature 2: Cost factor

bcrypt has a parameter called `rounds` that controls how many times the function iterates internally. The default is 12, which means it runs 2^12 = 4,096 iterations before producing a result.

This makes bcrypt deliberately slow. That's not a bug — it's the whole point.

### The hash string anatomy

Run the demo:

```bash
python3.12 hash_demo.py
```

Look at Section A's output:

```
Hash: $2b$12$uBMpac65irGoAIwbd4tuNet6OJuDk/QYlJdl8JZqVeXhbbzhVyjB6
```

That 60-character string contains everything needed to verify a password later. Break it down:

```
$2b       $12       uBMpac65irGoAIwbd4tuNe     t6OJuDk/QYlJdl8JZqVeXhbbzhVyjB6
 │         │                │                               │
version  cost factor    22-char salt                  31-char hash
         (rounds=12)   (random, unique                (the actual result)
                        per hash)
```

The salt is right there in the string — bcrypt stores it alongside the hash so it can repeat the exact same hashing process when verifying.

Everything lives in one column in your database. No separate salt table needed.

### Verification

Section B shows how verification works:

```python
bcrypt.checkpw(b"correct-horse-battery-staple", hashed)  # → True
bcrypt.checkpw(b"hunter2", hashed)                        # → False
```

When you call `checkpw()`, bcrypt:
1. Reads the salt out of the stored hash string
2. Hashes the candidate password using that same salt
3. Compares the result to the stored hash

**The server never stores your password** — only the hash. Even if someone steals the database, they can't reverse the hash back to the original password.

### Salt in action

Section C hashes `password123` three times. You get three completely different hashes — but all three verify correctly. Run it and watch:

```
Hash 1: $2b$12$ZKASZpFEvc4/...
Hash 2: $2b$12$WJCSeqS6L9RJ...
Hash 3: $2b$12$cACRJUp7p0kp...
```

Same password, completely different outputs. This is what defeats the rainbow table attack.

### Cost factor timing

Section D shows why the slowness is the point:

```
Cost  4:  0.001s   (too fast — never use this for passwords)
Cost  8:  0.015s
Cost 10:  0.059s
Cost 12:  0.237s   ← production default
```

At cost 12, hashing takes ~0.25 seconds. An attacker gets roughly **4 guesses per second**.

At SHA-256 speeds: **10 billion guesses per second**.

That's the difference between cracking a database in seconds versus thousands of years.

Rule of thumb: tune your cost factor so hashing takes around 250 milliseconds on your production hardware. As computers get faster over the years, bump it up.

## Part 2 — JWT Anatomy

### A quick detour: what is base64?

Before we look at JWTs, you need to understand base64, because JWTs are made of it.

Computers store everything as binary — ones and zeros. But some systems (email, URLs, JSON) were designed to handle only regular printable text, not raw binary data.

Base64 is a way to encode binary data as text. It takes any bytes and represents them using only 64 safe characters: `A-Z`, `a-z`, `0-9`, `+`, `/`. The name comes from the 64-character alphabet.

For example, the text `Hello` in base64 is `SGVsbG8=`.

You can decode it right now:

```bash
echo "SGVsbG8=" | base64 -d
```

Output: `Hello`

It's not encryption — it's just a different way of writing the same data. Anyone can decode it.

### What problem does JWT solve?

When you log into a website, the server needs to remember who you are for every request you make. There are two common approaches:

**Sessions (the old way):** The server creates a record in its database: "session #abc123 belongs to Alice." It gives your browser a cookie with that ID. On every request, your browser sends the cookie, the server looks it up in the database, and finds your info.

This works, but it means every single request requires a database lookup. And if you have multiple servers, they all need to share the same session database.

**JWTs (the newer way):** Instead of a database record, the server creates a small document that says "this is Alice, she has admin role, this expires in 1 hour" — and signs it cryptographically. It gives this document to your browser as a token. On every request, your browser sends the token, and the server verifies the signature. **No database lookup needed.**

The token carries its own proof of authenticity.

### The three-part structure

A JWT looks like:

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiJ1c2VyXzQyIn0.SflK...
          HEADER                               PAYLOAD             SIGNATURE
```

Three base64-encoded sections, separated by dots.

- **Header**: Metadata about the token — what signing algorithm was used
- **Payload**: The actual claims — who you are, what you're allowed to do, when it expires
- **Signature**: A cryptographic seal that proves the header and payload haven't been changed

Run the demo:

```bash
python3.12 jwt_demo.py
```

Section A creates a token and prints it. Copy the `TOKEN=...` line at the bottom.

### Decode it yourself — no library

Set the token as a shell variable:

```bash
TOKEN=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIi...  # paste your token
```

Decode the header:

```bash
echo $TOKEN | cut -d'.' -f1 | tr '_-' '/+' | base64 -d; echo
```

Breaking down that command:
- `cut -d'.' -f1` — split on dots, take the first piece (the header)
- `tr '_-' '/+'` — JWTs use a slightly modified version of base64 that swaps `+`→`-` and `/`→`_` so the token is safe to put in a URL. This command swaps them back.
- `base64 -d` — decode it

Output: `{"alg":"HS256","typ":"JWT"}`

Now decode the payload:

```bash
echo $TOKEN | cut -d'.' -f2 | tr '_-' '/+' | base64 -d | python3.12 -m json.tool
```

Output:

```json
{
    "sub": "user_42",
    "name": "Alice Demo",
    "email": "alice@example.com",
    "role": "admin",
    "iat": 1776127146,
    "exp": 1776130746
}
```

**You just read a JWT with two shell commands and zero libraries.** This is the most important thing to understand about JWTs: **the payload is not secret**. Anyone who has a copy of your token can read every claim in it. Never put passwords, credit card numbers, or anything sensitive in a JWT payload.

The `exp` field is a Unix timestamp — seconds since January 1, 1970. That's the expiry time. After that, the server will reject the token even if the signature is valid.

### What does the signature actually do?

The signature is what makes a JWT trustworthy. Here's how it works:

When the server creates a token, it takes the header and payload, runs them through a function called HMAC-SHA256 using a secret key that only the server knows, and appends the result as the third segment.

When the server receives a token back, it re-runs that same function over the header and payload. If the result matches the signature in the token, the data hasn't been tampered with. If anything changed — even one character — the signature won't match.

### Tampering in action

Section D of the demo attempts a **privilege escalation** attack — trying to upgrade your own permissions by modifying the token:

1. Original payload: `"role": "admin"`
2. We forge it to: `"role": "superadmin"`
3. We base64-encode the forged payload
4. We build a new token: `original_header.FORGED_PAYLOAD.original_signature`
5. We send it to the server

Result:

```
Server rejects tampered token → InvalidSignatureError
```

The signature was computed over the *original* payload bytes. We changed the payload. The signature no longer matches. Game over for the attacker — unless they know the server's secret key, they can't produce a valid signature for a modified payload.

### Exercise

Decode the third segment of the token — the signature — using the same shell commands:

```bash
echo $TOKEN | cut -d'.' -f3 | tr '_-' '/+' | base64 -d | xxd | head
```

`xxd` shows raw bytes in hex. You'll see random-looking binary data — not JSON. 

Then answer: why exactly does changing one character in the payload cause signature verification to fail? 

<details>
<summary>Answer</summary>

The signature is computed over the exact bytes of `header.payload`. If you change even one character in the payload, you get completely different bytes, which produces a completely different signature output. The stored signature was calculated from the original bytes, so it no longer matches the recalculated one. There's no way to fix the signature without knowing the server's secret key.

</details>

## Part 3 — OAuth 2.0 Authorization Code Flow

### The problem OAuth solves

Imagine you're building a fitness app that wants to read a user's Google Calendar to find free slots. You could ask the user for their Google password and log in as them.

But that's a terrible idea:
- Your app sees their full Google password
- Your app gets access to *everything* in their Google account, not just Calendar
- If your app gets hacked, their entire Google account is compromised
- The user can't revoke your access without changing their Google password

OAuth 2.0 is a standard that solves this. Instead of giving your app their password, the user grants your app a **limited, temporary token** that only works for the specific thing they approved. They can revoke it anytime without touching their password.

This is exactly what happens when you click "Sign in with Google" or "Continue with GitHub."

### The four players

Every OAuth flow involves four parties:

| Role | What it does | In this lab |
|------|-------------|-------------|
| **Resource Owner** | The user who owns the data | You |
| **Client** | Your app that wants access | `oauth_flow.sh` |
| **Authorization Server** | Verifies the user and issues tokens | `oauth_server.py` on port 8080 |
| **Resource Server** | The API holding the protected data | Also `oauth_server.py` |

In real life, the Authorization Server and Resource Server are usually run by the same company (Google, GitHub) but are logically separate.

### The flow, step by step

Here's the complete sequence before we run it:

```
Step 1: Your app sends the user's browser to the auth server's login page
        (with your app's ID and where to redirect back to)

Step 2: The user logs in and clicks "Allow"

Step 3: The auth server redirects the user's browser back to your app
        with a short-lived code in the URL

Step 4: Your app's server takes that code and swaps it for an access token
        (this happens server-to-server, not in the browser)

Step 5: Your app uses the access token to call the protected API
```

You might wonder: why the two-step code → token dance? Why not just put the token in the Step 3 redirect?

Because the redirect URL shows up in browser history, server logs, and the browser's address bar. A token sitting in a URL is easy to accidentally leak. The authorization code is useless on its own — an attacker would also need your app's secret password (`client_secret`) to exchange it for a real token. So the code can safely travel through the browser, and the actual token exchange happens privately, server-to-server.

### Walk through the flow

Open **Terminal 2**:

```bash
cd ~/src/courses/anyone-can-code/5.6
bash oauth_flow.sh
```

Watch both terminals as the script runs. Here's what each step is doing:

---

**Step 1 — Authorization Request**

The script makes a request to `/authorize`. In a real app, this is the URL your app would redirect the user's browser to:

```
GET /authorize
  ?client_id=demo-client-id          ← who is asking
  &redirect_uri=http://...9000/callback   ← where to send the user after approval
  &response_type=code                ← we want a code, not a token directly
  &state=random-csrf-token-...       ← a random value we made up
```

The server responds with a `302 Found` — a redirect. Look at the `Location` header:

```
Location: http://localhost:9000/callback?code=9LGqXKVury...&state=...
```

In a real browser flow, the user's browser would automatically follow this redirect to your app's callback URL. The `code` is in the URL as a query parameter.

The `state` value deserves a mention. Before redirecting the user, your app generated a random value and included it. When the redirect comes back, your app checks that the `state` value matches what it sent. This is protection against a specific attack: without `state`, a malicious site could trick your app into accepting *their* authorization code and associating it with *your* user's account. The random `state` token makes this impossible.

---

**Step 2 — Extract the Authorization Code**

The script pulls the code out of the `Location` header. It's a short random string — in this demo it expires in 60 seconds and can only be used once.

---

**Step 3 — Token Exchange**

Now the script makes a `POST` request to `/token` — this is the private server-to-server exchange:

```
POST /token
  grant_type=authorization_code   ← what kind of request this is
  code=...                        ← the code from Step 2
  client_id=demo-client-id        ← your app's ID
  client_secret=demo-client-secret ← your app's password (never goes to the browser)
```

Response:

```json
{
  "access_token": "ZZC_keMxWTHjaDDlIeKB7...",
  "token_type": "Bearer",
  "expires_in": 3600,
  "scope": "read:profile"
}
```

The `access_token` is what your app actually uses going forward. Notice `expires_in: 3600` — it's valid for one hour.

`client_secret` is essentially your app's password. It only goes in this POST request, server-to-server, over HTTPS. It never goes into a browser URL or a client-side JavaScript file.

---

**Step 4 — Access the Protected Resource**

Now the script uses the access token to call the protected endpoint:

```
GET /userinfo
Authorization: Bearer ZZC_keMxWTHjaDDlIeKB7...
```

Response:

```json
{
  "sub": "user_001",
  "name": "Alice Demo",
  "email": "alice@example.com"
}
```

The `Authorization: Bearer` header is how you present a token to an API. The resource server checks the token and returns the data.

---

**Step 5 — No token → denied**

```json
{ "error": "unauthorized" }
```

No token, no access. The resource is protected.

---

**Step 6 — Replayed code → denied**

The script tries to exchange the same authorization code a second time:

```json
{ "error": "invalid_grant" }
```

Codes are single-use. Once your app exchanges a code for a token, the code is destroyed. If an attacker somehow intercepts the code from the redirect URL and races to exchange it — they'll find your app already did, and the code is gone.

This exact flow runs behind "Sign in with Google," "Continue with GitHub," and "Log in with Spotify." The only differences in production are:

- The authorization server is Google/GitHub/Spotify instead of `localhost:8080`
- The connection uses real HTTPS
- The access token is often a JWT rather than a random string

## Putting it together

These three mechanisms show up together constantly in real systems:

| Mechanism | The problem it solves | The key idea |
|-----------|----------------------|--------------|
| **bcrypt** | How do you store passwords safely? | Hash + salt = useless if database is stolen. Slow = hard to brute-force. |
| **JWT** | How does the server know who you are without a database lookup? | A signed document that carries its own proof of authenticity. |
| **OAuth** | How does your app access someone's data on another service without their password? | A temporary, limited token the user explicitly grants — and can revoke. |

A typical login flow in a modern app uses all three:

```
1. User submits password
   → bcrypt.checkpw() verifies it against the stored hash

2. Server issues a JWT
   → jwt.encode({sub, role, exp}) → sent to the client as a Bearer token

3. User clicks "Connect Spotify"
   → OAuth flow → your server gets a Spotify access token
   → you can call Spotify's API on the user's behalf
```
