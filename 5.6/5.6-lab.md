## Part 1 — Password Hashing with bcrypt

### Why not SHA-256?

SHA-256 was designed to be *fast* — for TLS certificate verification, git commit hashing, file integrity checks. Fast is exactly what you don't want for passwords.

A modern GPU can compute **10+ billion SHA-256 hashes per second**. That's enough to try every 8-character lowercase+digit password in under a second.

bcrypt was designed in 1999 specifically for passwords. It has two key properties:

**1. Salt** — A random 128-bit value is generated and baked into every hash. Two users with the same password get completely different hashes.

**2. Cost factor** — A parameter controls how many iterations the function runs. Double the cost → double the time. You can crank it up as hardware gets faster.

### The hash string anatomy

Run the demo:

```bash
python3.12 hash_demo.py
```

Look at Section A's output:

```
Hash: $2b$12$uBMpac65irGoAIwbd4tuNet6OJuDk/QYlJdl8JZqVeXhbbzhVyjB6
```

Break it down:

```
$2b   $12   $uBMpac65irGoAIwbd4tuNe   t6OJuDk/QYlJdl8JZqVeXhbbzhVyjB6
 │     │              │                            │
algo  cost      22-char salt                  31-char hash
```

The entire verification state lives in this one string — the algorithm, the cost factor, and the salt. No separate database column needed.

### Verification

Section B shows the key insight: `checkpw()` works by extracting the salt from the stored hash, re-hashing the candidate password with it, and comparing. **The server never stores your password.** It only stores enough to verify it.

```python
bcrypt.checkpw(b"correct-horse-battery-staple", hashed)  # → True
bcrypt.checkpw(b"hunter2", hashed)                        # → False
```

### Salt prevents rainbow tables

Section C hashes the same password three times. Notice that you get **three completely different hashes**, but all three verify correctly. 

This is why salt defeats rainbow tables: a precomputed table maps `password123 → hash_value`. But with a unique salt per user, the attacker would need a separate table for every possible salt — computationally impossible.

### Cost factor is the point

Section D shows timing for rounds 4 through 12:

```
Cost  4:  0.001s   (toy; never use for passwords)
Cost  8:  0.015s
Cost 10:  0.059s
Cost 12:  0.237s   ← production default
```

At 0.237s per attempt, an attacker gets ~4 guesses per second. At SHA-256 speeds, they'd get billions. **The slowness is the feature.**

Rule of thumb: set the cost factor so hashing takes ~250ms on your production hardware. As CPUs get faster, bump it up.

## Part 2 — JWT Anatomy

### What problem does JWT solve?

Traditional sessions work like a coat-check: the user gets a ticket (session ID), the server keeps the coat (session data). Every request, the server looks up the ticket.

JWTs flip this: the coat travels with the user. The token carries its own claims (user ID, role, expiry) and a cryptographic signature that proves the claims haven't been tampered with. **The server needs no database lookup** — it just checks the signature.

### The three-part structure

A JWT looks like:

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiJ1c2VyXzQyIn0.SflK...
          HEADER                               PAYLOAD             SIGNATURE
```

Three base64url-encoded segments, separated by dots.

Run the demo:

```bash
python3.12 jwt_demo.py
```

Section A creates a token and prints it. Copy the `TOKEN=...` line at the bottom.

### Decode it yourself — no library

Set the token in your shell:

```bash
TOKEN=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIi...  # paste from output
```

Decode the header:

```bash
echo $TOKEN | cut -d'.' -f1 | tr '_-' '/+' | base64 -d; echo
```

Decode the payload:

```bash
echo $TOKEN | cut -d'.' -f2 | tr '_-' '/+' | base64 -d | python3.12 -m json.tool
```

What `tr '_-' '/+'` does: JWTs use base64**url** encoding, which swaps `+` → `-` and `/` → `_` (so the token is safe in URLs). The `tr` command reverses this so the standard `base64 -d` can decode it.

**You just decoded a JWT using two shell commands and zero libraries.** This is the critical insight: **the payload is not encrypted**. Anyone who holds a JWT can read its claims. Never put sensitive data in a JWT payload.

### What the signature actually protects

Section D of the demo performs a **privilege escalation attack**:

1. Original payload has `"role": "admin"`
2. We change it to `"role": "superadmin"`  
3. We base64-encode the forged payload
4. We assemble a new token: `header.FORGED_PAYLOAD.original_signature`
5. We try to verify it

```
Server rejects tampered token → InvalidSignatureError
```

The signature is HMAC-SHA256 over `header.payload`. Change one byte anywhere, and the signature no longer matches. The attacker would need the server's secret key to generate a valid new signature — and if they had that, they've already won a much bigger battle.

### The `alg: none` attack (don't do this in prod)

The JWT spec originally allowed `"alg": "none"` — no signature. Some early libraries accepted unsigned tokens. Modern libraries reject this by default, but it's why you always specify `algorithms=["HS256"]` explicitly when decoding — you don't want the token itself to choose its own verification algorithm.

## Part 3 — OAuth 2.0 Authorization Code Flow

### The problem OAuth solves

Imagine you're building an app that needs to read a user's GitHub repositories. You could ask them for their GitHub password — but that's terrible for several reasons:
- Your app sees their password
- Your app gets full access, not just repos
- If your app is breached, their GitHub account is too
- They can't revoke your access without changing their password

OAuth 2.0 solves this with **delegated authorization**: the user grants your app a limited, revocable token without ever giving you their password.

### The four players

| Role | Who | In this lab |
|------|-----|-------------|
| Resource Owner | The user | You |
| Client | Your app | `oauth_flow.sh` |
| Authorization Server | Issues tokens | `oauth_server.py` |
| Resource Server | The protected API | Also `oauth_server.py` |

### The flow

```
Your App                    Auth Server
    |                            |
    |-- GET /authorize? -------->|   "User, please log in and approve"
    |                            |
    |<-- 302 ?code=AUTH_CODE ----|   Short-lived one-time code
    |                            |
    |-- POST /token (+ secret) ->|   Back-channel exchange
    |                            |
    |<-- { access_token } -------|
    |                            |
    |-- GET /userinfo ---------->|   Use the token
    |  Authorization: Bearer ... |
```

The two-step design is deliberate: the authorization code arrives through the browser (where URLs get logged), but the access token is exchanged server-to-server (never touches a URL bar).

Watch both terminals. Let's trace what happens.

---

**Step 1 — Authorization Request**

```bash
curl -vs "http://localhost:8080/authorize?client_id=demo-client-id \
  &redirect_uri=http://localhost:9000/callback \
  &response_type=code \
  &state=random-csrf-token-..."
```

The server responds:
```
< HTTP/1.0 302 Found
< Location: http://localhost:9000/callback?code=OwZPy2DbG_9NwJEH...&state=...
```

In a real browser-based flow, the browser would follow this redirect to your app's callback URL. Here, curl sees the `302` and we extract the `code` from the `Location` header.

The `state` parameter is a CSRF token. When your callback receives the redirect, you verify `state` matches what you sent. This prevents an attacker from tricking your app into accepting *their* authorization code.

---

**Step 2 — Extract the Code**

We pull the auth code from the `Location` header using `grep` and `cut`. Notice it's a short-lived random string, not a structured token like a JWT.

---

**Step 3 — Token Exchange (the important part)**

```bash
curl -X POST http://localhost:8080/token \
  -d "grant_type=authorization_code" \
  -d "code=AUTH_CODE" \
  -d "client_id=demo-client-id" \
  -d "client_secret=demo-client-secret"
```

Response:
```json
{
  "access_token": "rc-ZT3RJWRAkN8TfGk9k-p_LBo8bfVJKOlLZfu4ueMg",
  "token_type": "Bearer",
  "expires_in": 3600,
  "scope": "read:profile"
}
```

This POST happens from your **server**, not from the browser. The `client_secret` is only known to your server and the auth server. This is the "back-channel" — the reason codes exist instead of handing the token directly in the redirect.

---

**Step 4 — Access the Protected Resource**

```bash
curl http://localhost:8080/userinfo \
  -H "Authorization: Bearer ACCESS_TOKEN"
```

Response:
```json
{
  "sub": "user_001",
  "name": "Alice Demo",
  "email": "alice@example.com",
  "role": "member"
}
```

The access token acts as a temporary, limited credential. The resource server validates it and returns the user's data.

---

**Step 5 & 6 — Edge cases**

The script also demonstrates:
- **No token**: `401 unauthorized` — the resource is protected
- **Replayed code**: `400 invalid_grant` — auth codes are single-use

### What you just saw happen

Every "Sign in with Google/GitHub/Spotify" button you've ever clicked runs this exact flow. The difference is:
- Steps 2–3 happen through the real user's browser and your real backend
- The auth server is Google/GitHub/Spotify instead of `localhost:8080`
- The token may be a JWT instead of an opaque random string

## Putting it together

These three mechanisms often appear together in a single system:

```
User Login:
  password → bcrypt.checkpw() → verified

Session Token:
  jwt.encode({sub, role, exp}) → JWT → stored in Authorization header

Third-Party Auth:
  OAuth flow → access_token → call their API on user's behalf
```

| Mechanism | What it protects | Main idea |
|-----------|-----------------|-----------|
| bcrypt | Stored passwords | Slow hash + salt = useless if DB stolen |
| JWT | Stateless identity | Self-contained claims + tamper-evident signature |
| OAuth | Delegated access | Limited token instead of full credentials |
