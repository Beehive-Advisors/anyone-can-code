## Part 1 — bcrypt Password Hashing

### What's wrong with fast hashing?

Imagine your app stores user passwords in a database. If the database is ever stolen — and breaches
happen — the attacker gets whatever you stored. If you stored plain text, they have every password
immediately. The obvious improvement: store a scrambled version of the password instead.

A hash function takes any input and produces a fixed-length "fingerprint." It's one-way — you can't
reverse it. At login, the server hashes what the user typed and compares fingerprints. If they
match, the password was correct. Common hash functions like SHA-256 work this way.

But SHA-256 has a problem for this use case: it's designed to be fast. A modern GPU can compute
billions of SHA-256 hashes per second. An attacker who steals a SHA-256 password database can
try every word in the dictionary, every common password, every variation — at billions of guesses
per second. Most passwords crack in minutes.

bcrypt solves this by being deliberately, purposefully slow. It's built on the Blowfish block cipher
and is configured to take tens of milliseconds per hash. That slowness is not a flaw — it's the
entire point. An attacker who could try billions of SHA-256 hashes per second can only try a few
thousand bcrypt hashes per second. A database that would crack in hours with SHA-256 would take
years with bcrypt.

The second problem bcrypt solves is rainbow tables. A rainbow table is a precomputed lookup: for
every common password, what is its SHA-256 hash? You don't need to guess — you just look up the
hash. bcrypt defeats this by automatically mixing in a random 16-byte "salt" unique to each user.
Even if two users have the same password, their bcrypt hashes are completely different. There's
nothing useful to precompute.

> **Before running:** open `hash_demo.py` in VS Code — the code walkthrough video explains each section before you run it.

**Conceptual question:**

> **Why does a wrong password still need to run through bcrypt's full computation before returning False?**

<details><summary>Answer</summary>

If the server returned "wrong password" faster than "correct password," an attacker could measure
the response time and learn whether their guess partially matched. bcrypt's `checkpw()` always runs
the full computation, making every response take the same amount of time regardless of outcome. This
is called a "timing-safe" comparison — it prevents timing side-channel attacks.

</details>

### Exercise

Write a short Python script that:
1. Prompts the user to type a password (use `input()`)
2. Hashes it with bcrypt at cost 12
3. Asks the user to type it again
4. Prints "Correct!" or "Wrong password." based on whether they match

<details>
<summary>Solution</summary>

```python
import bcrypt

password = input("Create a password: ").encode("utf-8")
hashed = bcrypt.hashpw(password, bcrypt.gensalt(rounds=12))
print(f"Stored hash: {hashed.decode()}")

attempt = input("Enter it again: ").encode("utf-8")
if bcrypt.checkpw(attempt, hashed):
    print("Correct!")
else:
    print("Wrong password.")
```

`input()` returns a `str`, so `.encode("utf-8")` converts it to `bytes` before passing to bcrypt.
`checkpw()` extracts the salt from `hashed` automatically — you pass the same `hashed` value you
stored, not the salt separately.

</details>

## Part 2 — JWT (JSON Web Tokens)

### The session problem

After a user logs in, every subsequent HTTP request arrives blank — HTTP has no memory. The server
needs to recognize the same user across many requests.

The traditional approach: the server creates a session record in a database and gives the user an
opaque random ID stored in a cookie. On every request, the server looks up that ID in the database
to find out who you are. This works, but every single request requires a database query just to
answer "who is this?"

A JWT is an alternative: instead of a pointer to data stored on the server, the token contains the
data itself. The user's ID, their role, when the token expires — all packed into the token string.
On every request, the server reads the token directly, with no database lookup.

The key question is: how does the server know the token wasn't forged? The answer: the token is
cryptographically signed with a secret key only the server knows. If anyone changes the payload —
say, upgrading their role from `"user"` to `"admin"` — the signature no longer matches, and the
server rejects the token. You cannot alter a valid JWT without the server's secret key.

One critical fact: a JWT is **signed**, not **encrypted**. Anyone can read the payload. Never put
passwords, credit card numbers, or secrets inside a JWT. If you need an encrypted token, that's a
different spec called JWE.

### Structure of a JWT

A JWT is three base64url-encoded strings joined by dots:

```
HEADER.PAYLOAD.SIGNATURE
```

base64url is a URL-safe variant of base64. It substitutes `+`→`-` and `/`→`_` and omits the `=`
padding character. This means a JWT can appear safely in a URL without percent-encoding.

Each of the three parts is independently encoded:

- **Header:** `{"alg": "HS256", "typ": "JWT"}` — which signing algorithm was used
- **Payload:** your data — user ID, role, expiration timestamp, etc.
- **Signature:** HMAC-SHA256 of `header.payload` using the server's secret key

> **Before running:** open `jwt_demo.py` in VS Code — the code walkthrough video walks through how the manual JWT construction in Section A maps to the PyJWT calls in Section B.

**Conceptual question:**

> **The JWT payload says `"role": "admin"` in plain text. If a user can read it, why can't they change it to escalate their own privileges?**

<details><summary>Answer</summary>

Reading the payload is easy — it's just base64url-encoded JSON. But changing it breaks the signature.
The signature was computed as `HMAC-SHA256(header.original_payload, server_secret)`. If you swap
in a new payload, the header-plus-payload string changes, which produces a completely different HMAC
output. The old signature doesn't match the new payload.

To produce a valid signature for the altered payload, you'd need the server's secret key. Without
it, you can read the token but you cannot forge a new valid one. This is exactly the security
guarantee the signature provides.

</details>

### Exercise

Decode the payload from a real JWT using only Python's standard library — no PyJWT.

Take the JWT from the Section B output. Copy the middle part (between the first and second dots).
Then write a Python one-liner to decode it:

```bash
python3.12 -c "
import base64, json
part = 'PASTE_THE_PAYLOAD_PART_HERE'
padded = part + '=' * (-len(part) % 4)
print(json.dumps(json.loads(base64.urlsafe_b64decode(padded)), indent=2))
"
```

You should get the raw JSON payload back. Try pasting the token into [jwt.io](https://jwt.io) to
verify you're reading the same data.

<details>
<summary>Why the padding calculation?</summary>

base64url strings in JWTs omit the `=` padding characters that standard base64 requires. Python's
`base64.urlsafe_b64decode()` needs the string to be a multiple of 4 bytes. The expression
`-len(part) % 4` computes exactly how many `=` characters to add: 0 if already aligned, 1, 2, or
3 otherwise. Adding extra `=` is harmless when the string was already aligned.

</details>

## Part 3 — OAuth 2.0 Authorization Code Flow

### The delegation problem

Imagine you're building an app that shows a user their GitHub repositories. You could ask the user
for their GitHub username and password, log in as them, and scrape the data. This is terrible:

- The user is trusting you with credentials that work on their entire GitHub account
- If your app is compromised, the attacker has their GitHub password
- The user can't revoke your access without changing their password
- GitHub might lock the account for automated login behavior

OAuth solves this. Instead of your app acting as the user, the user grants your app specific
permissions directly with GitHub. GitHub gives your app a temporary access token that works only
for reading repositories, not for anything else. The user can revoke it from their GitHub settings
at any time. Your app never sees the password.

### Front channel vs. back channel

The authorization code flow's key design is two separate communication channels:

**Front channel** — The user's browser. Every URL the browser visits ends up in browser history,
server logs, referrer headers, and potentially network logs. Nothing sensitive should travel here.

**Back channel** — A direct HTTPS request from your server to the auth server. The browser never
sees it. No logs. No history. This is where the actual access token travels.

The authorization code is the bridge between them. It's a short-lived, single-use code that travels
through the insecure front channel. Even if someone intercepts it, they'd need to exchange it at
the token endpoint — and that requires your app's secret credential. The token itself only ever
travels through the back channel.

### Step 1 — Hit the authorization endpoint

The client sends the user to `/authorize` with five parameters. Type this in Terminal 2:

```bash
curl -v --max-redirs 0 "http://localhost:5001/authorize?response_type=code&client_id=demo-client&redirect_uri=http://localhost:5001/callback&scope=read_profile&state=mystate123"
```

Look for the `Location:` header in the response — it contains the authorization code:

```
< HTTP/1.1 302 FOUND
< Location: http://localhost:5001/callback?code=mvA37xM0mULpIqdmdR2P-w&state=mystate123
```

*(Your code will be different — copy the value after `code=` up to the `&`.)*

The 302 redirect is the front channel. The code is in the URL — short-lived and single-use.

---

### Step 2 — Exchange the code for an access token

Copy the code value from Step 1. Replace `PASTE_CODE_HERE` with it and type in Terminal 2:

```bash
curl -X POST http://localhost:5001/token \
  -d "grant_type=authorization_code" \
  -d "code=PASTE_CODE_HERE" \
  -d "redirect_uri=http://localhost:5001/callback" \
  -d "client_id=demo-client" \
  -d "client_secret=demo-secret"
```

You should see:
```json
{
    "access_token": "Mu0rw7z61FMXocCMillVUDjzganG2z5IcIcF-oGhsxA",
    "expires_in": 3600,
    "scope": "read_profile",
    "token_type": "Bearer"
}
```

*(Your token will be different — copy the full `access_token` value.)*

This POST is the back channel — server to server, never visible in a browser URL or history.
Notice the body is form-encoded (`-d`), not JSON. OAuth requires `application/x-www-form-urlencoded`.

---

### Step 3 — Call the protected resource

Copy the `access_token` from Step 2. Replace `PASTE_TOKEN_HERE` and type in Terminal 2:

```bash
curl -H "Authorization: Bearer PASTE_TOKEN_HERE" http://localhost:5001/protected
```

You should see:
```json
{
    "message": "Access granted",
    "profile": {"email": "demo@example.com", "name": "Demo User"},
    "scope": "read_profile",
    "user": "demo-user"
}
```

**Conceptual question:**

> **Why does the authorization code flow use a code at all? Why not send the access token directly in the redirect URL?**

<details><summary>Answer</summary>

The redirect URL appears in browser history, server logs, and referrer headers. If the access token
traveled in a redirect URL, it would be logged in every system along the way and potentially
interceptable. The authorization code takes that risk instead — but the code is short-lived,
single-use, and requires the client's secret credential to exchange. Even if someone captures
the code from a URL log, they can't exchange it without the `client_secret` that never appears
in a URL.

The access token itself only travels in the back-channel POST response body — a private,
server-to-server connection that the browser never touches.

</details>

Modify the authorization request to use a redirect URI that isn't registered (`http://localhost:9999/callback`) and observe the server's response. What HTTP status does the server return, and what does the error say?

<summary>Answer</summary>

The server returns `400 Bad Request` with `{"error": "invalid_redirect_uri"}`.

If the server accepted any redirect URI, an attacker could send a victim a crafted authorization
URL with `redirect_uri=https://attacker.com/steal`. The victim would click "Allow" on the consent
screen, and the authorization code would be redirected to the attacker's server instead of the
legitimate app's callback. By only accepting pre-registered redirect URIs, the server ensures
the code can only go to a URL the app's developer explicitly registered.

</details>

## Putting it together

| Mechanism | Problem it solves | Key idea | Where you see it |
|-----------|------------------|----------|-----------------|
| bcrypt | Stolen password database | Deliberately slow hash + random salt | Every login form that stores passwords |
| JWT | Database query on every request | Signed, self-contained session token | API authentication; mobile apps |
| OAuth 2.0 | Sharing access without sharing passwords | Delegated authorization via short-lived codes | "Sign in with Google"; any third-party API integration |

In a typical login flow, all three appear together. When you first register, your password is hashed
with bcrypt and stored. When you log in, the server verifies your password with `checkpw()` and
issues a JWT. That JWT is attached to every subsequent API request. When a third-party app wants to
access your data, OAuth handles the delegation without your password ever leaving the original
service.
