## PART 1 — Password Hashing with bcrypt

---

### What is a hash function?

**SPEAK**
> "Before we run the demo, quick recap of the key ideas from the walkthrough."

> "A hash function is one-way and deterministic. One-way: you can go from 'password123' to a hash, but not back. Deterministic: same input always gives same output. That's how login works — hash what the user typed, compare to what's stored."

> "bcrypt adds two things on top of that. A random salt, so the same password hashes differently every time. And a tunable cost factor, so hashing is deliberately slow. The slowness is the security."

---

### Run the demo

**TYPE**
```bash
python3.12 hash_demo.py
```

**OUTPUT**
```
============================================================
SECTION A — Hashing a password
============================================================
Plaintext : correct-horse-battery-staple
Hash      : $2b$12$SNvBvPD4.MyCTd2hmVWOn.7MqxWPXQavjiGBrtqZwKINuxq30xora

Hash anatomy:
  $2b  → algorithm identifier (bcrypt v2b)
  $12  → cost factor (2^12 = 4,096 iterations)
  SNvBvPD4.MyCTd2hmVWOn.  → 22-char random salt (embedded in hash string)
  7MqxWPXQavjiGBrtqZwKINuxq30xora  → 31-char actual hash output

============================================================
SECTION B — Verifying passwords
============================================================
bcrypt.checkpw(correct_password, hash) → True
bcrypt.checkpw(wrong_password,   hash) → False

Note: checkpw() re-hashes the candidate using the salt embedded in the
stored hash, then does a constant-time comparison. No secret needed.

============================================================
SECTION C — Salt uniqueness (same password → different hash)
============================================================
Hash 1: $2b$12$Cf4gCXMUhWaKYCNtiLrFQexQErLiYGw6YWQHEf1r58XSvB/LPRSLq
Hash 2: $2b$12$seTdndvxsvMQJN3ITUNlRee3honjKVBaMvBRY/OJFiUe/yDVQIxZi
Hash 3: $2b$12$TricWyuBHgT.YHjKAIpIA.bi7.A/i17OOsRnS6/8l/kwF4RJUxawK

All three verify correctly against 'password123':
  Hash 1 matches: True
  Hash 2 matches: True
  Hash 3 matches: True

This defeats rainbow-table attacks: the attacker must crack each hash
individually, not build a single lookup table.

============================================================
SECTION D — Cost factor and time (the whole point of bcrypt)
============================================================
  Cost      Time  Iterations
----------------------------------------
     4     0.001s  2^4 =     16
     8     0.015s  2^8 =    256
    10     0.059s  2^10 =  1,024
    12     0.234s  2^12 =  4,096

Default (rounds=12): ~0.25s per hash → 4 attempts/sec for an attacker.
MD5 or plain SHA-256: billions of attempts per second.
The slowness is the point.
```

*(Your hash strings and timings will differ — hashes because the salt is random; timings because they depend on your hardware. Structure is always the same.)*

**EXPLAIN**
> "Section A. That 60-character string is the complete bcrypt output. The anatomy block is pulling it apart — dollar sign 2b is the version, dollar sign 12 is the cost factor, then 22 characters of salt, then 31 characters of hash. One string, everything needed."

> "Section B. `checkpw` returns True for the correct password and False for the wrong one. Notice the note: no secret needed. The salt is extracted from the stored hash. The library re-hashes the candidate with the same cost and salt and does a constant-time comparison. Constant-time means it can't be sped up by guessing: wrong passwords always take just as long as correct ones."

> "Section C. Same input password three times, three completely different hash strings. That's the salt. Without this, two users with 'password123' would have identical database entries — one crack would expose both. With salt, each hash is unique. All three still verify correctly, because each hash carries its own salt."

> "Section D. This table is the entire argument for bcrypt. Cost 4 is instant. Cost 12 takes a quarter of a second. That's 250ms per hash — 4 per second maximum for an attacker. SHA-256 can do billions per second on the same GPU. The slowness is not a bug. It's designed in."

### Exercise

**SPEAK**
> "The exercise is: hash `b'hunter2'` at cost 10 and verify it with two candidates. Let's write it."

**TYPE**
```python
# In a Python REPL or new file:
import bcrypt

password = b"hunter2"
hashed = bcrypt.hashpw(password, bcrypt.gensalt(rounds=10))

print(f"Hash: {hashed.decode()}")
print(f"Correct:  {bcrypt.checkpw(b'hunter2', hashed)}")
print(f"Wrong:    {bcrypt.checkpw(b'password', hashed)}")
```

```bash
"
```

```
Correct:  True
Wrong:    False
```

*(Your hash will differ. True and False won't.)*

**EXPLAIN**
> "Cost 10 — slightly faster than 12 but same structure. The hash starts with `$10` now. Both checkpw calls use the same stored hash, extract the salt themselves, re-hash, compare. True for the correct password, False for the wrong one."

## PART 2 — JWT: Signed Identity Tokens

---

### What is a JWT?

**SPEAK**
> "From the walkthrough: a JWT is three base64url-encoded strings joined by dots. Header — which algorithm was used. Payload — the claims, readable by anyone. Signature — HMAC-SHA256 over the first two parts using the secret. The payload is not encrypted. Anyone can decode it. But nobody can modify it without the secret, because the signature breaks."

---

### Run the demo

**TYPE**
```bash

```

**OUTPUT**

**EXPLAIN**
> "Section B is the manual decode — the `b64url_decode` function from the walkthrough, applied to the live token. Header decodes to exactly two fields: alg and typ. Payload decodes to our claims: sub, name, email, role, iat, exp. These are just JSON. Anyone who has the token can read this."

> "The signature at the bottom is raw bytes — it doesn't decode to JSON. It's the HMAC output. The comment explains it: the signature is over the string `header.payload` — the exact characters of the base64url header, a dot, the exact characters of the base64url payload. Change anything in those two fields and you get a different HMAC."


**EXPLAIN**
> "Section C: `jwt.decode` with the correct secret returns the payload as a Python dict. With the wrong secret — `InvalidSignatureError`. The library computed the HMAC of the token using the wrong key and it didn't match the stored signature."

**OUTPUT**
```
======================================================================
SECTION D — Tampering (privilege escalation attempt)
======================================================================
Original role:  admin
Forged role:    superadmin

Tampered token:
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiJ1c2VyXzQy...Nn0.Rr55PwnCO5z4DpNz11KEKWsRhI8bwXrspIMzDdJgY7k

Server rejects tampered token → InvalidSignatureError
```

**EXPLAIN**
> "Section D: we changed `role` from `admin` to `superadmin`, re-encoded the payload, and put the original signature back. We didn't touch the signature — we just swapped the payload section of the dot-separated string."

> "Rejected. The signature is HMAC of the original header + original payload. The forged payload produces different bytes, produces a different HMAC, and doesn't match the stored signature. You cannot privilege-escalate a JWT without the server's secret key."

## PART 3 — OAuth 2.0: Delegated Access

---

### What is OAuth?

**SPEAK**
> "OAuth solves: how does App B access your account on App A without you giving App B your App A password? The answer is a delegation protocol. App A issues a short-lived code. App B exchanges the code — using its own client secret — for an access token. The token is scoped to specific actions. You can revoke it without changing your password."

## PUTTING IT TOGETHER

**SPEAK**
> "Three mechanisms, one goal: secure authentication."

> "bcrypt: your users table stores hashes, not passwords. A stolen database is not an immediate catastrophe — the attacker has to brute-force each hash at 4 guesses per second."

> "JWT: after login, issue a signed token instead of storing session state. Any server with the secret can verify the token. The payload carries the claims — user ID, role, expiry — directly."

> "OAuth: when your app needs to access a user's account on another service, don't ask for their password. Use the delegation flow: code, exchange, access token. The user stays in control, the token is scoped, revocable."

> "In a real system: user registers — bcrypt stores the password. User logs in — bcrypt verifies, server issues a JWT. User clicks 'Connect Google Calendar' — OAuth handles the Google side, your app issues its own JWT when it's done. These three mechanisms appear together in almost every web app that has accounts."
