## PART 1 — Password Hashing with bcrypt (~9 min)

### What hashing is

> "Before we touch bcrypt, I want to make sure we're on the same page about what a hash function is."

> "A hash function takes any input — a word, a file, an entire database — and produces a fixed-length string of characters. Two rules: first, it's one-way. You can hash a password, but you can't un-hash it. There's no reverse function. Second, the same input always produces the same output."

> "This is how websites store your password. They don't store `password123` — they store the hash of `password123`. When you log in, they hash what you typed and compare to what they stored. If the hashes match, you're in. The actual password never sits in their database."

> "So why not just use SHA-256 for this? SHA-256 is everywhere — git uses it, TLS uses it, file downloads use it. The problem is it was designed to be *fast*. And fast is the enemy when you're storing passwords."

> "A modern GPU can compute 10 billion SHA-256 hashes per second. If someone steals a user database full of SHA-256 password hashes, they can try every common password in existence in seconds. bcrypt was built specifically to prevent this. Let's see how."

### Section A — Anatomy of a bcrypt hash

**On screen:** Highlight this line:

```
Hash: $2b$12$uBMpac65irGoAIwbd4tuNet6OJuDk/QYlJdl8JZqVeXhbbzhVyjB6
```

> "Look at this string. It's 60 characters, and every piece of information you'll ever need to verify this password later is encoded right in it. Let me read it left to right."

> "`$2b` — the bcrypt version identifier. Not important for us, just metadata."

> "`$12` — this is the cost factor. It means the function ran 2 to the power of 12 — 4,096 iterations — before producing a result. We'll come back to why this matters."

> "The next 22 characters — `uBMpac65irGoAIwbd4tuNe` — that's the salt. A random string that was generated specifically for this hash and mixed into the input before hashing. Every hash gets a new one."

> "The remaining 31 characters are the actual hash output."

> "This whole thing lives in one column of your users table. One column, everything you need."

---

### Section B — Verification

**On screen:** Section B output

```
bcrypt.checkpw(correct_password, hash) → True
bcrypt.checkpw(wrong_password,   hash) → False
```

> "Verification is one function call. What's happening under the hood: bcrypt reads the salt out of the stored hash string, runs the candidate password through the same hashing process with that salt, and compares the result. If they match, the password is correct."

> "Notice: the server never needs to store your password. Not even temporarily. Just the hash."

---

### Section C — Why salt matters

**On screen:** Three different hashes for `password123`

> "This is the salt doing its job. I hashed the same password three times. Three completely different outputs. But watch — "

**On screen:** All three verify as True

> "— all three verify correctly against 'password123'."

> "Here's why this matters. Imagine an attacker does the work ahead of time. They hash every word in the dictionary, every common password, every variation — and build a lookup table. `password123` maps to some SHA-256 hash. They steal your database and just look up each hash in their table. They know every weak password instantly."

> "With bcrypt's random salt, that lookup table is useless. Each user's hash was computed with a different random salt, so the same password produces a different hash for every user. The attacker would need to build a separate table for every possible salt — which is computationally impossible."

---

### Section D — Cost factor timing

**On screen:** Timing table — let it land for a beat

```
Cost  4:  0.001s
Cost  8:  0.015s
Cost 10:  0.059s
Cost 12:  0.237s
```

> "Here's the other weapon. At cost 4 — 2 to the 4th, 16 iterations — it's basically instant. At cost 12 — 4,096 iterations — it takes about a quarter second."

> "A quarter second feels like nothing to a legitimate user logging in. But to an attacker running millions of guesses? At cost 12, they get four attempts per second. At SHA-256 speed, they'd get 10 billion per second."

> "That's the difference between cracking a stolen database in an afternoon versus taking longer than the age of the universe."

> "The production default is 12. The recommendation is to tune your cost factor so hashing takes around 250 milliseconds on your hardware — and revisit it every few years as machines get faster."

## PART 2 — JWT Anatomy (~8 min)

### Base64 first

> "Before we look at JWTs, I need to explain base64 because JWTs are made of it."

> "Computers store everything as binary — ones and zeros. But some systems — HTTP headers, URLs, JSON — were designed to carry only printable text, not raw binary. Base64 is a way to take any binary data and represent it using only 64 safe characters: A through Z, a through z, 0 through 9, plus and slash."

> "It's not encryption. It's just a different way of writing the same data. Anyone can decode it."

**On screen:** Type this live

```bash
echo "SGVsbG8=" | base64 -d
```

> "That's 'Hello'. Base64-encoded, then decoded. Keep this in mind — JWTs use the same idea."

---

### Run the demo

```bash
python3.12 jwt_demo.py
```

**On screen:** Output — pause after Section A

---

### Section A — The token

**On screen:** Point at the token string

> "There it is. One long string with two dots in it. Three segments separated by dots. Header, payload, signature. Every JWT on the planet looks exactly like this."

> "And here's something important: the first two segments are just base64. Not encrypted. Not secret. I can read them right now."

---

### Shell decode

**On screen:** Copy the TOKEN= line from the bottom of the output and paste it

```bash
TOKEN=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOi...
```

> "I'm setting the token as a shell variable so I can work with it. Now let's decode the header — the first segment."

```bash
echo $TOKEN | cut -d'.' -f1 | tr '_-' '/+' | base64 -d; echo
```

> "Breaking this down: `cut -d'.' -f1` splits on dots and takes the first piece. Then `tr '_-' '/+'` — JWTs use a slightly modified version of base64 that replaces `+` with `-` and `/` with `_`, so the token is safe to put in URLs without any percent-encoding. This `tr` command swaps them back to normal base64 characters. Then `base64 -d` decodes it."

**On screen:** `{"alg":"HS256","typ":"JWT"}`

> "There's the header. Algorithm HS256 — that's the signing algorithm. Type JWT. That's it."

> "Now the payload — the part that actually matters."

```bash
echo $TOKEN | cut -d'.' -f2 | tr '_-' '/+' | base64 -d | python3.12 -m json.tool
```

**On screen:** The payload JSON

> "Name, email, role, expiry — all of it, readable. This is the critical thing to understand about JWTs: **the payload is not a secret**. Anyone who has a copy of the token can decode it with two shell commands. You just did it. Never put a password or anything sensitive in here."

> "So if anyone can read it, what stops someone from modifying it? That's what the third segment is for."

---

### Section D — Tampering

**On screen:** Section D output — show the original vs forged token

> "This is what the demo is doing in Section D. We take the token, change the role from 'admin' to 'superadmin' in the payload, base64-encode the new payload, and rebuild the token with the original signature still attached. Let's see if the server accepts it."

**On screen:** `Server rejects tampered token → InvalidSignatureError`

> "InvalidSignatureError. The signature was computed by the server over the exact bytes of the original header and payload. We changed bytes in the payload. When the server recomputes the signature over our modified payload, it gets a completely different result — it doesn't match what's in the token."

> "The attacker cannot fix the signature because that requires the server's secret key. Without the key, any modification to the payload is immediately detectable."

---

### The JWT tradeoff

> "Quick note before we move on. JWTs are convenient — no database lookup on every request. But there's a real tradeoff: because the server doesn't store anything, it also can't invalidate a specific token before it expires. If a user logs out, or you need to ban an account, their JWT keeps working until the `exp` timestamp passes."

> "This is why real systems often use short-lived JWTs — maybe 15 minutes — combined with a separate refresh token flow, or they maintain a small revocation list. Know the tradeoff before you reach for JWTs."

## PART 3 — OAuth 2.0 (~10 min)

### The problem

> "Let's talk about what OAuth actually solves. Imagine you're building a fitness app that needs to read a user's Google Calendar to find open time slots. One option: ask them for their Google password. Log in as them."

> "But that's terrible. Your app sees their full Google credentials. It gets access to everything in their account, not just Calendar. If your app gets breached, their entire Google account is compromised. They can't revoke your access without changing their password."

> "OAuth 2.0 is the solution. Instead of the user giving you their password, they authorize your app to get a limited, temporary token. The token only works for what they explicitly approved — just Calendar, not Gmail. They can revoke it at any time. Their password never leaves Google."

> "This is exactly what happens when you click 'Sign in with Google.'"

### Step 1 — Authorization Request

**On screen:** Step 1 output — both terminals simultaneously

```
< HTTP/1.0 302 Found
< Location: http://localhost:9000/callback?code=9LGqXKVury...&state=...
```

> "This first request goes to `/authorize`. In a real app, your server constructs this URL and redirects the user's browser to it — they'd see a Google or GitHub login page. Here, we're using curl with `-v` so we can see the raw HTTP response."

> "The server came back with a 302 — a redirect. Look at the Location header. There's the authorization code in the URL: `code=9LGqXKVury...`. This is a short-lived, one-time-use string. It expires in 60 seconds."

> "Also notice the `state` parameter. Before making this request, our app generated a random value and included it. When the redirect comes back, we check that the same value is there. This protects against a subtle attack: without `state`, a malicious website could trick your app into processing their authorization code and attaching it to your user's account. The random state token, which only your app knows, makes that impossible."

---

### Step 2 — Extract the code

**On screen:** `Authorization code: 9LGqXKVuryOwBn7jt9_Zew`

> "In a real app, this code would arrive as a query parameter at your callback URL — your app reads it from the request. Here we're just pulling it out of the Location header with grep."

---

### Step 3 — Token Exchange

**On screen:** Step 3 output — both terminals

```json
{
    "access_token": "ZZC_keMxWTHjaDDlIeKB7...",
    "token_type": "Bearer",
    "expires_in": 3600,
    "scope": "read:profile"
}
```

> "This is the most important step. Your server makes a POST request directly to the token endpoint — not through the browser, server-to-server. It sends the authorization code plus the `client_secret` — your app's password, essentially."

> "Here's why the two-step design matters. The authorization code traveled through the browser URL, which gets logged by web servers, stored in browser history, sometimes ends up in analytics. That's okay — a code alone is useless. To get a real token, you also need the `client_secret`, which only your server knows and never touches a URL bar."

> "The response gives us the access token — valid for an hour, scoped to read:profile only."

---

### Step 4 — Protected Resource

**On screen:** Step 4 output — userinfo JSON

> "Now we use the access token to call the protected endpoint. `Authorization: Bearer` — you'll see this header in every API that uses token-based auth. The resource server checks the token and returns the user's profile."

---

### Steps 5 & 6

**On screen:** 401, then invalid_grant

> "No token: 401. The resource is protected."

> "And here's the replay attack demo. We try to use the same authorization code a second time. The server returns `invalid_grant` — the code was already consumed when we exchanged it in Step 3. It no longer exists. If an attacker intercepted the code from the redirect URL and tried to race us to the token endpoint — they'd find it's already been used."

---

### The big picture

> "That's the complete OAuth Authorization Code Flow. Six steps. Every 'Sign in with Google' button runs this. The only differences between this demo and the real thing: Google's auth servers instead of localhost, real HTTPS instead of HTTP, and the access token is probably a JWT instead of a random string."
