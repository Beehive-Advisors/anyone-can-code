### Section A — Hash anatomy

**On screen:** Section A output:

```
Hash: $2b$12$uBMpac65irGoAIwbd4tuNet6OJuDk/QYlJdl8JZqVeXhbbzhVyjB6
```

> "Look at this string. Everything you need to verify the password later is right here — no side tables, no extra columns. Let me break it down."

**On screen:** Point to each segment as you say it:

> "`$2b` — bcrypt version. `$12` — cost factor, meaning 2 to the 12th, or four thousand iterations. Then 22 characters of salt — random, unique every time. Then 31 characters of actual hash output. That's it. This whole thing lives in one column in your users table."

---

### Section B — Verification

**On screen:** Section B output:

```
bcrypt.checkpw(correct_password, hash) → True
bcrypt.checkpw(wrong_password,   hash) → False
```

> "Verification is dead simple. bcrypt extracts the salt from the hash string, re-hashes your candidate, and compares. The server never stored your password — just enough state to check future guesses."

---

### Section C — Salt uniqueness

**On screen:** Three different hashes for `password123`

> "Same password, three completely different hashes. Run this for any user in your system and you'll see the same thing. This is why rainbow tables don't work. A rainbow table is a precomputed dictionary: `password123 → that_sha256_hash`. But with a random salt per user, the attacker would need a different table for every salt. That's not feasible."

---

### Section D — Cost factor timing

**On screen:** Timing table:

```
Cost  4:  0.001s
Cost  8:  0.015s
Cost 10:  0.059s
Cost 12:  0.234s
```

> "Here's the key insight. At cost 12, a single hash takes about a quarter second. An attacker gets maybe four guesses per second. SHA-256? Ten billion per second on a GPU. The slowness is not a bug. It's the entire point. Set your cost factor so hashing takes around 250ms on your production hardware — and bump it up every few years as CPUs get faster."

## PART 2 — JWT (7 min)

### Run the demo

```bash
python3.12 jwt_demo.py
```

**On screen:** Output scrolls — pause after Section A

---

### Section A — The token

**On screen:** The token string

> "That long string with two dots in it. Three segments. Header, payload, signature — separated by dots. Every JWT looks like this."

> "The header and payload are not encrypted. They're just base64-encoded. I can read them right now without any library."

---

### Shell decode exercise

**On screen:** Copy the `TOKEN=...` line from the bottom of the output, paste it into the shell

```bash
TOKEN=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOi...
```

> "Set it as a shell variable. Now decode the header."

```bash
echo $TOKEN | cut -d'.' -f1 | tr '_-' '/+' | base64 -d; echo
```

**On screen:** `{"alg":"HS256","typ":"JWT"}`

> "There's the header. Algorithm HS256, type JWT. Now the payload."

```bash
echo $TOKEN | cut -d'.' -f2 | tr '_-' '/+' | base64 -d | python3.12 -m json.tool
```

**On screen:**

```json
{
    "sub": "user_42",
    "name": "Alice Demo",
    "email": "alice@example.com",
    "role": "admin",
    ...
}
```

> "Name, email, role — all readable. This is why you never put a password or a credit card number in a JWT. Anyone who intercepts this token can read its contents. What they *cannot* do is change it without breaking the signature. That's what the third segment is for."

---

### Section D — Tampering

**On screen:** Section D output

> "We changed the role from `admin` to `superadmin` in the payload, re-encoded it, and swapped in the original signature. Let's see what happens when the server tries to verify it."

**On screen:**

```
Server rejects tampered token → InvalidSignatureError
```

> "InvalidSignatureError. The signature was computed over the original bytes of the header and payload. We changed a byte. The signature no longer matches. Done — the attacker can't escalate their privileges without knowing the server's secret."

---

### Why not put everything in the token?

> "Quick note: JWTs are convenient, but they have a trade-off. Because they're stateless, you can't invalidate a single token before it expires. If a user logs out or you ban them, their token still works until `exp`. This is why high-security systems often use short-lived JWTs combined with a token revocation list, or just use opaque tokens with a database lookup. Know the tradeoff before you reach for JWTs."

---

### Terminal 2: Run the flow

**Terminal 2:**

```bash
bash oauth_flow.sh
```

> "Watch both terminals."

---

### Step 1 — Authorization Request

**On screen:** Step 1 output — both terminals

```
< HTTP/1.0 302 Found
< Location: http://localhost:9000/callback?code=9LGqXKVury...&state=...
```

**Server terminal:**
```
[AUTH SERVER] /authorize → issued code 9LGqXKVury…
[AUTH SERVER] Redirecting → http://localhost:9000/callback?code=...
```

> "This is the redirect. In a real app, the user's browser would follow this URL to your callback endpoint. We're using curl to see the raw `302` with its `Location` header. The `code` parameter is a short-lived authorization code — it's not the token. It expires in 60 seconds and can only be used once."

> "Notice the `state` parameter too. It's a random nonce your app generated before the redirect. When your callback receives this, you check that `state` matches what you sent. That's CSRF protection — it prevents an attacker from tricking your app into accepting their code."

---

### Step 2 — Extract the code

**On screen:** `Authorization code: 9LGqXKVuryOwBn7jt9_Zew`

> "We extract it from the Location header. In a real callback, this would just be a query parameter."

---

### Step 3 — Token exchange

**On screen:** Step 3 — POST to /token

```json
{
    "access_token": "ZZC_keMxWTHjaDDlIeKB7...",
    "token_type": "Bearer",
    "expires_in": 3600,
    "scope": "read:profile"
}
```

> "This POST happens from your server — not from the browser. The client secret goes here, over HTTPS, server to server. This is the back-channel. This is why the two-step design exists: the authorization code shows up in the browser's URL bar and in server logs. The access token never does."

---

### Step 4 — Protected resource

**On screen:** Step 4 — userinfo response

```json
{
    "sub": "user_001",
    "name": "Alice Demo",
    "email": "alice@example.com"
}
```

> "Standard Bearer token auth. You've seen this `Authorization: Bearer` header in every API you've ever called."

---

### Steps 5 & 6 — Edge cases

**On screen:** 401 for no token, 400 for replayed code

> "No token — 401. Replayed code — 400 invalid_grant. Codes are single-use. If an attacker intercepts your code in the redirect URL and tries to exchange it, but your server already did — they get nothing. This is by design."

---

### The big picture

> "Every 'Sign in with Google' button runs this exact flow. The browser redirect, the code, the back-channel token exchange — that's all it is. The difference between this and a real OAuth provider is: GitHub's servers instead of localhost:8080, real HTTPS instead of HTTP, and probably a real JWT as the access token instead of a random string."
