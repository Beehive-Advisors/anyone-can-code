## PART 1 — bcrypt Password Hashing

---

### What is bcrypt?

**SPEAK**
> "Before we run anything — let's think about the problem. Every app with user accounts needs
> to store something that lets it verify your password at login. The naive approach is storing
> the password directly. That's catastrophic when — not if — the database is stolen."
>
> "The improvement: store a hash. A hash function takes any input and produces a fixed-length
> fingerprint. One-way — you can't reverse it. At login, hash what the user typed and compare.
> If they match, the password was correct."
>
> "The problem: fast hash functions like SHA-256 are designed for speed. A GPU can run
> billions of them per second. An attacker with a stolen database tries every common password
> at billions of guesses per second. Most databases crack in hours."
>
> "bcrypt is a hash function designed specifically for passwords. It's deliberately slow —
> around 250 milliseconds at cost 12. That slowness is not a flaw. It's the entire point.
> The same GPU gets a few thousand guesses per second instead of billions."

### Exercise — password verification

**SPEAK**
> "The exercise is to write a short script that prompts for a password, hashes it, asks for
> it again, and verifies it. The key things to remember: `input()` returns a string in Python.
> bcrypt needs bytes. You have to call `.encode('utf-8')`. And you pass the stored hash to
> `checkpw()`, not the salt separately."

**TYPE**
```python
import bcrypt
password = input("Create a password: ").encode("utf-8")
hashed = bcrypt.hashpw(password, bcrypt.gensalt(rounds=12))
print(f"Stored: {hashed.decode()}")
attempt = input("Enter it again: ").encode("utf-8")
print("Correct!" if bcrypt.checkpw(attempt, hashed) else "Wrong password.")
```

**TYPE**
```bash
python3.12 -c "
import bcrypt
password = input('Create a password: ').encode('utf-8')
hashed = bcrypt.hashpw(password, bcrypt.gensalt(rounds=12))
print(f'Stored: {hashed.decode()}')
attempt = input('Enter it again: ').encode('utf-8')
print('Correct!' if bcrypt.checkpw(attempt, hashed) else 'Wrong password.')
"
```

**EXPLAIN**
> "`.encode('utf-8')` converts the string from `input()` to bytes before bcrypt sees it.
> Passing a string raises TypeError. `checkpw()` extracts the salt automatically — you
> don't handle the salt at all."

## PART 2 — JWT (JSON Web Tokens)

---

### What is a JWT?

**SPEAK**
> "Every HTTP request arrives blank. After a user logs in, the server needs to recognize
> them on the next request without the user logging in again."
>
> "Traditional approach: the server stores a session record in a database and gives the user
> a random session ID in a cookie. Every request, the server looks up that ID. Works fine,
> but every single request costs a database query just to answer 'who is this?'"
>
> "A JWT is different: instead of a pointer to server-side data, you give the user the data
> directly, cryptographically signed. The user's ID, their role, when it expires — all in
> the token string. The server reads the token on every request, with no database lookup."
>
> "How does the server know nobody forged it? The token is signed with the server's secret
> key. Alter the payload, the signature no longer matches, the server rejects it."
>
> "Key fact before we run anything: a JWT is signed, not encrypted. Anyone can read the
> payload. Never put passwords or secrets inside a JWT."

>

> JWT you receive — but you cannot create or alter one. This is the security property."

## PART 3 — OAuth 2.0 Authorization Code Flow

---

### What is OAuth?

**SPEAK**
> "Imagine you're building an app that needs to read a user's GitHub repositories. You could
> ask for their GitHub password and log in as them. But then your app has full access to their
> entire account, the user can't revoke access without changing their password, and GitHub
> might lock the account for unusual login behavior."
>
> "OAuth solves this. Instead of acting as the user, your app asks the user to grant it
> specific permissions directly with GitHub. GitHub issues a temporary access token that works
> only for what the user authorized. The user can revoke it at any time. Your app never sees
> the password."
>
> "The authorization code flow is designed around two channels. The front channel is the
> user's browser — everything there ends up in browser history, server logs, referrer headers.
> The back channel is a direct server-to-server connection — the browser never touches it.
> The access token only travels through the back channel."

## PUTTING IT TOGETHER

**SPEAK**
> "Three mechanisms. One problem each."
>
> "bcrypt: what happens when the database is stolen. Slow hashing + random salt means attackers
> get a few thousand guesses per second instead of billions, and rainbow tables are useless."
>
> "JWT: what happens after the user logs in. The server can recognize the user on every request
> without touching the database, because all the necessary data is in a signed token."
>
> "OAuth: what happens when a third-party app wants to act on the user's behalf. Delegated
> authorization. The app gets a scoped, revocable token. The user's password never leaves the
> original service."
>
> "In a real login flow, all three work together. You register, your password is hashed with
> bcrypt. You log in, bcrypt verifies it and the server issues you a JWT. When a third-party
> app needs access, OAuth handles the delegation without your password leaving the auth server."
