## PART 1 — Password Hashing with bcrypt

---

### What is a hash function?

**SPEAK**
> "Before we touch bcrypt I want to make sure we're on the same page about what a hash function is, because it's the foundation of everything in this section."

> "A hash function takes any input — a word, a file, an entire hard drive — and produces a fixed-length string of characters. Two rules. First: it's one-way. You can go from the word 'password123' to a hash, but you cannot go from the hash back to 'password123'. There's no reverse. Second: the same input always produces the same output."

> "This is how websites store your password. They don't store 'password123' in a column called 'password'. They store the hash of it. When you log in, they hash what you typed and compare it to what they stored. Same hash? You're in."

> "So why not just use SHA-256? SHA-256 is everywhere — git uses it, TLS uses it, file integrity checks use it. The problem is it was designed to be fast. Really fast. A modern GPU can compute ten billion SHA-256 hashes per second. If someone steals your user database, they can try every common password in existence in seconds. bcrypt was built specifically to prevent this. Let's see how."

> "Here's why this matters. Imagine an attacker builds a lookup table before they ever steal anything. ## PART 2 — JWT Anatomy

---

### What is base64?

**SPEAK**
> "Before we touch JWTs, I need to explain base64, because JWTs are built from it."

> "Computers store everything as binary — ones and zeros. But some systems — HTTP headers, URLs, JSON — were designed to carry only regular printable text, not raw binary. Base64 is a way to take any binary data and write it using only 64 safe characters: A through Z, lowercase a through z, 0 through 9, plus, and slash."

> "It's not encryption. There's no key. Anyone can decode it. It's just a different way of writing the same data — like how you can write the number seven in Arabic numerals, Roman numerals, or spelled out as a word. Same thing, different representation."

> "Let me show you."

**TYPE**
```bash
echo "SGVsbG8=" | base64 -d
```

**OUTPUT**
```
Hello
```

**EXPLAIN**
> "That was 'Hello' encoded in base64, then decoded. Keep that in mind. JWTs use the exact same idea."

---

### What problem does JWT solve?

**SPEAK**
> "When you log into a website, the server needs to remember who you are on every request you make. One approach: the server stores a record in its database — 'session number abc123 belongs to Alice' — and gives your browser a cookie with that ID. Every request, your browser sends the cookie, the server does a database lookup, finds your info, continues."

> "That works. But it means every single request — every page load, every API call — hits the database. And if you have multiple servers, they all need to share the same session database."

> "JWTs are a different approach. Instead of a database record, the server creates a small document that says 'this is Alice, she has admin role, this token expires in one hour' — and signs it cryptographically. Your browser stores this token and sends it with every request. The server checks the signature and reads the claims directly from the token — no database lookup at all."

---

### What does the signature actually do?

**SPEAK**
> "Before we look at tampering, let me explain how the signature works mechanically, because it's the whole point."

> "When the server creates a JWT, it takes the header and the payload — the first two segments — and runs them through a function called HMAC-SHA256 using a secret key that only the server knows. HMAC stands for Hash-based Message Authentication Code — but the name isn't important. What matters is this: the function produces a unique fingerprint of the input. Change anything in the input — one character, one byte — and you get a completely different fingerprint."

> "That fingerprint becomes the third segment: the signature."

> "When the server receives a token back, it recomputes that fingerprint over the header and payload it received. If the result matches the signature in the token, the data is untouched. If anything changed, the fingerprints don't match, and the token is rejected. The attacker can't forge a valid signature without the server's secret key."

> "Here's why. When the server originally created this token, it ran the header and payload through a function called HMAC-SHA256 using a secret key that only the server knows. If the result matches the signature in the token, the data is untouched. If anything changed — even one character — the function produces a completely different result, and the comparison fails."

> "The attacker can modify the payload all they want. Without the server's secret key, they cannot produce a valid signature for their modified payload. The token is tamper-evident."

### JWT tradeoff

**SPEAK**
> "One thing to know before we move on. JWTs are stateless — the server doesn't store anything. That's the advantage. The tradeoff is: the server also can't cancel a specific token before it expires. If a user logs out, or you need to lock out an account, their JWT keeps working until the exp timestamp passes. Real systems deal with this by using short expiry times — like 15 minutes — paired with a refresh token, or by maintaining a small blocklist of revoked tokens. Know the tradeoff."

## PART 3 — OAuth 2.0

---

### The problem OAuth solves

**SPEAK**
> "Here's the scenario. You're building a fitness app that needs to read a user's Google Calendar to find open time slots. Option one: ask the user for their Google password and log in as them."

> "That's a bad idea for several reasons. Your app sees their full Google credentials. Your app gets access to everything in their account — not just Calendar, but Gmail, Drive, all of it. If your app gets hacked, their entire Google account is compromised. And they can't revoke your access without changing their Google password."

> "OAuth 2.0 solves this. Instead of giving you their password, the user authorizes your app to receive a limited, temporary token. That token only works for the specific thing they approved — read-only Calendar access, nothing else. They can revoke it at any time without touching their password. And your app never sees their Google credentials at all."

> "This is exactly what happens when you click 'Sign in with Google.'"

---

### The four players

**SPEAK**
> "Before we run anything, let me name the four parties involved in every OAuth flow, because the script uses all of them."

> "First: the Resource Owner — that's the user who owns the data. Second: the Client — that's your app, the thing that wants access. Third: the Authorization Server — the service that verifies the user's identity and issues tokens. Fourth: the Resource Server — the API that holds the protected data."

> "In this demo, our bash script is the Client. `oauth_server.py` plays both the Authorization Server and the Resource Server. In real life, those last two are run by Google or GitHub, but they're logically separate roles."

> "The flow has five steps. Your app sends the user to the auth server to approve access. The user approves. The auth server redirects back with a short-lived code. Your server exchanges that code for an access token — privately, server-to-server. Your server uses the access token to call the API. That's it."

**EXPLAIN**
